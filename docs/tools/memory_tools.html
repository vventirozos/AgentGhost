unified_forget — four-store sweep

forget(target) sweeps four stores in order: disk (sandbox files), vector memory (fuzzy filename match + semantic sweep), the profile, and the knowledge graph. The profile pass is the one users notice, because the profile is injected into the system prompt every turn — so anything it misses keeps being "remembered".

The profile pass now runs in two stages:

  1. Key sweep — exact key match, then word-boundary substring match on keys only (so forget('age') can't wipe a language key). Matched keys are deleted whole.
  2. Value sweep — for every key not already handled, match the values with _value_mentions_target (token/word-boundary aware). A matching scalar deletes the key; matching list items are pruned via ProfileMemory.prune_value, dropping the key/category when emptied.

Stage 2 fixes the long-standing case where a forgotten entity lived as a value (e.g. assets.pets = ["Hanzo the dog", "Mortimer the iguana"]): the key-only sweep never matched forget('mortimer'), so the model kept reporting the pet as owned — often annotated "(removed)" from an earlier soft-delete tombstone the merge semantics left behind. The token-boundary matcher avoids the destructive greedy-substring behaviour the key-only rule originally guarded against. Covered by tests/test_forget_profile_value_sweep.py.

Vector sweep — literal-mention override

The semantic sweep originally deleted a candidate only when its L2 distance fell under a type-dependent threshold (0.8 auto / 0.6 identity). That silently missed facts that name the entity outright — e.g. forgetting iguana left "user previously had an iguana that was removed" in place because the sentence-vs-word distance exceeded the bar. The sweep now also deletes any candidate whose text mentions the target as a word-boundary token, regardless of distance. n_results was raised 10 → 20 so more candidates are examined.

Entity-aware expansion (alias tombstones)

A deleted entity often survives under a synonym: forget('mortimer') can't reach a fact stored under iguana. Before severing graph edges, unified_forget calls GraphMemory.get_connected_entities(target) to pull the target's direct (1-hop) neighbours — mortimer → iguana via a mortimer IS_A iguana edge — filtering generic hubs (user, pronouns). It then runs a literal-mention-only secondary sweep (vector + graph + profile values) for each neighbour, so the alias tombstone goes too without semantic over-reach.

Stopping tombstone regeneration

The deepest cause was a feedback loop: each time the user asked "what pets do I own?", the model answered "you previously had an iguana that was removed", and the smart-memory consolidator (run_smart_memory_task) extracted that sentence back into a positive fact + graph triplet — so the tombstone regenerated itself faster than forget could remove it. Consolidation now drops removal/negation statements: is_removal_or_negation_text() bails before a removal fact is embedded, and is_removal_triplet() filters past-ownership predicates (PREVIOUSLY_OWNED, NO_LONGER_HAS, …) and (removed)-tagged entities before they reach the graph. A removal must delete knowledge, never insert a memory about the deletion. The post_mortem learning pass is also skipped on a forget turn (forget_was_called). Covered by tests/test_forget_hardening.py.

Intent guard on self_play / self_play_loop

Both tools refuse to run unless the current turn's user text explicitly asks for self-play, a practice round, or a training cycle. handle_chat stashes the turn's message on context.last_user_content right after parsing the request body; _user_asked_for_self_play(context) matches a conservative allow-list of phrases ("run self-play", "train until stopped", "practice cycle", "synthetic self-play", etc. — see _SELF_PLAY_INTENT_PHRASES). If the guard trips, the tool returns _SELF_PLAY_INTENT_REFUSAL without invoking the Dreamer. Rationale: the 2026-04-24 webOS incident showed the 30B-A3 model hallucinating "The user wants me to run self-play" 33 minutes into a webOS-building session where the user had never mentioned self-play. The tool description says "Use this EVERY TIME the user asks to practice, train, or do self-play" but that's LLM-side prose — the guard makes it enforceable.

The biological watchdog bypasses this check: phase 3 of _biological_tick calls Dreamer.synthetic_self_play directly, not through tool_self_play, so legitimate background self-play fires unaffected. Covered by tests/test_self_play_intent_guard.py.

Self-play loop internals

  • Cool-off between cycles is adaptive (5–180 s clamped) via FrontierTracker.adaptive_cooldown. Falls back to a 30 s baseline if no tracker is available.
  • The loop task + stop event are stashed on context.selfplay_loop_task and context.selfplay_loop_stop. handle_chat checks these on every user message and sets the stop event — the loop checks it at each cycle boundary and during the cool-off asyncio.wait_for.
  • Between cycles the loop explicitly drains the short-term journal via context.agent.process_journal_queue() so hippocampus doesn't fall behind during long runs. Cheap no-op when the journal is empty.
  • Inside the isolated sub-agent's context, selfplay_loop_* attributes are stripped to prevent the inner solver's handle_chat from tripping the user-message interrupt on the outer loop.

Memory bus pattern

When a MemoryBus is attached, insert_fact calls publish_fact("insert_fact", ...) rather than writing the vector store directly. The bus fans the write out to vector + graph (and other tiers) in parallel, and the dedup-LRU gate prevents re-publishes within a 256-unique-event window.
