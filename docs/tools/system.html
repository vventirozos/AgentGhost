2026-07-14 audit fixes

  • Null-valued profile keys no longer crash location lookup. _find_location_in_profile used data.get("root", {}).get(...) — a profile with an explicit "root": null (key present, value null) made the default-arg form return None and the chained .get raised AttributeError (crashing check_location, and silently killing profile-based weather lookup via the bare except). Now (data.get("root") or {}).
  • Unknown location gives a clear message, not failed: None. When the geocoder answers 200 with no results and both sources fall through, the tool now says could not find a location named 'X' (and tells the model not to retry the same name) instead of the opaque last_error=None string.
  • Tor mode detection accepts localhost/::1, not just the 127.0.0.1 literal (a socks5://localhost:9050 proxy was silently running in WEB mode — no identity rotation). The bare except: in the profile-location path was narrowed to Exception, and psutil.cpu_percent(interval=0.1) (a blocking 100ms sleep) moved off the event-loop thread. Covered by tests/test_tools_batch_audit_fixes.py.

Reviewed residual — verify=False on the HTTPS checks. The weather / 1.1.1.1 / check.torproject.org fetches disable TLS verification (a Tor-exit-compatibility choice). This means the Tor-anonymity confirmation is forgeable by a malicious exit and forged weather bodies enter model context. Left as-is deliberately: flipping verify on the LIVE Tor path risks regressing the weather/health tools against exits with cert-handling quirks (the usual reason it was set), the threat model (an adversary controlling the operator's own Tor exit) is narrow, weather text is already treated as untrusted external content (same trust boundary as web_search), and the wrong-anonymity-label consequence is low-stakes. Revisit if the endpoints prove stable with verification on.
