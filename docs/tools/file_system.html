Tool: file_system

One LLM-facing tool that dispatches to a sub-handler by the operation argument.

Replace matching ladder & the full-rewrite footgun (2026-06-07)

tool_replace_text tries progressively looser matches before giving up: (1) exact substring, (2) flexible whitespace (tokens joined by \s+, with replacement indentation re-anchored to the match — see the 2026-06-12 section), (3) a fuzzy contiguous-block match (_fuzzy_block_match), and only then (4) a failure with a nearest-snippet hint. Every landing path is wrapped by a syntax-regression rollback guard.

Why fuzzy matching was added. The flexible matcher only forgives whitespace; a single misremembered character or stray token makes it miss. The documented next move was then "rewrite the whole file with write" — which, on a large index.html, is a 200–380s full regeneration from the model's loose memory and a fresh chance to reintroduce bugs (this is literally how a spawn/spawnPos identifier drift and a black-blocks regression got re-introduced in production). _fuzzy_block_match slides a window of the target's line-count over the file, compares each window to the target with difflib after stripping per-line whitespace, and applies the replacement only when the best window clears ratio ≥ 0.92 and beats the runner-up by a margin — so an ambiguous edit never silently lands in the wrong place. A trailing newline on the matched block is preserved so a single-line fuzzy replace can't weld the following line on.

Anchor-block match (2026-06-09). Fuzzy compares the whole block, so when a model rebuilds a large old_text from paginated reads and mis-transcribes the middle (the live addFace edit — signature and shape remembered, vertex math wrong), the ratio falls below threshold and fuzzy misses too. _anchor_block_match rescues exactly this by keying on the block's boundaries: (a) if the first AND last non-blank lines of old_text each occur exactly once, replace the file bytes spanning them; or (b) if old_text contains a unique block-opener line (the first line ending in { that occurs exactly once — so a leading comment or context line the model included doesn't defeat it), replace from that line through the matching closing brace. Both require a unique anchor and cap the matched span relative to the target size, so a bad anchor can never swallow a huge region. This turned the live ~4-minute, 7-attempt replace thrash (which ended in a full-file rewrite) into a single surgical edit.

Param aliases (2026-06-09). The live run's first replace failed purely because the model passed old_text=/replace_with= and old_text wasn't an alias for content. The dispatch now accepts the variants the model reaches for: old text via content/data/text/old_text/old_string/search/old; new text via replace_with/new_text/new_string/new/replacement.

Size-aware failure guidance. When all matchers miss, the recovery advice now scales with file size. For a large file (> 16 KB or > 250 lines) the tool forbids the whole-file rewrite — "DO NOT use operation='write' … a full regeneration is slow and routinely introduces NEW bugs" — and steers the model to copy the exact current text from the (line-numbered) nearest snippet and emit a tight single-line replace. Small files still get the old "a full write is fine" advice. This directly removes the 200s-per-edit spiral. Covered by tests/test_file_replace_fuzzy_and_size_guidance.py.

Loop-breaking "not found" for reads (2026-06-07)

A missing read/inspect target used to return a bare Error: '<f>' not found.. That caused an infinite loop when stale cross-project context disagreed with the live sandbox: a fresh "create a new Minecraft project" request kept re-reading minecraft-clone/index.html — which existed only in a prior project — because the injected workspace narrative / DYNAMIC SYSTEM STATE asserted it existed while the (correctly empty, newly-scoped) sandbox said "not found", and nothing reconciled the two.

_missing_file_message(filename, sandbox_dir) now returns a reconciling, exit-providing message: it states the file isn't in the current project's sandbox, lists the files that do exist (or "EMPTY"), declares the live sandbox authoritative over any workspace-narrative / memory / prior-session hint, and tells the model to stop re-reading and instead write the file or pick a real one. Wired into tool_read_file and tool_inspect_file. Paired with the agent-loop's repeated-identical-failure breaker (see core/agent) so the loop terminates even if the stale context keeps re-asserting itself. Covered by tests/test_not_found_loop_breaker.py.

Path-safety

_get_safe_path resolves all paths relative to sandbox_dir and rejects path traversal via Path.is_relative_to() (on the resolved, symlink-followed path). Leading / is stripped and treated as relative.

Destructive ops cannot target the root (2026-07-03). All the heals above intentionally collapse ., /, workspace, /workspace, an empty string, and (when project-scoped) projects/<active-id> down to the sandbox/project ROOT. That's correct for a read or a listing, but a delete / rename-move / copy-source that resolved to the root would rmtree/relocate the entire workspace. Those call sites now pass allow_root=False, which makes _get_safe_path raise a refusing to run a destructive operation on the sandbox/project root Security Error when the resolved target equals the root. Reads and lists are unaffected (they use the default allow_root=True).

/workspace/ prefix is stripped (2026-04-24). The sandbox bind-mounts the host sandbox root at /workspace inside the container, so /workspace/foo.py in the container is the same file as foo.py at the host's sandbox root. When the LLM round-trips container-visible paths (prompts, shell commands, and docs all encourage /workspace/...) to host-facing tools like file_system, the tool now also strips a leading workspace/ segment. Without the strip, /workspace/skills/foo.py resolved to $SANDBOX_DIR/workspace/skills/foo.py — a phantom workspace/ subdir — and later execute calls using the same path got ENOENT because the real file lived at /workspace/workspace/skills/foo.py inside the container. Confirmed in the 2026-04-24 in_gr_news session, which burned ~60 s rediscovering the mismatch. The strip is exact-segment only: workspaces/ (plural) and workspace_backup/ are not touched, and traversal inside a /workspace/ path (/workspace/../escape) is still blocked.

Redundant project-prefix heal — now covers sandbox/projects/<id>/ (2026-06-20). When a project is active the sandbox is scoped to <root>/projects/<id>/, and the model reaches for a file via whatever route it has seen it referenced. The heal collapses every such redundant prefix to the project-relative remainder, longest match first: projects/<id>/X (container-root-relative) and <root>/projects/<id>/X — e.g. sandbox/projects/<id>/X (the sandbox dir is literally named sandbox on disk and surfaces that way in listings / workspace_dir). Without the latter, a model that built file:///workspace/sandbox/projects/<id>/browser-os.html double-nested to /workspace/projects/<id>/sandbox/projects/<id>/browser-os.html → a net::ERR_FILE_NOT_FOUND the browser tool hit three times before the loop-breaker fired (observed live, webOS build). The root segment is derived from sandbox_dir.parent.parent.name (not hardcoded). A real sandbox/ or projects/ subdir that is not the redundant <root>/projects/<id>/ form is still honoured literally, and traversal is still blocked. Covered by tests/test_browser_path_scoping.py.

Guessed-slug project-prefix heal (2026-06-20). The heal above strips only the active id's prefix. But the model also guesses a project dir by name/title: live, a single-file build wrote projects/DeskMiniX3/index.html (the title) the same turn it created the project, before the canonical id (3163a85ac4fc) was active. Left literal, that spawns an orphan projects/<title>/ tree that no project record owns — the file, the store's workspace_dir, and the cleanup sweep (which keys on the canonical id) then point at three different places, and the sweep can never manage that workspace. So while a project is active, any leading projects/<slug>/ (a one-segment slug followed by a slash) is now collapsed into the active project's dir, so all three agree on the canonical id. Trade-off: a literal projects/<sub>/ directory nested inside a project workspace is not addressable this way — acceptable, the model should not nest one (a bare projects/notes.txt with no further segment is untouched). The cleanup sweep also gained an empty-keep-set recovery so a missing registration can no longer let this (or anything) delete the deliverable.

Container-side path translation for search / find (2026-04-26)

tool_file_search (operation=search) and tool_find_files (operation=find) shell out to rg / find via sandbox_manager.execute, which runs commands inside the Docker container at workdir=/workspace. The host filesystem is otherwise opaque to the container. Until the 2026-04-26 fix, both tools resolved their path argument with _get_safe_path (which returns a HOST-absolute path like /Users/me/sandbox/webos/app.js) and passed that path straight into the container — which silently produced "no matches" because the container can't see /Users/....

The new helper _to_container_path(sandbox_dir, host_path) translates a host-absolute path to its container-visible /workspace/<rel> equivalent. tool_file_search applies it unconditionally; tool_find_files applies it when given sandbox_dir via the dispatcher (legacy callers without sandbox_dir still work but skip translation). A path that resolves outside the sandbox is rejected explicitly rather than silently searching the wrong tree.

Incident: a 2026-04-26 webOS session burned ~70 minutes because six consecutive rg lockScreen.style.display webos/app.js calls all returned empty. The agent (correctly) concluded the searches showed no match and (incorrectly) concluded the file edit hadn't been applied — when in fact the search was running against a path the container couldn't see. Coverage: tests/test_file_system_search_container_path.py.

Binary sniffing

_looks_like_binary: NUL byte → binary; otherwise count non-text control chars (everything < 0x20 except TAB/LF/CR/FF) — > 5 % → binary.

PDF handling

  • read blocks PDFs with a hint to use read_chunked.
  • read_chunked uses PyMuPDF; max 1000 pages.
  • For ingestion via knowledge_base, the PDF is chunked into the vector store.

Chunked-read pagination — out-of-range handling (2026-06-11)

Section/page counts are computed dynamically from file size (total_pages = ceil(file_size / effective_chunk)), so a small file is 1–2 sections. The model frequently OVER-ESTIMATED and requested a wildly out-of-range page (e.g. section 75 of 2), which returned a bare Error: → a wasted strike with no recovery path. Two causes were fixed:

  • End-aware footer. Every section used to end with “Use page=N+1 to continue reading” — even on the last section, literally instructing the model to request a page that doesn't exist. The final in-range section now says it is the LAST section and not to request page+1; only non-last sections offer the continue hint.
  • Graceful out-of-range. An out-of-range page no longer strikes. For a single-section file the whole file is served as section 1 (the model jumped past it), so the turn makes real progress; for a genuinely chunked file that overshot the end, a terminal non-erroring NOTE states the valid range and tells the model to stop paginating (so the failure-classifier doesn't count it and no higher-page retry loop forms). PDF pages get the same treatment.

Covered by tests/test_file_system_chunked.py.

Per-batch read budget — parallel-read overflow guard (2026-06-23)

The per-file read cap judges each read in isolation, which let the model overflow the context window by issuing several whole-file reads from a single assistant message. Observed live: asked to synthesise three experiment JSON files into a report, the agent read them in parallel. The 285 KB file was correctly refused, but experiment1 (178 KB) and experiment3 (168 KB) each cleared the cap on their own — together they injected ~99 K tokens of raw JSON, producing a 136,469-token request against a 131,072-token window. The upstream returned HTTP 400 exceed_context_size, emergency pruning fired, and the prune-retry then crashed on an empty upstream body (see llm.py) — a recoverable overflow became a hard failure.

Two changes close the hole:

  • Lower the per-file factor. read_byte_budget(max_context) = max(150000, int(max_context * 3.5 * 0.40)) — down from 0.5 (which let one read eat ~70% of the window). At a 131 K window the per-file cap drops from ~229 KB to ~183 KB.
  • Cumulative ReadBudget per batch. The agent resets a ReadBudget (same byte allowance) before dispatching each assistant message's tool calls; every file_system read charges it by the content length actually injected (wired through the registry closure as context._read_budget). The FIRST read always proceeds (already bounded by the per-file cap), but a later read whose size exceeds the remaining allowance is refused before injecting — with a message steering the model to execute (load + summarise the data in a script), read_chunked, search, or inspect. Scope is one batch; cross-turn accumulation is handled by context compaction, so the budget deliberately does not persist. Passing no budget (read_budget=None) preserves the legacy per-file-only behaviour.

Result on the live scenario: experiment1 reads (charges ~178 KB), then experiment3 is refused (would breach the ~183 KB batch budget) instead of overflowing the window. Covered by tests/test_read_budget_overflow.py.

The budget is the backstop; the report-path prompt steering is the proactive half — it tells the model to execute a digest script for report/synthesis tasks so it never reaches the refusal.

Path resolution

_get_safe_path resolves every path against the sandbox root and rejects traversal. Beyond the leading-slash and /workspace/ handling, it also accepts a host-absolute sandbox path: if the LLM echoes back the full host path of a sandbox file (e.g. /Users/x/.../sandbox/report.md — which it sees in a sandbox-tree listing or a prior error and passes verbatim), the sandbox-root prefix is stripped and the remainder honoured as sandbox-relative. Without this the path joined onto the sandbox root into a phantom nested path → ENOENT → a wasted strike + retry. (2026-07-05) A leading ~ is expanded first (the model echoes the run command it just gave the user, e.g. ~/Data/AI/Data/sandbox/chess_client.py) and then flows through the same host-absolute mapping; a ~ expanding outside the sandbox falls through to the normal relative handling, preserving the traversal-safety invariant. Covered by tests/test_file_system_sandbox_path.py.

Project-scope fallback to the conversation binding (2026-06-20)

project_scoped_sandbox is the single source of truth for per-project file scoping. It reads context.current_project_id — but that is process-global and can be cleared MID-REQUEST by a concurrent conversation's reconcile_conversation. Observed live: a ~700s autoadvance-then-manual build had its scoping wiped partway through, so the agent's file_system writes silently landed in the sandbox ROOT; it then couldn't find its own files in the project workspace, tripped the loop-breaker, and thrashed for minutes. When current_project_id is empty, scoping now falls back to _conversation_bound_project(context) — the project bound to THIS conversation in the projects-tool scratchpad sentinels (a binding a concurrent conversation does not stomp; it only clears the global). A binding owned by another conversation is ignored, and stateful kernel sessions never scope. Covered by tests/test_project_scope_fallback.py.

Re-bind on referenced reactivation (2026-06-22). The above fallback only works when the binding's conversation matches THIS request's — but the chat API ships no stable conversation id, so a project keeps deactivating across requests and the fallback would still miss, dropping a report_pdf / file write to the bare sandbox ROOT instead of projects/<id>/ (observed live: a report PDF and stray project directories at the sandbox root). Now when reconcile_conversation reactivates a project because the request references it (title/id in the message or its Context preamble), it also re-binds the conversation to this request's key — so the _conversation_bound_project fallback resolves it for the rest of the request even if current_project_id is cleared mid-request. Covered by tests/test_project_conversation_scope.py.

Post-write syntax check (2026-06-11)

Every successful write path — tool_write_file plus all content-landing branches of tool_replace_text (aider blocks, exact, flexible, fuzzy, anchor, and the replace→write auto-promote) — now runs _syntax_feedback(path, filename) on the result and appends any failure to the SUCCESS message:

SUCCESS: Wrote 9241 chars to 'data.js'. …
⚠ SYNTAX CHECK FAILED: 'data.js' was written but does NOT parse. Fix this
BEFORE any other step — a browser loads a broken script silently and every
downstream symptom (dead buttons, blank page) traces back here:
/…/data.js:35
    'every decision you didn't make…',
SyntaxError: Unexpected identifier 't'
  • Checkers: .py → in-process ast.parse; .json → json.loads; .js/.mjs/.cjs → node --check (skipped silently when node isn't on PATH; 10s timeout); .html/.htm → each inline <script> block is extracted and node --checked, and a parse error is remapped from the block-local line back to the HTML file line (e.g. line 100: Invalid shorthand property initializer) so the model edits the right line. External (src=), type=module, and non-JS (text/babel, JSON) blocks are skipped to avoid false positives. Other extensions are not checked.
  • A fresh write always stands. The file is on disk either way — the agent needs to read its own broken file to fix it. The warning rides the same tool result, so the fix lands on the very next turn. A replace that introduces a syntax error does NOT stand — see the regression-rollback section below.
  • Fail-safe: any checker error returns "" — a broken checker can never fail a good write. A pretty_log WARNING mirrors the failure to the operator stream.

Production motivation (req EA → req 70, 2026-06-11): data.js/game.js shipped with unescaped apostrophes in single-quoted strings; four bare SUCCESS messages later the verifier confirmed a build that throws on load, and the bug cost an 807-second debugging session that still misdiagnosed it. Covered by tests/test_write_syntax_check.py.

Inline-<script> extension (2026-06-20): the single-file browser-OS benchmark builds in one turn now (a cohesive deliverable is no longer split per-feature — see manage_projects), so the whole app lives in one index.html with all JS inline. The .js checker never saw it: a one-character typo (content='…' written content= → Invalid shorthand property initializer) wrote a bare SUCCESS, the browser loaded a blank page, and the model spent ~760s + a thinking-loop abort re-reading 32KB chunks unable to localise it. _syntax_feedback now parses inline <script> blocks and names the offending HTML line at write time. Helpers _inline_js_blocks / _node_check_source / _remap_node_diag; covered by tests/test_html_syntax_feedback.py.

Replace indentation re-anchoring & regression rollback (2026-06-12)

Two defects in tool_replace_text, both surfaced by the GHOSTNET training session (session D4) where a single one-line fix snowballed into a ~10-turn replace → syntax error → re-read → full rewrite spiral that eventually dropped a whole optimizer class. They compounded each other.

Bug 1 — flexible match mangled indentation. The flexible matcher (r'\s+'.join(words)) matches starting at the first non-whitespace token, so the matched span excludes the leading indent of its first line — that indent stays in the file, before the insertion point. A raw str.replace(match, new_text) then placed line 1 of the replacement after the file's indent but every continuation line exactly as the model typed it. Whatever absolute indentation the model chose, the result broke: dedented → unindent does not match any outer indentation level; absolute → unexpected indent — the exact errors in the D4 log, four times in a row. _reindent_replacement() now drops the model's absolute indentation and rebuilds it from the anchor: line 1 gets none (the file supplies it), each continuation line gets the anchor indent plus its own indentation relative to the block's first line, so internal nesting is preserved and the block lands at the right column. Fires only when the match starts at column 0 of its line and the replacement is multi-line. Applied to both flexible paths (the main matcher and the aider-block fallback); exact/fuzzy/anchor matches already carry their own indentation and are untouched.

Root cause behind Bug 1 — unconditional markdown stripping. The replace path ran extract_code_from_markdown on old_text/new_text for every code extension, even with no fences present; its no-fence fallback .strip() dedented the first line only of an indented replacement, corrupting the block's relative structure before the matcher saw it. The extractor now runs only when ``` is actually present (the same guard the replace→write auto-promote path already used), preserving the block byte-for-byte otherwise.

Bug 2 — no rollback on introduced breakage. Every replace path wrote first and syntax-checked second, so a replace that turned a parsing file into a non-parsing one was persisted (reported SUCCESS + a warning). That destroyed the known-good anchor state the model needed for its next surgical edit — every subsequent SEARCH block then failed to match the now-corrupted file, which is what turned one broken edit into a multi-turn rewrite spiral. _write_replace_guarded() now checks the result in-memory before writing: if it would introduce a syntax error that the previous content didn't have (.py via ast.parse, .json via json.loads), the file is left unchanged and the model gets a REJECTED message steering it to a tight single-line edit instead of a rewrite. An already-broken file is still editable (the guard only fires on parse → no-parse), and non-Python/JSON files are never blocked. All five content-landing paths (aider, exact, flexible, fuzzy, anchor) route through the guard. Covered by tests/test_file_replace_indent_and_rollback.py and the updated tests/test_write_syntax_check.py.

Replace-argument corruption guard + block-form parity (2026-07-05)

Trajectory measurement over 5 days (2026-07-01 → 07-05): of 99 replace calls, 34 failed "search block NOT found" — and 27 of those 34 arrived with content and replace_with byte-identical. The upstream native-tools argument transport (the same llama-server parser that once merged multi-tool replies into one call's args — the introspect incident) delivers the model's NEW block in both arguments, so the tool searches for text that doesn't exist in the file yet. No matcher improvement can rescue that. Worse, ~23 more identical-args calls "succeeded" as silent no-ops: the identical text still matched the file and was replaced with itself, leaving the model convinced a fix landed that never did. The operator-visible symptom was misdiagnosed for days as "the model writes sloppy SEARCH blocks".

  • Identical-args guard. content == replace_with (non-aider form) is rejected before any file I/O — covering both the NOT-found path and the silent no-op path — with a ⚠️ File Replace Corruption Guard log line and a SYSTEM INSTRUCTION that teaches the single-argument form verbatim. replace_with="" (a legitimate deletion) is unaffected.
  • Single-argument form promoted to primary. The registry schema now steers the model to put ONE <<<< SEARCH … ==== … >>>> block in content and omit replace_with: the old→new pair rides in a single argument, which an argument-merge cannot corrupt. The two-argument form remains as legacy.
  • Matching-ladder parity for the block form. The aider-block loop previously only had exact + whitespace-flexible matching; it now runs the same full ladder as the two-argument form via _locate_block() (exact → flexible-unique → fuzzy window → anchor block), reports which blocks needed tolerant matching ("rescued by … — VERIFY"), and flags per-block no-ops (SEARCH == REPLACE, the same corruption signature) without sinking the remaining blocks.
  • Indentation-preserving block parse. The old block regex's \s* swallowed the leading whitespace of each block's first line, which (a) made exact matches impossible for indented blocks and (b) mis-computed _reindent_replacement's indent base, so fuzzy/anchor insertions landed a dedented first line. Blocks are now captured starting on the line after each marker (first-line indent survives; exact matches fire); the loose regex remains as fallback for marker-line content.

Covered by tests/test_replace_arg_corruption_guard.py.
