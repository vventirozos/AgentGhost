tools / execute.py

Sandboxed code execution with multiple validation guards and stateful Jupyter support.

Tool: execute

Three execution modes selected from arguments:

1. Bash command (line 32-117)

  • Auto-strips invalid cd /sandbox / cd /home/user prefixes.
  • Handles substantive inline -c scripts (≥120 chars, 2+ statements, or import+nested-quotes). The match fires after an optional command-separator prefix — cd <dir> && python3 -c "…", not only a start-anchored python3 -c — because the model almost always works inside a project subdir. (2026-06-30) Auto-convert, don't just reject. The block used to reject every substantive body and force the model to re-emit as file_system(write)+execute — a wasted turn each, fired on perfectly well-formed scripts (the operator saw a steady stream of 🛡️ Inline Script Blocked warnings on multi-statement one-liners and long bodies that would have run fine). Now a well-formed body is auto-converted into an in-sandbox file run: the body is extracted with shlex (byte-identical to bash's own unquoting) and shipped into the container via base64 (printf %s <b64> | base64 -d > /tmp/_ghost_inline_*.py && python3 /tmp/_ghost_inline_*.py), so bash quoting can't corrupt it. Any cd … && prefix is preserved so relative paths resolve as before; bash -c bodies convert to a .sh file. The auto-convert fires only when it is provably quote-safe: the captured body contains no unescaped copy of the outer delimiter quote (an escaped \" is fine — shlex unescapes it correctly). This precise check replaces the old import+both-quote-types heuristic, which over-blocked valid escaped bodies. Covered by tests/test_inline_c_autoconvert.py and tests/test_inline_c_guard_cd_prefix.py.
  • Still hard-BLOCK (no auto-convert) only for the two shapes a redirect genuinely helps: a body with an unescaped nested delimiter quote (the real bash-split corruption shape) or that shlex can't parse / has a trailing pipe-or-redirect we won't faithfully reconstruct — re-emitting cleanly via write+execute is the fix; and an acquired-skill call wrapped in -c (see next bullet).
  • Acquired-skill invocation nudge (2026-04-24). When the blocked body looks like a failed attempt to call an acquired skill — from X import X where module and symbol names match, or acquired_skills/X.py in a subprocess call — the SYSTEM BLOCK error appends a targeted HINT: pointing at the canonical invocation: "Acquired skills are TOP-LEVEL TOOLS. Invoke it directly as X(...)." This shortcut exists because after the skill-storage relocation (skills live under $GHOST_HOME/system/memory/acquired_skills/, not inside the sandbox), the LLM can no longer see the skill via ls and its first instinct is often to wrap the call in python -c / subprocess. Hint fires only on the narrow patterns above; a body that blocks for a non-skill reason keeps the generic message, and an unrelated quote-safe body now auto-converts and runs (no block at all). Covered by tests/test_acquired_skill_invocation_nudge.py.
  • Executes via sandbox_manager.execute(bash -c ...) with 300 s timeout.

Path & exit-code healing (2026-07-02)

Four fixes from the 2026-07-01 chess-game trace (rounds a4e1b33d / a4a2ce04 in ghost-agent.log), where the agent looped 10+ turns on files it had just written. All covered by tests/test_execute_path_and_exit1_heal.py.

  • File-run path derives from the healed host path. execute(filename=...) used to derive the container-side run path with a hand-rolled strip (lstrip("/") + drop a sandbox/ prefix) that had diverged from _get_safe_path's healing: filename="/workspace/game.py" under an active project WROTE <proj>/game.py but RAN python3 -u workspace/game.py → ENOENT, every retry identical. The run path is now host_path.relative_to(sandbox_dir), so write and run always target the same file (sandbox/foo.py symmetry included — the old code stripped that prefix on run but not on write).
  • /workspace → project-workdir remap fallback. When a project is active, file_system heals /workspace/game.py into the project dir, but a shell command's absolute /workspace/game.py stayed at the sandbox root — so file read succeeded while python3 /workspace/game.py ENOENT'd, an asymmetry the model cannot see. Now, if a project-scoped shell command fails with a file-not-found signature and references /workspace (word-boundary match, and not already the scoped dir), it is retried once with /workspace remapped to /workspace/projects/<id>. On success the result carries a [SYSTEM NOTE: … remapped …] so the model learns the real layout; on failure the ORIGINAL error (the path the model asked for) is kept. Commands with no /workspace reference keep the pre-existing root-cwd retry.
  • grep-family exit 1 with no output = "no matches", not a strike. grep/egrep/fgrep/zgrep/rg/pgrep exit 1 to mean "no matches" — the agent loop scored that as an execution failure (Strike 1/6 … [SYSTEM ERROR]: Process failed (Exit 1) with no output, observed on a grep that was correctly proving a fix had landed). Exit exactly 1 with no output (including the docker layer's [SYSTEM ERROR]…with no output. sentinel), where the LAST pipeline segment's command is in the grep family, now returns a success-shaped result (EXIT CODE: 0) stating that exit 1 meant NOT FOUND. Genuine grep errors (exit 2, stderr output) still fail.
  • Inline -c auto-convert keeps cwd import semantics. python -c "…" puts the CWD on sys.path; the converted /tmp/_ghost_inline_*.py file run put /tmp there instead, so a body importing a module in the working directory (from chess_engine import Board) failed with ModuleNotFoundError — only BECAUSE of the conversion. The converted python run now carries PYTHONPATH="$PWD${PYTHONPATH:+:$PYTHONPATH}"; bash -c conversions are untouched.

2. File execution (line 119-494)

  • Validates extension: only .py, .sh, .js.
  • Syntax check via sanitize_code().
  • Forbidden module guard: rejects imports of native tool modules (knowledge_base, execute, postgres_admin, …).
  • HTML guard: HTML/CSS detected in a .py file → redirect user to file_system.
  • "Stubbornness" guard: skip rewriting an unchanged file.

3. Stateful Jupyter (line 198-306)

  • Boots persistent ipykernel_launcher if not running.
  • Wraps via jupyter_client.BlockingKernelClient.
  • Calls kc.wait_for_ready(timeout=10) BEFORE kc.execute() — without this, is_alive() false-negatives for ~1 s after start_channels() while the heartbeat channel hasn't exchanged yet. The runner's except queue.Empty branch reads is_alive() as a "kernel died?" probe, so every FIRST stateful execute of a session used to emit [SYSTEM ERROR: Kernel Terminated Abruptly (Did the script call os._exit()?)] even though the script hadn't run. wait_for_ready closes the race; a genuine launch failure surfaces as a distinct [SYSTEM ERROR: Kernel did not become ready within 10s] message so reflection can distinguish launch failure from runtime crash. Covered by tests/test_execute_jupyter_wait_for_ready.py.
  • Streams stdout / stderr; detects 5-min timeout or mid-execution kernel death.

Output handling — one policy + spill-to-file (2026-07-07)

Truncation is owned by the sandbox layer via one shared helper, utils.text_truncate.truncate_head_tail(text, budget, label) (head 25% / tail 75% by default — tracebacks print the exception at the end). Before, four call sites each had their own budget and message shape (docker 256 KB, execute.py's dead 512 KB layer that never fired behind docker, file_search 40 KB, browser 64 KB); at --max-context 240000 docker's 256 KB was the only real gate, so one noisy pip install / test run could inject ~70 K tokens that then persisted in history and re-prefilled every subsequent turn.

The execute tool now runs in spill mode (DockerSandbox.execute(spill_large_output=True)): the returned view is small (~24 KB head+tail) and the FULL output is written to a workspace run-log .ghost_runs/run_<n>.log (disk-bounded, 10 MB ceiling) with a pointer telling the model to inspect it via file_system (operation='search' for lines, operation='read' with start_line/end_line). Truncation becomes an affordance instead of information loss. Both the script path and the bare-bash path (which previously had no tool-level trim at all) use it. Non-execute callers of sandbox_manager.execute (ripgrep search, find, browser) keep the legacy 256 KB head+tail and do not spill. The dead 512 KB execute.py layer is removed. Covered by tests/test_execute_output_spill.py.

Safety guards summary

  • Forbidden imports — regex rejection in Python source.
  • HTML/CSS guard — detects and rejects.
  • SQL injection — out of scope (database tool uses parameterised queries).
  • Inline script handling — auto-converts substantive -c usage to a base64-transported file run; hard-blocks only unsafe/skill-wrap shapes.
  • Output truncation — head + tail capping.

Import guards relaxed off the small-model era (2026-06-22)

Both import-related guards over-fired on a strong model's legitimate code:

  • Inline -c: an import is no longer a block trigger by itself. The real corruption mode is NESTED QUOTES in the inline body mangled by bash escaping, so an import only triggers the block when the body ALSO mixes both quote types. A clean probe like python3 -c "import sys; print(sys.path)" — which the block message itself used to advertise as allowed, a self-contradiction — now passes. (2026-06-30 superseded) the 120-char / 2-semicolon / import+nested-quote cases no longer redirect at all when they're quote-safe — they auto-convert to a file run (see the Bash-command guards above). Only an unescaped-nested-quote body, an unparseable/piped form, or an acquired-skill wrap still blocks.
  • Native-tool import blocklist narrowed. The generic names file_system (a very common module name), scratchpad, recall, replan were REMOVED — they collide with local modules (a local file_system.py helper was getting hard-blocked). browser is KEPT (a prominent native tool the model does hallucinate importing, with a dedicated guard test). The other unambiguous tool names (knowledge_base, web_search, vision_analysis, …) remain. Covered by tests/test_inline_c_guard_cd_prefix.py and tests/test_acquired_skill_invocation_nudge.py.

Sandbox egress guard — the agent's own ports (2026-07-05)

The sandbox container has its own loopback, so 127.0.0.1:8000 in there is NOT the agent's API and :8088 is not the upstream LLM. Three separate incidents (2026-07-02, 07-04, 07-05 request C5) followed the same script: the model curl-tests the port from inside the sandbox, reads the connection failure as "the server is down on the user's machine", and writes a mock/stand-in server — the forbidden engine. The CRITICAL SANDBOX FACT prompt rule did not prevent the third incident (the model quoted it in its own thinking, then trusted its curl anyway), so the probe itself is now intercepted.

tool_execute scans BOTH the shell command and inline code content for (127.0.0.1|localhost|0.0.0.0):(8000|8088) before any execution. A hit returns a ground-truth explanation (🛡️ Sandbox Egress Guard log line) instead of running: the servers are fine on the host, a failure in here proves nothing, verify via the browser tool or ask the user, bind a different port (e.g. 8081) for genuine in-sandbox servers, and NEVER write a mock. Returned as a plain instruction — not an EXIT CODE failure — so it teaches without burning a strike. Deliberately narrow: colon-forms only (an nc -z 0.0.0.0 8000 without a colon passes — acceptable false negative; the guard exists to break the curl-misdiagnosis chain, not to firewall). Covered by tests/test_sandbox_agent_port_guard.py.

Host-process blind spot (2026-07-08)

The sandbox has its own PID namespace: a process the USER started on the host (their python app.py) is invisible and unkillable from inside. A pkill -f app.py therefore succeeds — exit 0, no output, nothing killed — and the model concludes it restarted the server, then keeps debugging against stale code (observed twice in the chess session). Same family as the loopback blind spot: the exit code lies, so the tool must tell the truth. A name-based kill (pkill/killall/kill $(pgrep …)) now appends a ground-truth note to the result on BOTH outcomes — the "successful" one is the dangerous case — telling the model to ask the user to restart the process instead. The execute schema warns up front. Covered by tests/test_host_process_and_verdict_clarity.py.

Egress guard: usage-context matching (2026-07-08)

The agent-port egress guard previously blocked any command or content mentioning 127.0.0.1:8000/:8088. Observed false positive (chess session): a heredoc writing a file whose text legitimately references the agent's API was blocked — composing with the pre-flight guard to leave the model zero legal write paths. Shell commands are now judged by _command_probes_agent_port: heredoc bodies are stripped (they are data, not execution) and a network-client token (curl/wget/nc/requests/urllib/httpx/…) must co-occur with the URL. Direct probes stay blocked, including after a heredoc; echo/sed text manipulation is allowed. Inline content (code that executes) keeps the strict any-match rule, and the guard message now names the legal path for file-writing (file_system write / heredoc). Covered by tests/test_guard_box_fixes.py.
