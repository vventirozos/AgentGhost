tools / composed_skills.py

Compile recurring tool-call sequences into reusable macros with conditional branching and parameter templating.

Data shapes

TypeFields
SkillSteptool_name · description · param_template · branch_condition · branch_target · optional
ComposedSkillname · trigger_description · steps · branches · execution_mode · status · usage_count · success_count · last_used · created_at

Templates

param_template uses $variable syntax; resolved against the runtime params when the macro is dispatched.

Registry

  • Up to 50 skills in memory; persisted to composed_skills.json.
  • On overflow the least-used skill is evicted — but proposed drafts are evicted before any active macro, so a flood of auto-proposals can never push out a macro the user approved or uses.
  • compile_from_pattern(pattern_name, tool_sequence, description, *, status="proposed", execution_mode="sequential") creates a macro from a mined sequence. It defaults to proposed status (see auto-proposals).
  • execute(skill_name, executor_fn, params) dispatches on execution_mode (see below) and returns a result dict with success · results · steps_completed · total_steps · mode.
  • record_usage(skill_name, success) updates counters.

Status: active vs. proposed

The status field gates visibility:

  • active (default for user-defined macros) — advertised to the LLM via to_tool_definitions() and dispatchable via register_composed_skill_runners().
  • proposed — an auto-discovered draft. Stored and listable, but deliberately hidden from the tool catalogue and not dispatchable until the user approves it with manage_composed_skills(action="approve"). This keeps possibly-stale mined macros out of the live tool list until a human signs off.

Execution modes

execution_mode selects how execute() runs the steps:

  • sequential (default) — steps run in order; branch_condition is evaluated against each result to optionally jump into a named branch; an optional=true step that fails does not abort the macro. Best for pipelines where a later step consumes an earlier step's output.
  • parallel — every step is fanned out concurrently via asyncio.gather and all results are collected. Branching is ignored (there is no ordered result to test). A non-optional step failing marks the macro partial/failed but never aborts the fan-out. Best for an independent, read-only bundle like a morning briefing.

Per-step result cap

Each step's result body is bounded to MAX_STEP_RESULT_CHARS (4000 chars) by _cap_step_result() before the per-step blocks are assembled, so one chatty step can't blow the context budget when the combined macro output is handed back to the LLM.

Truncation is explicit, never silent. When a step exceeds the cap, the body is sliced and an …[truncated N chars — re-run this step's tool standalone…] marker is appended. The model (and the verifier gate) can then see that content was dropped and re-fetch the step on its own, instead of believing it received the whole result and delivering a short answer.

Regression fixed: the cap was previously a fixed [:1000] slice applied silently. A list-bearing step — the morning_briefing macro's "latest 10 headlines" fetch (well over 1000 chars for 10 items) — was truncated down to ~2 headlines, and the model, with no signal that anything was missing, delivered a 2-headline briefing. Raising the cap to carry a realistic 10-item list and marking any truncation explicitly closes both halves of that bug. Tests: tests/test_composed_skill_step_truncation.py.

Defining a macro: manage_composed_skills

The manage_composed_skills tool lets the agent (or user) declare a named bundle on demand — action ∈ define · list · approve · delete. A defined (or approved) macro becomes a top-level tool the agent invokes by name; its steps run and the combined results return for the model to synthesise. list shows active macros and, separately, any auto-discovered proposed ones; approve activates a proposed macro; delete removes (or rejects) one. Macro names must be bare identifiers (they become tool names), and a step may only call a built-in tool or an acquired skill — not another composed skill (no nesting / recursion).

Example — a parallel morning_briefing:

manage_composed_skills(
  action="define",
  name="morning_briefing",
  description="daily briefing: weather, headlines, diagnostics, lessons",
  mode="parallel",
  steps=[
    {"tool": "system_utility", "description": "Weather",     "params": {"action": "check_weather"}},
    {"tool": "web_search",     "description": "Headlines",   "params": {"query": "latest news headlines"}},
    {"tool": "system_utility", "description": "Diagnostics", "params": {"action": "check_health"}},
    {"tool": "list_lessons",   "description": "Lessons",     "params": {"scope": "today"}},
  ],
)
# → 'morning_briefing' is now a tool. Calling it fans the four steps out
#   concurrently and returns all four panels for synthesis.

A step's params value of "$varname" is substituted from the macro's own runtime arguments at call time, so a macro can take parameters too.

Auto-discovered macro proposals

The dream cycle mines the agent's own trajectory log (distill/ — ordered tool calls with arguments, across sessions) for recurring tool-call sequences and registers the strongest as proposed macros. The user reviews them with manage_composed_skills(action="list") and activates the good ones with action="approve".

  • mine_recurring_tool_sequences(trajectories, …) (in core/dream.py) finds contiguous tool-name windows (length 2–4) that appear in ≥ min_support distinct trajectories. Failed trajectories are skipped, single-tool repeats and all-meta windows are dropped, and a sub-window subsumed by a same-support longer window is discarded.
  • Each step's params is the most common argument set observed at that position — so an approved macro replays realistic arguments, not an empty skeleton.
  • Dreamer._propose_macros_sync() walks the collector, mines, and calls compile_from_pattern(..., status="proposed") for the top few — skipping any signature already on file so the same sequence isn't re-proposed every REM cycle. Each proposal is surfaced via a pretty_log line and counted in the dream summary ("N macros proposed").

Collector source. The live recording collector is wired onto the context by main.py during lifespan (context.trajectory_collector = TrajectoryCollector(...), under $GHOST_HOME/trajectories, disabled by --no-trajectories). When the dream cycle runs on a context that lacks the live collector (headless / cron, or a context built outside lifespan), _propose_macros_sync falls back to a read-only collector at the same canonical on-disk root, so mining still runs over the recorded corpus — unless --no-trajectories is set, which it honours.

LLM exposure & dispatch wiring

  • to_tool_definitions() renders each active composed skill as an OpenAI-style tool definition (proposed drafts are skipped).
  • register_composed_skills(tool_defs, context) appends those definitions into the active tool list (called from registry.get_active_tool_definitions) — this is what makes the LLM see the macro.
  • register_composed_skill_runners(tools, context) wires a callable runner per active macro into the dispatch dict (called from registry.get_available_tools) — this is what makes the macro executable. Each runner dispatches steps through build_step_executor, which blocks nested composed-skill calls.
  • After a define / approve / delete, the agent loop calls invalidate_tool_defs() so the macro appears (or disappears) in the very next turn-iteration rather than waiting for the next user message.

Macros persist to composed_skills.json under memory_dir, so they survive sandbox wipes (same rationale as acquired skills).
