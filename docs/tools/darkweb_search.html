tools / darkweb_search.py

Discovery of Tor hidden services (.onion) via dedicated onion search engines, with per-attempt circuit rotation, cross-engine corroboration ranking, and an optional deep-read research mode.

Why a separate tool

The clearnet search tool (search.py) queries scraper backends — DuckDuckGo, Brave, Mojeek — that do not index Tor hidden services. Discovering .onion content requires querying dedicated onion search engines (Ahmia, Torch, Haystak, …) and parsing their result pages. The transport is the same Tor SOCKS proxy the rest of the agent already uses; what is new is the engine set and the result parsing.

Tools

ToolPurpose
darkweb_searchFan a query across the onion-engine set over Tor, parse the .onion result links/titles/snippets, de-dup by onion host, rank by cross-engine corroboration, and return a ranked list. List-only — pair with browser / darkweb_research to actually read a result.
darkweb_researchThe dark-web analogue of deep_research: runs darkweb_search, then fetches the top onion pages over Tor and distils each via the worker LLM into a synthesised report.

Onion-engine registry (_ONION_ENGINES)

Onion search-engine endpoints rotate and die far more often than clearnet ones, so the set is config-driven and intentionally wide — breadth plus circuit rotation wins, the same philosophy as search.py's _TOR_BACKENDS. The built-in default is:

  • ahmia — the clearnet mirror ahmia.fi (still fetched over Tor); indexes onion services and filters known-abuse material at the index level.
  • ahmia-onion — Ahmia's own .onion endpoint. Listed alongside the clearnet mirror on purpose: the two have independent reachability, so one is often up when the other is blocked. They serve the same search index, though, so both carry index: "ahmia" — reaching Ahmia over two transports must not be mistaken for independent corroboration (see ranking below).
  • torch — a broad unfiltered onion index, for reach.

On dead endpoints: haystak was in the initial default set but was dropped after live functional testing — its v3 endpoint failed on every attempt across many rotated Tor circuits (SOCKS5 connect / timeout), i.e. the hidden service is down rather than exit-node blocked, and its retry could burn a full _ONION_TIMEOUT and inflate the tail latency of the whole (concurrent) call. It — or any replacement onion engine — can be re-added via GHOST_ONION_ENGINES once a live endpoint is known. This is the expected maintenance mode for onion engines, which rotate and die far more often than clearnet ones.

The whole set is overridable at runtime via the GHOST_ONION_ENGINES env var (a JSON list of {"name","url"[,"index"]} objects, where url contains a single {q} placeholder). The optional index marks endpoints that share one underlying search index so they count once in corroboration ranking; omitted, it defaults to the entry's own name (treated as an independent index). A malformed override falls back to the built-in default rather than silently disabling dark-web search, and an individual entry whose URL carries a stray placeholder other than {q} (which would raise a KeyError in .format on every query and be silently skipped) is rejected at load with a warning instead.

Circuit rotation + timeouts (reused from search.py)

Onion search engine reachability is exit-node dependent in exactly the same way clearnet search is, so the same per-attempt circuit rotation applies: _proxy_for_attempt (imported from search.py) folds the attempt index into the SOCKS username:password, and Tor's IsolateSOCKSAuth hands each attempt a different exit node. Each engine is tried up to 2 times, each on a fresh circuit, and all engines are queried concurrently.

Timeouts are deliberately higher than clearnet search. Onion round-trips are materially slower: _ONION_TIMEOUT = 30 s per engine request (vs. search.py's _DDGS_TOR_TIMEOUT = 18 s), and _ONION_PAGE_TIMEOUT = 35 s for each onion page fetch in the research phase. These clear a healthy onion service while still letting dead endpoints fail fast on connect error. Like the clearnet timeout, they are empirically tunable.

Per-engine deadline (dead-engine hardening). Unlike search.py — which hands its whole backend set to ddgs in a single call bounded by one shared timeout — this tool fans the engines out itself with asyncio.gather, so the search can only finish once the slowest engine returns. A dead or hung engine that burns its per-attempt timeout twice (~2×_ONION_TIMEOUT) would therefore dominate the gather and inflate every call's latency. To bound this structurally (not just by curating the engine list), each engine runs under an overall _ONION_ENGINE_DEADLINE = _ONION_TIMEOUT + 8 s via asyncio.wait_for: it allows one full attempt (a slow-but-alive engine legitimately needs up to _ONION_TIMEOUT) plus a short window for a fast second circuit, then skips the engine so the gather proceeds. The underlying fetch runs in a worker thread with its own curl/httpx timeout, so it is not force-killed — cancelling the await just releases the gather. This makes engine curation a tuning aid rather than the only thing standing between a newly-dead engine and a slow tool. Covered by tests/test_darkweb_search.py::test_slow_engine_bounded_by_deadline.

socks5h — mandatory for .onion

_normalize_tor_proxy forces the proxy to socks5h:// form (note the h), which routes DNS resolution through the proxy. This is mandatory: .onion has no clearnet DNS, so a plain socks5:// proxy (local resolution) would fail to resolve every hidden service. The SSRF guard shared by the fetch helpers treats a DNS-resolution failure as allow, so .onion hosts pass it cleanly.

Result parsing & ranking

Each engine renders results differently, so _parse_onion_results is deliberately tolerant: it walks anchors to harvest (onion-url, title) pairs with a nearby snippet, unwraps Ahmia-style redirect_url= wrappers, then does a regex sweep of the whole document to catch onion URLs that appeared as plain text. Addresses are matched as base32 v3 (56 chars) or legacy v2 (16 chars). Results are de-duped by onion host.

Self-link filtering. A tolerant whole-page harvest also picks up the nav/footer/sidebar links every engine page carries — including the engine's own onion address and those of sibling engines. Those are excluded (_engine_onion_hosts) so an engine's home link can't masquerade as a result and, worse, ride the corroboration ranking to the top.

Corroboration by independent index, not by endpoint. Onions surfaced by more than one independent index sort first — cross-engine corroboration is the only cheap relevance signal available for unindexed hidden services. Crucially the count is over distinct index tags, not engine names: Ahmia's clearnet and onion endpoints share index: "ahmia", so a hit reached over both Ahmia transports counts as one source, not two. Counting endpoints instead would have let any Ahmia-only result outrank a genuinely Ahmia-plus-Torch-corroborated one — turning "corroboration" into "is it in Ahmia."

Raw-HTML fetch vs. text fetch

The search phase needs the raw result HTML to parse out links, so it uses a dedicated _fetch_raw_html (curl_cffi with TLS impersonation, falling back to httpx). The research phase wants page text, so it wraps the same fetch in _fetch_onion_text, which strips tags after the fact. It deliberately does not reuse the shared helper_fetch_url_content, for two reasons: (1) that helper ignores the passed proxy and reads TOR_PROXY from the environment, silently discarding both a caller-supplied proxy and the per-query anonymity identity tag; and (2) on any fetch error or 503 it calls request_new_tor_identity, which does a global NEWNYM or — if the control port is closed — a tor service restart. Since onion fetches fail often and research fans them out concurrently, that would re-circuit or restart Tor mid-run and sabotage the sibling fetches. _fetch_onion_text keeps the shared SSRF guard but honours the passed proxy and never re-circuits.

Untrusted-body cap. Onion engines are adversarial by this tool's own posture, so _fetch_raw_html caps every response at _MAX_ONION_BODY_BYTES (5 MB): it refuses binary content-types and bodies whose declared Content-Length exceeds the ceiling, and truncates the decoded text as a backstop against a server that lied about its size or streamed chunked (mirroring the cap helper_fetch_url_content enforces on clearnet). Without it a hostile engine could stream a multi-GB body straight into memory and then into BeautifulSoup plus a full-document regex sweep.

Dedicated fetch pool. The blocking fetch runs on a bounded _ONION_EXECUTOR rather than the default thread pool. When the per-engine deadline fires, asyncio.wait_for cancels the await but cannot kill the worker thread — it runs on until curl's own timeout. Isolating those threads means a lingering post-deadline fetch can queue behind the pool but can never exhaust the process-wide executor the rest of the agent depends on.

Anonymous mode

Under --anonymous, the same stylometric egress scrubbing as clearnet search applies: darkweb_search uses the deterministic scrub_query plus a per-query SOCKS identity tag; darkweb_research uses the stronger LLM-based neutralize_query. Tor anonymises the network; this anonymises the content fingerprint of the query before it leaves the box. Because the research deep-read now fetches onion pages through the caller's proxy (via _fetch_onion_text) rather than the environment default, the per-query identity tag actually reaches the page fetches too, instead of being discarded at the transport boundary.

Deep-read sizing & caching

The per-source extract in darkweb_research is sized to the worker model's context window (max_context) rather than a fixed 40k chars: it reserves room for the prompt scaffold plus max_tokens and shrinks the source slice accordingly (≈4 chars/token, with a 4k-char floor and the 40k ceiling), so a small-context worker can't overflow on the distill call. The synthesised report is cached under an onion-research:: key (distinct from the list search's onion::), so a repeated identical research request is served from cache instead of re-fetching every onion over Tor.

Safety posture

Hidden services are unverified and frequently malicious. The tools never auto-execute or auto-download anything — darkweb_search returns links only, and darkweb_research returns extracted text with an explicit instruction to treat all claims as unverified and corroborate before relying on them. Every reached onion host is logged to the operator's live stream via pretty_log with a distinct 🧅 icon.

Tests

Parsing (anchors, redirect unwrapping, plain-text sweep, de-dup), independent-index corroboration ranking (the two Ahmia endpoints are not double-counted), engine self-link filtering, the per-engine deadline, the untrusted-body size cap (_cap_body: binary refusal, declared-oversize refusal, truncation), the actionable zero-results error, anonymous-mode scrubbing, the GHOST_ONION_ENGINES override + malformed fallback + stray-placeholder rejection, socks5h normalisation, result caching, the research deep-read path honouring the passed proxy, research caching, and max_context-bounded extraction are covered by tests/test_darkweb_search.py.
