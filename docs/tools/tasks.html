tools / tasks.py

Cron + interval background-job scheduling via APScheduler.

Tool: manage_tasks

ActionArgsBehaviour
createtask_name, cron_expression, promptAdd a TIME-based job: "0 9 * * *" → CronTrigger.from_crontab(); "interval:60" → add_job(..., 'interval', seconds=60). A malformed interval: spec (non-integer, e.g. interval:5m, or ≤0) is rejected with an error — it no longer silently falls back to 60 s while reporting SUCCESS with the original expression (2026-07-03).
watch 2026-07-16task_name, check_command, prompt, interval_secsREACTIVE task — fires on a CONDITION, not a clock. Polls check_command every interval_secs (≥10) in the agent's sandbox; when it first exits 0 (shell if semantics — the command succeeds when the thing to react to is true), it fires prompt as a background turn with the check's output attached. Edge-triggered: fires on the transition to true and re-arms only when the condition clears, so a persistent condition doesn't spam. Runs in the sandbox (reaches LAN/tailnet directly per the egress guard), so ops checks work: "grep -q ' ERROR ' /path/app.log", "! curl -sf https://host/health" (down), "[ $(metric) -gt 90 ]" (threshold). The edge state (last_fired) persists in the task store, so a restart doesn't re-fire a condition that was already true. Why: the scheduler could only fire on time; watching external state (a log, an endpoint, a metric) is a distinct, ops-shaped capability. Tests: tests/test_watch_condition.py.
list—Returns all active jobs (time-based and watches).
stoptask_identifierRemove a single job by id.
stop_all—Wipe all scheduled jobs.

The action value is normalised (.strip().lower()) like the other action-dispatch tools, so "Create" / " list " dispatch correctly instead of falling through to "unknown action". A failure to write the bookkeeping memory entry after a job is scheduled is logged as a warning but does not turn a successful schedule into a reported ERROR.

The job body is the prompt that will be re-injected into the agent on the trigger schedule. APScheduler is bound to the FastAPI lifespan and shut down with wait=False so server restart doesn't block on long-running jobs.

Persistence across restarts (2026-07-14)

The bug this closes: the AsyncIOScheduler jobstore is in-memory, and the operator deploys by killing the agent — so every deploy silently WIPED all user cron tasks, while the task X is running note written to vector memory kept asserting they were alive. Scheduled tasks run with nobody watching the output, so nothing ever surfaced the loss.

Every create now also writes the task (name, prompt, expression) to $GHOST_HOME/system/scheduled_tasks.json (atomic tmp+replace; persisted only AFTER the live registration succeeds, so the store never holds an unschedulable task). stop/stop_all unpersist. At lifespan start, restore_persisted_tasks() re-registers everything in the store through the same _add_job trigger-builder the create path uses — a malformed record is skipped with a warning and dropped, never aborting the rest. No memory_dir (degraded/test contexts) ⇒ persistence is a clean no-op with the old semantics. Covered by tests/test_tasks_persistence.py.

Timezone

The scheduler runs on UTC (AsyncIOScheduler(timezone="UTC")): cron expressions are interpreted in UTC and list output is labelled accordingly — a Next Run: 06:00 read as local time would make a correctly-firing task look 2-3 hours late (or early) from Athens. The manage_tasks schema now tells the model to convert local-time requests to UTC when building the expression. (Deliberately NOT switched to the host timezone: existing expressions were authored under UTC semantics, and silently shifting their firing times on upgrade would be its own invisible bug.)
