tools / report_pdf.py

Native PDF report generator. Produces a multi-page A4 document from markdown sections and writes it into the sandbox so it can be downloaded through /api/download/<file>.

Tool: report_pdf

Always advertised — there is no remote-service gate (unlike image_generation, which depends on the GPU swarm). Rendering happens locally and offline. The tool slots into the same artefact lifecycle as image generation: write a binary file into sandbox_dir, return a markdown link the LLM is told to emit verbatim so the user can open it.

Why a native tool instead of execute

  • One LLM turn instead of three (no Python-authoring detour).
  • A structured sections=[{heading, body}] schema forces report-shaped output rather than a wall of prose on a single page.
  • The web client and Slack bot already understand the /api/download/<file> markdown convention; PDFs slot into the same auth-fetch + visualizer pipeline as images for free.

Engine

PyMuPDF (fitz) + the pure-Python markdown library:

  • markdown.markdown(body, extensions=["extra","sane_lists","tables","nl2br"]) renders each section body into HTML.
  • A small inline CSS block styles headings, code, tables, lists, blockquotes.
  • fitz.Story(html=...) + fitz.DocumentWriter flow the assembled HTML across as many A4 pages as needed (hard-capped at 200 pages to guard against runaway outputs).

PyMuPDF is already a hard dependency (it backs vision_analysis and file_system's read_chunked path), so this adds no new system-level libraries — unlike WeasyPrint, which would pull in Pango and Cairo.

Parameters

  • title — H1 on page 1. REQUIRED.
  • sections — list of {heading, body}. body is markdown. May also be a single markdown string; the healer wraps it in one section. REQUIRED.
  • subtitle — optional tagline under the title.
  • author — optional byline.

Parameter healing

Mirrors the kwarg-aliasing in tool_generate_image. Self-play logs show the LLM regularly hallucinates parameter names (body, content, text, report_title, name). The tool accepts all of them rather than failing hard. HTML-escapes title, subtitle, and author so a stray < can't break the document.

Return value

SUCCESS: PDF report generated (N page(s), KB.K KB). DO NOT CALL THIS TOOL
AGAIN with the same title. Respond DIRECTLY to the user by including this
exact markdown so the file is offered for download:

[📄 <title> (PDF)](/api/download/report_XXXXXXXX.pdf)

Failure modes return SYSTEM ERROR: ... strings (missing title, missing sections, bodies over 200 000 chars, render exceptions). On render failure the partial file is unlinked so the returned link never points at a corrupt artefact.

Example prompt

>>> Look up the top 5 upcoming tech trends for next year, compile a
    market analysis report, and save it as a PDF.

The agent fans out research via deep_research / delegate_to_swarm, assembles findings in the scratchpad, then makes one report_pdf call with section-shaped {heading, body} entries. The reply ends with the markdown download link, which the web client auto-promotes into a clickable card with inline PDF preview.
