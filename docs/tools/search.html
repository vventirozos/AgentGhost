tools / search.py

Anonymous web search via Tor, over a Tor-reliable backend set, with operator sanitization, a short result cache, reformulated fallbacks and multi-source synthesis.

Tools

web_searchSearch via ddgs over a Tor SOCKS proxy, against a curated backend set (mojeek,duckduckgo,yandex,brave,google). Backend set + circuit rotation (Tor reliability)

ddgs ≥ 9 defaults to backend="auto", which fans every query across all text engines. Two are excluded:

  • wikipedia — its engine reads region="wt-wt" as language wt and builds the host wt.wikipedia.org, which doesn't exist, so it always raises a ConnectError over Tor.
  • yahoo — never returned useful results in testing and is a known hang-until-timeout offender; pure latency risk, no upside.

The current set is _TOR_BACKENDS = "mojeek,duckduckgo,yandex,brave,google". The crucial empirical finding (measured directly over Tor): no single engine is reliable — reachability is exit-node-dependent. On one circuit mojeek and yandex return results while duckduckgo/brave/google are CAPTCHA/empty; on the next circuit it flips. The engines that fast-fail (“No results found” in ~1.5 s) are cheap lottery tickets, so breadth is kept; only genuine hangers are excluded.

Because reachability is per-exit, the real fix is rotating the Tor circuit on every attempt. _proxy_for_attempt folds the attempt index into the SOCKS username:password, so Tor's IsolateSOCKSAuth hands each attempt a different exit node (verified: distinct tags → distinct exit IPs). Previously every attempt reused the same per-query circuit, so a retry on a blocked exit was pointless — the most common reason a search would burn three tries and still return nothing. There are 3 primary attempts (1 s backoff) plus 2 reformulations, every one on its own circuit. This replaces the old global NEWNYM, which re-circuited all of Tor and cost seconds per call.

Per-request timeout — the dominant reliability lever (measured 2026-06-11). The ddgs request timeout is _DDGS_TOR_TIMEOUT = 18 s. This was previously 8 s, and that single number was responsible for most search failures: measured directly over Tor, the engine that actually returns results (usually mojeek) responds in ~10–18 s through a circuit, while the others fail fast (~1–6 s). At 8 s, mojeek was killed mid-request (the production log filled with error sending request for url (mojeek…)), so every engine came back empty and the search failed even on circuits where results were reachable — the agent then looped for minutes on retries that could never win. Raising the ceiling to 18 s lets a successful search complete in ~12–18 s; the fast-failing engines still fail fast, so only an all-circuits-blocked search pays the full budget. Validated live: queries that returned zero in the field now return 8 sources. Covered by tests/test_search_tor_timeout.py.

Query sanitization

The DuckDuckGo / Brave / Mojeek scraper backends do not honour Google-style operators the way a real search API does: at best they're ignored, at worst the whole query returns zero results. The LLM is nonetheless prone to emitting them (the tool descriptions previously even encouraged site:). _sanitize_query strips them at the boundary, before a Tor round trip is spent:

  • site: / inurl: / intitle: / filetype: / ext: operators (with their argument token).
  • Standalone boolean OR / AND (boundary-gated and case-insensitive, so an or buried inside a word survives, and dropping a stopword from natural prose is invisible to the scrapers).
  • Quotation marks (the words inside are kept).

If stripping would empty the query (a pure-operator query with no keywords), the original is returned unchanged rather than sending an empty search. The tool descriptions in registry.py now instruct plain keywords only to match.

Result cache

The model habitually fires many near-identical queries in one turn. A small in-process TTL cache (_SEARCH_CACHE, 300 s, 64 entries, FIFO eviction) keyed on the normalized query serves the repeats from memory instead of re-paying the Tor round trip. Only successful results are cached — error strings never are, so a transient failure is always retried fresh.

Reformulation strategy

  1. Drop years / version numbers / numbers from the query (broaden).
  2. Convert to a question form ("how to X").
  3. Append tutorial / guide / example.

Output cleaning

_clean_for_cpp preserves valid JSON but strips control characters and structural braces from non-JSON text. This prevents the C++ JSON parser inside llama.cpp from crashing on stray {…} in scraped page bodies.

Stylometric egress scrubbing

Tor anonymises the network layer; it does nothing about the content layer. The agent's prose — politeness habits, first-person framing, punctuation idiosyncrasies, verbosity — is a stable author fingerprint that would otherwise egress verbatim to every search engine. utils/stylometry.py normalises the outbound query into a neutral keyword form before it leaves the box, under --anonymous (content anonymisation travels with network anonymisation).

  • scrub_query — deterministic, always-safe lexical pass used by web_search: peels stacked request/politeness frames ("please can you find me…", "I'm looking for…"), lowercases, drops trailing sentence punctuation, removes politeness tokens — while preserving every content keyword and in-word technical symbol (c#, c++, .net survive). Idempotent.
  • neutralize_query — stronger tier used by deep_research (already LLM-heavy): re-authors the query into a neutral keyword search via Ghost's own upstream, falling back to scrub_query on any failure. No third party.

Covered by tests/test_stylometry.py.

Per-identity Tor circuit isolation

Under --anonymous, each search also rides a distinct Tor circuit: helpers.socks_url_with_identity injects a SOCKS username:password into the proxy URL, and Tor's IsolateSOCKSAuth (on by default) maps distinct credentials to distinct circuits — so a colluding set of exit nodes / sites can't trivially link a sequence of different searches into one session. No NEWNYM thrash, no control port needed. Covered by tests/test_quick_wins.py.

The search path takes this one step further: _proxy_for_attempt folds both a query hash and the attempt index into the SOCKS tag, so retries of the same query land on different exits. This serves reliability (escape a blocked exit) and anonymity (a single logical search isn't pinned to one exit) at once.

This is also why the search retry path no longer calls request_new_tor_identity(): a global NEWNYM re-circuits all of Tor (slow) and fights the per-attempt isolation above. The previous behaviour — 3 attempts, each rotating the identity and sleeping 5 s, then two reformulations — could stretch a single failing search past 200 s. The retry path now pauses 1 s and rotates to a fresh circuit instead. (Page fetches in helpers.helper_fetch_url_content still rotate on 503, where the exit node itself is rate-limited — that's the right place for it.)
