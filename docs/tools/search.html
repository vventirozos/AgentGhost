tools / search.py

Anonymous web search via Tor, racing a wide engine set with one Tor circuit per engine, with operator sanitization, a short result cache, reformulated fallbacks and multi-source synthesis.

Tools

ToolPurpose
web_searchSearch via ddgs over a Tor SOCKS proxy. The query is sanitized first, then served from a short TTL cache on repeat. 2 racing waves: each wave fires one single-engine call per engine, each on its own Tor circuit, first non-empty result wins and the losers are cancelled; if both waves miss, two reformulated queries get a wave each before giving up. No NEWNYM rotation — per-engine SOCKS circuit isolation handles freshness (see below).
deep_researchSame racing search front-end (2 waves, no reformulation), then fetches the top URLs and synthesises results via the LLM. Falls back to web_search when synthesis fails. Per-URL fetch racing (2026-07-08): each URL rides its OWN Tor circuit and a circuit-retryable failure (timeout / 503 / connection error) retries on a FRESH exit — page-fetch reachability is exit-node-dependent just like search, so sharing one circuit across all URLs correlated their failures. The outer per-attempt timeout sits ABOVE the client's 20s (was 15 s, which guillotined slow-but-live Tor fetches — the mojeek-timeout bug's twin), and concurrency rose 2→3. Measured live: 6/8 → 8/8 sources, 21s → 14s wall. Still suppresses the global NEWNYM: a global re-circuit from one fetch would sabotage its concurrent siblings. Definitive errors (binary, 401/403, SSRF, 4xx) are NOT retried. The per-source extract is sized to max_context (not a hardcoded 40k) so a small worker can't overflow, and the distill-failure fallback is _clean_for_cpp-scrubbed.
fact_checkVerifies a claim: runs deep_research on the claim, then a single verify call judges it strictly against the evidence (verdict-first prompt: TRUE / FALSE / PARTIALLY TRUE / UNVERIFIABLE). 2026-07-14 rewrite: the old flow spent a whole LLM round forcing a deep_research tool call via tool_choice just to rephrase the claim — when the model answered in plain content instead (the native-tools transport corruption family), the function fell off the end and returned None to the dispatcher, and when tool_definitions lacked deep_research (subagent allowlists) it forced a function that wasn't in tools. The research call is now made directly (one less LLM round, no forced-tool fragility, no None path); the evidence is capped against max_context (previously accepted-but-unused) before the verify call; a verify-call failure degrades to FACT CHECK PARTIAL carrying the raw research results instead of dropping them; content: null from the server is coerced (was rendered as the literal None). Covered by tests/test_fact_check.py.

Engine race: one circuit per engine (Tor reliability, re-measured 2026-07-08)

Re-measured over Tor on 2026-07-08 (42 probes: 7 engines × 2 queries × 3 fresh circuits each): per-(engine, circuit) success is ~10% — brave 2/6, yahoo 1/6, mojeek 1/6, everything else 0/6 — and which engine wins flips from circuit to circuit. Even the trivial query python asyncio tutorial scored 1/21. Failures are driven by the exit IP's reputation, not the engine.

That kills the previous design: ddgs's multi-backend mode (backend="m,d,y,b,g") runs every engine through the one proxy set on the DDGS instance — so all engines shared a circuit and failed together whenever that exit was blocked (~90% of the time). Worse, with max_results=20 ddgs caps its internal fan-out at 3 concurrent engine workers, so the "5-engine breadth" was really ~3 engines on 1 exit.

_race_search_wave decorrelates the failures: one single-engine ddgs call per engine, each tagged onto its own Tor circuit (_proxy_for_attempt with a per-engine salt folded into the SOCKS username:password; Tor's IsolateSOCKSAuth maps distinct credentials to distinct exits). The first engine returning non-empty junk-filtered results wins the wave and pending losers are cancelled. That turns one correlated ~10% attempt into len(_RACE_ENGINES) independent tickets (~47% per wave); 2 waves + 2 reformulation waves ≈ 24 tickets ≈ 90%+ overall. A typical success now lands in ~2–5 s (the fast engines win fast); only a fully-blocked wave pays the ddgs timeout + a small grace (_RACE_WAVE_GRACE).

The race set _RACE_ENGINES = (mojeek, duckduckgo, yandex, brave, google, yahoo) is deliberately wide — at ~10% per ticket every fast-failing engine is a cheap independent lottery ticket. yahoo is back in (re-measured 2026-07-08: fails fast at ~1.4–2.2 s instead of hanging, and won a probe). Still excluded: wikipedia (reads region="wt-wt" as language wt and builds the non-existent host wt.wikipedia.org — always a ConnectError) and grokipedia (typeahead API, 0/6 on real queries). A per-engine StopIteration is converted to RuntimeError before it can cross the thread→Future boundary (PEP 479 corrupts asyncio future chaining).

Per-request timeout (measured 2026-06-11). The ddgs request timeout is _DDGS_TOR_TIMEOUT = 18 s. This was previously 8 s, and that single number caused a wave of failures of its own: measured over Tor, mojeek responds in ~10–18 s through a circuit while the others fail fast (~1–6 s). At 8 s, mojeek was killed mid-request (error sending request for url (mojeek…)), so every engine came back empty even on circuits where results were reachable. With the engine race, the slow-but-real engine can win a wave without making anyone wait when a fast engine wins first. Covered by tests/test_search_tor_timeout.py.

Query sanitization

The DuckDuckGo / Brave / Mojeek scraper backends do not honour Google-style operators the way a real search API does: at best they're ignored, at worst the whole query returns zero results. The LLM is nonetheless prone to emitting them (the tool descriptions previously even encouraged site:). _sanitize_query strips them at the boundary, before a Tor round trip is spent:

  • site: / inurl: / intitle: / filetype: / ext: operators (with their argument token).
  • Standalone boolean OR / AND (boundary-gated and case-insensitive, so an or buried inside a word survives, and dropping a stopword from natural prose is invisible to the scrapers).
  • Quotation marks (the words inside are kept).

If stripping would empty the query (a pure-operator query with no keywords), the original is returned unchanged rather than sending an empty search. The tool descriptions in registry.py now instruct plain keywords only to match.

Result cache

The model habitually fires many near-identical queries in one turn. A small in-process TTL cache (_SEARCH_CACHE, 300 s, 64 entries, FIFO eviction) keyed on the normalized query serves the repeats from memory instead of re-paying the Tor round trip. Only successful results are cached — error strings never are, so a transient failure is always retried fresh.

Reformulation strategy

  1. Drop years / version numbers / numbers from the query (broaden).
  2. Long queries (>6 words): hard-trim to the first 5 words of the broadened form. A keyword-stuffed query has near-zero organic hits anywhere, so no circuit can save it — and "how to {query}" keeps every rare term and fails identically (observed live 2026-07-08: the only total strike-out of the session was an 11-word query whose reformulations both preserved all its rare terms). Short queries still get the question form ("how to X").
  3. Append tutorial / guide / example as padding when the above yield fewer than 2 variants.

Wave logging

Several searches can race concurrently in one agent turn, so every wave log line — winner (mojeek won wave 0 in 1.8s (10 results) ‹query…›) and failure — carries a truncated-query tag to keep interleaved output readable (an apparent "won wave 10" right before "ZERO results" is two different searches finishing together).

A failed wave emits one terse categorized line (operator request 2026-07-08 — even grouped exception reprs were a wall of text): wave 0 ‹postgresql 20 features…›: no winner — 5 empty; mojeek conn-error. _failure_category collapses each losing engine into empty (bare count — it's the boring default), conn-error, or timeout (engines named); an unknown error keeps a 48-char snippet so a new failure shape is never hidden. The full sanitized per-engine detail (URLs stripped via _brief_engine_error) goes to logger.debug for forensics.

Output cleaning

_clean_for_cpp preserves valid JSON but strips control characters and structural braces from non-JSON text. This prevents the C++ JSON parser inside llama.cpp from crashing on stray {…} in scraped page bodies.

Stylometric egress scrubbing

Tor anonymises the network layer; it does nothing about the content layer. The agent's prose — politeness habits, first-person framing, punctuation idiosyncrasies, verbosity — is a stable author fingerprint that would otherwise egress verbatim to every search engine. utils/stylometry.py normalises the outbound query into a neutral keyword form before it leaves the box, under --anonymous (content anonymisation travels with network anonymisation).

  • scrub_query — deterministic, always-safe lexical pass used by web_search: peels stacked request/politeness frames ("please can you find me…", "I'm looking for…"), lowercases, drops trailing sentence punctuation, removes politeness tokens — while preserving every content keyword and in-word technical symbol (c#, c++, .net survive). Bare search/lookup are not treated as frames (they lead real content phrases: "search algorithms in python", "lookup table sql"); only the framed forms ("search for", "look up") are stripped. Gratitude tokens are stripped only from the query tail ("… thanks" goes; "thank you card template" survives intact). Idempotent.
  • neutralize_query — stronger tier used by deep_research (already LLM-heavy): re-authors the query into a neutral keyword search via Ghost's own upstream, falling back to scrub_query on any failure. No third party.

Covered by tests/test_stylometry.py.

Per-identity Tor circuit isolation

Under --anonymous, each search also rides a distinct Tor circuit: helpers.socks_url_with_identity injects a SOCKS username:password into the proxy URL, and Tor's IsolateSOCKSAuth (on by default) maps distinct credentials to distinct circuits — so a colluding set of exit nodes / sites can't trivially link a sequence of different searches into one session. No NEWNYM thrash, no control port needed. Covered by tests/test_quick_wins.py.

The search path takes this one step further: _proxy_for_attempt folds a query hash, the wave index and a per-engine salt into the SOCKS tag, so within one wave every engine rides its own exit, and every wave rotates all of them. This serves reliability (uncorrelated engine failures, escape from a blocked exit) and anonymity (a single logical search isn't pinned to one exit, and no single exit sees the query hit every engine) at once.

This is also why the search path never calls request_new_tor_identity(): a global NEWNYM re-circuits all of Tor (slow) and fights the per-engine isolation above. The old behaviour — 3 attempts, each rotating the identity and sleeping 5 s, then two reformulations — could stretch a single failing search past 200 s. (Page fetches in helpers.helper_fetch_url_content still rotate on 503, where the exit node itself is rate-limited — that's the right place for it.)

Tests
