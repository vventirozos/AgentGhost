Tool: manage_projects

Create gating — don't spin up a project unprompted (2026-06-21)

Observed live: the prompt "Create a single-file browser OS with desktop, taskbar, 5+ apps…" made the agent call action=create and build a task tree, even though nobody asked for a project and the whole deliverable is one index.html. Project creation is 100% model-driven — no code path auto-creates one — so the cause was guidance, not logic: the system prompt's TOOL ORCHESTRATION triggers had explicit rules for self-play, lessons, skills, RAG and web, but nothing on when to enter project mode vs. stay in free-chat; the tool description's "a new named multi-step effort" read as a match for any multi-feature build; and memory/RAG surfaced prior browser-OS projects, reinforcing the bias (visible in the model's own thought: "I can see references to previous browser OS projects. Let me create a new project").

The fix is explicit gating in two places. The system prompt (prompts.py) gains a PROJECT MODE GATING trigger, and the manage_projects tool description gains a GATING for create clause. Both state the same policy: create a project ONLY when (1) the user explicitly asks to start/track a project, OR (2) the deliverable genuinely spans multiple files/modules AND multiple turns. A self-contained single-file deliverable — even a large one (a one-file browser OS, game, or script) — is a one-shot: build it directly with file_system write, no project, no task tree. Past similar projects in memory are explicitly disqualified as a reason to create a new one. (This complements the pre-existing _is_cohesive_single_file detection, which only collapsed an already-created single-file project to one task — it never stopped the project being created.) Covered by tests/test_project_create_gating.py.

Project-level actions

  • create · list · get · switch · exit · update · archive · delete · resume · status

Delete vs. archive

  • archive — soft, reversible. Flips status to ARCHIVED; the row and the on-disk workspace are kept, and resume brings it back. This is what the old delete used to do.
  • delete — hard, permanent. Removes the project row, all its tasks/artifacts/events (FK ON DELETE CASCADE with PRAGMA foreign_keys=ON, including scratchpad-snapshot events), and the project's workspace directory on disk (<sandbox>/projects/<id>/). The on-disk removal only fires for a path strictly inside the configured sandbox root, so a stray/custom workspace_dir can never wipe an arbitrary directory. The tool schema steers the model toward archive unless the user clearly means to erase. Covered by tests/test_projects_store.py and tests/test_projects_tool.py.

Note: the web API DELETE /api/projects/{pid} takes an explicit ?hard= query flag (default soft/archive) — the frontend chooses; it is independent of the chat tool's delete (which is always hard).

Eligibility gate for unconfirmed hard deletes (2026-06-12)

Observed live: one "i don't really like it, delete it an make something else" cascaded into six successive hard deletes — five of them of projects the agent had created seconds earlier in the same request and the user never saw — because each new turn re-read the user's instruction as unfulfilled and deleted the project it had just created. The agent's loop-breakers never fired: they are keyed on failures, and every delete/create succeeded.

_delete_eligibility_error() now refuses any hard delete whose target the user cannot plausibly mean by "delete it". Eligible targets:

  • the project that was active when the user's message arrived — handle_chat snapshots it as context.request_start_project_id right after conversation reconcile, so "delete it" always resolves against something the user has actually seen;
  • a project the user's own message names (case-insensitive title or id match against context.last_user_content).

Everything else — most importantly a project created mid-request — is refused with an instruction to stop cycling ideas and BUILD the current project, pointing at archive (reversible) as the escape hatch. archive itself is not gated. Contexts without the snapshot attribute (direct tool use, tests, the web API route) bypass the gate, so non-chat surfaces keep legacy behavior. Covered by tests/test_project_delete_eligibility.py.

Task-level actions

  • task_add · task_update · task_decompose · task_next · task_list

Task ordering (prerequisites)

  • task_add accepts depends_on=[ids] — prerequisite sibling task ids that must be DONE before the new task becomes eligible for task_next / autoadvance. A peer-ordering edge, distinct from parent/child dependency_type.
  • task_decompose accepts sequential=true — chains the generated subtasks so each runs only after the previous is DONE (works for both the under-a-task and top-level fan-out forms). Default false = all subtasks independently runnable.

Other

  • artifact_add · event_log
  • ledger — append one durable fact to the project's design ledger (ledger="…"), or omit it to read the ledger back. See "Working memory" below.
  • config 1B — upsert one durable setting (config_key="GHOST_MODEL" config_value="qwen-3.6-35b-a3"): env vars, key flags, dependency versions, model, ports, DB URIs. Empty config_value deletes the key; omit both to read the config map back. Bounded + surfaced in the project briefing's CONFIG section every turn. Backed by ProjectStore.get_config / set_config_value.
  • autoadvance — run the autonomous advancer. count selects how many tasks: a number ("3") or "all" to run to completion; default 1. Drives advance_many(), a bounded loop that checkpoints each task and stops at the first of done / a task needing you / budget / a FAILED task. Returns {advanced, count, requested, stop_reason}.

Working memory & decomposition granularity (2026-06-20)

Two changes target the failure where a long, per-turn project felt clunky and lossy (a single "proceed" turn re-deriving state for ~96 s, then thrashing on a 14-task / one-file decomposition):

  • Design ledger — action=ledger ledger="single HTML file at browser-os.html; windows are .window divs opened via openApp(id)" records a durable fact (file layout, key API/function name, convention). It is bounded + dedup'd in project metadata and surfaced in the project briefing every turn, so later turns inherit it instead of re-reading files to rediscover it. A ledger="…" may also ride along on a task_update status=DONE to capture the decision as the task closes. Backed by ProjectStore.get_ledger / append_ledger / set_ledger.
  • Per-file decomposition — the subtasks description and the tool/create instructions now steer toward tasks that each own a FILE or a bounded function you can build and verify alone (a thin shell + one file per feature), and explicitly warn against N tasks that all edit the same file — that forces whole-file context every turn and doesn't scale.
  • create honours subtasks (2026-06-20) — the model routinely calls action=create WITH subtasks, believing create decomposes too. Those used to be silently dropped, leaving the project with ZERO tasks while the model believed they existed — so a later "proceed" found "no plan" and the flow derailed (observed live, twice). create now decomposes the subtasks in the same call (sequential-aware) and returns tasks_created; the success instruction reflects whether tasks already exist or still need task_decompose.

Together with the DONE SO FAR briefing digest (auto-surfacing completed tasks' result_summary), these give the next turn "what exists and how it works" cheaply. Covered by tests/test_project_working_memory.py.

Pacing — one task per go-ahead (2026-06-20)

Projects advance one task at a time, under user direction, not in an automatic sprint through the whole tree. After create or task_decompose, the tool returns a stop-and-await agent_instruction telling the agent to present the plan and wait — do not begin executing. The agent then advances exactly the next task on each explicit user go-ahead ("proceed to next task"), closing just that id with task_update status=DONE. This is the counterpart to the briefing's ONE TASK AT A TIME directive; it exists because grinding every task in one turn floods the context window on large projects. Covered by tests/test_project_pacing.py.

Batch ("proceed with all", 2026-06-20). When the user explicitly asks for MORE than one task — "do the next 3", "proceed with all remaining tasks", "finish the project" — the agent routes to autoadvance count=<N|"all"> instead of grinding them itself. That runs advance_many, a bounded autonomous loop: each task is a separate checkpointed tick (so context stays bounded however many run), and it pauses at the first of done / a task needing the user / budget / a FAILED task, then reports. This is how "do everything" stays safe on a large project — a loop of bounded ticks, not one context-busting turn. A single "proceed"/"next" still runs as a full agent turn (higher per-task quality). Each coding tick BUILDS for real via build_coding_task (write files + verify), failing loudly on a hard task rather than marking shallow work done. Covered by tests/test_advance_pacing.py and tests/test_coding_executor.py.

list always surfaces the ids (2026-07-05)

The ids were always present in action=list's JSON, but the model's user-facing summaries dropped them — the user was left referencing projects by title for follow-up commands (delete/switch), which is exactly the ambiguity-prone path the delete-resolution machinery below exists to defend against. The list result now additionally carries a ready-made display block — one line per project, id first, status, title, with the conversation-bound project marked <- ACTIVE — plus a note directing the model to always show the ids when presenting projects ("the id is the only unambiguous reference for follow-up get/switch/update/delete calls"). Empty store renders (no projects). Covered by tests/test_projects_tool.py::test_list_display_block_leads_with_ids.

get miss → recall fallback (2026-07-14)

Journal §4C recall-routing variance: "when does project Kestrel ship?" routed the model to manage_projects, got a bare project not found, and it gave up — while the answer sat in vector memory at 0.76 relevance. A get whose reference resolves to no tracked project now runs _not_found_with_recall: a search_advanced pass over vector memory (MEDIUM-or-better band, score < 1.15). Hits come back as a non-error payload — {"project": null, "note": ..., "memory_recall": [facts]} — so the model answers from the facts without a strike or a second unprompted tool hop; no hits keeps the plain error plus a redirect to the recall tool. The recall tool description carries the mirror-image nudge. Tests: tests/test_iterative_recall_expand.py.

switch / get resolve a title → id (2026-06-22)

The model routinely passes a project's title where an id is expected (switch petai), which hard-errored with "project not found" and burned a strike before the model recovered via list+switch-by-id (observed live). switch and get now run the reference through the shared _resolve_project_ref (id first, then a case-insensitive title match) before erroring, so switch petai resolves to the real id. A genuinely unknown reference still errors (the destructive-action contract is unchanged), and an ambiguous title surfaces the candidate ids instead of silently picking one. Covered by tests/test_projects_tool.py.

Read with no active project: guide, don't strike (2026-06-22)

A READ action — get / task_list / task_next — invoked with no resolvable project (none passed, none active) used to return ERROR: project_id is required…. The agent loop scores an ERROR-prefixed tool result as a failure and burns a strike — harsh for a recoverable usage slip (the model just forgot to say WHICH project), and it happened at the very start of a "resume my project" turn before the model recovered with list+switch (observed live). These reads now return a non-error {"no_active_project": true, "projects": [...], "agent_instruction": "pick one and switch"} — no strike, and the project list is right there so the model can switch in one step instead of an extra list round-trip. Mutations (task_add, research, …) keep the hard error: you must not write to an unspecified project. Covered by tests/test_projects_tool.py.

On a failed/paused batch, STOP and report (2026-06-20). When the loop stops early (a task FAILED, budget, or needs-user), the result instruction now tells the agent to report exactly which tasks completed vs not and hand back to the user — explicitly NOT to keep building the remaining/failed tasks by hand. That manual fallback is what thrashed for ~700s live and exposed the project-scoping race; re-running autoadvance after a fix is the way to continue.

Auto-research

  • research — research a topic and persist a summarised brief in the project workspace. Pass topic="…" for a single topic, or omit it (optionally with max_topics) to auto-derive several from the project goal + open tasks. Each brief lands at research/<slug>.md, is indexed in project metadata, and is surfaced under RESEARCH NOTES in the project briefing. See core.project_research.
  • research_list — return the project's research index (topic → path → preview). Read a brief with file_system before re-researching the same thing.

Duplicate guard

_DUPLICATE_CREATE_WINDOW_SECONDS = None — a same-title project that is still in flight (status ACTIVE / PAUSED / NEEDS_USER) is reused rather than re-created, which stops the model spawning duplicates of work that's already underway. A terminal same-title project (DONE / FAILED / BLOCKED / ARCHIVED, the _TERMINAL_PROJECT_STATUSES set) is not reused — it's superseded by a fresh create. Previously any non-ARCHIVED match was reused, which resurrected a finished project (with its stale tasks and on-disk files) when the user asked to start a new one of the same name. Covered by tests/test_project_delete_and_guard.py.

Delete / archive resolution

Users refer to a project by NAME ("delete the minecraft clone project"), but the destructive actions key on the opaque hex id. delete and archive now run the reference through _resolve_project_ref: a real id is used directly; otherwise a case-insensitive title match is tried (the model can pass the name as project_id, or an explicit title). If nothing matches, the action fails loudly with an ERROR rather than silently no-op'ing — a silent deleted:false is what previously let a "delete" be reported as done while the project survived (and was then resurrected by the duplicate guard). Several projects sharing a title resolve to an ambiguous error listing the candidate ids, so the caller must disambiguate instead of deleting the wrong one. Covered by tests/test_project_delete_and_guard.py.

DONE-gate for visual/runnable tasks (2026-06-09)

task_update status=DONE on a task that builds a runnable/visual artifact is held when no verification evidence is given (no result and no existing result_summary). The gated ids + an instruction are returned: verify with the browser tool (screenshot, click_center=true for a game), read the RENDER_CHECK / PRE_INTERACTION lines, then re-call with a result describing what rendered. This targets the live anti-pattern where the agent wrote index.html, marked every task DONE before testing, and claimed a blank game "works".

Precision (2026-06-09, after a false-positive). The matcher (_is_visual_artifact_task) is HIGH-precision and word-boundary anchored. The first cut used broad words and misfired on a philosophy project — a text reflection "…whether the scene is pre-existing or being constructed" matched " scene", and "game theory" matched "game" — gating ordinary research tasks for ~6 wasted turns. The keyword set is now limited to tokens unambiguous in software work (webgl, three.js, canvas, shader, hud, sprite, playable, gameplay, game loop, render loop, minecraft, frontend, …), matched on word boundaries (so "hud" can't fire inside "shudder", nor "3d" inside "add"). Escape hatch: the gate message offers two clears — verify (for a real visual artifact) or, for a text/research/markdown task the gate misfired on, simply re-call with a one-line result. The handler also accepts result_summary/summary as aliases for result (the model's natural names), so the bypass works first try. Covered by tests/test_run_failure_fixes.py.

Deliverables & end-of-project cleanup

When a project completes (transitions to DONE), its workspace is swept by core.workspace_cleanup: registered file artifacts are kept, everything else (screenshots, helper scripts, temp files) is deleted. So task_update status=DONE takes a deliverables=[...] argument — the project-relative paths the user should keep. Each is registered via register_file_artifact (deduped) the moment the task is marked DONE; a legacy artifact_kind="file" + payload pair is registered the same way. The DONE-gate instruction now reminds the agent to pass deliverables so nothing important is swept. Covered by tests/test_workspace_cleanup.py.

Per-project sandbox scoping

When a project is active (context.current_project_id set on create/switch/resume), file ops and code execution are rooted at <sandbox>/projects/<id>/ instead of the sandbox root — so a whole project's scratch space cleans up with a single rm -rf sandbox/projects/<id>. The same path is what ProjectStore._default_workspace() records as a project's workspace_dir.

Single source of truth. All scoping routes through file_system.project_scoped_sandbox(context, stateful) → (host_dir, container_workdir). The registry's _proj_ws delegates to it, and so does every other file surface, so they can't drift: the tool registry, the ambient sandbox-state listing injected into each turn (core/agent.py — previously listed the root while file ops were scoped, which is what made the model insist "the file is at sandbox root"), the user-attached-image save, the /api/upload route, knowledge_base's source-file reads, and the (dormant) Qwen-Agent qwen_bridge runtime.

The per-tool wiring threads that helper into the artefact-producing tools:

  • file_system + report_pdf + image_generation — sandbox_dir is the scoped host dir. report_pdf and image_generation emit a root-relative download link (/api/download/projects/<id>/…) via the shared file_system.project_download_prefix() helper, because the /api/download/<path> route resolves against the sandbox ROOT.
  • execute — also receives container_workdir = /workspace/projects/<id>, threaded down to DockerSandbox.execute(workdir=…), so a script written via file_system reads back when run by execute (the write-then-read symmetry invariant). The workdir kwarg is passed ONLY when scoping is active, so sandbox managers without the parameter are unaffected.
  • vision_analysis — scoped to read images that the scoped execute/image_generation wrote (the generate-plot → analyse loop). As a read-only tool it additionally falls back to the sandbox root if the scoped path misses.
  • browser — scoped too: it runs its in-container runner from /workspace/projects/<id> (via container_workdir), writes screenshots into the scoped dir (container path via _to_container_path), and heals file:// URLs — a model-emitted file:///workspace/page.html or the host-root-relative file:///sandbox/projects/<id>/page.html is rewritten to file:///workspace/projects/<id>/page.html (with a root fallback), so the browser can open files the scoped file_system just wrote. The sandbox/projects/<id>/ case was a live 404 loop fixed 2026-06-20 (see file_system path heal). The persistent profile stays shared at /workspace/.browser_profile. The create/switch workspace note now also explicitly tells the model to use BARE names and never prefix with /workspace/, sandbox/, or projects/<id>/.

Consistency across the handoff surface. Because execute must be scoped (to preserve file_system↔execute symmetry), every tool it hands files to is scoped to match: a plot written by execute is found by vision_analysis; an image from image_generation is found by vision_analysis and downloadable. The /api/download route also falls back to the active project dir for a bare filename, so a model-emitted /api/download/plot.png link to a scoped file still resolves.

Switch-race handling. The model often emits a file_system write before the switch tool call in the SAME turn, so the file lands at the root just before scoping turns on; a later execute then runs from the scoped cwd and can't see it. Two mitigations: the create/switch/resume result carries a note telling the model its working directory is now projects/<id>/ (reference files by bare name) — create was added to this set on 2026-06-09 after a live run where the model wrote index.html expecting the sandbox root right after creating a project, then burned ~4 turns discovering the write had been project-scoped and "moving" it; and execute's command path retries once from the sandbox root when a scoped run fails with a file-not-found signature (safe — that error means nothing executed). Net effect: the race recovers silently with zero strikes (verified live: a generate-plot-then-analyse prompt that previously burned 2 strikes now completes clean).

Guards: scoping activates only for a genuine string project id (a MagicMock context attribute can't trigger it); _get_safe_path heals the model's redundant projects/<id>/ path prefix so it never double-nests, and _to_container_path un-scopes to the root before translating (the bind mount is at the root). Stateful Jupyter-kernel sessions opt out (stateful=True) and stay at /workspace, since the kernel's connection file is pinned to /workspace/.kernel.json.

Intentionally NOT scoped: create_skill/manage_skills and acquired-skill execution stay at the root because skills are global, cross-project resources (the knowledge base's storage is the global memory system — only its source-file reads are scoped). The browser profile (/workspace/.browser_profile) is deliberately left shared across projects so session/cookie state persists. Self-play runs in its own throwaway sandbox and is explicitly project-less (see dream). Covered by tests/test_project_sandbox_scoping.py and tests/test_dream_synthetic.py.

Conversation-scoped activation (2026-06-12)

context.current_project_id is process-global, and before this fix it was also sticky across conversations: a project activated in one chat stayed active for every other conversation hitting the same process, so an unrelated request's file writes silently landed in <sandbox>/projects/<id>/ (observed live: a "just give me the SQL" chat's migration.sql was written into a project created days earlier in a different conversation, then several turns were burned explaining and undoing it).

The activation is now bound to the conversation that asked for it:

  • Identity. The chat API ships the full message history and no conversation id, so conversation_fingerprint(messages) hashes the FIRST user message — constant for a conversation's lifetime, different across conversations. Returns "" (owns nothing) when there is no user turn.
  • Binding. _set_current writes a second sentinel, __current_project_conv__, next to __current_project__, recording the owning conversation's fingerprint (taken from context.conversation_key, which is set per-request).
  • Reconcile. reconcile_conversation(context, conv_key) runs at request start in handle_chat, BEFORE the ambient sandbox listing or any file op. Owning conversation → (re)activate the bound project (returning to that chat resumes it, scratchpad hydrated). Any other conversation — including a legacy bare sentinel written before the conv sentinel existed, and an empty fingerprint — → deactivate for this request, snapshotting the project scratchpad and PRESERVING the binding so the owner gets it back.

Autonomous flows (dream, project advancer) that activate a project outside a user request write an unbound ("") binding, so their activation can never capture the next user conversation. Covered by tests/test_project_conversation_scope.py.

Explicit user constraints — capture, gate, correction detection (2026-07-02)

Observed live (the chess-game incident): the user asked for "a turn by turn game where YOU will play against me — don't come up with some random AI". The agent acknowledged the clause in its reasoning, then shipped a minimax engine anyway: a stale pending task from an earlier (pre-correction) decomposition, a 70×-reinforced "single-file HTML game" skill prior, and nothing re-surfacing the constraint after turn 1 combined to erase it. Three mechanisms now make explicit constraints durable and load-bearing:

  • Capture. action=create and task_decompose merge the model-passed constraints=[…] param with clauses extracted deterministically from context.last_user_content by utils/constraints.py (negations, "YOU will…" role assertions, CAPS emphasis — no LLM call). They persist in project metadata.constraints and per-task constraints_json, and are stamped on every task created in the same call. Operator-visible via the ⛓️ Constraints log line.
  • Re-surfacing. build_project_briefing renders an EXPLICIT USER CONSTRAINTS section every turn (above the design ledger), and the agent injects the current request's extracted constraints into the dynamic state even without a project (see agent).
  • DONE-gate. task_update status=DONE on a task whose task-level or project-level constraint set is non-empty and that carries no result evidence is HELD (reported as gated_constraints + an instruction to state, per constraint, how the work honors it — or fix the deliverable first). Judging whether the evidence is truthful stays the verifier's job: the agent prepends the active project's constraints to the verifier's request view, and constraint violation is the verifier prompt's highest-priority refute rule.

Delete-then-recreate = correction, reuse-with-new-constraints = correction

Deleting a project and re-asking is a rejection signal, but a hard delete used to erase all evidence the project ever existed (FK cascade). delete_project(hard=True) now writes a tombstone (deleted_projects table: id/title/kind/goal/created_at/deleted_at) before the cascade. On the next create, find_deleted_similar() (stopword-insensitive title Jaccard ≥ 0.5, 24 h window) flags the recreate: the response instruction opens with CORRECTION CONTEXT ("plan from the CURRENT message only; do not reuse the remembered decomposition"), and metadata.correction_of records the tombstone id. Symmetrically, when the duplicate-title guard reuses an in-flight project and the new message carries constraints the project doesn't have yet, they are merged in and the response opens with CORRECTION DETECTED + an instruction to revise conflicting pending task descriptions before proceeding — the exact anchoring failure of the chess incident (the pending "AI opponent" task was read as "matches exactly what the user wants" after the user had excluded it).

Covered by tests/test_project_constraints.py and tests/test_correction_detection.py.

Persistence model: see memory.projects.

Constraint judgment gate on task_update→DONE (2026-07-08)

The existing constraint DONE-gate is evidence-based — it only demands that result text exists. The chess session showed the model supplies compliant-sounding evidence while the artifact violates the constraint (it built a heuristic engine after restating "Ghost plays directly"). Marking a constrained task DONE now also runs core/build_gates.constraint_gate — one LLM audit of the project workspace files per call. A confirmed violation refuses the transition and returns agent_instruction_violation with the evidence; a user-approved exception can pass a result beginning with CONSTRAINT-OVERRIDE:. Fails open on infra errors.
