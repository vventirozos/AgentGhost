memory / projects.py — ProjectStore

Multi-session project & task store. SQLite-only; no ORM. Designed for cross-process access from API, Slack bot, dream consolidator.

Storage

SQLite at memory_dir/projects.db; threading.RLock; PRAGMA foreign_keys=ON. Each connection also sets PRAGMA journal_mode=WAL and PRAGMA busy_timeout=5000 so the cross-process readers/writers this store is designed for (API server, Slack bot, dream consolidator — each opens its own connection, often its own process, beyond the reach of the in-process RLock) get concurrent-reader + waiting-writer semantics instead of an immediate "database is locked".

Schema upgrades for pre-existing DBs are handled by additive, idempotent ALTER TABLE ADD COLUMN calls in _migrate (guarded by a PRAGMA table_info presence check), since CREATE TABLE IF NOT EXISTS never alters an existing table.

Snapshot pruning: only the most recent scratchpad_snapshot event is ever read back (_hydrate_scratchpad uses limit=1), but each one carries the full free-chat key/value dict, so on a long-lived project they were the dominant source of unbounded event growth. log_event now deletes superseded scratchpad_snapshot rows for the project as it writes the new one, keeping just the latest. Other event types are never pruned.

Status rollup

_maybe_rollup_project_status transitions the project when its tasks settle (task terminal set = DONE / FAILED / BLOCKED):

  • all tasks DONE → project DONE;
  • all terminal, ≥1 FAILED/BLOCKED → project FAILED (previously this collapsed to DONE because ProjectStatus had no FAILED member — a wholly-failed project reported as completed);
  • only-remaining-open work is NEEDS_USER → project NEEDS_USER;
  • any other open work → no-op (stays ACTIVE).

DONE and ARCHIVED are locked (never auto-undone). FAILED and NEEDS_USER are not locked, so a revised or answered task rolls the project forward to DONE on a later update. Covered by tests/test_project_mgmt_fixes.py and tests/test_audit_bug_fixes.py.

ID canonicalisation

IDs are generated as lowercase 12-char hex (uuid4().hex[:12]), but an LLM that echoes one back in a manage_projects tool call routinely mangles the case of an opaque hex token — e.g. 9b5bd5cd812b → 9B5Bd5Cd812B — which used to make the case-sensitive WHERE id = ? lookups miss with "project not found". Every store method that accepts a project_id / task_id now normalises it via _canon_id (strip + lowercase), and the manage_projects tool canonicalises model-supplied ids at the boundary, so generation and resolution always agree regardless of how the id was transmitted. Covered by tests/test_project_id_canon.py.

Public API

Projects

  • create_project(title, kind, goal, metadata, workspace_dir) -> str
  • list_projects(status_filter) / get_project(id)
  • update_project(id, **fields)
  • delete_project(id, hard=False) — soft = ARCHIVED, hard = cascade delete.
  • ensure_workspace(project_id) — create / return workspace path.

Tasks

  • add_task(...) -> str
  • get_task(id) / list_tasks(project_id, status_filter)
  • update_task(id, **fields)
  • delete_task(id) — cascade delete descendants.

Artifacts & events

  • add_artifact(task_id, kind, payload) -> str
  • list_artifacts(project_id=None, task_id=None)
  • log_event(project_id, task_id, event_type, payload) -> int
  • list_events(project_id, limit, event_type)

No automatic eviction; archiving via status='ARCHIVED' preserves history. Hard delete cascades via FK ON DELETE CASCADE.
