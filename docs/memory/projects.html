memory / projects.py — ProjectStore

Multi-session project & task store. SQLite-only; no ORM. Designed for cross-process access from API, Slack bot, dream consolidator.

Storage

SQLite at memory_dir/projects.db; threading.RLock; PRAGMA foreign_keys=ON. Each connection also sets PRAGMA journal_mode=WAL and PRAGMA busy_timeout=5000 so the cross-process readers/writers this store is designed for (API server, Slack bot, dream consolidator — each opens its own connection, often its own process, beyond the reach of the in-process RLock) get concurrent-reader + waiting-writer semantics instead of an immediate "database is locked".

Schema upgrades for pre-existing DBs are handled by additive, idempotent ALTER TABLE ADD COLUMN calls in _migrate (guarded by a PRAGMA table_info presence check), since CREATE TABLE IF NOT EXISTS never alters an existing table.

Snapshot pruning: only the most recent scratchpad_snapshot event is ever read back (_hydrate_scratchpad uses limit=1), but each one carries the full free-chat key/value dict, so on a long-lived project they were the dominant source of unbounded event growth. log_event now deletes superseded scratchpad_snapshot rows for the project as it writes the new one, keeping just the latest. Other event types are never pruned.

Status rollup

_maybe_rollup_project_status transitions the project when its tasks settle (task terminal set = DONE / FAILED / BLOCKED):

  • all tasks DONE → project DONE;
  • all terminal, ≥1 FAILED/BLOCKED → project FAILED (previously this collapsed to DONE because ProjectStatus had no FAILED member — a wholly-failed project reported as completed);
  • only-remaining-open work is NEEDS_USER → project NEEDS_USER;
  • any other open work → no-op (stays ACTIVE).

DONE and ARCHIVED are locked (never auto-undone). FAILED and NEEDS_USER are not locked, so a revised or answered task rolls the project forward to DONE on a later update. Covered by tests/test_project_mgmt_fixes.py and tests/test_audit_bug_fixes.py.

Completion hook (workspace cleanup)

On the transition to DONE — and only DONE — the store calls _fire_project_done(project_id), which invokes the optional on_project_done hook (set in main.py to core.workspace_cleanup's sweep). It fires from both _maybe_rollup_project_status (auto-rollup) and a manual update_project(status="DONE"), exactly once per transition, and is fully guarded so a cleanup failure never propagates back into the status update. FAILED / NEEDS_USER stay resumable, so they do not fire it. The hook lets a finished project's scratch files be removed automatically while registered deliverables are kept.

ID canonicalisation

IDs are generated as lowercase 12-char hex (uuid4().hex[:12]), but an LLM that echoes one back in a manage_projects tool call routinely mangles the case of an opaque hex token — e.g. 9b5bd5cd812b → 9B5Bd5Cd812B — which used to make the case-sensitive WHERE id = ? lookups miss with "project not found". Every store method that accepts a project_id / task_id now normalises it via _canon_id (strip + lowercase), and the manage_projects tool canonicalises model-supplied ids at the boundary, so generation and resolution always agree regardless of how the id was transmitted. Covered by tests/test_project_id_canon.py.

Public API

Projects

  • create_project(title, kind, goal, metadata, workspace_dir) -> str
  • list_projects(status_filter) / get_project(id)
  • update_project(id, **fields)
  • delete_project(id, hard=False) — soft = ARCHIVED, hard = cascade delete.
  • ensure_workspace(project_id) — create / return workspace path.

Tasks

  • add_task(...) -> str
  • get_task(id) / list_tasks(project_id, status_filter)
  • update_task(id, **fields)
  • delete_task(id) — cascade delete descendants.

Artifacts & events

  • add_artifact(task_id, kind, payload) -> str
  • register_file_artifact(task_id, rel_path) -> str | None — register a deliverable file artifact, deduplicated within the project (re-registering the same path returns the existing id). This is the durable "keep me" marker the workspace cleanup sweep reads.
  • list_artifacts(project_id=None, task_id=None)

Design ledger (cross-turn working memory)

  • get_ledger(project_id) -> str / append_ledger(project_id, line) -> str / set_ledger(project_id, text) -> str — the project's durable, compact working memory (file layout, key APIs, conventions) stored in metadata and surfaced in the briefing every turn. append_ledger collapses whitespace, dedups an identical trailing line, and bounds to LEDGER_MAX_LINES (30) / LEDGER_MAX_CHARS (2400). Written via _write_metadata (no project_updated event) so frequent appends don't spam the event log.
  • log_event(project_id, task_id, event_type, payload) -> int
  • list_events(project_id, limit, event_type)

Config slot (durable settings) feature 1B

  • get_config(project_id) -> dict / set_config_value(project_id, key, value) -> dict — the project's durable settings map (env vars, key flags, dependency versions, model, ports, DB URIs), stored as metadata.config and surfaced in the briefing's CONFIG section every turn. Where the design ledger answers what exists and where, the config slot answers under what settings — the things a fresh turn would otherwise re-discover from requirements.txt / env / argv. set_config_value normalises the key (trim + whitespace-collapse), treats an empty value as a delete, is last-write-wins, and bounds the map to CONFIG_MAX_KEYS (30) / CONFIG_MAX_VALUE_CHARS (200) / CONFIG_MAX_CHARS (2000), dropping the oldest key first on overflow. Written via _write_metadata (no project_updated event), same as the ledger. Recorded by the agent with manage_projects action=config key="…" value="…" (omit both to read the map back). Covered by tests/test_project_working_memory.py.

No automatic eviction; archiving via status='ARCHIVED' preserves history. Hard delete cascades via FK ON DELETE CASCADE.

Constraints column + deletion tombstones (2026-07-02)

  • tasks.constraints_json — explicit user constraints a task must honor (verbatim clauses like "don't come up with some random AI"). Unlike postconditions (which describe the success state), constraints are standing prohibitions/requirements: re-rendered into the briefing every turn, replayed to the verifier, and gating task_update → DONE (see tools.projects). Additive migration in _migrate; plumbed through add_task / update_task / _row_to_task / TaskNode / ProjectPlan. Project-level constraints live in metadata.constraints (no schema change).
  • deleted_projects table — a tombstone (id, title, kind, goal, created_at, deleted_at) written inside delete_project(hard=True) BEFORE the row is removed, because the FK cascade otherwise erases every trace that the project existed. find_deleted_similar(title, within_secs=86400) matches a new title against recent tombstones (stopword-insensitive token Jaccard ≥ 0.5) so create can detect the delete-then-recreate correction pattern. Soft delete (archive) writes no tombstone.

Covered by tests/test_project_constraints.py and tests/test_correction_detection.py.
