memory / competence.py — CompetenceProfile

Per-domain capability prior. The Level-1 (cross-domain transfer) component of the metacognition uplift.

Why per-(domain, tool)?

Aggregating success rates by raw tool name fragments the signal: list_files and read_file share a domain but live in different cells. Aggregating by tool alone loses the cross-tool generalisation MCTS needs. The dual key gives both: every record updates the leaf, the per-domain roll-up, AND the global cell — so estimate(domain, tool) can fall through whichever level has data.

Domain taxonomy

shell · sql · fetch · code · fs · memory · vision · other. Common synonyms (postgres → sql, bash → shell, python → code) are canonicalised at write time. Unknown domains land in other.

Prior

Beta(1, 1) — uniform. After ~100 samples the prior contributes <2% of the posterior mass; that's only load-bearing during the cold-start window. The CompositeConfidence scorer additionally shrinks toward neutral when the cell has <5 observations.

Persistence

Single JSON under the memory directory with atomic-rename writes. The format is keyed by "domain|tool" so a future schema change can be detected at load time.

Load hardening (2026-07-22). _load() now distinguishes three outcomes instead of collapsing them into "empty" — the old except Exception: return treated an unreadable file exactly like an absent one, so the very next record() atomically overwrote a hundred-observation profile with a single-tool file (total history loss, no sidecar, only a debug line):

  • absent → start empty (normal cold start);
  • corrupt (bad JSON / wrong type / undecodable bytes) → move the raw bytes to a timestamped competence_profile.corrupt-*.json sidecar, WARNING, start clean;
  • present-but-unreadable (OSError — EIO / EACCES …) → keep in-memory neutral priors but set _degraded (logged at ERROR); while set, _save() REFUSES to write, so a transient read failure can't wipe the intact file.

Sample counting & optional debounce (2026-07-22)

_Cell now carries an integer samples count rather than deriving n from the Beta mass. Deriving it — int((alpha-1)+(beta-1)) — truncates: ten records at weight=0.05 add 0.5 of mass and floor to n=0, so a low-weight cell stayed invisible to the n >= 1 gate in estimate() and reported n=0 in the prompt for ever. The derived value survives only as the fallback for a legacy file that predates the field.

CompetenceProfile(memory_dir, flush_interval=0.0) gained an opt-in write debounce: with flush_interval > 0, record() marks the profile dirty and flushes at most once per that many seconds (plus every _FLUSH_EVERY = 10 records, on reset(), and on an explicit flush()). The DEFAULT stays 0.0 = write-through, because the durability contract is "an outcome recorded is an outcome persisted" and the deploy procedure is a plain kill (no atexit / signal handler) — a debounce would silently drop the tail. Tests: tests/test_memory_store_durability.py.
