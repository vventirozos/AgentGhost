memory / skills.py — SkillMemory

Lessons learned from past mistakes. Vector-backed retrieval with BM25 re-rank; utility-driven retention; verified-first pinning.

Storage

  • JSON at memory_dir/skills_playbook.json.
  • Vector embeddings stored alongside in VectorMemory with metadata.type = "skill".
  • threading.RLock for JSON reads/writes; lazy-init fallback for tests.

Atomic save (cross-process)

_save_playbook_unlocked writes to a temp file and then os.replaces it over the live playbook (atomic on POSIX). Two problems the in-process RLock alone could not solve — Slack, web and CLI each run their own process — are now closed:

  • Torn temp file. The temp path was FIXED (skills_playbook.tmp), so two processes writing at once interleaved into one corrupt temp that os.replace then promoted. The temp name now carries the writer's PID (skills_playbook.<pid>.tmp), so each process promotes only its own fully-written file. Any leftover temp on the error path is unlinked in a finally.
  • Concurrent promotion. The write + replace is wrapped in an fcntl advisory lock (_crossproc_lock) on a sibling skills_playbook.lock file — mirroring FrontierTracker — so at most one writer (across processes, or two SkillMemory instances in one process) is inside the critical section at a time. On platforms without fcntl the lock degrades to a graceful no-op (the PID-unique temp still prevents cross-process tearing). Covered by tests/test_memory_crossproc_locking.py.

source_refs (2026-07-14): evidence handles the lesson was generalized FROM — e.g. ["ep:12", "ep:15"] for the episode rows behind a dream episode-strategy. Unioned (deduped, capped 20) when a duplicate lesson is merged/reinforced; mirrored onto the vector twin as a comma-joined source_refs metadata string so tool_recall hits can print an EVIDENCE REFS: line. Distinct from source (free-form origin tag) and source_trajectory_id (retraction key). Tests: tests/test_memory_provenance.py.

dimension (2026-07-19): the harness layer the recorded failure was attributed to — see failure_dimension. Auto-classified at the write chokepoint for real-mistake lessons (heuristic regex over trigger + anti_pattern; kill switch GHOST_FAILURE_DIM=0); empty for mistake-less rules, legacy rows, and unclassifiable records (the dream distillation pass adjudicates those offline via the worker). An explicit dimension= kwarg wins. Back-filled onto an existing row on dedup merge; mirrored onto the vector twin metadata. source="distilled" marks the cross-case pattern lessons the dream-side failure_distill pass writes — their distilled(<dim>/<cluster>): trigger prefix is a dedup contract: re-distillation reuses the trigger verbatim so frequency accumulates on one row. Tests: tests/test_lesson_dimension.py.

Lesson-quality gate at the write chokepoint (2026-07-16)

The dream heuristics loop had an actionability gate since 2026-07-13, but it lived inside dream — so self-play and other producers bypassed it, and observations still leaked in. Measured on the live playbook: 23 of 50 lessons were mistake-less pseudo-entries and they drew 28% of all retrievals — the single most-retrieved item being a dream chess-persona note ("When playing live chess against Vasilis, provide continuous coaching commentary", 84 retrievals), alongside self-play observations ("On a regex_parse task that has a familiar shape…") and user preferences ("The user prefers ripgrep") that match no real query.

The gate now runs at learn_lesson — the single write chokepoint, so it covers every producer (dream, self-play, reflection, episode consolidation). The logic (memory/lesson_quality.py, a leaf module shared with core.dream to avoid an import cycle): a lesson that records a real mistake is a genuine correction and always passes; a mistake-less entry is a rule/observation, so its solution must read as an actionable heuristic (_is_actionable_heuristic — imperative starter, or conditional + modal). verified gets no bypass — it would only ever exempt a verified observation (a verified fix has a mistake; a verified actionable rule passes anyway). The 11 non-actionable lessons on the live playbook were pruned retroactively (50 → 39; backup at skills_playbook.json.pre-quality-prune-20260716), and their orphaned vector twins were then deleted in-process via VectorMemory.delete_skill_twins (POST /api/memory/delete_skill_twin — a second PersistentClient against the live Chroma dir risks HNSW corruption, so the delete must run inside the owning process; 11 twins removed, 7315 → 7304 docs, verified idempotent). This is the content-quality complement to the 2026-07-15 retrieval-routing fix: routing now surfaces lessons (mediation ~1% → ~85% in B4), and this stops ~a quarter of what surfaces from being noise. Tests: tests/test_lesson_quality_gate.py, tests/test_delete_skill_twins.py.

Conversational-trigger gate (2026-07-18)

The 2026-07-16 gate waved through any lesson carrying a real mistake. The 2026-07-17 overnight REM cycle showed the gap: genuine mistakes attached to TRIGGERS lifted verbatim from user chat — "proceed with the next task", "it still does the same. the game never starts, notify me in slack when…". A trigger is the lesson's retrieval key; raw conversation fragments are keys no future query will match, so such entries are permanent playbook noise however real the mistake was.

_is_conversational_trigger (in memory/lesson_quality.py) now also gates mistake-bearing lessons. Default-ACCEPT — only unambiguous chat tells reject: pronoun-initial fragments with no antecedent ("it still does…", "that didn't work…"), short bare turn-level commands ("proceed", "try again", ≤ 60 chars), and direct-address phrases ("notify me in slack", "let me know"). Deliberately NOT rejected: user-question triggers ("How do I parse JSON?", "Please parse JSON!") — the paraphrase-normalised dedup counts on them recurring — and "you"-phrasing, which is legitimate rule language addressed to the agent. Tests: tests/test_lesson_quality_gate.py::TestConversationalTrigger.

Retention & eviction

PLAYBOOK_MAX = 50. Trim policy:

  1. Newest lesson is always kept (head pin).
  2. Verified lessons pinned next.
  3. Remaining slots filled by unverified, ranked by utility.

Utility formula

utility = confidence·0.5 + hit_rate·0.8 + (0.3 if verified) + log(freq)·0.1 + stale_penalty

Retrieval (three-tier, picked by query + memory_system presence)

get_playbook_context(query, memory_system, distance_threshold=0.45, limit, record_retrievals) dispatches to one of three paths:

  1. Vector path — memory_system AND query both present. Runs a Chroma query over type="skill" docs, tightens by DEFAULT_RETRIEVAL_DISTANCE = 0.45, then re-ranks with a BM25-lite keyword-overlap bonus. Best path; used whenever the bus wires a vector tier. Domain rescue (2026-07-09): the 0.45 floor demands near-verbatim similarity (~cosine 0.78 on the L2-normalised MiniLM embeddings), which a GENERALIZED lesson can never reach against a concrete task prompt — measured live on the B4 run: the arm's real self-play lesson sat at distance 1.056 from a matching probe, so the skill tier filtered it on all 96 probe turns and the learning loop was write-only (mediation ≈ 0). A candidate past the strict floor is now admitted up to _DOMAIN_RELAXED_DISTANCE = 1.25 when its domains metadata contains the query's cluster per _explicit_query_cluster() — which requires an explicit CLUSTER_KEYWORDS hit (NOT classify_cluster, whose python_general fallback would let every small-talk turn "match"). Untagged lessons (e.g. reflection writes) derive a domain from their trigger text. Semantic near-match OR domain match — never a blind dump. Tests: tests/test_lesson_retrieval_domain_gate.py (incl. the forensic B4 reproduction).
  2. BM25 fallback — query is present but memory_system is None (vector store offline, init failed, or a bus is wired without a vector tier). Does pure token-overlap against each lesson's trigger. Returns "" if no lesson shares tokens — never falls through to recency.
  3. Recency fallback — no query supplied (system-prompt-injection usage). Returns the N most recent lessons regardless of content, under the ## RECENT LESSONS header.

Before the fix, the recency fallback fired whenever the vector path was unavailable — including on a real query. A question like "what's the capital of France?" would surface an unrelated Python-syntax lesson just because it happened to be the most recent entry, polluting context with junk. The new three-tier dispatch preserves the recency path ONLY for the no-query case (the SYSTEM_PROMPT cold-inject). Covered by tests/test_skills_bm25_fallback.py.

When a lesson is surfaced, record_retrieval increments the counter and emits a debug log keyed by source + source_challenge_hash — so the "is this self-play lesson ever actually used?" question has data to answer.

Dedup

Vector similarity check (distance < 0.15; < 0.25 for mistake-less rules since 2026-07-20 — dream/reflection re-emit the same "When X, do Y" heuristic reworded every cycle, and rewordings measure 0.07–0.17 apart on the live bge-small store while genuinely distinct rules stay ≥ 0.29, so the single 0.15 cutoff let the reworded tail through: 10+ copies of two rules saved overnight 07-19→20; override with GHOST_RULE_DEDUP_DIST, tests tests/test_skill_dedup.py::TestMistakeLessWideThreshold) or a normalized trigger match merges frequency + solution instead of inserting a duplicate. The JSON fallback no longer requires an exact lowercased string: _normalize_trigger() lowercases, strips punctuation, drops stopwords and sorts the remaining significant tokens, so paraphrased triggers ("How do I parse JSON?" ≡ "parse JSON") collapse onto the same key and frequency actually accumulates toward graduation.

Listing lessons (user-facing surface)

list_lessons(scope, source, limit) returns lessons filtered by time window and source, most-recent first. Boundaries use local wall-clock:

  • scope="today" — since local midnight.
  • scope="week" / "7d" — last 7 days.
  • scope="all" — entire playbook.
  • source="self_play" (etc.) — filter by provenance.

Surfaced to the LLM via the list_lessons tool (see tools / memory). Phrases like "what did you learn today?" / "what have you learned so far?" are routed to this tool via SYSTEM_PROMPT.

Lesson quarantine + hydration side-channel (2026-07-17)

quarantine_lesson(trigger, reason) marks matching playbook entries quarantined — excluded from BOTH retrieval surfaces (get_playbook_context and get_playbook_items share the _filter_quarantined chokepoint, covering the vector branch too) but kept on disk with reason + timestamp for operator review/reinstatement. Written for counterfactual replay's regression verdicts — a false regression must never silently delete a good lesson. Additionally, get_playbook_context stashes last_playbook_triggers (the lessons that actually entered the prompt this turn; turns are globally serialized) and _record_turn_trajectory stamps them into the trajectory's extra.hydrated_lessons — the attribution set a future regression points at.
