memory / readonly.py — read-only store façades

Isolate an agent that may READ the operator's memory but must never WRITE it — today the delegated sub-agent.

Model (hardened 2026-07-14 bug hunt)

Default-allow-reads / explicit-deny-writes. A base _ReadOnlyProxy proxies every attribute through to the real store EXCEPT: names in the subclass _MUTATORS set (return a no-op) and names in _BLOCKED_ATTRS (return None). is_read_only = True is exposed so callers that branch on it (dream's Perfect-It skip) see the wrap. The real store is held as _real, so proxy.real hits __getattr__ and is blocked.

Why the rewrite

The previous version no-op'd guessed method names (add_memory, delete_memory, add_triplet, insert_fact) that do not exist on the real stores, while the REAL mutators fell straight through __getattr__ and executed against the operator's live memory:

  • Vector: add, ingest_document, process_batch, smart_update, bump_retrievals, bump_helpful, forget_episode, delete_document_by_name, correct_fragment, delete_fragment, delete_by_query — all unblocked.
  • Graph: add_triplets (plural), delete_by_target, wipe_all, prune_stale_edges, bump — all unblocked.
  • Skill: remove_by_trigger, record_retrieval, record_helpful_retrieval, record_retrievals_bulk, mark_verified, retract_lessons_from_trajectory — all unblocked.
  • And the raw writable handles .real / chromadb .collection / .client / graph .nx_graph were reachable, so a caller could write around the façade entirely.

Combined with the sub-agent tool-surface escape (see delegation), a delegate could permanently rewrite the operator's vector DB and knowledge graph — the exact outcome the read-only wrap exists to prevent.

Guard against drift

The _MUTATORS sets are pinned to the real method names and covered by tests/test_readonly_memory.py, which introspects each real store and fails if a store grows a new writer that isn't blocked. search is special-cased on the vector proxy to force record_retrievals=False — a read that reinforces retrieval stats is still a write to operator memory.

NOTE: core.dream's self-play path defines its own equivalent classes inline (they predate this module and carry self-play-specific behaviour). Keep the semantics in sync if you change them there.
