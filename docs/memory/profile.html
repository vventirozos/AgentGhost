memory / profile.py — ProfileMemory

Canonical user profile with multi-value merge semantics.

Storage

JSON at path/user_profile.json; atomic temp + os.replace; threading.RLock guards reads / writes.

Schema

{
  "root":          { "name": str },
  "relationships": { "wife|husband|son|daughter": str | list[str] },
  "interests":     { "science|general|...": str | list[str] },
  "assets":        { "car|...": str | list[str] }
}

Canonicalisation

Common synonyms remap at write time: wife → relationships.wife, vehicle → assets.car, etc. The caller receives the rewritten key in the return message so the agent can keep its mental model synced.

Merge semantics

Scalar → scalar: replace if different, else promote both to a list. Lists are deduped and order-preserved.

Robustness — profile_update shape

The smart-memory consolidation prompt asks the LLM to return profile_update as an object ({category, key, value}). The model occasionally ignores this and emits a bare string instead. The agent now coerces any non-dict profile_update to None at parse time: such a fact is still stored as a normal auto vector memory but skips the identity / profile_memory.update path entirely. This prevents the previous crash — 'str' object has no attribute 'get' — which fired (and was swallowed) when a high-score string slipped into the identity branch.
