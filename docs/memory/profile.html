memory / profile.py — ProfileMemory

Canonical user profile with multi-value merge semantics.

Storage

JSON at path/user_profile.json; atomic temp + os.replace; threading.RLock guards reads / writes.

Schema

{
  "root":          { "name": str },
  "relationships": { "wife|husband|son|daughter": str | list[str] },
  "interests":     { "science|general|...": str | list[str] },
  "assets":        { "car|...": str | list[str] }
}

Canonicalisation

Common synonyms remap at write time: wife → relationships.wife, vehicle → assets.car, etc. The caller receives the rewritten key in the return message so the agent can keep its mental model synced.

Merge semantics

Scalar → scalar: replace if different, else promote both to a list. Lists are deduped and order-preserved. Singular keys are exempt (they replace) and merged lists are now bounded — see below.

Singleton keys — corrections replace, not accumulate (2026-07-22)

Singular relationship / possession / identity keys — wife, husband, spouse, son, daughter, car, home, employer, name, … (module-level _SINGLETON_KEYS) — are now SINGLETONS: a second write REPLACES rather than merges. The merge branch exists for facts that genuinely coexist (python AND rust are both interests), but applied to a singular noun it turned every correction into an accumulation: update("relationships","wife","Anna") then (…,"wife","Maria") produced - wife: Anna, Maria, injected into the system prompt every turn for ever with nothing anywhere able to reconcile it. These are exactly the keys the canonicalisation table CREATES, so a correction was guaranteed to hit the merge branch. Rule of thumb: a genuinely multi-valued fact belongs under a PLURAL key (pets, children, languages, topics), which keeps merge semantics untouched.

Bounded merge — no unbounded prompt growth (2026-07-22)

Merging keys were previously uncapped, and every merged key is rendered inline by get_context_string() into every system prompt, so an unbounded key is a slow context-pressure leak. A merging key is now capped at _MAX_VALUES_PER_KEY = 8, dropping the OLDEST first (the newest statement is the most likely to be current) with the trim logged at WARNING. notes.info — the DEFAULT sink a malformed profile_update lands in (the callers default category="notes", key="info", so any dict the model emits without those keys dumps the whole extracted fact here) — gets a much tighter ring: _SINK_MAX_VALUES = 3 values of _SINK_MAX_VALUE_CHARS = 200 chars each. It had no cap, no TTL and no dedup beyond exact-string match, growing straight into the prompt. Tests: tests/test_memory_store_durability.py.

Forgetting values (not just keys)

The profile is injected into the system prompt every turn via get_context_string(), so it is the authoritative source the model answers from — vector and graph stores are secondary. This made one deletion bug user-visible: pets / interests / assets are stored as values (e.g. assets.pets = ["Hanzo the dog", "Mortimer the iguana"]), but unified_forget's profile sweep only matched keys. forget('mortimer') therefore never reached the row — it kept being injected, so the model kept "remembering" the deleted pet even after reporting success. The merge semantics above compounded it: a "Mortimer is gone" interaction appended a soft-delete tombstone string ("Mortimer the iguana (removed)") rather than deleting anything.

prune_value(category, key, target) closes this: it removes every list item (or a matching scalar) whose text mentions the target, collapsing a surviving singleton list back to a scalar and dropping the key — and the category — when nothing remains. Matching is token/word-boundary aware so forget('age') does not strip a value of 'language', while forget('mortimer') still matches 'Mortimer the iguana (removed)'. unified_forget now runs this value sweep after its key sweep (see memory tools).

Robustness — profile_update shape

The smart-memory consolidation prompt asks the LLM to return profile_update as an object ({category, key, value}). The model occasionally ignores this and emits a bare string instead. The agent now coerces any non-dict profile_update to None at parse time: such a fact is still stored as a normal auto vector memory but skips the identity / profile_memory.update path entirely. This prevents the previous crash — 'str' object has no attribute 'get' — which fired (and was swallowed) when a high-score string slipped into the identity branch.

API

MethodEffect
load() / save(data)Whole-file read / write.
update(category, key, value) -> strMerge-semantic update; returns sync message describing canonical mapping.
delete(category, key) -> strRemove a whole key; status string.
prune_value(category, key, target) -> strRemove list items / a scalar that mention target (token-boundary match); drops the key/category when emptied. Lets forget reach entities stored as values.
get_context_string() -> strFormat for LLM injection.
