memory / journal.py — MemoryJournal

FIFO work queue consumed by the dream consolidator.

Storage

JSON at path/memory_journal.json; atomic temp + fsync + os.replace (the fsync before the rename closes a torn-/zero-length-file window on power loss); threading.RLock. A sibling path/memory_journal.inflight.json stages a taken batch for crash recovery — see the TAKE / ACK lifecycle below.

Crash-safe TAKE / ACK lifecycle (2026-07-22)

pop_all() / drain() used to empty the on-disk queue in one atomic write and hand the items to the dream drain, which then spends up to ~90 s of LLM time per item. A deploy kill landing mid-drain raises asyncio.CancelledError — a BaseException the drain's except Exception never catches, with no finally — so up to max_capacity (50) buffered facts / lessons were destroyed silently: the file was already [], so nothing on restart could tell that anything had been lost.

The take now happens in two on-disk steps. pop_all() / drain() copy the batch to memory_journal.inflight.json before clearing the queue, then return the items exactly as before — signature and semantics unchanged, so the un-editable drain in core.agent keeps working verbatim. recover_inflight() folds an orphaned staging batch back to the front of the queue and runs automatically on construction and on the first load() of the process, so recovery needs no cooperation from any caller. Because the current drain cannot ack, the staging file is also rotated on the next take (a fresh take implies the previous, serially-awaited drain returned).

Net effect: crash / cancel mid-drain → items come back on next boot (at-least-once; an item whose consolidation had already been written may be re-consolidated once — the memory layer dedups facts, and re-doing a consolidation is cheap next to losing 50); normal operation → the staging file is rotated away, no replay, no growth. A crash in the ~1 ms window between staging and clearing leaves an item in both files; recover_inflight() de-dups against the queue contents, so that window cannot double-deliver either.

New ack() / commit() (alias) / inflight() exist for future callers that want explicit at-most-once confirmation: ack() clears the whole staging batch, ack(items) clears only those entries (partial ack for an item-by-item drain), and inflight() returns the un-acked batch for diagnostics.

Upstream-transient classification widened (2026-07-22)

is_upstream_transient(exc) now decides in a fixed order, so a permanently-broken item can't retry until JOURNAL_MAX_RETRIES burns out on every drain:

  1. Structurally-definitive types short-circuit to False first, before any text matching — ValueError (which json.JSONDecodeError subclasses), TypeError, KeyError, AttributeError, IndexError, NotImplementedError — so a decoder error whose message happens to contain "timeout" can't sneak through.
  2. httpx types keep their authoritative meaning (5xx → transient, everything else incl. 4xx → definitive, no fall-through to the heuristics).
  3. Builtin transport failures — TimeoutError (which asyncio.TimeoutError aliases on 3.11+) and ConnectionError.
  4. Only then a message allow-list, for the plain RuntimeError / Exception that core.llm re-raises after exhausting its OWN retries ("Max retries exceeded", "the request did not complete", empty/non-JSON, and a 5xx rendered into a string) — while a 4xx-in-a-string stays definitive.

Those two llm.py RuntimeErrors were previously classified definitive, so the consolidation was dropped instead of retried.

Overflow spill — the buffer never drops (2026-07-23)

The drain runs only in idle windows (each consolidation spends up to ~90 s of LLM and must not compete with the user's live turn). Under sustained back-to-back load with no idle gap the hot buffer fills — and it used to silently drop the oldest items past the cap of 50 (~25 turns overflowed it; a Track-B ablation driver, firing hundreds of turns with zero idle, lost far more — found 2026-07-23 in the B4 pilot).

The logical queue is now overflow + hot (oldest → newest). append() past max_capacity (raised to 256) spills the oldest surplus to a sibling memory_journal.overflow.json instead of dropping it; load() still returns only the hot buffer, but pop_all()/drain() take overflow + hot so the next idle drain consolidates everything owed, oldest-first. push_front() (a drain requeuing work it couldn't finish) folds to the overflow head — drained first — and the old capacity-bounded _merge_front that discarded surplus is gone. Nothing here ever drops an item; the overflow is small text, unbounded in principle but drained by any idle gap in practice.

Consequence for the "is there work?" checks: the biological watchdog's phase-1 gate and the self-play inter-cycle drain now ask pending_count() (hot + overflow), not len(load()) — otherwise overflow-only work (a burst that spilled, or a preemption requeue) would sit undrained forever. Tests: tests/test_journal.py (burst-loses-nothing, push-front-to-overflow, crash-recovery-to-overflow), tests/test_memory_audit_fixes.py, tests/test_smart_memory_requeue.py.

pending_count() -> intTotal owed a consolidation = overflow + hot. "Is there work?" checks MUST use this, not len(load()) (which sees only the hot buffer).
push_front(items)Re-queue items the drain couldn't finish to the overflow HEAD (drained first). Lossless — absorbs any amount, no tail-drop.

RetryableConsolidationError / is_upstream_transient(exc) / JOURNAL_MAX_RETRIESThe bounded re-queue contract for the drain loop (2026-07-09). A journal item is removed by pop_all() before processing, so a task failure after that used to be a permanent, invisible loss: an upstream 503 during the smart-memory consolidation was retried at the HTTP layer (worker failover + one 2s 5xx retry in the client) but on final failure the item was swallowed by a bare logger.error — and a main-node timeout got no HTTP-layer retry at all. Now run_smart_memory_task classifies upstream-transient failures (5xx / timeout / connection, via is_upstream_transient) and raises RetryableConsolidationError before any memory write; process_journal_queue catches it, re-queues the item with an incremented retries counter (visible WARNING + 🔄), and drops it — with a visible WARNING + 🔶 — only past JOURNAL_MAX_RETRIES (2). Definitive failures (4xx, parse errors) keep the old log-and-drop path since a re-run would fail identically. Tests: tests/test_smart_memory_requeue.py.
