memory / vector.py — VectorMemory

ChromaDB-backed semantic memory with adaptive spaced repetition, BM25 hybrid re-ranking, temporal decay, and document ingestion.

Storage

  • ChromaDB PersistentClient at memory_dir/ with collection "agent_memory".
  • Library index JSON at memory_dir/library_index.json (atomic temp + os.replace).
  • Embedding via a local SentenceTransformerEmbeddingFunction — BAAI/bge-small-en-v1.5 since 2026-07-13 (was all-MiniLM-L6-v2), overridable with GHOST_EMBED_MODEL.

Embedder swap (2026-07-13). MiniLM is trained for SYMMETRIC similarity with a 256-token window and is weak on technical/SQL text; document QA is ASYMMETRIC (short question vs long passage) — precisely its failure mode, which the retrieval code conceded by relaxing the document distance threshold to 1.25 "for Asymmetric QA". bge-small-en-v1.5 is trained for that task, takes an optional query instruction (applied to QUERIES only, via VectorMemory.embed_query — Chroma's query_texts= path reuses the DOCUMENT embedder and cannot express the asymmetry, so the doc-QA path embeds the question itself and passes query_embeddings=), and is still 384-d — so the Chroma schema is unchanged.

Embedder-fingerprint guard. Because both models are 384-d and L2-normalised, swapping them raises NOTHING: the vectors merely stop meaning anything, and retrieval returns plausible-looking garbage. The store therefore carries an embedder.json sidecar naming the model that embedded it; __init__ refuses to boot on a mismatch (including a legacy store with no sidecar) and points at scripts/reembed_memory.py, which snapshots the store to JSONL, recreates the collection with the new embedder and re-adds every fragment (161 fragments re-embedded in 0.7 s on the live store). The guard fired for real during the migration — launchd's respawn hit the un-migrated store and was correctly refused.

Conflict-recovery refuses a populated store (2026-07-22). The delete_collection+recreate recovery for an "Embedding function conflict" / "already exists" error lives in the except of the raising get_or_create_collection — so it runs BEFORE the fingerprint guard above can, and a chromadb upgrade or an embedding-function class change (both produce a matching message) would have silently wiped all ~7,368 fragments. It now counts the collection first and sys.exit(1)s with a re-embed instruction when it is non-empty; the auto drop-and-recreate fires ONLY for an EMPTY collection (nothing to lose). Covered by tests/test_vector_memory_integrity_fixes.py.

Boot self-check (2026-06-11): sentence-transformers does not raise when it can't resolve the model config (HF unreachable / not forced offline — common under default-on fail-closed Tor); it silently builds an untrained mean-pooling model that returns non-normalised 384-d vectors and poisons retrieval. __init__ now probes the embedder once and sys.exit(1)s with a remediation message if the output isn't the trained, L2-normalised model (_embedding_degradation_reason). Fix on a cold install: pre-cache the model by booting once with --no-mandatory-tor.

Leaked-semaphore fix (2026-07-15). transformers renders a "Loading weights" tqdm bar when the embedder loads, and tqdm's default get_lock() creates a multiprocessing RLock — a named posix semaphore the resource_tracker never reclaims, so every SIGTERM (a plain-kill deploy) printed resource_tracker: There appear to be 1 leaked semaphore objects. Traced to tqdm/std.py get_lock. memory/vector.py now calls tqdm.tqdm.set_lock(threading.RLock()) at import (before any bar): we never drive tqdm bars ACROSS PROCESSES, so a thread lock is sufficient and the bars still render — the process-lock semaphore is simply never created. Verified: 0 leaked semaphores under SIGTERM (was 1). Pinned by tests/test_embedder_semaphore_leak.py.

Document QA — ingest & scoped retrieval (2026-07-13)

Target: load the full PostgreSQL manual (15 MB, 3,075 pages, ~10 M chars) and ask it questions. The old pipeline could not — it refused PDFs over 1,000 pages, then silently truncated at 5 M chars, held the whole text + chunk list + an enriched copy in RAM, and had no document-scoped retrieval at all: the only path to the model was ambient hydration, where doc chunks competed with episodes and skills for a shared 6–12 k-char budget.

  • Streaming ingest (memory/pdf_ingest.py): pages are read one at a time, accumulated per TOC section, chunked, and flushed to the store in 256-chunk batches — peak memory is one batch, not one document. Caps raised to 6,000 pages / 40 M chars; chunk size 600 → 1,200 (a 600-char chunk shredded parameter descriptions and syntax blocks).
  • TOC breadcrumbs — the biggest single retrieval win on a reference manual. PDF text has no markdown headers, so semantic_split_text's header-prepending never fired and a chunk about wal_level had no idea it lived under "19.5. Write Ahead Log". build_page_breadcrumbs walks PyMuPDF's outline with a level stack to map every page → Part III › Chapter 19 › 19.5. Write Ahead Log › 19.5.1. Settings, and that breadcrumb is stamped into the EMBEDDED text of every chunk.
  • Scoped retrieval: VectorMemory.search_document(filename, question) → Chroma where={"source": filename}, 60-candidate pool, BM25 rerank, top-k. No priority tiers and no time decay (meaningless inside one document) and no distance gate — the user explicitly asked THIS document, so return the best k and let the model judge.
  • Tool surface: knowledge_base(action="query", filename=…, question=…) returns the ranked passages as TOOL OUTPUT for the model to reason over and iterate on (search → read → refine → search again). The system prompt and recall's description now both steer document questions here.

Measured live: manual ingested in 2 m 29 s → 3,075 pages, 1,897 sections, 7,131 chunks. A 6-question eval (wal_level, VACUUM vs VACUUM FULL, range partitioning syntax, pg_stat_activity, MVCC isolation levels, shared_buffers) retrieved the correct section 6/6, with the agent citing breadcrumbs like Chapter 27 › 27.2. The Cumulative Statistics System › 27.2.3. pg_stat_activity. Unprompted routing verified: a plain "what does archive_command do?" made the agent choose the scoped tool on its own. Tests: tests/test_rag_document_qa.py (20).

Scoring

For each candidate:

combined = (priority_score · _TIER_WEIGHT) + distance + time_penalty
time_penalty = age_days / (30 · log(retrieval_count + e))

Then a cross-encoder re-rank subtracts up to 0.3 from combined based on BM25 keyword overlap (normalised 0-1).

Prior weight, not absolute gate (2026-07-07): _TIER_WEIGHT was cut from 10 to 0.3. At ×10 the category tiers sat ±10 apart while distance + time_penalty spans only ~2.3 and the BM25 rerank ±0.3 — so category priority was absolute: a barely-over-threshold name-memory (dist 1.4) could never lose to a dist-0.1 exact-match document, making cross-tier semantic relevance cosmetic. At 0.3 the prior still separates distant tiers (identity vs auto ≈ 3.3 apart) but a decisively closer match can cross an adjacent tier (manual↔auto = 0.3, document↔manual = 1.5).

Priority scores

Priority + threshold are now taken from metadata.type. The deliberately-written identity and synthesis types used to have no case in the scorer and fell into the generic auto else-branch (lowest priority, 0.55 threshold) unless their text happened to match the name-string heuristics — so the highest-curation tiers ranked below raw document chunks. Fixed 2026-07-07: identity scores with the identity tier, synthesis with the summary tier.

Ambient hydration EXCLUDES the document corpus (2026-07-22). _search_selection now queries with where={"type": {"$ne": "document"}}, so the document row above is reached only via the scoped search_document path — never ambient recall. After the 2026-07-13 manual ingest 96.8% of the live store (7,130 of 7,366 fragments) is document chunks, and the tiering worked against ambient recall on two compounding axes: (1) documents carry the 1.25 threshold (2× everything else) and p_score=−5 (−1.5 after _TIER_WEIGHT), so — since lower combined_score wins — a barely-related chunk at dist 1.0 (−0.5) beat a strong auto memory at dist 0.30 (+0.60); (2) with 96.8% doc chunks the 30-candidate pool was essentially all documents, which the MemoryBus then rejected wholesale at its 0.42 match floor (vs doc distances of 0.8-1.2) — so the vector tier returned [] and ambient memory went DARK since the manual ingest. Measured after the fix: a personal query went from 30/30 documents to 18 episodes + 6 identity + 6 skills. Covered by tests/test_vector_memory_integrity_fixes.py.

Concurrency

threading.RLock guards Chroma mutations + library writes. Library JSON written via temp + os.replace for crash safety.

correct_fragment(match, replacement) — surgical text fix (2026-07-02)

Built for correcting a poisoned auto-memory: the hippocampus consolidated the chess request as "single-file, turn-based chess game…" — "single-file" was the agent's own distortion, and the stored fragment would bias every future retry. There was no update-text API (ids are md5(text), so text changes mean a new id), and the Chroma persist dir must never be opened by a second process (the HNSW segment files are mmap'd without cross-process locking), so the fix runs in-process, exposed as POST /api/memory/correct.

  • match is tried as the EXACT stored text (direct id lookup) first, then as a case-insensitive substring over the non-document fragments — deterministic, unlike a semantic query which can land on a neighbor. The type != document filter is pushed INTO the collection.get query (2026-07-22); it used to run in Python after a full-collection fetch, materialising ~7k document chunks (~75 MB) into RAM on every substring correction. Identical semantics, a fraction of the cost; delete_fragment shares the same scoped scan.
  • Ambiguous (>1 hit) or empty matches are REFUSED with previews rather than guessed at; the fuzzy multi-store forget sweep stays the wrong tool for this job.
  • Delete-then-upsert under the instance lock, preserving the original metadata (an auto memory stays auto, timestamp kept). 📝 Memory Correct log line.
  • id-collision loss fixed (2026-07-07): ids are md5(text) and Chroma's add() silently no-ops on an id that already exists — so if the corrected text hashed to an id already in the collection, the old fragment was deleted and the new one never written: the correction was lost. The write now uses upsert (the same guarantee ingest_document relies on), so the replacement always lands.

Covered by tests/test_game_move_api.py::TestCorrectFragmentUnit and tests/test_vector_memory_dataloss.py.
