memory / scratchpad.py — Scratchpad

In-memory LRU + optional SQLite persistence with per-entry TTL (default 24 h).

Storage

  • OrderedDict in process memory (LRU).
  • Optional SQLite at persist_path/scratchpad.db with table scratchpad(key PK, value JSON, created_at REAL, accessed_at REAL, namespace TEXT). The namespace column (2026-07-22) is added to pre-scope DBs in place via ALTER TABLE at construction, so the per-entry scope tags survive restarts.
  • Prod is persistent since 2026-07-14: main() builds the context scratchpad with persist_path=$GHOST_HOME/system/memory/scratchpad.db, so working state (incl. the __current_project__ resume sentinel) survives the plain-kill deploy path. --no-memory keeps it purely in-memory; self-play isolation contexts (core/dream.py) still build ephemeral Scratchpad()s.
  • A corrupt/unreadable DB at construction degrades to in-memory with a warning instead of raising (an unguarded raise at boot would respawn-loop the launchd supervisor).
  • Every SQLite connection is wrapped in contextlib.closing — with sqlite3.connect(...) alone only scopes the transaction, never closes the handle.

Eviction model

  • Age-based: any read computes cutoff = now − ttl; expired entries are deleted lazily.
  • Capacity-based: oldest LRU entry evicted when capacity exceeded.
  • Initial load on startup re-reads only non-expired entries.

threading.RLock guards in-memory dict + SQLite.

Namespace scopes & durability (2026-07-22)

Covered by tests/test_scratchpad_scope_fixes.py.

Per-entry namespace scope

One process-wide scratchpad is shared by every conversation, project and background job, and it used to be a single flat namespace. So tools/projects.py::_hydrate_scratchpad's "clear the previous project's keys" meant "delete every key that isn't a proj:: key or a sentinel" — and it runs at REQUEST START for any conversation that doesn't own the bound project, so one unrelated chat message wiped in-flight swarm results owned by another conversation. Verified in prod: the live DB had been reduced to its 2 sentinel rows (the detached worker then re-created the key in whatever scope was live, and jobs(action='collect')'s bare get(output_key) returned None — an empty result with no error).

Each entry now carries a NAMESPACE tag in a _scopes dict parallel to _data (None == global/free-chat). set() tags writes with active_namespace unless a namespace= is passed. The tag is metadata, NOT folded into the key, so every cross-subsystem bare-key lookup still resolves (swarm output_key, the job registry's resolver, recall, the proj::/sentinel conventions). New clear_namespace(ns, protect=...) deletes exactly ONE scope, so a project switch (_clear_other_project_scopes) now clears only OTHER projects' scopes and spares: the incoming project's scope, the global scope, sentinels, proj:: keys, and in-flight background-job keys — a job's output_key and its _swarm_task_id:: marker are NEVER dropped (the worker writes them minutes later, in whatever scope is live). Per-project isolation is preserved.

A park whose reversibility snapshot FAILED (_snapshot_scratchpad returned False) does NOT clear the outgoing project's keys: leaking is recoverable, deleting is not, and this runs at request start for conversations that merely share the process. The keys are left tagged with the old project (the next successful switch clears them) and only the write scope is reset, with a warning.

Persistence-failure visibility & bounded echo

_persist_entry now logs at warning (was debug) when the DB write fails while set() keeps serving the value from memory — a locked or full DB was silently making swarm results memory-only, a loss that surfaced only after the next restart. The operator watches the live stream, so this must be visible the moment it starts.

Separately, the string set() echoes back is bounded by max_echo_chars (default 500): the ack is handed straight to the model, so echoing a multi-KB swarm result doubled it into context for nothing (the model just wrote it). The STORED value is never truncated — only the acknowledgement is capped.
