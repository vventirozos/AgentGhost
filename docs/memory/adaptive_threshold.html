memory / adaptive_threshold.py — AdaptiveThreshold

Self-tuning recall acceptance threshold. Trades precision for recall based on observed usefulness.

Storage

  • JSON at memory_dir/adaptive_threshold.json.
  • Atomic temp + os.replace; threading.RLock.

Load hardening (2026-07-22)

_load() distinguishes three outcomes rather than the old except Exception: pass that silently started from the initial threshold and let the very next record() atomically overwrite the intact file — the whole learned window gone, at no log level:

  • absent → keep the seeded initial threshold (normal cold start);
  • corrupt (bad JSON / wrong type / non-list window / undecodable bytes) → move the raw bytes to a timestamped adaptive_threshold.corrupt-*.json sidecar, WARNING, restart from the initial threshold;
  • present-but-unreadable (OSError — EIO / EACCES) → run on the initial threshold but set _degraded (logged at ERROR); while set, _save() REFUSES to write over the on-disk state.

Tests: tests/test_memory_store_durability.py.

Recalculation rule

The threshold is set to 90 % of the minimum useful score, but never below the median of useless scores. Floor 0.3, ceiling 0.95.

Known limitation — the learned threshold is dead weight under --smart-memory 0.9

The 2026-07-22 durability pass did not change the retention numbers — this is a documented caveat, not a change. Under the live flag the learned value is effectively inert: the consumer takes max(cli_selectivity, learned), so with --smart-memory 0.9 the effective bar is already >= 0.9; the learned value can never exceed 0.9 (the upward candidate is min(useful_scores) * 0.9 <= 0.9) and its update rule can only ratchet DOWN toward the 0.3 floor when the window starves — so it can neither raise nor lower the bar. Compounding it, was_useful is self-referential as wired in core.agent: the signal fed back is "the score cleared the bar", not "the memory was later retrieved and helped", so the loop measures its own gate rather than utility. The class becomes load-bearing again only for a selectivity below ~0.9. Changing the arithmetic alters what the agent remembers, so it is a retention decision that was deliberately kept out of a persistence fix.

Concurrency

threading.RLock guards both reads and writes. Writes are atomic via temp file.
