interface / externals / cli

Terminal client: streaming Markdown chat, durable sessions, real turn cancellation, files/workspace, projects, and ops — one PEP-723 script.

Location & launch

One extensionless executable script, interface/externals/cli/ghost (moved into the repo 2026-07-17; previously it lived only at ~/Data/AI/bin/ghost, which is now a symlink to the repo copy — that directory is on the operator's PATH, so the command stays ghost and edits land in exactly one place). Dependencies are declared PEP-723 inline (requests, prompt_toolkit, rich) so uv run ghost works anywhere; on the operator's machine the system python has them installed.

Connection

Talks DIRECTLY to the agent (GHOST_URL, default http://eva:8000) — not through the web-interface proxy. The key rides X-Ghost-Key, resolved env-first then ~/Data/AI/.ghost_api_key (blank env never shadows the file — the Slack-bot " " incident class). Each turn sends an X-Request-ID keying the agent's turn registry, so Ctrl-C cancels the turn server-side (releasing the global turn lock) instead of just closing the socket.

Inline images (2026-07-17)

After a reply settles, ![…](name.png) references are fetched from the agent sandbox (GET /api/download/{name}, in-memory, 25 MB cap, ≤4 per reply, de-duplicated; http/data schemes skipped — the CLI only fetches from the agent) and drawn in the terminal below the reply. Protocol auto-detection: iTerm2/WezTerm (OSC 1337 inline escape; LC_TERMINAL honoured over ssh) → kitty (graphics protocol, f=100 PNG-only so non-PNG transcodes via Pillow or falls through) → half-block (universal truecolor fallback: two vertical pixels per cell via ▀, needs Pillow) → none. Escape-protocol failures fall through to half-block. Images can't be drawn INSIDE the rich Live region (the escapes are raw byte streams a repaint would shred), hence after-the-reply placement. /download of an image also renders it. Override with GHOST_CLI_IMAGES=off|iterm|kitty|halfblock. Pillow rides the PEP-723 deps but is optional at runtime (protocol paths ship raw bytes without it).

Ref normalization + tmux (2026-07-17, first live use). Replies embed whatever path shape the model chose — the first live image arrived as the FULL API path (![…](/api/download/gen_x.png)), which double-prefixed the fetch → 404. _normalize_image_ref strips /api/download/, /workspace/ and sandbox: prefixes (anchored, not mid-path); a pathed ref that still 404s is retried as its flat basename (generated images land at the sandbox root). And under tmux, auto-detection picks half-block instead of an escape protocol — tmux swallows OSC-1337/kitty sequences unless allow-passthrough is on (off by default since tmux 3.3), and a swallowed escape looks like no image at all. Forcing GHOST_CLI_IMAGES=iterm under tmux wraps sequences in DCS passthrough framing (ESC-doubled, the imgcat trick) — full-resolution images inside tmux panes, provided set -g allow-passthrough on is in .tmux.conf.

Sixel — redraw-persistent images in tmux (2026-07-17, second live report). Passthrough images are OVERLAYS: tmux repaints panes from its character grid, so any redraw — a resize, a pane switch — erases them (operator: "disappears if I slightly resize"). Sixel fixes this structurally: a sixel-built tmux (Homebrew builds it in since 3.4; the operator runs 3.7b) consumes the sequence itself, stores the image in its own model, and repaints it across resizes. New pure-python encoder (_render_sixel: Pillow 256-color quantize → 6-row bands, RLE, ~0.4s for 640×480) — emitted RAW even under tmux, never passthrough-wrapped (tmux must SEE it to own it). Auto-detection under tmux now prefers sixel when the stack supports it (tmux ≥3.4 + iTerm2/WezTerm outer + Pillow), falling back to half-block; GHOST_CLI_IMAGES=sixel forces it.

Tests
