interface / externals / slack_bot / main.py

OWNER-LOCKED async Slack client (slack_bolt Socket Mode), rewritten 2026-07-11. Replies to exactly ONE Slack user, fail-closed; also the delivery leg for the agent's outbound notifications. Tests: tests/test_slack_bot_owner_lock.py (31).

Owner lock (2026-07-11)

The bot previously answered any workspace member — full agent (and tool) access for anyone who could DM it. Now:

  • Startup is fail-closed. The owner is resolved from GHOST_SLACK_OWNER (a Slack user id, U…) or looked up from GHOST_SLACK_OWNER_EMAIL via users.lookupByEmail (needs the users:read.email scope). With neither resolvable the bot refuses to start — an unlocked bot must never run by accident.
  • Every event passes is_owner_message: authored by the owner, not bot-authored, not a message subtype (edits/deletes/joins carry mutated payloads). Everyone else is ignored silently — a reply would confirm the bot exists and invite probing; attempts are logged (user + channel) as the audit trail.
  • Thread context is owner-filtered too — only the owner's messages and the bot's own replies are forwarded to the agent. This is part of the authorization boundary, not a convenience: without it a third party could seed a shared-channel thread with prompt content (or file attachments) that the owner's next mention would forward as trusted history. Files are ingested from the OWNER only.

Per-message flow

  1. Generate an 8-char request_id, sent as X-Request-ID.
  2. Spawn tail_logs for live emoji status (below).
  3. Ingest the owner's attachments via POST /api/upload (multipart, X-Ghost-Key) — the old path wrote directly into a locally-mounted sandbox dir, which required the bot to share a filesystem (and exact path) with the agent; the API enforces its own containment and project scoping. Slack filenames are basename()d as defense-in-depth (attacker-controlled metadata).
  4. POST to $GHOST_API_URL. No model field — the agent 404s any mismatched name (the old pinned model name broke every request after a model upgrade); omitting it always matches the configured model.
  5. Strip image-markdown from the reply, fetch each image via /api/download (authed), upload to Slack with files_upload_v2 from memory (no local temp files).

Live status (fixed 2026-07-11)

The old tailer grepped for [{request_id}], which the pretty stream never prints — the full id appears ONLY on the request's BEGIN frame (then a 2-char tag), so the status feature had never matched a single line. scan_log_line (pure, tested) now arms on the BEGIN frame carrying our 8-char id, attributes emoji lines to our request until the END frame, and disarms — sound because agent turns are globally serialized (exactly one foreground request per BEGIN/END window). Default log path corrected to the agent's live stream (/Users/vasilis/Data/AI/Logs/ghost-agent.log; override with GHOST_AGENT_LOG or --log-file). The emoji map is synced to the current Icons set.

Outbound notifications

notification_poller polls /api/notifications/pending (consumer "slack") and posts notify-severity autonomous events (needs-user project tasks, scheduled-turn conclusions — see core.autonomous_activity), acking AFTER chat_postMessage so a mid-delivery crash re-serves rather than drops. Destination: GHOST_NOTIFY_SLACK_CHANNEL if set (C…/U…; off disables), otherwise a DM to the owner — push works out of the box on an owner-locked bot.

Slash command router

interface/slack_project_commands.py is a stateless shlex-based router for /project ... commands:

  • list [status] · new <title> · resume <pid> · switch <pid> · exit · status
  • task add <desc> · task done <tid> [summary] · events [limit]
  • advance — async-only; the placeholder response triggers advance_async() in the bot handler, which calls core.project_advancer.advance_once().

Launcher (run.sh, fixed 2026-07-11)

Runs the bot on the agent venv's python (the bare homebrew python3 lacks slack_bolt/httpx — same class of gotcha as the agent launcher's uvicorn; override with GHOST_BOT_PYTHON). Loads .env from the script's directory via set -a + source (the old export $(cat .env | xargs) word-split any value containing spaces, and resolved .env against whatever directory you invoked it from). Pre-flights every hard requirement — both Slack tokens, GHOST_API_KEY, and the owner lock — so a misconfigured launch fails with instructions instead of a traceback, and execs python so supervisor signals reach the bot directly.

Autostart (launchd, 2026-07-11) — LIVE

com.local.ghost-slackbot.plist ships next to the bot and is installed as a system LaunchDaemon (/Library/LaunchDaemons/), so it starts at boot with no login session — matching the prod agent's job — but runs as vasilis via UserName, never root (the venv, .env and logs are all user-owned; the bot needs no privileges). KeepAlive gives the same ops model as the agent: a plain kill is a restart/deploy; to truly stop it, bootout. ThrottleInterval=30 stops a misconfigured launch from tight-looping. Secrets live in the sibling .env (chmod 600, from .env.example) — never in the plist, never committed.

cp .env.example .env && $EDITOR .env && chmod 600 .env
sudo cp com.local.ghost-slackbot.plist /Library/LaunchDaemons/
sudo chown root:wheel /Library/LaunchDaemons/com.local.ghost-slackbot.plist
sudo chmod 644       /Library/LaunchDaemons/com.local.ghost-slackbot.plist
# logs MUST be writable by UserName — see the trap below
sudo touch /Users/vasilis/Data/AI/Logs/ghost-slack-bot.{log,err}
sudo chown vasilis:staff /Users/vasilis/Data/AI/Logs/ghost-slack-bot.{log,err}
sudo launchctl bootstrap system /Library/LaunchDaemons/com.local.ghost-slackbot.plist

Status: sudo launchctl print system/com.local.ghost-slackbot · deploy: sudo launchctl kickstart -k system/com.local.ghost-slackbot · stop: sudo launchctl bootout system/com.local.ghost-slackbot.

Trap: root-owned log files silently kill a UserName daemon. If StandardOutPath/StandardErrorPath are not writable by the UserName user (e.g. left root:staff 644 by an earlier root-run install), launchd cannot open stderr for the job and it exits EX_CONFIG (78) — before producing any output, then respawns forever. Diagnosis is otherwise baffling: the bot appears never to start, the log shows nothing new, and only launchctl print … | grep -E 'runs|last exit' reveals runs = 10, last exit code = 78. Bit us live on this exact install. Fix: sudo chown vasilis:staff the two log files.
