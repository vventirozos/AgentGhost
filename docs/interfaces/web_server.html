interface / server.py

Browser-facing FastAPI proxy on port 8080. Handles auth, streaming, file I/O, log tailing, voice forwarding.

Streaming model

  • Per-task state dict keyed by UUID in global active_chat_tasks.
  • Background worker spawned on POST /api/chat with stream=true, sharing a pooled httpx.AsyncClient.
  • The worker reads upstream chunks and appends bytes to task["buffer"]; an asyncio.Event signals new data.
  • The stream generator yields by chunk index (not byte index) and clears the event before draining to avoid races.
  • Truncation is detected via task["truncated"] and surfaces as an SSE error marker.
  • Resume endpoint accepts an offset for reconnection.
  • A janitor task evicts done tasks past 10-min TTL or hard cap of 200.

Request timeouts

Chat proxying to the agent backend uses a split httpx.Timeout built by _chat_timeout(), applied to the shared pooled client, the streaming client.stream() call, and the non-streaming client.post() call.

  • Read / write / pool: CHAT_TIMEOUT_S (env GHOST_CHAT_TIMEOUT, default 1800s / 30 min). This is the ceiling that matters for long turns. httpx's read timeout is the max gap between received bytes, not total wall-clock — a long agent turn (deep browser automation, many sequential tool calls) can run for minutes without flushing any SSE bytes. The old flat timeout=600.0 fired the read timeout at 600s, aborted the stream, and the UI rendered a bare "No response" because no assistant tokens had arrived yet. Raising this prevents that premature cut.
  • Connect: CHAT_CONNECT_TIMEOUT_S (env GHOST_CHAT_CONNECT_TIMEOUT, default 10s). Kept short and separate so a genuinely down backend still fails fast instead of hanging for the full read window.
  • The browser front-end (app.js) imposes no timeout of its own — it streams via fetch until the reader reports done, and only aborts on explicit user action (stop button). The server-side ceiling above is therefore the effective limit.

Buffer & payload limits

  • Per-task stream buffer cap 50 MB (GHOST_INTERFACE_STREAM_CAP).
  • Request body: max 500 chat messages.
  • Upload body: 100 MB hard cap via _read_capped_upload(); 64 KB chunks; rejects non-bytes.

Document caching

The root GET / reads index.html from disk on every request (so front-end edits need no server restart) and serves it with Cache-Control: no-cache, must-revalidate, private. Static assets are cache-busted via a ?v= query (e.g. style.css?v=2.7); without no-cache on the document, a browser holding a cached copy of the HTML would keep referencing the old ?v= and never fetch updated CSS/JS until a manual hard refresh. private stops shared proxies caching the document, which carries the injected GHOST_API_KEY. Guarded by tests/test_interface_index_no_cache.py. Note: bumping the ?v= on changed assets in index.html is still required for those assets themselves.

Log streaming

A log_streamer() task spawns tail -f on the agent log and broadcasts to connected websockets. Detects ERROR/Exception and tags is_error for client reactivity. Subprocess is cleaned up with timeout-then-kill on shutdown.

Auth

  • GHOST_API_KEY mandatory; the module fails to import without it.
  • All state-mutating endpoints depend on verify_interface_key() which compares X-Ghost-Key.
  • The page itself is gated by a query parameter check (browsers send keys in URL on first load only).
  • CORS: allow_origins=*, allow_credentials=False.
  • Key provenance (2026-07-13 rotation): the prod key comes from the canonical mode-600 secret file ~/Data/AI/.ghost_api_key, exported by bin/start-ghost-client.sh — this deliberately OVERRIDES the stale GHOST_API_KEY=ghost-secret-123 still present in /Library/LaunchDaemons/com.local.ghost-client.plist (plist edit needs sudo; the old public key no longer works anywhere). The same value is the AGENT's key (agent auth enabled the same day), which is why the baked upstream X-Ghost-Key keeps working. Bookmarks using ?key=ghost-secret-123 must be re-saved with the new key.

Concurrency

  • asyncio throughout; one shared httpx.AsyncClient pool.
  • FastAPI Depends(verify_interface_key) for auth.
  • StreamingResponse with custom generator for back-pressure.
  • Background tasks for upstream polling + log streaming.

Live log stream — path + resilience (2026-07-12)

The tail target defaults to /Users/vasilis/Data/AI/Logs/ghost-agent.log, overridable via the GHOST_AGENT_LOG env var — the --agent-log flag can never arrive in the prod deployment because uvicorn server:app owns sys.argv. The previous default was missing the Data/ segment, so the live-log stream (face pulses, planner monologue) tailed a nonexistent file and was silently dead in production (tail -F retries forever without erroring). The streamer is also now a restart loop: if the tail subprocess ever exits, it respawns after 5 s instead of ending the stream until the next server restart. Hardening in the same pass: the stream-buffer cap now closes the upstream connection when hit (it previously kept draining a discarded stream for up to 30 min), and a text form field named file on /api/stt is a clean 400 instead of an AttributeError 502. Regression-guarded by tests/test_interface_log_and_mobile.py.

Mobile / touch

The UI is touch-ready: 100dvh layout (no Safari address-bar jump), ≥16 px inputs (no iOS zoom-on-focus), 44–52 px touch targets, hold-to-talk with the full touchstart/touchend/touchcancel lifecycle + touch-action:none on the mic (the long-press belongs to the recording gesture, not text-selection or the iOS callout), wss: auto-selected under TLS, and an iOS getUserMedia permission timeout. Note: the service worker does not register over a self-signed cert (browser policy) — harmless, it only provides PWA caching.

Static assets

interface/static/ holds index.html (three.js, marked, dompurify, mermaid, papaparse, chart.js via CDN), app.js (auth wrapper that injects X-Ghost-Key on every /api/ fetch, voice queue management, monologue display, renderer iframes), style.css (cyan/magenta cyber theme — directional message entrances, streaming edge-pulse, animated input glow, gradient hero wordmark), matrix_graph.js (the background "face" — a three.js particle system with depth-faded nodes/links, a slow orbit, and envelope-reactive bloom; state-reactive via setWorkingState/triggerSpike/triggerPulse/setAudioLevel. Dark-multicolor theme since 2026-07-13: every node holds a stable position on a 5-stop jewel wheel — deep violet / electric blue / teal / emerald / magenta, all tuned dark for additive+bloom — declared as COLORS.palette; the wheel drifts slowly (~50s idle, ~15s busy) via a uHueDrift uniform, each line gradients between its endpoint nodes' hues (per-line aLineHue attribute fed from the per-node aSeed instanced attribute), and the crimson error tint stays unique because warm hues are deliberately absent from the wheel. Palette contract pinned by tests/test_interface_face_palette.py), cyber_face.png (the notification icon), and sw.js (service worker). All decorative motion (CSS animations and the 3D sphere's orbit/morph) is damped under prefers-reduced-motion. CDN failures degrade gracefully: marked/DOMPurify fall back to escaped plain text, and a missing mermaid only disables diagram rendering rather than breaking the page.

Chat area layout

The #ui-layer is a full-height flex column: header (flex-shrink:0), main#chat-container (flex-grow:1), then the input footer. The dominant cause of "the top third isn't showing chats" was the header height, not the chat area: #header-controls stacked its 5 icon buttons in a flex-direction:column, making the non-shrinking header ~300px tall and reserving the top third (its transparent gradient background made that band look like empty sphere backdrop with floating buttons). The buttons are now laid out in a flex-direction:row so the header is one button tall.

main#chat-container fills the space from the header down to the input using the robust flex pattern flex:1 1 0; min-height:0. An earlier attempt used margin-top:auto on top of flex-grow:1 to bottom-anchor a max-height-capped block — but since the container is position:relative, on iOS Safari that auto-margin + grow combination let it overlap the (statically-positioned) header band and, because #chat-log has pointer-events:auto, swallow taps meant for the header buttons. The fill pattern can't overflow its siblings, and the header is given position:relative; z-index:20 so it always paints above the chat and stays tappable. Two other red herrings on the chat container were cleaned up along the way: a 55vh/78vh max-height cap and a top mask-image gradient that faded the top 15% of messages until hovered — both removed. Regression-guarded by tests/test_interface_chat_layout.py (header buttons horizontal, header above chat, fill pattern with no auto-margin/cap, no fade).

Mobile (iOS Safari) note: the uConsole optimisation media query (max-width:1280px and max-height:720px) also matches portrait phones, where the ≤480px rule stacks the header into a two-row column (status + buttons). That query originally pinned header { height:50px } — a fixed height that clipped the taller column header, so #header-controls overflowed below the header box straight into the chat (the buttons appeared overlapped by chat). It is now min-height:50px so the header grows to contain its buttons while staying compact on the uConsole. The test suite forbids any fixed height on a header rule. Verified at iPhone width with a WebKit (Playwright) render: header 110px tall, buttons fully inside it, chat starts 8px below.

Two more leaks from that same broad query were split out into a uConsole-only @media (min-width:1000px) and (max-height:720px) block (phones are <1000px wide, even in landscape, so they no longer match): (1) the 36px shrink of .icon-btn/#send-btn/#mic-btn — phones now keep the 44px base touch target; and (2) #chat-input { font-size:0.95rem } — at <16px iOS Safari auto-zooms the page when the field is focused, so phones now keep ≥16px. A WebKit sweep (iPhone SE / 13 portrait+landscape / 15 Pro Max landscape / uConsole) confirms phones report 44px controls and 16px input while the uConsole still gets its compact 36px/0.95rem. Guarded by tests/test_interface_chat_layout.py. Pinch-zoom remains intentionally disabled via the viewport meta (user-scalable=no).

Visualizer iframes & sandboxing

The floating "VISUALIZER" window uses two iframes. #render-iframe (sandbox="allow-scripts allow-same-origin") is used only to display our own blob previews — images and PDFs — which need the parent to write into contentDocument. Untrusted agent-authored HTML/CSS/JS from "Visualize" buttons goes into a separate #render-iframe-sandboxed (sandbox="allow-scripts", no allow-same-origin) via srcdoc, so it runs in an opaque origin and cannot reach window.parent or read the injected GHOST_API_KEY. Mixing allow-scripts + allow-same-origin on a single iframe rendering agent code would let it escape the sandbox and exfiltrate the key.

Agent reasoning (<think>…</think>) and <tool_call> blocks are stripped from the displayed transcript (via _stripInternalTags) so the visible chat matches what the TTS engine speaks.

Inline media handling for /api/download/…

A MutationObserver in app.js watches #chat-log for tool-emitted artefact links. Two cases:

  • Images (<img src="/api/download/gen_*.png">): the unauthed <img> load 401s because browsers don't attach custom headers to plain image loads. _handleChatImage fetches through the authed wrapper, caches a blob URL in an LRU (cap 100), and swaps img.src to the blob. Failed fetches surface a "broken image" badge.
  • PDF reports (<a href="/api/download/report_*.pdf">): _handleChatPdfLink decorates the anchor with the .pdf-link card style, intercepts the click, fetches the bytes through the same authed wrapper / blob cache, opens the blob in the right-rail visualizer iframe (browsers render application/pdf blobs natively), and simultaneously triggers a real download via a hidden <a download>.

Both code paths share _toAuthedBlobUrl and the same blob cache, so an artefact opened once doesn't refetch. Because a MutationObserver only reports future mutations, the same processing (_processChatLogArtifacts) is also run once on startup against the chat history restored from localStorage, so restored images/PDF links get the authed-blob treatment instead of 401-ing. Auto-opening the visualizer is suppressed while history is being restored (live replies still auto-open).
