Metacognition uplift — calibrated confidence, telemetry, and dual-solver arbitration

Six new modules that close the four metacognitive gaps identified by the AgentGhost Meta-Cognition Evaluation: token-level calibration, grounded host telemetry, trigger-driven replanning, and confidence-gated dual-solver arbitration.

Status: WIRED + AUTO-ROUTING

As of 2026-05-23, the uplift modules are integrated into the live turn loop behind the --enable-metacog flag (off by default — legacy behaviour is unchanged when the flag is absent). The single source of truth for wired-state is core/metacog.py:MetacogBundle, hung off GhostContext.metacog. Every wire-point in the agent reads context.metacog with a single attribute check; None short-circuits the integration immediately.

The dual-solver arbiter now auto-routes mid-turn. When the agent is about to dispatch a mutating-host tool (shell or SQL) AND the most recent composite confidence reading was below threshold, the arbiter fires on the user's prompt. If the two candidate plans diverge enough that the arbiter returns ask_user, the dispatch is REPLACED with a synthetic SYSTEM PAUSE tool result — the model's next turn then produces a clarification question instead of executing the action.

Mid-turn arbiter gate

Five gates govern when the auto-route fires. All five must pass; otherwise the dispatch proceeds unchanged. The gates are evaluated in this order (cheapest first) inside MetacogBundle.arbitrate_tool_calls:

  1. Bundle enabled. --enable-metacog set and the bundle's arbiter field is wired (i.e. --metacog-disable-arbiter NOT set).
  2. Tool is in GATED_DOMAINS. Only shell and sql — the host-mutating domains where a wrong call is irrecoverable. file_system, web_search, update_profile, vision tools, etc. always pass through.
  3. Per-request arbitration cap. MAX_ARBITRATIONS_PER_REQUEST = 1. The cap is reset by reset_arbitration_counter() at the top of every handle_chat call, so each user turn gets a fresh budget.
  4. Composite confidence below threshold. Pulled from bundle._last_confidence, populated by the post-stream confidence step. Cold start (no prior reading) passes through — we don't pay the doubled inference cost on the first turn of a fresh session. The force=True parameter on arbitrate_tool_calls bypasses this gate, mostly used by tests.
  5. User prompt present. The arbiter needs SOMETHING to arbitrate over. The bundle walks messages in reverse and uses the most recent role=="user" content as the prompt.

Decision routing

DecisionAgent behaviour
executeLogged; original dispatch proceeds. Arbiter agreed with the original plan.
validateLogged; original dispatch proceeds. Arbiter saw divergence but no validator was attached. Caller may inspect decision.other for the alternative.
ask_userHard block. Dispatch is replaced with a synthetic tool result that surfaces the divergence to the model. The model's next turn produces a clarification request.
skippedEmpty prompt; treated as if the gate returned None.

Pre-flight repeat-failure guard feature 1A

Live counterpart to the offline post-mortem repeated-error fingerprint — blocks a known-failing action before dispatch instead of mining the loop after the turns are spent.

The idle post-mortem (reflection/postmortem.py) computes a TranscriptSignature whose dominant pathology is usually the SAME error from the same tool recurred N×. That insight only lands after a failed run has already burned its turns. The pre-flight guard promotes the same notion into the live tool loop: RecentFailureGuard (core/triggers.py) keeps a bounded rolling memory of recently-FAILED calls keyed by (tool, primary target) — the same primary_target_from_args key the offline read-loop detector and the in-run no-progress loop-breaker use — with the error text normalised to an 80-char lowercased prefix so two failures that differ only in a trailing variable count as one recurring failure.

Before each tool is dispatched (core/agent.py, immediately after the metacog arbiter gate), the loop asks would_repeat(tool, target). If the same action has already failed the same way at least repeat_threshold times in the window, the call is skipped and replaced with a synthetic SYSTEM BLOCK tool message carrying the prior error, telling the model to change approach (or re-issue with an explicit modification if it believes the failure was transient). Every FAILED result is fed back into the guard in the per-call outcome loop.

Default repeat_threshold = 2 is deliberate: keying on (tool, target) alone would block a legitimate retry — e.g. re-running execute on a script after editing it to fix the bug (the exact case the idempotency guard already exempts execute/file_system.write for). Requiring the identical error to recur means a post-fix re-run that now succeeds, or fails a new way, never trips the guard — only a genuinely stuck loop does. Idempotent setters (update_profile, learn_skill, knowledge_base insert/forget) are exempt entirely, since the idempotency guard above already covers them.

The guard is independent of --enable-metacog: it ships on by default and is governed by its own --enable-preflight-guard / --no-enable-preflight-guard flag. When disabled, the guard object still records failures (cheap) but is never consulted as a hard block. Covered by tests/test_preflight_guard.py.

Host-signal dedup

The 1Hz host poller used to emit a signal on every poll that crossed a threshold. On a deployment where the LLM server pins RAM at 95% as steady state, that produced ~86,400 signals per day — useless spam. The poller now dedups on (metric-set, severity): identical consecutive states are suppressed until either the state changes (a metric clears, a new metric trips, or severity escalates) or --metacog-host-heartbeat-s elapses. Steady-state 95% RAM → one log line, plus a heartbeat every 5 min.

The ReplanBridge similarly suppresses log output for every noop:* action (no_plan / no_request_revision / info-severity) — those used to log once per host signal. The full audit still lives in bridge.revisions for programmatic inspection.

Why

The pre-uplift agent already had hierarchical planning with postconditions (core/planning.py), MCTS over candidate actions (core/mcts.py), a Process Reward Model (prm/), idle-time reflection (reflection/loop.py), and runtime circuit-breakers — strong Level-2/Level-3 metacognitive infrastructure. The four remaining gaps were:

  1. Calibration — no signal of generation uncertainty beyond verbalised hedges; specifically no token-level Shannon entropy.
  2. Grounded monitoring — host telemetry existed only as a callable tool; it never fed the live turn loop.
  3. Trigger replanning — existing detectors retried-or-aborted but did not switch strategy.
  4. Dual-solver arbitration — no two-shot consensus check on low-confidence turns.

core/confidence.py 2.4 — Composite confidence CompositeConfidence fuses entropy and competence into a single C ∈ [0, 1] with operator-tunable weights and threshold. Includes prior-shrinkage when the competence cell has fewer than five observations.
core/triggers.py 2.5/2.6 — Triggers & replan TriggerBus async pub/sub with three first-class event kinds (LoopDetected, ResourceExhausted, ExecutionAnomaly), helpers ToolRuntimeBudget (p95 × multiplier window) and RepetitionCounter (sequential streaks), and ReplanBridge which routes warning/critical events to ProjectPlan.request_revision.
core/arbiter.py 3 — Dual-solver arbitration DualSolverArbiter runs two scripted-temperature candidates in parallel, computes cosine similarity (Jaccard fallback when no embedder is wired), and returns an ArbitrationDecision with action ∈ {execute, validate, ask_user, skipped}. The confidence_gated helper enforces "arbitration runs only on low-confidence turns".
core/mcts.py 1.5 — PRM uncertainty downweighting New uncertainty_penalty constructor arg on MCTSReasoner. When the attached scorer exposes uncertainty(state, action), candidates are penalised by penalty × u so MCTS prefers branches the PRM is confident about. Default 0.0 preserves legacy behaviour.

Operational notes

  • Edge cost discipline. The arbiter is opt-in via confidence gating; the host poller runs at 1 Hz; entropy work is in the existing streaming path. None of these add a second round-trip on the easy-prompt fast path.
  • Backward compatibility. Every module integrates via narrow hooks. MCTSReasoner's default uncertainty_penalty=0 keeps the pre-uplift behaviour unchanged for callers that haven't been re-wired.
  • Failure modes. Every numeric input runs through a saturating clamp; non-finite inputs collapse to neutral 0.5 rather than NaN. Validators that themselves crash return safe-reject. Trigger handlers are isolated — a misbehaving subscriber does not halt the bus.

Logging contract

Every uplift event reaches the operator in ONE consistent shape. Single grep prefix, structured fields. See core/metacog_log.py. 

metacog <subsystem>    key=value key=value ...

🧮metacog arbiterEach mid-turn arbiter decisionWARNING when action=ask_user, INFO otherwise

Each subsystem gets a distinct emoji so a quick visual scan separates metacog activity from the agent's own brain-think (💭), planner (📋), and tool-execution lines. The previous unified-but-monotonic 💭 made metacog events indistinguishable from the model's thinking trace.

Field conventions

  • Floats render at %.2f (so C=0\.\d{2} matches every confidence line).
  • Booleans render as yes / no (cheaper to scan, matches operator phrasing).
  • None renders as -.
  • Strings with spaces are auto double-quoted so a parser can split on whitespace without losing words.
