Cross-project dependency map feature 3

Collapsing every project's libraries and techniques onto SHARED graph nodes so the agent can reuse what it solved in one project while working on another.

The gap it closes

The knowledge graph already received a node per project and per task, but the only edges were project → its own tasks → a truncated description string. So PetAI uses GRU and Genesis uses recurrent nets became two unrelated nodes under two separate project subtrees — they never shared a node, and spreading activation could not connect them. There was no USES_LIBRARY / USES_TECHNIQUE predicate, no cross-project retrieval, and the per-turn briefing was strictly single-project. The agent re-learned in one project what it already knew in another.

Concept extraction 3A

core/project_concepts.py turns a project's durable text (title, goal, design ledger, config) and its workspace requirements.txt into canonical concept names via a curated lexicon: extract_libraries (alias-collapsing — pytorch→torch, sklearn→scikit-learn — plus a requirements parser) and extract_techniques (recurrent-net, transformer, cnn, reinforcement-learning, diffusion, …). Short/ambiguous tokens (rnn, gru, gan, rl) are matched on word boundaries so they don't fire inside unrelated words.

link_project_concepts(graph_memory, project) writes the edges to shared canonical nodes: 

project:<id>  --USES_LIBRARY-->   library:torch
project:<id>  --USES_TECHNIQUE--> technique:recurrent-net

Because GraphMemory.add_triplets lowercases subjects/objects and USES_* are multi-valued (non-functional) predicates, two projects emitting technique:recurrent-net land on the same node — the bridge the old design lacked. Re-adds are idempotent (they only bump edge weight). The agent links concepts at project create and whenever the ledger or config changes (tools/projects.py _link_concepts_in_graph); scripts/backfill_project_concepts.py backfills projects created before the wiring (idempotent; supports --dry-run).

Retrieval & briefing 3B

find_related_projects(graph_memory, store, project_id) walks the graph's in-memory mirror: the project's concept nodes are its library:*/technique:* successors, and any other project:* predecessor of those nodes is related. Results are ranked by shared-concept count. render_related_work formats them, and build_project_briefing (given the graph) appends a RELATED WORK block every turn a project is active: 

RELATED WORK (other projects sharing tech — reuse their lessons instead of re-solving):
  - Genesis [ab12cd · ACTIVE] — shares: torch, recurrent-net

The agent passes context.graph_memory at the briefing call site (core/agent.py); API / tool callers may omit it, in which case the block is simply absent (no crash). The section is self-scoped — a project that shares no concepts shows nothing.

Scope notes

  • Extraction is rule-based by design — cheap, deterministic, good enough to bridge on the obvious shared tech. New surface forms are added to the lexicons as they come up; it is not exhaustive NLP.
  • The standing surfacing is the per-turn RELATED WORK block. A separate "while you were away" digest flag for shared-library adoption was considered and deferred as redundant with the briefing.

Test surface

  • tests/test_project_concepts.py — extraction, alias collapse, word-boundary matching, triplet shape, graph linking, idempotency, tool wiring.
  • tests/test_cross_project_retrieval.py — overlap ranking, self-scoping, RELATED WORK in the briefing, graph-absent fallback.
