Off-main node offload

A second box is a second inference SLOT — not a smarter or faster model. That is the whole value, and it is substantial.

The constraint

llama-server runs with -np 1 (ONE inference slot) and agent turns are serialized behind Semaphore(1). So every auxiliary LLM call the agent makes — verifier, conversation compaction, constraint audit, task classification, memory consolidation — does one of three bad things:

  • blocks the user's turn (an inline, awaited call);
  • queues behind it (a background call waiting on foreground_tasks == 0);
  • disturbs the main model's KV prefix cache, so the next user turn pays a re-prefill (the llama log is full of "restored context checkpoint" lines — each restore costs real time).

A secondary node — even a SMALL, SLOWER model — removes that work from the critical path entirely. It does not need to be good at the hard work; it needs to be good enough at the small work that is currently stealing the big model's slot.

Pools

LLMClient carries six pools, selected per call via chat_completion(..., use_worker=/use_critic=/use_coding=/…). Every selector falls back to the main node when its pool is empty, so opting a call site in is a safe no-op until a node is configured.

--worker-nodes url|modelConversation compaction/summarization (inline — the most painful blocking call on a 240k-context agent), the mid-turn "shield" summarizer, follow-up query expansion (entirely disabled without a worker pool), smart-memory consolidation, the constraint-gate screen, the autoadvance task classifier.

Screen-then-confirm (the constraint gate, 2026-07-11)

The constraint gate is the ideal offload target — it is a blocking LLM call on the user's turn — but it is also a gate whose false positive blocks work (an over-broad version of it deadlocked a real project; see tools/projects.py). Handing a veto to a small model makes that failure mode worse. So the gate splits the decision:

  • Screen on the worker pool. The common case — "no violation" — is answered off-main and costs the 35B nothing.
  • A "violates" verdict is not trusted on its own: it is re-run and CONFIRMED on the main model before it blocks anything, and the main model's constraint/evidence text is what the agent sees. The expensive call happens only on the rare positive.
  • A false NEGATIVE needs no confirmation — it simply passes the gate, which matches this function's fail-open posture everywhere else.

With no worker pool, the screen already was the main model, so the confirm pass is skipped and behaviour is byte-identical to before (no wasteful double call).

Deliberately NOT offloaded

Reflection critique and post-mortem analysis stay on the main model. They WRITE lessons and classify defects, so a weak judge poisons the learning stack — and they run at idle, when the main model is free anyway. The offload rule: a call is a candidate only when the small model is competent at it and it currently costs the user latency. Idle-time, quality-critical work fails both halves. Pinned by a test.

Verifying it works

GET /api/health now reports a nodes map (per pool → configured URLs). Without it there was no way to confirm from outside that a node was actually wired — you had to read the boot log. An empty pool means that work runs on the main model.

curl -s localhost:8000/api/health | jq .nodes
# {"worker": ["http://mini:8088"], "critic": [], …}
