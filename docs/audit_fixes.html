Functional hunt (2026-07-04): async-critic mode shipped six untested writes — the chess post-mortem

A live request — play chess against each other. with YOU, not a generated chess AI … turn based from the terminal — failed comprehensively: five distinct crashes shipped to the user (a chess.py that shadowed the python-chess library, a rank off-by-one IndexError, a NameError from a half-renamed function, a hallucinated chess.square_index()), and the final deliverable violated the one constraint the agent had itself captured on create.

Root cause 1 — the unverified-write guard was OFF in the deployed mode. The in-loop VERIFIER-GATE AUTO-REPAIR (which forces an actually RUN it re-entry when a turn finalises on an untested file write) was gated behind not self._critic_async_enabled() — and the production config runs --critic-nodes (async critic). So all six finalised on an UNVERIFIED file write — flagged INCOMPLETE warnings fired, appended a note, and shipped anyway: the user became the test harness. The check itself (_is_unverified_mutation) is a pure predicate over the turn's tool records — no LLM call — so there was never a latency reason to skip it in async mode. Fix: async mode now runs the deterministic check inline and triggers the same bounded repair re-entry; the LLM verdict itself stays deferred to the post-loop gate, preserving async's non-blocking contract. Tests: tests/test_async_unverified_repair.py (5).

Root cause 2 — the captured constraint never reached the code generator, and the steer had no teeth. The project record stored with YOU - Ghost plays directly, not a generated chess AI, but the autoadvance coding executor's spec prompt contained no trace of it — the first engine violation was written by that exact path. And when the post-write constraint steer did fire, it carried no architecture guidance, so the model rationalised the evaluation function IS me. Meanwhile the correct infrastructure already existed: POST /api/game/move (built after the 2026-07-02 chess incident) has the agent's LLM choose the move at inference time with no engine fallback — and a terminal script run on the user's machine reaches 127.0.0.1:8000 fine (only the agent's sandbox cannot; the model over-generalised the sandbox-loopback fact and dismissed the design). Fixes: (a) project_advancer now passes the project's stored constraints into build_coding_task → _generate_build_spec, which renders them as an EXPLICIT USER CONSTRAINTS (PROJECT-WIDE) block; (b) new has_participant_constraint() + PARTICIPANT_STEER (utils/constraints.py) detect player-role constraints and append a binding architecture directive — state-file + per-chat-turn reasoning, or a thin client POSTing to /api/game/move — to both the post-write steer and the executor spec prompt; (c) the PARTICIPANT MODE prompt section now states explicitly that a user-run terminal client can reach the API and is the correct terminal-game design. Tests: tests/test_participant_constraint_steer.py (15).

Live regression (2026-07-04): native tool_call corruption broke introspect

A live hey ghost, tell me about yourself turn wedged: the agent called introspect and got, every turn, SYSTEM ERROR: 'action' must be one of ['narrative', 'recall', 'recent', 'stats', 'summary'] — even though its own reasoning clearly passed a valid action (summary / narrative / stats). The model was not at fault; the argument was being corrupted before it reached the tool.

Root cause. With --native-tools on (the Qwen 3.6+ default), the upstream server parses the tool-call XML itself and hands us structured message.tool_calls, bypassing Ghost's own robust XML parser. On a multi-tool reply using the equals-format <function=name> / <parameter=name> shape, the server closes the first <parameter> correctly but then leaks the entire serialized XML of every following tool call into the first argument's string value: 

{"action": "summary</parameter>\n</function>\n</tool_call>\n<tool_call>\n<function=list_lessons>\n<parameter=scope>\nall",
 "limit": 10}

So action becomes a giant invalid string and the turn is wasted (and the pre-flight dedup guard then blocks the retry). The trace above is verbatim from session e260ae3e.

ItemResolutionSeverity
Corrupt native tool_calls merged a multi-tool reply into one call's arguments New _repair_native_tool_calls (in core/agent) runs on the native-tool_call fallback path. It detects the leak by its unambiguous CLOSE-then-OPEN signature (a </parameter>/</function>/</tool_call> close followed by a new <tool_call>/<function=…> opening), and — only when the leaked tail names at least one known tool — truncates the value at the first framing token and recovers the merged calls as separate tool_calls. The example above becomes a valid introspect(action=summary, limit=10) plus a recovered list_lessons(scope=all). high (broke every multi-tool native turn)

Why it's safe. Clean tool_calls pass through byte-for-byte. Content that merely mentions framing text (code, docs) has no close-then-open pair, or names no real tool, so it is never mangled — verified by tests/test_native_toolcall_repair.py (16 tests) and the probe:native_toolcall_repair invariant in the offline eval gate. Ghost's own XML parser already handled this reply shape correctly; this fix brings the native path up to the same robustness rather than disabling native tools (which would forfeit the ~7 K-token-per-turn schema saving).

Follow-up (functional hunt unit 5). A live file_system read exposed a gap: when the native parser dumps the entire merged reply into the first argument, the value STARTS with framing (no clean prefix, m.start()==0). The first version left that value untouched and dispatched a tool call whose argument was raw tag-soup — a wasted strike. Now that "phantom" primary is dropped and the first recovered call is promoted in its place (its real content was in the tail all along). Pinned by an added case in tests/test_native_toolcall_repair.py.

Follow-up (functional hunt unit 7). A live manage_tasks stop showed a third shape: the SECOND parameter of the same call leaked into the first — {"action": "stop</parameter><parameter=task_identifier>task_123"} — i.e. a </parameter> followed by a sibling <parameter=…>, not a new <function>. The repair was generalised from "close-then-new-call" to "close-then-continuation": a leaked sibling <parameter> now folds back into the same call's arguments ({action: stop, task_identifier: task_123}), so the call succeeds in one turn instead of striking with an "Unknown action".

Functional hunt (2026-07-04): a quoted <tool_call> mention truncated the <think> strip

First finding of the live functional bug hunt (BUGHUNT_FUNCTIONAL.md — driving the running agent on :8000 and scanning the log). The app log showed a recurring system_parse_error on coding tasks: 

Parse target truncated. len=66 chars. tool_call=1/0 function=0/0 ...
HEAD: <tool_call>` per turn, I'll start by writing the script.\n\n</think>

Cause. When the model reasons about the guidance emit exactly one <tool_call> per turn, it quotes <tool_call> inside its <think> block. The stripper — <think>.*?(?:</think>|(?=<tool_call|function)|$) with a lazy .*? — cut the block at that quoted mention instead of the real </think>, leaving <tool_call>` per turn…</think>\n<tool_call>… as the parse target. The tool parser read the leading tag-soup as a malformed call → system_parse_error, a wasted turn, and an execution strike.

ItemResolutionSeverity
Lazy <think> strip truncated at a quoted <tool_call> mention New _strip_think_blocks (core/agent, applied at all four strip sites) prefers the real </think>: closed blocks are removed whole (a quoted mention inside can no longer truncate anything), and only a genuinely unclosed <think> is stripped up to a REAL tool-call opening (a wrapped <tool_call><function> or a named <function …>) — never a bare quoted mention. med (recurring wasted turns on coding tasks)

Pinned by tests/test_think_strip_toolcall_mention.py (12 tests, incl. the verbatim log shape and the unclosed-think design case).

Functional hunt (2026-07-04): insert_fact hung the turn and dropped the fact

Unit 3 (memory). Add this to your knowledge base: … Bluefin … hung the turn to the client timeout (>150 s); a follow-up query then could not find Bluefin at all. The log showed 📝 memory store and then nothing — no request finished.

Cause. tool_remember's bus-aware path awaited the graph-triplet extraction LLM call inline, BEFORE the publish_fact() that stores the fact — and with is_background=True. The background path parks on _wait_for_foreground_clear(), which waits until foreground_requests==0; but the request it waits for is THIS turn — a self-deadlock that parks for the 600 s ceiling. So the turn hung and, because the hang was before the publish, the fact was never written (only the raw journal logged it, so recall found nothing). The legacy direct path had always backgrounded this extraction; the bus path had regressed to a blocking inline await.

ItemResolutionSeverity
Inline, self-deadlocking graph-triplet extraction in insert_fact Moved the extraction OFF the turn's critical path (tools/memory): publish_fact() now stores the fact immediately with empty triplets, and a fire-and-forget background task extracts triplets (bounded by asyncio.wait_for(_GRAPH_EXTRACT_TIMEOUT_S=20s)) and adds them to the graph — mirroring the legacy path. There is_background=True is finally correct, because the turn no longer awaits it, so it can't self-deadlock. The turn returns in ~5 s (was >150 s hang), the fact is stored and immediately recall-able, and the graph is enriched asynchronously. high (turn hang + silent data loss)

Verified live: "store … Kestrel …" returned in 7 s and a follow-up recall quoted the fact back (match 0.76). Pinned by tests/test_insert_fact_hang.py (4 tests: a 1-hour-hang client returns the turn fast with the fact published; a fast extractor lands triplets in the graph via the background task with is_background=True; no-LLM and empty-text paths).
