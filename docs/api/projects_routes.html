api / projects_routes.py

REST surface over the ProjectStore. Used by the Slack project commands and the web UI's Projects tab.

All endpoints require X-Ghost-Key.

Project endpoints

Method · PathBehaviour
GET /api/projectsList, optionally filtered by status.
POST /api/projects201 Create. Required: title; optional: kind (default GENERAL), goal.
GET /api/projects/{pid}Fetch.
PATCH /api/projects/{pid}Update title, goal, kind, status, metadata. metadata is shallow-merged into the existing blob (2026-07-20 — see below).
DELETE /api/projects/{pid}Soft delete (status=ARCHIVED). ?hard=true for cascade hard delete.
POST /api/projects/{pid}/switchSet as current (via the conversation-binding setter — see 2026-07-20 note); returns project briefing.
POST /api/projects/{pid}/resumeResume + log a project event (same binding path as switch).
POST /api/projects/{pid}/advanceRun one task-advancement cycle via advance_once() with a project-pinned tool runner; 503 when no runner can be built (2026-07-20 — see below).
GET /api/projects/{pid}/eventsList project events. Query: limit, type.

Route hardening (2026-07-20)

  • /advance builds a real, project-pinned tool runner — or refuses. The route used to call advance_once with no runner, i.e. the degraded classify-only path, which marked a research leaf DONE having done nothing — theatrical completion on a network-reachable authed route. It now builds the SAME registry-backed runner the manage_projects autoadvance action uses: tools constructed from pinned_project_context (file writes resolve to projects/<pid>/), the run wrapped in pinned_event_project (workspace events stamp the advanced project, not whatever the chat side has active), plus the LLM classifier / code generator and the real coding executor. When no runner can be built the route returns 503 instead of failing open. The Slack advance command's async path (interface/slack_project_commands.py · advance_async) got the identical treatment — same pinned runner and pinned_event_project wrap, refusing with a tool runner unavailable warning reply (a chat surface has no status code). Covered by tests/test_slack_project_commands.py.
  • switch / resume / delete route through _set_current. A raw current_project_id attribute write skipped the conversation sentinel + workspace-model pointer + scratchpad snapshot/hydrate, so the next chat turn's reconcile silently undid the switch — and a delete left the sentinel naming the dead id, so reconcile reactivated the deleted project and the sandbox recreated projects/<deleted-id>/. All three now go through the projects-tool setter.
  • Canonical pids. Every pid-bearing route canonicalizes the path pid at entry (_canon_id: strip + lowercase — store ids are canonical); a case-mangled pid used to 404 on ownership checks and diverge the workspace path.
  • PATCH metadata merges, never replaces. ProjectStore.update_project is merge-by-default since 2026-07-20 inside one BEGIN IMMEDIATE txn — the blob carries system state (design ledger, config, step counters) that a whole-dict replace silently destroyed, so the documented budget-raise {"metadata": {"steps_cap": 100}} no longer wipes the rest. The store's metadata_replace=True escape hatch is for deliberate callers only and is not exposed over REST (ProjectUpdate has no such field).
  • Blocking store work is off-loop. PATCH /{pid}, DELETE /{pid}, and the task-status PATCH run their store calls via asyncio.to_thread: a DONE transition fires the workspace-cleanup sweep (a synchronous filesystem walk) from the store hook, and a hard delete rmtree's the workspace — neither may block the process-wide event loop. /events also clamps its limit (1…1000; a negative limit means "no limit" in SQLite).

Covered by tests/test_projects_api.py.
