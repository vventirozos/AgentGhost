Delegation, jobs & pipelines

core/subagent.py · core/jobs.py · tools/delegate.py · the save_as data-flow in tools/composed_skills.py — 2026-07-11. Three gaps, one theme: the agent could dispatch work but never COMPOSE it.

The gap

The 2026-07 feature survey found that no orchestration primitive could pass a value from one step to the next:

  • composed_skills._resolve_args resolved $var only against the macro's initial params — a step could never consume the previous step's output, so "fetch → transform → act on the fetched value" was inexpressible and the skill-graduation loop minted macros that were structurally incapable of being pipelines.
  • delegate_to_swarm's workers are stateless single completions with no tools — delegation could summarise text you already had, never go and get it.
  • Fire-and-forget work (swarm ids, scheduled tasks) had no status surface: results appeared only if the model remembered to read a specific scratchpad key.

1. Composed-skill data-flow (save_as)

A sequential step may bind its result to a name: {"tool": "browser", "save_as": "page"}. Later steps interpolate it as $page — either as a whole value ({"text": "$page"}, substituted with its native type) or inside surrounding text ({"prompt": "Summarise: $page"}, ${page} also accepted).

_execute_sequential keeps a live binding scope — the initial params plus every completed step's save_as — and resolves each step against the scope as it stands when that step runs, so bindings only flow FORWARD. The caller's params dict is copied, never mutated. A bound value is capped at MAX_BOUND_VALUE_CHARS (16 KB — larger than the 4 KB display cap, since a downstream step may legitimately need a big fetched body). An optional step that FAILED still binds, so a later step can deliberately inspect the error text.

Authoring is validated, not silently degraded (_validate_dataflow, run at define time): a forward reference, a self-reference, a duplicate binding, or any save_as in a parallel macro is REJECTED with an explanatory error — parallel steps start simultaneously, so no sibling can observe another's result. A name that no step produces is fine: that's a runtime param the caller supplies (the pre-existing contract). The advertised schema subtracts step-produced names from the macro's parameter list, so the LLM is never asked to supply a value the pipeline computes for itself.

2. Tool-using sub-agents (delegate)

core.subagent.run_subagent runs the REAL agent loop (tools, sandbox, memory reads) in an isolated context, bounded by a tool allowlist and a turn cap, and returns its final answer as a job result. Isolation follows the pattern dream's self-play temp agent established — for the same reasons, with the same failure modes already paid for:

  • workspace_model = None — a sub-agent runs under its OWN semaphore; leaving the shared model attached would let it clobber the live turn's event-stamp pointer and write synthetic outcomes into the real activity log (the 2026-07-09 stamping race).
  • trajectory_collector / episodic_memory / journal = None — a sub-agent's turns must not be mined for auto-macros or reflected on as if they were user work.
  • Memory stores wrapped read-only (memory.readonly) and memory_bus = None — a delegate has no hydration bus at all, so a stray publish can't reach the production stores; a delegate researches, it does not rewrite the operator's memory.
  • The LLM client is forced to background mode, so a delegated job can never starve a live user turn of the single upstream slot.

wait=True deadlock downgrade (2026-07-22 LLM-stack review). Because a sub-agent's LLM calls are forced background, they park on _wait_for_foreground_clear for as long as a user request is active. delegate(wait=True) blocks the parent request inside reg.wait while that request still holds foreground_requests up — so the sub-agent parks on the very request awaiting it: a circular wait that produced ~600s of dead air, then a timed-out FAILED job. tool_delegate now treats foreground_requests > 0 (which is true precisely on an interactive turn, and false in an idle/autoadvance context) as the deadlock condition and downgrades wait=True to fire-and-forget there: it spawns the jobs, returns immediately with the job ids and an explanatory note, and the model collects them on a later turn (by which point the foreground request has ended and the sub-agents can actually run). In an idle/background context wait=True is honoured unchanged. Tests: test_delegate_swarm_review_fixes.py.

Containment hardening (2026-07-14 bug hunt). Two escapes made the isolation above nominal-only and are now closed: (1) tool surface — the old code filtered only the dispatch DICT and left _subagent_allowed_tools unused, so the SCHEMA the model saw was the full registry (it was shown delegate/ jobs/manage_* and invited to call them), and a dispatch miss healed the dict back to the full registry. Fixed: run_subagent sets disabled_tools = advertised − allowlist (filters the schema AND blocks dispatch by name), and GhostAgent._rebuild_available_tools re-narrows to the allowlist on every miss. (2) read-only memory — the old readonly.py no-op'd GUESSED method names that don't exist on the stores, while the real mutators (add, ingest_document, add_triplets, delete_by_target, remove_by_trigger …) and the raw .real/ .collection handles passed straight through. Rewritten as a default-allow-reads / explicit-deny-writes proxy pinned to the real method names, with a test that fails if a store grows an unblocked writer. Tests: test_subagent_containment.py, test_readonly_memory.py.

Rebuild re-drops the disabled set 2026-07-22. _rebuild_available_tools (which heals the dispatch dict back from the registry after a hallucinated-name miss) re-narrowed only via _subagent_allowed_tools. But dream's self-play temp agent contains differently — it SETS disabled_tools and pops them from available_tools, and never sets the allowlist — so a dispatch miss healed the popped tools back in, and an aliased variant ("websearch"→"web_search") could then reach the network-egress tools self-play disables. The rebuild now also drops self.disabled_tools; since dispatch requires fname in available_tools and canonicalisation only returns names already in it, a disabled tool can never fire post-rebuild. Also hardens the sub-agent path (it sets disabled_tools too). Test: test_subagent_containment.py::TestRebuild.

Containment now fails CLOSED 2026-07-22. The three-gate restriction block sat in one broad try/except that only logger.debug'd on failure — so if the registry import or the defs shape ever threw, the sub-agent ran with the full schema and dispatch dict: a containment boundary that failed OPEN. It now logger.errors and raises, landing a FAILED job via the same path as the timeout bound (JobRegistry._on_done converts the task exception to STATUS_FAILED). Deny-all was rejected because gate 1 (disabled_tools) is a denylist that requires enumerating the registry — impossible precisely when the registry can't be read — so aborting is the only guaranteed-safe state. Test: test_subagent_failclosed.py.

Rails. FORBIDDEN_TOOLS is absolute — a sub-agent can never delegate (no recursive fan-out), schedule (manage_tasks), run daemons (manage_services), mutate projects, author selfhood, or write profile memory. Turn cap MAX_SUBAGENT_TURNS (12), wall-clock timeout, and at most 4 parallel delegates (they share one upstream slot — more is slower, not faster). Its sandbox is a per-job subdir under the REAL sandbox (delegated/<job-id>/), not a temp dir like self-play: the point of delegation is to produce artifacts the main agent can then read.

3. The job registry (jobs)

core.jobs.JobRegistry is the status surface every fire-and-forget mechanism was missing. delegate and delegate_to_swarm both register into it, so the jobs tool reports them together: status (running + recently finished), collect (a finished job's full result — or every unread result), cancel. In-memory by design: jobs are per-process (their asyncio tasks die with the process), so persisting them would only resurrect zombies. Completed jobs are retained (bounded at 50, FIFO-evicted, running jobs never evicted) so a later turn can still collect a result the model never read. Tasks are spawned via spawn_bg — the codebase's one fire-and-forget primitive (strong ref + contextvar propagation + exception logging), never a bare create_task.

Collect read-marking 2026-07-22. jobs(action='collect') with no job_id (the "collect all" case) previously re-dumped every retained finished job — up to 50 × 8000 chars ≈ 400 KB — back into context on every call, and the status text steered the model to keep doing exactly that (a real context-pressure hazard). Job gained a collected flag: a no-id collect now returns only un-collected finished jobs and marks them collected via JobRegistry.mark_collected; a repeat says "no NEW results" with a by-id re-fetch hint. An explicit collect(job_id=…) still returns that job's result unconditionally (explicit re-fetch is never hidden). Test: test_jobs_collect_readmark.py.

Swarm result surfacing (2026-07-14 bug hunt). A swarm worker returns a bool while its real output lands in the scratchpad, so jobs(action='collect') on a swarm job returned "True". Job gained an optional result_resolver that maps the raw task result to the real payload; delegate_to_swarm wires it to read the scratchpad output_key, so collect now returns the content (or the failure ALERT). Test: test_bughunt_20260714.py.

Why this matters beyond convenience

It is the fix for the deepest finding of the ablation program: "recalled skills are prose never executed", and B4's mediation ≈ 0. A learned lesson injected as advisory text measurably does nothing; a learned skill that is an executable pipeline is load-bearing by construction. Data-flow turns the learning stack's output from advice into capability.
