core / llm.py

Multi-pool LLM orchestrator. Owns httpx async clients, round-robin scheduling, and per-node circuit breakers.

Six logical pools, each backed by zero-or-more (url, model) nodes plus the always-present foreground client.

Routing tasks

The RoutingTask enum at line 71 advertises the labels worker pools can fulfil:

  • VALIDATE_TOOL_ARGS — quick JSON-schema sanity check.
  • EXPAND_QUERY — sub-query decomposition for memory hydration.
  • CLASSIFY_INTENT — factual vs procedural vs contextual.
  • SCORE_RELEVANCE — re-rank candidate documents.
  • REPAIR_JSON — coerce a malformed payload back to valid JSON.
  • VERIFY — used by Verifier.

Per node, compute_tor_proxy(url, tor_proxy) decides: a node is reached via the Tor SOCKS proxy (normalised to socks5h://) only when its host is globally routable (not ip.is_global ⇒ direct). Loopback, RFC1918, link-local, IPv6 ULA and CGNAT/Tailscale (100.64.0.0/10) connect directly — the same predicate the mandatory-Tor guard uses, so a tailnet GPU node is never wrongly forced through (and unreachable via) a Tor exit.

Concurrency model

  • Fully async; asyncio.Semaphore(3) caps concurrent background routing tasks.
  • Per-node httpx AsyncClient means a single failing node cannot starve the others.
  • The breaker is process-local — restart resets state.

Streaming idle timeouts

_do_stream_chat_completion guards each SSE read with asyncio.wait_for so a genuinely hung upstream can't hold the event-loop slot forever (which keeps foreground_tasks > 0 and parks the biological watchdog). The budget is split because the first token of a turn is legitimately slow — the upstream must prefill (prompt-eval) the whole context before emitting any bytes, and on a large context (e.g. 120k tokens) or a loaded/CPU node that prefill routinely takes longer than the inter-token gap, with zero bytes during it:

ConstantEnv varDefaultApplies to
_STREAM_FIRST_BYTE_TIMEOUTGHOST_STREAM_FIRST_BYTE_TIMEOUT180sTime to the FIRST byte (prompt prefill / time-to-first-token).
_STREAM_IDLE_TIMEOUTGHOST_STREAM_IDLE_TIMEOUT60sGap between subsequent bytes (catches a real mid-stream hang).

Any received line — including SSE keepalive/blank lines — ends the first-byte wait and resets the gap timer. A timeout yields an error SSE event + [DONE]; the agent then sees a turn with thinking but no actionable output and forces a retry. The earlier flat 30s tripped that false "stall → forced retry" whenever a turn's prefill exceeded 30s, wastefully re-prefilling the whole (growing) context. The httpx client timeout (1200s) bounds the overall request above these. Tune the env vars for very slow or very fast nodes. Covered by tests/test_stream_idle_timeout.py.

Foreground-request yield (2026-06-11)

Background LLM callers (is_background=True) used to wait only on foreground_tasks — LLM calls in flight — capped at 30s. But a user turn spends much of its wall-clock between LLM calls (tools, file I/O, browser); in those gaps the counter reads 0, a background turn grabs the single llama slot, and the user's next turn queues behind a full background generation. Post-req-70 this starved the prompt for ~12 minutes while consolidation/self-play churned.

Two-signal policy in _wait_for_foreground_clear (shared by chat_completion and stream_chat_completion):

SignalMeaningBackground waits
foreground_tasks > 0a foreground LLM call is in flightup to 30s, then proceed (stale result beats none)
foreground_requests > 0a USER REQUEST is anywhere in its handle_chat lifecycleas long as it takes (600s hard ceiling against a leaked counter)

foreground_requests is bracketed at the API layer by routes._mark_foreground(agent, ±1) around both chat handlers (the streaming one spans the whole SSE generator, agent loop + final-answer streaming). The helper only touches a counter that is actually an int, so mocked test apps and partial contexts no-op instead of raising. Self-play's isolated handle_chat runs never pass the API layer, so background cycles cannot deadlock themselves. Covered by tests/test_foreground_yield.py.

Priority routing fixes — inline self-stall & off-main skip (2026-07-07)

The wait above protects exactly one resource: the main inference slot. Two classes of caller were misclassified against that rule (IMPROVEMENTS.md #2/#19):

  • Inline self-stall (fixed): three calls awaited inline by the user's own turn carried is_background=True and parked in _wait_for_foreground_clear against their own still-active request — a deterministic stall up to the 600s ceiling that read as “agent went silent” on long-output turns. Fixed to foreground: the Context Shield summarizer (fires on any oversized non-exempt tool result), the _prune_context history condenser, and the --perfect-it inline optimization (its shared helper takes foreground=; the deferred internal-learning path keeps background priority).
  • Foreground learning work (fixed): the reflection closures (_critique_fn, _verify_plan_fn) and post-mortem closures (_analyze_fn, _patch_fn) in main.py ran at foreground priority — contending with the user's next turn for the 35B slot and inflating foreground_tasks so the “is a user live?” checks misread an idle reflection as an active user. Now is_background=True; if a user turn is running, the Reflector's per-call timeout simply defers the trajectory to the idle backstop.
  • Off-main skip (new): a background call that will be served by an existing off-main pool (use_worker/use_critic/use_vision/use_swarm with a non-empty pool) skips the wait entirely — it never touches the main slot, so parking it was pure latency. If every pool node fails, _do_chat_completion may still fall back to main: a rare, already-bounded edge, same as proceeding after the 600s ceiling.

Observability: a background caller parked 120s behind an active request now logs one BG Queue Wait line, so the operator's stream shows the park instead of a silent multi-minute gap. Covered by the priority-routing section of tests/test_foreground_yield.py.

Empty / non-JSON upstream body — crash-proof decode (2026-06-23)

On the non-streaming path _do_chat_completion returned resp.json() directly. A 200 with an empty body therefore crashed as json.JSONDecodeError: Expecting value: line 1 column 1 (char 0), which the generic handler logged as “Upstream Fatal” and re-raised. Observed live right after a context overflow: while the agent's emergency-prune retry was in flight the upstream returned 0 bytes, turning a recoverable state into a hard failure with an opaque message.

The decode is now wrapped: an empty/non-JSON body is retried once (a transient empty body followed by a good body self-heals), and on a second failure it raises a clean, explanatory RuntimeError (“Upstream returned an empty/non-JSON response …”) instead of a bare decoder traceback. The handler catches ValueError (which json.JSONDecodeError subclasses) rather than naming json — conditional import json in the swarm branches above makes that name function-local and thus unbound on the no-swarm main path.

The caller side is gentler too: when the context-overflow emergency-prune retry fails, the agent now surfaces a calm, actionable message (“I hit my context limit … the task and its files are preserved … ask me to retry”) instead of a raw CRITICAL: dump. Covered by tests/test_upstream_empty_body.py.

Off-main node warmup + keepalive (2026-07-12)

Two background routines keep worker/critic nodes hot so the short route() timeout never eats a cold path: warm_up_workers() fires 3 tiny thinking-off generations per node at boot (covers model-load / first-request latency), and keepalive_workers(interval_s) (spawn_bg'd from main.py, default 45 s via GHOST_WORKER_KEEPALIVE_S, ≤0 disables) pings each node forever — covering Tailscale/WireGuard idle cool-down mid-session, which boot warmup can't.

Heartbeats log transitions, not ticks. Healthy pings are fully silent in the live stream (the per-45s keepalive → Worker Node line was pure spam); a node that stops answering logs one WARNING, and its recovery logs one line. Per-ping traffic stays at logger.debug. Keepalive also passes off_main_only=True: a failed ping must never "fall back" to the main upstream (that was a max_tokens=1 hit on the single 35B slot every 45 s for as long as a node stayed down). Covered by tests/test_worker_warmup.py.

Main-node prefix warmup (2026-07-14)

The worker warmup above covers the off-main nodes; the main node's cost is different — the FIRST request of a session prefills the ~30k-token rendered head (system slot ≈ 14 KB + native tool schemas ≈ 63 KB, rendered by the chat template immediately after the system text) at the measured ~450 tok/s ≈ 70 s on the user's critical path. Both blocks are byte-stable across conversations (all volatile continuity was moved to the tail injection precisely for cache stability), so GhostAgent.warm_up_main_prefix() (spawn_bg'd from main.py at lifespan start; opt-out GHOST_MAIN_PREFIX_WARMUP=0) sends one background max_tokens=1 request carrying that exact head. The server's prefix cache (cache_prompt + --cache-ram) then serves it, and the first real request pays only its unique tail (hydrated memory + dynamic state + query ≈ 20-25 s).

Byte-exactness is the contract: the warmup builds its system slot and tools through the SAME code paths a live request uses (_RequestState.get_profile_str() → SYSTEM_PROMPT splice + the perfect_it mutation when configured; get_active_tool_defs for the tools) so it cannot drift. The acquired-skill tail of the tool list is query-routed and may differ per conversation — that only shortens the cache match (schemas render in list order, static tools first), never breaks it. Main-slot etiquette: is_background=True with no off-main flags targets main but waits for any live foreground request, so a user who beats the warmup is never queued behind it; failures are debug-logged and never escape. Covered by tests/test_main_prefix_warmup.py.

Non-JSON-body retry strips a leftover stream flag (2026-07-18)

A 200 response whose large body starts with data: is SSE — the caller passed a payload still carrying the turn loop's stream: true into the NON-streaming chat_completion (observed: the context-overflow recovery got 102 KB of SSE frames, read as "empty/non-JSON" → fatal). The retry now forces stream=False and says so in the log line; harmless when the body was genuinely empty.

LLM-boundary recording & stub-replay (2026-07-17)

core/llm_recording.py. With GHOST_LLM_RECORD=1 every chat_completion result (foreground + background branches) and every route() verdict appends one JSONL line — request id, per-process ordinal, exact payload, response, routing flags — under $GHOST_HOME/system/llm_recordings/<day>.jsonl. OFF by default, deliberately: payloads carry the rendered system prompt (memory + user profile) verbatim, bypassing the corpus redaction contract — enable for a debugging session, harvest fixtures, turn it off. ReplayLLMClient serves a recording back in order (fingerprint of messages verified; mismatches flagged on .mismatches, exhaustion raises) — deterministic stub-replay for harness tests and forensics. Live same-seed re-generation was REJECTED as a goal: llama.cpp on Metal with prefix-cache reuse is not byte-stable run-to-run. Tests: tests/test_llm_recording.py.
