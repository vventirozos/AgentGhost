core / llm.py

Multi-pool LLM orchestrator. Owns httpx async clients, round-robin scheduling, and per-node circuit breakers.

Routing tasks

The RoutingTask enum at line 71 advertises the labels worker pools can fulfil:

  • VALIDATE_TOOL_ARGS — quick JSON-schema sanity check.
  • EXPAND_QUERY — sub-query decomposition for memory hydration.
  • CLASSIFY_INTENT — factual vs procedural vs contextual.
  • SCORE_RELEVANCE — re-rank candidate documents.
  • REPAIR_JSON — coerce a malformed payload back to valid JSON.
  • VERIFY — used by Verifier.

Concurrency model

  • Fully async; asyncio.Semaphore(3) caps concurrent background routing tasks.
  • Per-node httpx AsyncClient means a single failing node cannot starve the others.
  • The breaker is process-local — restart resets state.

Streaming idle timeouts

_do_stream_chat_completion guards each SSE read with asyncio.wait_for so a genuinely hung upstream can't hold the event-loop slot forever (which keeps foreground_tasks > 0 and parks the biological watchdog). The budget is split because the first token of a turn is legitimately slow — the upstream must prefill (prompt-eval) the whole context before emitting any bytes, and on a large context (e.g. 120k tokens) or a loaded/CPU node that prefill routinely takes longer than the inter-token gap, with zero bytes during it:

ConstantEnv varDefaultApplies to
_STREAM_FIRST_BYTE_TIMEOUTGHOST_STREAM_FIRST_BYTE_TIMEOUT180sTime to the FIRST byte (prompt prefill / time-to-first-token).
_STREAM_IDLE_TIMEOUTGHOST_STREAM_IDLE_TIMEOUT60sGap between subsequent bytes (catches a real mid-stream hang).

Any received line — including SSE keepalive/blank lines — ends the first-byte wait and resets the gap timer. A timeout yields an error SSE event + [DONE]; the agent then sees a turn with thinking but no actionable output and forces a retry. The earlier flat 30s tripped that false "stall → forced retry" whenever a turn's prefill exceeded 30s, wastefully re-prefilling the whole (growing) context. The httpx client timeout (1200s) bounds the overall request above these. Tune the env vars for very slow or very fast nodes. Covered by tests/test_stream_idle_timeout.py.

Foreground-request yield (2026-06-11)

Background LLM callers (is_background=True) used to wait only on foreground_tasks — LLM calls in flight — capped at 30s. But a user turn spends much of its wall-clock between LLM calls (tools, file I/O, browser); in those gaps the counter reads 0, a background turn grabs the single llama slot, and the user's next turn queues behind a full background generation. Post-req-70 this starved the prompt for ~12 minutes while consolidation/self-play churned.

Two-signal policy in _wait_for_foreground_clear (shared by chat_completion and stream_chat_completion):

SignalMeaningBackground waits
foreground_tasks > 0a foreground LLM call is in flightup to 30s, then proceed (stale result beats none)
foreground_requests > 0a USER REQUEST is anywhere in its handle_chat lifecycleas long as it takes (600s hard ceiling against a leaked counter)

foreground_requests is bracketed at the API layer by routes._mark_foreground(agent, ±1) around both chat handlers (the streaming one spans the whole SSE generator, agent loop + final-answer streaming). The helper only touches a counter that is actually an int, so mocked test apps and partial contexts no-op instead of raising. Self-play's isolated handle_chat runs never pass the API layer, so background cycles cannot deadlock themselves. Covered by tests/test_foreground_yield.py.
