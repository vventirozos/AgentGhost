core / dream.py

Idle-time consolidation, synthetic challenge generation, and the full self-play orchestration pipeline.

Responsibilities

  • Mine failed tasks from the journal into standalone, self-contained challenges via journal_challenges.
  • Validate LLM-generated challenges before running them — quality gate, preflight, and validator self-test gate.
  • Detect cross-episode tool patterns (advisory lessons) and mine recurring tool-call sequences from the trajectory log into proposed composed-skill macros for the user to approve.
  • Orchestrate the self-play simulation loop: spawn an isolated temp agent, run the challenge, score with correctness_weighted_score, verify and persist any new lesson via SkillMemory.

Validation stack (three gates, in order)

1. validate_challenge_quality(setup, validator) → (bool, reason)

Pattern-based quality gate (line 102). Rejects:

  • Validators that call random.seed, random.randint, random.uniform, random.choice or np.random — data-generation markers that would make scoring non-deterministic.
  • Validators that depend on dynamic discovery (os.listdir, glob.glob, pathlib.Path) when the setup writes files with explicit names — they must reference at least one shared filename.
  • SQL setup scripts where CREATE TABLE column count doesn't match INSERT value count, or CSV headers that don't match row field count.
  • The unwinnable split pattern: .strip().split('\n') combined with randomness and a len(act) != len(exp) check.
  • Datetime import-style misuse (2026-07-18, _datetime_misuse): AST lint on BOTH scripts for the crash family that killed two overnight cycles — from datetime import datetime followed by datetime.timedelta(...) (the name is the CLASS), and datetime.datetime.<module-attr> spellings. Deterministic runtime AttributeErrors the syntax check can't see; the reject reason spells out the two valid import styles for the regen loop. Generation prompt rule 13 states the same discipline up front.
  • Near-duplicate theme (2026-07-18, diversity guard): the candidate challenge is compared against the frontier tracker's window of recently generated challenge heads (token-set containment + a shared-mock-filename bonus); ≥ 0.60 rejects with the offending prior challenge quoted. The generator mode-collapses onto a pet theme when left alone — 4 of 6 challenges in the 2026-07-17 overnight run were the same transaction_log.csv fraud scan, reworded. Recent heads are also fed FORWARD into the generation prompt as explicit negative examples, so the first attempt usually avoids the reject.

Rejection kinds are returned so the regeneration loop can route targeted feedback (files-mismatch → targeted "open file X" hint; data-gen → "don't call random.seed" hint; etc.). Up to 3 generation attempts; a files-mismatch rejection triggers the validator repair path which regenerates only the validator with a focused prompt.

2. Preflight (.preflight.py)

Runs the validator in the sandbox with __name__ == "__dry_run__" (bypasses any if __name__ == "__main__" guard). Catches module-scope NameError / ImportError / ModuleNotFoundError / AttributeError (the last added 2026-07-18: attribute-resolution bugs are deterministic — they can't depend on solution.py existing — and swallowing them let a broken validator reach score time) — unrunnable validators that used to burn a whole solver attempt.

3. Validator self-test gate (.validator_selftest.py)

Catches internal-contradiction bugs like the canonical float('60.00%') trap, where a validator formats an expected field with a unit suffix and then calls float() on it. Flow:

  1. _instrument_validator_for_self_test(src) — AST-find the first subprocess.run(...solution.py...) statement and prepend a probe that dumps the first-resolved expected_* variable between <<<__GHOST_SELFTEST_EXPECTED_START__>>> / <<<__GHOST_SELFTEST_EXPECTED_END__>>> sentinels, then raise SystemExit(42).
  2. _extract_selftest_dump(stdout) — pull the dumped block out of the probe's stdout.
  3. Write a solution.py that sys.stdout.write(<dumped>); restore mocks from the post-setup snapshot.
  4. Run the original validator. _looks_like_validator_crash(out) returns True iff the traceback's innermost frame is .validator.py — in which case the challenge is rejected as unwinnable.

Candidate variable names scanned (in order): expected_output, expected_lines, expected, expected_text, expected_str, expected_result, golden_output, golden, correct_output, answer. The gate is best-effort: unparseable validators or those without a matching subprocess.run skip cleanly.

4. Reference-solution consistency gate (2026-07-08)

The echo self-test (gate 3) only catches validators that crash on their own expected output. It cannot catch the unwinnable shape observed live: a validator whose hardcoded expectations disagree with the data the setup script actually wrote (validator expected duration=10 while tasks.json yields 25) — the probe echoes the expected output back, so the validator happily passes it, and the solver then fails 3/3 attempts on correct code, records a bogus -1.0 frontier delta on the cluster, and learns a misleading "you skimmed a constraint" lesson.

LLM-generated challenges now must emit a fourth XML block, <reference_solution> — a runnable solution.py that computes the answer from the setup files at runtime. Two checks enforce it:

  1. Static (validate_reference_solution, gen loop): the reference must open at least one file the setup script creates (same literal/dynamic-path logic as gate 1) — a reference that prints hardcoded values would agree with an equally-hardcoded validator, making the sandbox check vacuous. Rejection feeds the regeneration loop.
  2. Sandbox (after the M5 snapshot restore, before the solver starts): write the reference as solution.py, run it, then run .validator.py against the real setup data. Any non-zero exit (reference crash or validator rejection) discards the challenge as internally inconsistent; the snapshot is restored and solution.py unlinked either way, so the solver always starts from a clean sandbox.

Deterministic templates and journal-mined challenges are pre-verified shapes and skip this gate; an LLM generation that omits the block is accepted with a logged warning (soft requirement — the gate is skipped, not the challenge).

Tests: tests/test_selfplay_reference_gate.py (static gate semantics + wiring pins).

Runtime crash detector (widened)

Even past the three gates, a validator can raise during comparison. The attempt-loop circuit breaker classifies a traceback as a validator crash when the tail frame is .validator.py, solution.py is absent from the feedback, and the exception type is one of:

  • Structural: SyntaxError, IndentationError, ImportError, ModuleNotFoundError, NameError
  • Internal-contradiction: ValueError, TypeError, KeyError, IndexError, AttributeError

Detection aborts the cycle after attempt 1 instead of burning all 3 on the same broken validator.

Infra abort, not agent failure (2026-07-18): a detected validator crash now concludes the run as INFRA_ABORT instead of FAILURE. The frontier record_run is skipped (no −1.0 compression delta lands on the cluster), the correctness-weighted score is not computed, the skill gate suppresses any lesson, and the adversarial-generator tracker does not record the fingerprint (a crash-prone prompt would otherwise read as a "hard challenge" and get reinforced). Observed 2026-07-18 04:50: the agent's solution.py exited 0, the validator died on a module-scope datetime AttributeError, and the run was recorded as a real failure with data_analysis delta −1.0 — generator noise wearing an agent label. Test: tests/test_dream_synthetic.py::test_synthetic_self_play_validator_crash_is_infra_abort.

Lesson pipeline

_extract_structured_lesson

Meta-cognitive LLM call that returns {trigger, anti_pattern, correct_pattern, domains, confidence, task/mistake/solution (legacy mirrors)}. The prompt explicitly requires task-class triggers, forbids copying fixture literals, and mandates a non-empty taxonomy domain set.

_generalization_guard

Last line of defence against overfit lessons. Uses n-gram token overlap (_GENERALIZATION_MIN_NGRAM = 6) to reject lessons whose:

  • trigger copies a 6-token run from the challenge text
  • correct_pattern copies a 6-token run from setup_script or the validator
  • domains is empty or contains nothing from _VALID_LESSON_DOMAINS ({data_analysis, regex_parse, sql, concurrency, algo, bash, python_general})
  • trigger or correct_pattern is empty

_verify_lesson_helpful

For struggled-then-won and failure cases, re-runs the solver once with the lesson prepended under the production ### SKILL PLAYBOOK: header. Keeps only if the outcome strictly improves.

REM-cycle heuristics — per-content keying (2026-07-07)

Consolidation heuristics from the dream JSON are written with task = <first 80 chars of the heuristic> and source="dream". Previously every heuristic used the constant task="[System] Dream Heuristic"; since SkillMemory dedups on the normalized trigger, all heuristics from all REM cycles collapsed into a single playbook slot — a new one overwrote the previous (if longer) or was dropped, its frequency inflated toward bogus graduation, and its constant trigger matched no user query. Per-content keying makes each a first-class lesson governed by the normal dedup / cap / utility machinery, and the source="dream" tag lets the B3 idle-loop ablation count dream's real contribution by provenance.

REM-cycle heuristics — actionability gate (2026-07-13)

Observed live: dreaming over trajectory digests stored observations as skills — actor profiles ("The agent exhibits a tendency to engage in inappropriate requests" — actually the operator's boundary-test prompts misattributed to the agent; "The user has a strong preference for role-playing scenarios involving coaching a beginner chess player (Vasilis)" — the operator profiled as a persona) and operational trivia ("service management commands are frequently used for chess-v4"). All landed in SkillMemory as mistake="none" pseudo-lessons that match no real query. Two layers now prevent this:

  • Prompt contract — the REM prompt demands imperative rules only (verb-first, or condition + verb), explicitly forbids observation/summary/profile sentences shaped like "The agent… / The user… / The system…", and states that raw memories quote the operator's messages, which must never be attributed to the agent nor turned into rules.
  • Deterministic gate — _is_actionable_heuristic() default-REJECTS anything that doesn't read as a rule, because prompt instructions alone don't hold against a small worker model. Accepts: imperative starters (Always… / Use… / Verify… / Don't…) or conditional starters (When/If…) followed by an imperative/modal. Rejects: subject-opener observations (The agent/user/system…, There is/are…, Requests…), non-strings, and texts outside 12–600 chars. Dropped heuristics are logged as Dream Skip and counted in the completion message ("N non-actionable heuristics dropped"); the "extracted N heuristics" count now reports only what actually reached the playbook. Moved 2026-07-16 to the shared leaf module memory/lesson_quality.py and also enforced at the learn_lesson write chokepoint, so self-play and every other producer are gated too (see memory / skills).

Dream heuristics are a bonus learning channel (the reflector and the self-play lesson pipeline carry the primary signal), so the default-reject posture costs little; a false accept pollutes retrieval until utility pruning catches it. Tests: tests/test_dream_heuristic_gate.py.

RRF-weight refit step (2026-07-14)

Dreamer._refit_rrf_weights(min_observations=30) runs after graph compression: it reads the usefulness-observations ledger (rrf/observations.jsonl, written by the bus's post-turn hydration judge — see bus), fits rrf_weights.fit_intent_weights anchored on the bus's current effective matrix (thin cells keep their base weight; everything clamped to [0.1, 3.0]), persists rrf/weights.json for the next boot, hot-swaps bus._intent_weights in-process, and trims the ledger past 5000 lines. Below the observation floor it's a no-op — the hand-tuned defaults stay untouched until real usage data accumulates.

Provenance on abstractions (2026-07-14)

Dream's outputs now carry drill-down evidence. Syntheses: the consolidation writer stamps {"type": "synthesis", "provenance": "[{\"id\", \"excerpt\"}, ...]"} — captured before the merged source fragments are deleted, so the excerpt is the only surviving evidence a synthesis can be checked against. Episode strategies: lessons from the episodic pass carry source_refs=["ep:<id>", ...] (resolvable via EpisodicMemory.get_episode). tool_recall renders both as EVIDENCE lines on matching hits. Tests: tests/test_memory_provenance.py.

Episodic consolidation pass (2026-07-14)

Dreamer._consolidate_episodes(model_name, min_episodes=3, max_episodes=40) runs right after the journal drain and before the REM entropy gate, so a thin auto-fragment pool cannot starve it (the two early-return paths append its result to their message). It drains EpisodicMemory.get_unconsolidated — episodes are the trajectory-shaped substrate the B3 postmortem said dream needs: trigger → tool-call chain (with per-action FAILED markers) → outcome → lesson. One batched worker call (temperature 0, JSON {"strategies": [...]}) generalizes across episodes into imperative strategies; each must pass _is_actionable_heuristic before learn_lesson(..., source="episode"). Failure contract (mirrors the smart-memory requeue fix): the batch is marked consolidated only after a successful worker parse — a transient failure leaves it queued for the next cycle; a successful parse marks the batch even when zero strategies survive the gate, so the same rows don't re-process forever. Short-circuits below min_episodes, so most cycles cost zero LLM calls. Reported as "(N episode strategies learned)". Tests: tests/test_dream_episode_consolidation.py.

Graph compression step (2026-07-14)

After the stale-edge prune, the REM cycle runs Dreamer._compress_graph_nodes(model_name, max_merges=8) — the live caller execute_graph_compression was hardened for on 2026-07-07 but never had. Candidates come from the read-only, deterministic GraphMemory.propose_merge_candidates: "safe" pairs (punctuation/whitespace variants of one name) are merged directly; "fuzzy" pairs (plurals/typos by string similarity) are merged only when a single batched worker call confirms the names denote the same entity (temperature 0, JSON {"same_entity": [...]} — when unsure, exclude). A worker failure skips the fuzzy pairs but keeps the safe merges; the whole step is best-effort and never fails a dream; merges are capped per cycle so one bad batch can't rewrite the graph. Self-play isolation is unaffected — the ReadOnly graph wrapper no-ops execute_graph_compression. Reported in the Dream-Complete line as "(N duplicate graph nodes merged)". Tests: tests/test_dream_graph_compression_wiring.py.

Trajectory-seeded REM (2026-07-09)

The REM entropy gate counted only type:"auto" vector fragments, which nothing organically feeds: the smart-memory consolidator only stores facts scoring ≥0.9, which neither fact-chat seeding (B3) nor task-shaped seeding (B4) ever produced — 0 auto-memories across 12 instrumented arm-runs. trajectory_dream_fragments(context) digests the newest trajectories (task, outcome, tool names, first error, failure reason — one line each, ids as traj:<id>) and dream() falls back to them when the auto pool has <3 fragments; the biological-watchdog eligibility gate mirrors the same fallback (otherwise it would be dead code). In trajectory mode the merge/delete consolidation pass is disabled — traj: ids are not vector-store ids and must never reach collection.delete; the path's whole value is the heuristics harvest (lessons with source="dream"). The fragment-set idempotency guard works unchanged on trajectory ids. Tests: tests/test_dream_trajectory_seeds.py.

Isolation markers

The isolated sub-context sets several attributes to prevent production-state writes:

  • ReadOnlySkillMemory.is_read_only = True — class marker that agent.py checks (via is True) at two points: (a) to skip the entire ~15 s Perfect-It follow-up LLM call during self-play, and (b) to short-circuit the confirmation turn that the solver would otherwise spend re-deriving that solution.py just ran clean. When the solver's last execute tool call exits 0 on solution.py with non-empty stdout, the turn loop sets force_stop = True and synthesises a minimal final message, skipping the ~15–25 s "task complete" thinking turn. The outer validator re-runs solution.py directly and never reads the agent's reasoning, so the confirmation turn was pure dead time.
  • isolated_context.args.perfect_it = False, smart_memory = 0.0, native_tools = True.
  • isolated_context.workspace_model = None (2026-07-09) — the shallow copy kept the real WorkspaceModel, so the temp agent's turns set the process-global event-stamp pointer to "" mid-flight (the temp agent has its OWN semaphore, so this raced a concurrent real turn's stamping) and recorded synthetic self-play command/browser outcomes into the real activity log. Every record/prefix site guards on workspace_model is None, so None is the supported "no workspace" state.
  • selfplay_loop_task / selfplay_loop_stop / selfplay_loop_started_at are stripped — otherwise the inner sub-agent's handle_chat would trip the outer loop's user-message interrupt hook.
  • verifier, uncertainty_tracker, mcts_reasoner, hypothesis_tester, frontier_tracker set to None.

Rejection prompt contract (retry attempts)

When an attempt fails validation, the retry prompt injected for the next attempt contains the validator's feedback string (the FAIL line with expected-vs-actual output) plus optional float-formatting hints. It does not contain the .validator.py source. An earlier revision pasted the full validator script into the retry prompt so the agent could "debug the validator's logic", but this turned every struggled-then-won cycle into an answer-key lookup — the agent copied the validator's constants (multipliers, SQL query shape) instead of reasoning from the expected-vs-actual diff. Skill-gate lessons from those cycles were memorised constants, not transferable knowledge.

Retry prompts now force the solver to reason from the diff and the original task description. Some complex challenges will fail their second attempt that previously "succeeded" via copying — that is the intended behaviour: a genuine failure is better training signal than a cheated pass.

Concurrency

Async orchestration; the temp agent loop runs synchronously inside the simulation. Triggered by the lifespan-spawned biological watchdog, by the self_play tool (one-shot), or by self_play_loop (continuous). Validator probe / self-test / solver run on the sandbox via asyncio.to_thread.

Frontier-aware cluster selection

Before the source-selection stage, synthetic_self_play chooses which cluster to target. The default path calls FrontierTracker.pick_seed (brittle-pool weighted). When --frontier-selfplay is on (default) AND both ctx.prm_scorer is a real PRMScorer with has_model=True AND ctx.trajectory_collector is a real TrajectoryCollector (strict isinstance checks — MagicMock-backed test contexts fail closed), the dream loop instead:

  1. Builds the candidate cluster pool as set(challenge_templates.TEMPLATES.keys()) ∪ tracker.clusters.
  2. Computes per-cluster signals via core/frontier_selection.py: compute_cluster_uncertainty (PRM boundary-distance) and compute_cluster_rarity (log-decay of Trajectory.cluster counts).
  3. Calls frontier_tracker.pick_frontier_seed(uncertainty_by_cluster=…, rarity_by_cluster=…, uniform_sample_prob=args.frontier_uniform_sample_prob).

Any exception in the frontier-aware block is logged at debug and falls through to pick_seed — frontier weighting must never block a self-play cycle. The selected seed's hint is appended to the challenge-generation prompt under ### FRONTIER SEED regardless of which picker produced it. The new path's hint begins with FRONTIER TARGET (PRM-weighted) so logs and tests can attribute the source.

Covered by tests/test_dream_frontier_weighted.py (real PRM + collector + tracker, mocked LLM/sandbox) and tests/test_dream_synthetic_curiosity.py (legacy path regression).

Counterfactual replay — regression-of-learning, phase 1 (2026-07-17)

core/counterfactual.py. Every concluded self-play sim now persists its challenge spec (prompt + setup + validator + outcome) to $GHOST_HOME/system/counterfactual/challenges.jsonl; ~1 idle self-play slot in 4 (when a backlog exists) REPLAYS a past challenge against the CURRENT lessons via the new synthetic_self_play(injected_challenge=…) seam — same isolated sandbox, same validator, generation paths skipped; the only variable is what the agent has learned since. Verdicts: past FAILURE→SUCCESS = generalized (the learning works, measurably); past SUCCESS→FAILURE = regression — the lessons hydrated into the failing replay are QUARANTINED (kept on disk with reason, excluded from prompts — never auto-deleted; see skills) and the operator is notified at notify-severity. Replays are not re-persisted (no ledger compounding); results land in results.jsonl so each challenge replays once. Phase-1 scope is deliberately validator-backed tasks only — user-turn counterfactuals need pre-turn workspace snapshots, and verifier-scored replays would inherit the verifier's own failure modes. Attribution substrate: every chat trajectory now stamps extra.hydrated_lessons (the playbook triggers actually injected that turn). Tests: tests/test_counterfactual.py.
