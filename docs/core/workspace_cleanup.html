core / workspace_cleanup.py

Mark-and-sweep of a finished project's scratch space. When a project completes, registered deliverables are kept and everything else under its workspace is deleted.

Why

While a project is in flight the agent litters its workspace (<sandbox>/projects/<id>/) with transient files — screenshots, helper scripts, downloaded fixtures, __pycache__, temp output. Historically these survived completion and accumulated. This module removes them automatically the moment a project reaches DONE, leaving only the files the user actually wanted.

The contract

When deliverables are registered, registered file artifacts are kept and everything else is deleted. Anything that should survive cleanup should be registered as a deliverable — either explicitly (the deliverables=[...] argument of the manage_projects task_update action, or ProjectStore.register_file_artifact) or automatically (the research subsystem registers each brief).

Empty-keep-set recovery (2026-06-20). Registration depends on the agent remembering to do it, and it doesn't always: a one-turn single-file build wrote index.html, marked its task DONE without deliverables=[...], and the old "nothing registered ⇒ everything is scratch" rule wiped the project's only file (observed live — the deliverable was gone before the verifier even loaded it). So an empty keep-set is now read as missing registration, not as permission to delete the workspace. The sweep recovers the deliverables (_recover_deliverables): it keeps — and back-registers — every non-debris, non-symlink file, and deletes only categorical debris (_is_debris: bytecode caches, .browser_profile/.browser_runner.py scaffolding, swap/backup files, dotfiles, temp_/tmp_/scratch_/debug_ names). Recovery is deliberately conservative — an image or stray file is kept rather than risk-deleted — because losing a real deliverable is far worse than leaving a stray scratch file. A workspace holding nothing but debris still nets an empty keep-set, so a lone __pycache__ is still swept. The summary gains a recovered list, and an operator WARNING fires so the missing registration is visible.

API

FunctionPurpose
sweep_project_workspace(store, project_id, *, dry_run=False) -> dictDelete every file under the project workspace not registered as a file artifact, then prune the directories left empty. Returns {project_id, status, deleted, kept, dirs_removed, freed_bytes, kept_referenced}. Never raises. dry_run=True reports what it would delete without touching disk.

Safety guards

  • Scoped — only ever walks <sandbox_root>/projects/<id>/; the project dir itself is preserved so a resume still has a workspace.
  • No symlink escape — the walk never follows symlinks; a symlink is removed as scratch but its target outside the tree is untouched.
  • Fail-safe — if the keep-set (the artifact list) can't be read, the sweep deletes nothing and returns status="skipped: artifact read failed".
  • Non-fatal — wrapped end to end; a cleanup failure can never strand a project mid-transition.
  • DONE only — never runs on FAILED / NEEDS_USER, which stay resumable (a revised task can roll them forward to DONE later), so their files survive.

Trigger wiring

The sweep is hung off ProjectStore.on_project_done, a hook fired by _fire_project_done exactly on the transition to DONE — from both _maybe_rollup_project_status (autonomous and task-driven completion both funnel through here) and a manual update_project(status="DONE"). main.py wires the hook to sweep_project_workspace when it constructs the store. Because every completion path ends at store.update_task → rollup, one store-level hook covers them all.

Partial keep-set recovery (2026-07-02)

Observed live (the chess-game build): the agent built four files but registered only one (js/game-state.js) as a deliverable. The old rule — "keep registered, delete everything else" — took that sparse keep-set at face value and deleted index.html and two of the three JS modules at the moment the project completed. The empty-keep-set recovery never fired because the keep-set wasn't empty; one registered file made the sweep destructive. (The build was reconstructed byte-faithfully by replaying the trajectory JSONL's recorded file_system writes/replaces — the trajectory log stores full content / replace_with payloads plus result text with anchored line ranges.)

The contract now treats registration as evidence of what IS a deliverable, never proof of what isn't:

  • Registered file artifacts — always kept (unchanged).
  • Empty keep-set — unchanged: _recover_deliverables keeps + registers every non-debris file, including ambiguous media.
  • Non-empty (partial) keep-set — new: _recover_unregistered_sources keeps + registers every unregistered source/document file (_is_source_like: code, markup, docs, config, text data — .html .js .py .md .json … plus Makefile/README-style names). Unregistered media/binary files (screenshots, renders) stay deletable — they are the scratch the sweep exists to remove; a media deliverable is protected by registering it. The debris classifier outranks the source classifier (temp_*.py, *.log, dotfiles still go). Recovery registers what it keeps, so the record becomes truthful and the next sweep needs no recovery pass. Operator-visible as a ⚠️ Cleanup … keep-set was PARTIAL warning; summary["recovered"] lists the saved files.

Rationale for source-vs-media asymmetry: losing a stray kept file is cheap; losing the build is catastrophic and non-obvious (it happens at the success moment, when nobody is watching for damage).

Recurring tidy — post-completion debris (2026-07-18)

The DONE sweep fires exactly once, on the transition — but verification and post-completion debugging keep producing screenshots and scaffolding after it. Live case: the game project rolled DONE at 21:41 and had six unswept screenshots by the next morning; the operator was deleting them by hand. tidy_project_workspace(store, project_id, min_age_hours=24, dry_run=False) is the recurring counterpart — safe on ANY project status, repeatedly. It deletes only: categorical debris (_is_debris) and unregistered media files (_MEDIA_SUFFIXES) that are older than the age gate, not in the keep-set, AND not referenced by any source file (_referenced_media — basename scan of source-like files ≤512 KB, so a sprite sheet an index.html points at is treated as an asset, never a screenshot). Source/document files are NEVER deleted here regardless of registration — on a non-terminal project today's unregistered helper script may be tomorrow's deliverable; judging unregistered scratch scripts remains the DONE sweep's job. One workspace_tidy project event records what was removed.

Triggers: biological-watchdog phase 2.7d (idle, every _WORKSPACE_TIDY_COOLDOWN = 6 h, walks all projects, 24 h age gate) and the explicit manage_projects action=cleanup (no age gate — the user asked now). Note the tidy walks only projects/<id>/ trees: stray directories at the sandbox ROOT (repo clones, ad-hoc analysis dirs) are the free-chat workspace and deliberately stay untouched.

Containment + never-debris guards (2026-07-20)

  • Path-escape containment (sweep AND tidy). Both walks now assert the resolved root is strictly inside <sandbox>/projects/ before touching anything (_escapes_projects_root): a project id bearing /, \ or .. — which made _project_dir resolve OUTSIDE the sandbox, so the walk would have deleted the parent tree — deletes nothing and returns status="skipped: path escape". Defense in depth: the tool boundary validates ids too, and any doubt (including an unexpected exception in the check) reads as an escape.
  • .git/ is never cleanup's to delete. _in_git_tree makes both the recurring tidy and the DONE sweep skip the whole subtree — files and legitimately-empty dirs (refs/tags, …) alike — because tidying an in-flight clone corrupts the repo. .git stays in _SCRATCH_DIRS only so the recovery passes don't register its internals as deliverables; the guard is what keeps the files on disk.
  • Config-dotfile variants are not debris. The kept-dotfile set now matches format variants too (_is_kept_dotfile: .eslintrc.json, .babelrc.js, .env.local) — real projects spell them that way, and the exact-match set classified them debris and deleted them.
  • The tidy's age gate covers ALL debris. Categorical debris used to be deleted at any age; on an ACTIVE project a fresh debris-shaped file may be an in-flight tool's live state (a dotfile being written, a log mid-append), so debris now gets the same min_age_hours grace period media always had. The explicit manage_projects action=cleanup still runs with no age gate — the user asked now.
  • _normalize_rel strips workspace/. Artifact payloads are reduced to the exact project-relative key the walk produces — now including a leading /workspace/ / workspace/ segment, and register_file_artifact stores the same normalized form (see memory.projects). An absolute /workspace/projects/<id>/x.png deliverable previously matched no walked rel and was deleted by the very sweep it was registered to survive.
  • The DONE sweep protects referenced media too. The tidy's _referenced_media scan now also runs in the sweep: an unregistered media file that a kept source file points at (sprite sheet, texture, favicon) is an asset of the build, not a stray screenshot — deleting it broke the deliverable the sweep had just protected. The summary gains a kept_referenced list; symlinks stay deletable.

Covered by tests/test_workspace_cleanup.py (path-escape id, git-tree, referenced-media, _normalize_rel) and tests/test_workspace_tidy.py (age-gate-on-debris, config-dotfile variants, path-escape id).
