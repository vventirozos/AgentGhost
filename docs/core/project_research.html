core / project_research.py

Auto-research for long-term projects (any domain, not just code): search → summarise → persist a Markdown brief in the project workspace → index it → surface it in the briefing.

Why

Projects are multi-session efforts of any kind. While working one — on demand or during background auto-advance — the agent should be able to research relevant topics, keep the findings (not just raw search dumps), and stay aware of what it has already looked into. This module turns a web search into a durable, summarised brief inside the project's own workspace_dir, and records it so every later turn sees it.

Entry points

FunctionEffect
async research_topic(context, project_id, topic, *, tool_runner=None, llm_summarizer=None, queries=None, task_id=None)Research ONE topic: run web search(es), summarise, persist the brief. Falls back to the tool registry for search and to context.llm_client for summarisation; with neither it still writes a (sourceless) brief so the topic is recorded.
async research_project(context, project_id, *, topics=None, max_topics=5, ...)Research several topics. When topics is empty, auto-derives them via propose_topics.
async propose_topics(context, project_id, n=5, ...)LLM-derive up to n distinct topics from the project goal + open tasks. Fallback when no LLM: the open task descriptions, then the goal.
async persist_research_from_output(context, project_id, topic, search_output, *, task_id=None)Turn a search output already in hand into a brief — used by the autoadvancer so it doesn't search twice.
get_research_index(store, project_id) -> list[dict]The project's research index (read by the briefing).

What a run produces

  • A Markdown brief at <workspace>/research/<slug>.md — title, ## Summary + ## Key findings (LLM-written, or a heuristic title list when no LLM), a ## Sources section parsed from the search tool's [Source: url] markers, and a timestamp. The slug is alphanumerics + dashes only, so a topic can never escape the research dir.
  • A research index entry in project metadata (metadata.research_index): {topic, slug, path, ts, summary_preview, sources_count}, replace-by-slug (re-researching refreshes rather than duplicates), capped at 50 entries.
  • A research/INDEX.md human-readable mirror of the index.
  • A file artifact (when tied to a task) + a research_added event on the project log.

Triggers

  • On demand — manage_projects action=research (a single topic, or omit it to auto-derive several from the goal) and action=research_list.
  • Automatic during auto-advance — a research-classified task in advance_once that ran a real web_search and got usable output calls persist_research_from_output on that output (no second search), so background ticks leave durable findings, not just a transient tool_call artifact. The synthesised summary becomes the task's result_summary; the brief path is added to the autoadvance_step event and the result summary.

Awareness

build_project_briefing reads the metadata index (no file I/O) and appends a RESEARCH NOTES section listing recent topics → file paths with a one-line preview, so the agent sees what it has already researched on every turn it works the project — and is told to read a brief with file_system before re-researching it. The manage_projects create/switch/resume briefing dict carries a compact research list too.

Robustness

Every path degrades and never raises into the caller: missing LLM → heuristic summary (the top result titles); missing search runner → sourceless brief; missing workspace → a clean ok=False result. The autoadvance hook is wrapped so a research failure can't break a tick.

The summariser and topic-proposer issue a no-think utility completion (_llm_complete): a plain foreground chat_completion with thinking disabled — the /no_think soft-switch + chat_template_kwargs:{enable_thinking:false} hard-switch + a system nudge, mirroring core.dream. This matters because the upstream runs as a reasoning model under --deep-reason: a utility call with thinking left ON burns the entire token budget inside <think> and returns an empty content (verified live: finish_reason=length, all completion tokens spent on reasoning_content), which silently dropped the summary to the title-list fallback. Responses are still defensively cleaned — _strip_think removes any stray <think> block and _message_text salvages reasoning_content if content is empty. The index summary_preview skips Markdown headings so it reads as a real sentence. Covered by tests/test_project_research.py and tests/test_project_advancer.py.
