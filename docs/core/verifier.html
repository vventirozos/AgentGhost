core / verifier.py — Verifier

Reflective self-evaluation: a separate LLM call (worker pool when possible) confirms / refutes the agent's conclusion before it reaches the user.

Routing

Tries llm_client.route(task="VERIFY", max_tokens=2048, ...) against the worker pool first — both cheaper and a different "perspective" than the foreground model. Falls back to a direct foreground call if the worker pool is empty or its circuit is open. When a dedicated critic pool is configured (--critic-nodes) it takes precedence over both, with its own budget (GHOST_CRITIC_CALL_TIMEOUT, default 120 s).

Verify-sized worker budget (2026-07-16). The worker-route call used to inherit route()'s default _ROUTE_TIMEOUT_S (12 s — sized for sub-second routing chores). A verdict takes 7–11 s uncontended on the live worker (Gemma 4 E4B), so any contention on the node pushed it past the ceiling: Nova: ReadTimeout → no gate verdict → a hallucinated answer shipped unchecked (req 738c/35, the "Everest pizza" turn), caught only by the late async verdict a full turn later. _call_llm now passes its own budget — _VERIFY_WORKER_TIMEOUT_S, default 45 s, env GHOST_VERIFY_WORKER_TIMEOUT — through route(timeout=…). The loop-exit repair budget (25 s) and the late-verdict handler already tolerate a verdict that lands after the reply, so the longer ceiling costs the user nothing; a genuinely sick node still fails bounded. Covered by tests/test_verify_worker_timeout.py.

JSON parse robustness

The verifier strips any <think>…</think> blocks the worker model emits, then attempts: direct parse → reversed greedy match → final greedy match. Returns None if no parse succeeds — callers treat None as "skip verification" rather than "verdict unknown".

Concurrency

Async; safe to call concurrently per request because the underlying httpx client per node already pools connections.

Executed-code reconstruction from tool_call_id

When the verifier gate in handle_chat classifies a turn as code-shape (last substantive tool was execute / postgres_*), it calls verify_code_output(code, output, intent). The code slot MUST be the actual program that was submitted.

Prior to the fix, the call site at agent.py:4835 passed code=tool_name — literally the string "execute". With nothing real to audit, the verifier LLM hallucinated plausible-sounding rejections ("appears to be a directory listing", "missing expected field") and emitted REFUTED (95%) verdicts on CORRECT turns. The user then saw Verifier note: … contradicting their right answer.

The fix is _reconstruct_executed_code(messages, tool_msg) in agent.py: walk messages backwards from the tool result, match tool_call_id to the assistant's tool_calls[i].function.arguments, and extract content / code / script_content / text / command / cmd (first non-empty wins). Capped at 4000 chars. When the lookup fails — malformed arguments, missing id, non-dict messages — the gate falls back to verify_claim, which doesn't need a code slot.

Covered by tests/test_verifier_code_reconstruction.py (14 cases including dict-vs-string arguments, alias preference, traversal past unrelated messages, oversize cap, identity-based dedup on duplicate tool-call ids).

Gate authority: REFUTED / unverified-mutation lower confidence (2026-06-07)

The gate used to be advisory-only: a high-confidence REFUTED verdict appended a **Verifier note:** to the reply but otherwise let the turn finalise as a success, and the finalize confidence reading — competence-driven — still read C ≈ 0.92–0.96, below=no. So the agent could mark a broken build "done" with high confidence (observed: a request that finished "REFUTED (95%)" on a game that did not run, and another that finalised an untested file write with the gate skipped at C=0.96).

The gate now has teeth without re-running the whole turn (which would double latency):

  • REFUTED ≥ 0.7 already sets verifier_backfill = ("failed", …); the finalize confidence block now reads that and passes outcome_penalty = 0.8 into CompositeConfidence.score(), pulling the reading below threshold so below=yes and the calibration spine records a "confidently-wrong" negative.
  • Unverified mutation — when the verifier is unavailable (v_result is None) and the turn's final substantive action was a successful file_system write/replace (_is_unverified_mutation), the turn is marked outcome="failed" with an honest "⚠ Unverified — run/preview it before relying on it" note, rather than a silent success. This is the untested-write case.

Net effect: a refuted or untested result no longer reports high confidence and is recorded as a failed outcome, so downstream task-completion logic stops marking broken work "done". Covered by tests/test_verifier_gate_teeth.py.

Autonomous repair re-entry (SHIPPED 2026-06-10)

The full autonomous repair re-entry — re-running tool turns against the verifier's critique — is now implemented (it was previously deferred as needing a core-loop refactor). When the verifier returns high-confidence REFUTED on the final answer (or the turn finalised on an unverified file_system write), the agent gets up to GhostAgent._MAX_VERIFIER_REPAIRS (default 1) bounded in-loop re-attempts to FIX it instead of only annotating and shipping.

  • The repair reuses the EXISTING for turn loop: at the normal-success finalisation (the if not tool_calls: branch) the verdict is computed, and on refute/unverified the critique is injected as a user message and the loop continues — no ~3,900-line re-indent.
  • Verdict computation is extracted to GhostAgent._compute_verifier_verdict (pure, no side-effects), cached at finalisation (_verdict_is_fresh) and REUSED by the post-loop gate, so a clean success still costs exactly ONE verifier pass.
  • The verdict cache is reset each turn (_verdict_is_fresh = False at turn start) and explicitly invalidated on a repair continue (_verifier_verdict_cache = None), so an error or abort on the repair iteration can never reuse a stale verdict from the prior answer.
  • Anti-loop accumulators and strike caps are PRESERVED across repair (bounded); only force_final_response is reset on re-entry. The trajectory records once, post-loop, on the final answer.

Covered by tests/test_verifier_auto_repair.py.

Async-critic repair — bounded verdict await at loop-exit (2026-07-07, #18)

The repair above originally fired only in SYNC mode. In production the launcher runs GHOST_CRITIC_ASYNC=1 (verdict on the off-host critic-nodes model), which used to skip the in-loop verdict entirely — so a REFUTED answer shipped with only a next-turn correction note. Because the critic runs on a second model, waiting for it briefly costs the MAIN inference slot nothing. So async mode now AWAITS the verdict at loop-exit for a bounded budget (_critic_repair_await_budget, default 25s, GHOST_CRITIC_REPAIR_BUDGET overrides, 0 disables) when the last tool is substantive and the deterministic unverified-mutation guard hasn't already tripped. If the verdict lands REFUTED within the budget, the same in-loop repair fires and the corrected answer REPLACES the refuted one; the verdict is cached so the post-loop gate doesn't recompute it. On timeout it falls back to the pre-2026-07-07 defer path (ship + queue a next-turn correction; the background verdict still scrubs poisoned lessons). This closes ledger item #4 as landed (sequential repair, critic-judged) — parallel best-of-N was rejected as strictly worse on a single-slot box. Covered by tests/test_critic_async.py.

Widened evidence window for claim-shaped verdicts (2026-07-16)

The gate used to judge the ENTIRE final answer against only the last substantive tool output (4000 chars). A multi-source research answer synthesises many tool outputs, so whichever tool happened to run last decided the verdict: on the souvlaki turn (req 738c/73) the last fetch was a 403 Forbidden while the answer came from two earlier successful page loads — the verifier truthfully reported "evidence does not support the claim" about the sliver it was shown and REFUTED a correct answer, burning an auto-repair round re-fetching sources the turn already had. The same keyhole produced the "corrected list is unsupported" refute one turn later (prior-turn evidence is invisible to the gate) and near-miss request-alignment refutes on multi-wave search turns.

_collect_verifier_evidence(tools_run, max_items=3, budget=4000) in agent.py now assembles the last three substantive tool outputs — selection mirrors _find_substantive_tool_for_verifier exactly (synthetic and bookkeeping entries skipped) — rendered chronologically, each prefixed [tool_name], with newest-heavy budgets (50/30/20; a single-tool turn keeps the full budget, byte-equivalent to the old path) and the total capped including labels so verify_claim's own [:4000] guard can never truncate the newest item away. Both claim-shaped gate paths use it; the code-shaped path (verify_code_output) deliberately keeps the single-output view — it audits one specific run. _VERIFY_CLAIM_PROMPT declares the multi-tool shape and adds two rubric lines: one failed tool does NOT refute parts of the claim supported by other outputs, and specifics (names, dates, awards, rankings) appearing in NO output are fabrications — still REFUTED, so the widened window does not defang the gate (the "Everest / world champion June 2026" fabrication remains refutable). last_tool semantics are unchanged everywhere else (mutation guard, empty-verdict diagnostics, code-shape classification). Covered by tests/test_verifier_evidence_window.py.

Slack redistribution + URL squeeze — the starved-evidence refute (2026-07-17)

The one-pass newest-heavy split had a dual failure mode to the souvlaki keyhole: a tiny newest output could starve a large older one. On the naftemporiki turn (req 4dab5067) a 106-char weather report — the newest tool — held 65% of the 4000-char budget and left it unused, while the 4KB RSS feed was cut mid-item #4; the verifier (correctly, on what it was shown) REFUTED the answer's items #5–#10 as "not in the evidence" — twice, sync then late — so a fully-supported answer burned a repair round, had its lessons scrubbed, was backfilled failed into the corpus, and queued a bogus next-turn correction. (Initially misread as the worker model failing Greek: the refuted items were simply the ones past the cut.)

Two changes in _collect_verifier_evidence: (1) allocation is now two-pass — weighted caps as before, then unused slack is redistributed to still-truncated items, newest first, so a small output can never hold budget it doesn't use; (2) _squeeze_evidence_noise first trims tracking-length URLs (https?://\S{72,} → 64-char stub + ellipsis) — they carry zero entailment value and were ~70% of the RSS payload. The total stays capped including labels. On the failing turn's real data this brings all 10 headlines plus the weather into 2212/4000 chars. Regression tests in tests/test_verifier_evidence_window.py (2026-07-17 section).

Repair directive demands a standalone rewrite (2026-07-17)

The auto-repair directive rides a user-role message while the refuted draft is discarded — the real user never saw what is being corrected. The model treated the alert conversationally, so its "corrected" reply opened with checker-facing dialogue ("You're right — I was interpreting and embellishing…", req 4dab5067) addressed to nobody the user knows about. Both repair directives (REFUTED and unverified-mutation) now append _REPAIR_STANDALONE_SUFFIX: the user never saw the draft or the alert; write a clean standalone reply; no acknowledgements, apologies, or mention of any verifier/review/correction. Wiring guarded by tests/test_verifier_evidence_window.py.

Turn duration stamped into the trajectory corpus (2026-07-17)

Same incident, adjacent gap: _record_turn_trajectory left duration_s at the schema default 0.0 on every chat turn. It now stamps the wall-clock from the pretty-log request clock via ghost_agent.utils.logging.request_elapsed_s(req_id) (None — closed request / sim contexts — keeps the default), so corpus consumers (PRM features, reflection) can see per-turn latency. tokens_in/out remain 0: no usage accumulation exists on the streaming client path — a deliberate scope-out, not an oversight. Finalize-burst stagger — verify vs hydration-judge (2026-07-16)

At end-of-turn the deferred async verdict and the memory bus's hydration-usefulness judge were dispatched to the single worker node in the same second; the node shares compute across its slots, so the pair reliably pushed the verdict past its ceiling (Nova: ReadTimeout on effectively every substantive finalize — 4 of 17 verify calls in the 2026-07-16 log window died this way). _attach_late_verdict_handler now publishes its task as agent._deferred_verdict_task (cleared by the done-callback), and _judge_hydration_safe waits — bounded by _HYDRATION_JUDGE_STAGGER_S (90 s, above the 45 s verify budget, far inside the judge's 600 s stash-staleness guard) — for the in-flight verdict before spawning the judge. The verdict is the safety signal, so the judge yields; when no verdict is in flight the judge starts immediately, and a failing/cancelled verdict task never takes the judge down with it (asyncio.wait shields it). Covered by tests/test_finalize_stagger.py.

Visual verification — closing the pixels gap (2026-06-08)

The text verifier audits words: it checks whether the agent's reply is on-topic and supported by tool output. For a visual bug report ("the blocks look transparent", "the layout is broken") that is structurally blind — the truth lives in the rendered frame, so a self-claim like "I split the meshes to fix it" passes as CONFIRMED while the screenshot still shows the defect. This was the live-log failure where a still-broken game was confirmed at 90%.

verify_visual(symptom, claim, after_image, before_image=None) closes it by looking at the actual pixels via the vision node (_call_llm_vision → chat_completion(..., use_vision=True); it does NOT use the text VERIFY worker pool, which isn't multimodal). When a before image is available the model does a before/after comparison, which is far more reliable than judging a fresh frame cold.

Verdict semantics — claim vs. pixels, not symptom-presence. The judgement is whether the agent's RESPONSE is honest about what's on screen: a "fixed it" claim while the image is still broken (or blank / stuck on a loading screen) → REFUTED; a claim that matches the pixels → CONFIRMED. Crucially, an honest "it's still broken, I couldn't fix it" is ACCURATE and stays CONFIRMED — the gate catches false success, it does not punish truthful failure reports. (Validated live: false-fix claim on a red circle → REFUTED 1.0; honest "still red" → CONFIRMED 1.0.)

Start/menu screen = not running (2026-06-09). The prompt now treats a visible start menu, a "Click to Play"/"Start"/"Press to start" button, an instructions modal, or a loading screen as proof the app has NOT started — it is showing its MENU, not its running state. A claim that the app/game "works", "renders correctly", "is fully functional", or "is playable" made over such a frame is REFUTED. This closes the live hole where the agent screenshotted a Minecraft clone with its "Click to Play" modal still up (blocks visibly broken behind it) and the claim was CONFIRMED at 90%; the browser tool's PRE_INTERACTION signal flags the same state in the tool evidence.

  • Gate (cheap, no LLM). _is_visual_intent(user_text) — keyword/extension match — keeps the extra vision call off the ~99% of turns that aren't about how something looks.
  • Evidence. _select_visual_evidence(messages, …, sandbox_dir) picks before = an image the user referenced, and after = a DIFFERENT screenshot the agent produced this turn. If the only image in play is the user's own (the agent never re-rendered), after is None and the check is skipped — no post-fix evidence means no verdict. Container paths (/sandbox/…, /workspace/…) and bare filenames resolve to host paths via _resolve_image_path.
  • Override. A confident (≥ 0.7) visual REFUTED overrides an optimistic text CONFIRMED — pixels win over the self-claim — and flows into the same verifier_backfill / outcome_penalty = 0.8 machinery above. Visual UNCERTAIN never overrides; a visual verdict only fills in when the text path produced none.
  • Conservative by construction. No browser / unreadable / oversized image → None (skipped, no penalty): the agent is never punished for infra it can't control. The vision model is itself fallible, so the bar to penalize is the same 0.7 used everywhere else.

Covered by tests/test_verifier_visual.py. Runs only under --enable-metacog (the verifier gate). A proactive re-render step (screenshot the edited artifact when the agent didn't) is a natural follow-on; today it reuses a screenshot the turn already produced.

Web-artifact execution check (2026-06-11)

Sibling of the visual override, living in agent._compute_verifier_verdict and running after it (execution is harder ground truth than pixels). When the finishing turn wrote web files (_web_artifacts_written parses file_system SUCCESS messages for quoted .html/.js/.mjs/.cjs names), _execute_web_artifact loads the entry page headless through the regular browser tool: the written .html itself, or for JS-only edits the conventional index.html next to the edited file.

  • Uncaught JS exception on load → REFUTED (0.95) with the captured exception block in the reasoning, overriding any text-level CONFIRMED. This feeds the existing auto-repair / backfill machinery, so the agent fixes the artifact instead of shipping it.
  • Clean load → text verdict stands (logged as WEB-EXEC clean).
  • Inconclusive ≠ clean — and inconclusive now caps a CONFIRMED (2026-07-05). No entry page on disk, no browser tool, or a failed navigation returns None; a broken probe must never certify a build. Previously "changes nothing" meant a text/vision CONFIRMED (even 100%) stood on a build that was never executed — the fail-open the 2026-06-20 live run exposed (verifier CONFIRMED 100% on vision alone right after WEB-EXEC check skipped). Now, when the turn wrote web artifacts and the probe was inconclusive (skipped or crashed), a CONFIRMED verdict keeps its verdict but has its confidence capped at _WEB_EXEC_SKIP_CONF_CAP (0.6) — below every ≥ 0.7 consumption gate (backfill "passed", auditor note, calibration) — with the cap noted in the reasoning and logged (WEB-EXEC inconclusive → CONFIRMED capped). REFUTED is never weakened (refuting is already the fail-safe direction). In sync-critic mode the in-loop repair gate now also treats a sub-0.7 CONFIRMED on a final untested write as unverified and fires the bounded "actually RUN it" re-entry, mirroring the async path's pure-predicate behaviour.

Production motivation (req EA, 2026-06-11): the text verifier said CONFIRMED (95%) on "The Infinite Corridor" — claim, evidence and request all aligned — but data.js had a parse error and the page threw on load; the user found out by clicking a dead button. Covered by tests/test_verifier_web_exec.py.

Reliability fix (2026-06-20): the probe was navigating to file://{page_rel} with a relative filename (e.g. file://index.html) — which a browser parses as a host with an empty path, so it never loaded. Every project-scoped build therefore logged WEB-EXEC check skipped and a throwing page kept its text-level CONFIRMED (observed live on the single-file browser-OS build: verifier CONFIRMED 100% right after WEB-EXEC check skipped). Two changes: (1) the entry page is now resolved to a real host file and navigated via an absolute container URL (file:///workspace/<rel-from-mount-root>) built with _to_container_path, so the scope's projects/<id>/ segment is preserved; (2) a new _locate_entry_page helper searches the scoped sandbox, the root mount, and finally a freshest-wins basename search — so when a project-REUSE turn leaves the deliverable in projects/<id>/index.html while the sandbox binding reads as un-scoped, the check still finds and loads the file the agent actually wrote instead of skipping. The basename fallback is gated on recency (15-min window): a live run surfaced the opposite failure — the turn's deliverable never landed on disk and the fallback loaded a 25-minute-old index.html from an UNRELATED project and reported it clean (a false CONFIRM). A fallback match older than the window is rejected, so a deliverable that didn't actually land this turn stays inconclusive rather than being certified by a stale stand-in. The mocked tests previously never exercised the URL string; new cases assert it is absolute, carries the scope segment, survives the binding gap, and refuses a stale stand-in.

Upstream caveat: the binding gap itself (a project-reuse turn writing to / reasoning about the wrong project id, so the just-written deliverable isn't where the scope expects it) is a separate project-binding defect — WEB-EXEC now fails honest (inconclusive) under it rather than masking it, but does not fix it.

File-artifact ground-truth check (2026-07-16)

The general, non-web form of the "execute, don't trust" overrides. The agent's #1 most-retrieved real lesson (ret=55) is that it "prematurely declared task completion … without showing the actual content" — it claims a file deliverable that is missing or empty. A text verifier can't catch this (the claim reads fine); re-reading the file is hard ground truth. B4 independently showed the same: scattered off-by-a-few outputs that the LLM verifier CONFIRMED at 100%.

Running last in _compute_verifier_verdict (after web-exec — a missing/empty file is as hard a fact as a throwing page): _claimed_deliverable_files(final_ai_content) extracts filenames the answer presents as a produced deliverable — anchored on completion verbs (saved / wrote / created / generated / produced / exported / stored / placed / dumped) so it ignores files the agent merely READ and present-tense descriptions of what a script does ("the script writes to X"). _verify_file_artifacts then re-reads each under the project-scoped sandbox host path (direct FS read, no docker exec); any claimed deliverable that is MISSING or EMPTY yields a REFUTED (0.9) that overrides a plausible text CONFIRMED and feeds the same bounded auto-repair loop, so the agent gets a chance to actually produce it. Low false-positive by construction: an input file the agent read still exists → passes; only a claimed-but-absent deliverable refutes. Tests: tests/test_grounded_file_verify.py.
