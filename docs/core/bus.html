RRF (Reciprocal Rank Fusion)

For each ranker (vector, graph, skill, episodic) the bus computes:

score(doc) = Σ_ranker  weight_ranker / (k + rank_in_ranker + 1)

k = rrf_k: _reciprocal_rank_fusion keeps 60 as its classmethod default, but the hydration path now passes _RRF_K = 10 (2026-07-22, see the review section below). Per-source weights depend on a query intent classification; the table below is the hand-set prior that a learned matrix (core.rrf_weights) now scales multiplicatively within a bounded band:

Global budget + per-item emission (2026-07-07)

There are no fixed per-source budgets. _format_markdown_with_survivors emits items strictly in descending fused-score order under one global max_chars (headers printed lazily on first sight of a source), with a _PER_SOURCE_CAP (6) so one tier can't monopolise the budget. An earlier redesign moved off the old 25/40/20/15 % section split because fixed quotas discarded the fused ranking; this section previously still documented the dead quotas.

Per-item, not per-tier. The skill and episodic fetchers now return one RRF item per lesson / per episode (SkillMemory.get_playbook_items, EpisodicMemory.format_episode) instead of a single whole-tier blob. A blob was always rank 1 in its ranker regardless of any individual lesson's relevance, so the per-item relevance floor and _PER_SOURCE_CAP were no-ops on it, and under sub-query fan-out several near-identical multi-KB blobs each consumed the global budget. Per-item, each lesson/episode is ranked, gated, capped, and deduped on its own.

Session tier — raw conversations (2026-07-14). _fetch_session is the fifth fetcher: SessionStore.search_messages keyword-searches the 50 most-recently-updated stored sessions (mtime-cached parses; a 2-second summaries memo shares one scan across the sub-query fan-out; multi-term queries must match ≥2 distinct non-stopword terms). One RRF item per matching message, rendered [session title] role: snippet under a ### PAST CONVERSATIONS header with the same reference-only framing as episodes. Intent weights: factual 0.8 / procedural 0.5 / contextual 1.2 — conversations answer "what did we discuss/decide" queries, not how-to ones. This closes the raw-layer gap: sessions were durable (2026-07-11) but replay-only, unreachable by every retrieval path. Tests: tests/test_session_hydration_tier.py.

Episodic semantic recall is wired. _fetch_episodic passes the bus's vector store to search_similar, so the embedding-based episode retrieval path fires (previously it always fell back to substring matching over the 100 most-recent episodes, leaving ~half the collection unreachable). Stubs with a 2-arg search_similar still work (the bus catches the TypeError and retries without the kwarg).

Recall regression eval (2026-07-14). tests/test_recall_regression_eval.py seeds a golden corpus across all five tiers and asserts end-to-end hit-rates through the REAL pipeline (real BGE embeddings + Chroma + fusion) — the guard the retrieval stack never had (the embedder fingerprint catches a wrong model, nothing caught worse retrieval). Measured 2026-07-14: paraphrase recall 100% (floors set at 75% headroom); every tier-specific case passes.

Off-topic gate (2026-07-15). The recall eval's former xfail — an off-topic query hydrating a small store — is now fixed and asserted. The fix is not the RRF _RELEVANCE_FLOOR: measured that it CANNOT reject an off-topic query, because RRF scores are a function of (rank, tier weight, intent) and discard the underlying embedding distance, so normalising against the top item makes the best match 1.0 whether the query is on- or off-topic (off-topic scores are in fact flatter). The real signal is the best match's absolute distance: on BGE-small a genuine match lands < 0.40 while an off-topic query's closest item is ≥ 0.44. So _fetch_vector now passes _VECTOR_MATCH_FLOOR (0.42, env GHOST_VECTOR_MATCH_FLOOR) to VectorMemory.search_items: if the closest candidate exceeds it, the vector tier injects nothing. On-topic hydration is untouched (its true match is always well under the gate) so recall cannot regress; the gate is on the hydration path only — the recall tool stays best-effort for an explicit "what do you know about X".

Usefulness feedback loop (2026-07-14). Surfaced-credit alone is circular — items get credit for entering the prompt, so popular memories only get more popular. hydrate_context now stashes the turn's survivors (+ intent + timestamp) on bus.last_hydration; after the reply is final both agent finalization paths spawn judge_hydration_usefulness (fire-and-forget via spawn_bg, worker-hosted, 60 s cap): one JSON call marks which snippets the reply actually drew on ("on-topic is NOT enough; when unsure, exclude"). Used vector items get VectorMemory.bump_helpful (helpful_count weighs 2× a plain retrieval in the spaced-repetition half-life), used skills get record_helpful_retrieval, and EVERY survivor — used or not — appends an (intent, source, used) line to $GHOST_HOME/system/rrf/observations.jsonl. The dream cycle's _refit_rrf_weights fits fit_intent_weights on ≥30 observations, persists rrf/weights.json (loaded at boot), hot-swaps the live matrix, and trims the ledger — turning the learned fusion weights from a static offline artifact into an online loop keyed to real usefulness. Stale stashes (>10 min, from turns that never finalized) are discarded; a worker failure or unparseable verdict credits nothing and writes nothing. Since 2026-07-16 the judge's spawn is staggered behind any in-flight deferred verifier verdict (bounded 90 s wait — see verifier "Finalize-burst stagger"): the two used to hit the single worker node in the same second and the loser blew the route timeout. Tests: tests/test_hydration_usefulness_loop.py, tests/test_finalize_stagger.py.

Retrieval credit is deferred to post-fusion. The fetchers use side-effect-free reads (search_items, get_playbook_items); _credit_surfaced then bumps reinforcement counters exactly once per turn, only for the items that actually entered the prompt, via VectorMemory.bump_retrievals(ids) + SkillMemory.record_retrievals_bulk(triggers). Previously every candidate of every sub-query was credited inside the search call — inflating retrievals ~4× per turn while helpful_retrievals credits at most once per window, which structurally crushed hit_rate and mis-flagged useful lessons as stale-and-prunable, at a cost of up to ~20 full skills_playbook.json rewrites per turn.

Memory-substrate review fixes (2026-07-22)

RRF intent-weight fit is now relative + multiplicative on the prior (CRIT). rrf_weights.fit_intent_weights computes weight = prior * clamp(lift ** (GAMMA * shrink)), where lift is the cell's usefulness relative to its OWN empirical base rate (the observed judged-used rate, not an assumed 0.5), shrink = n / (n + PRIOR_STRENGTH) makes the pull evidence-proportional, and a hard MAX_DEVIATION = 2.0 band keeps a 2.0-prior cell inside [1.0, 3.0] — so a learned matrix can re-order tiers only mildly, never INVERT them. lift == 1 (the cell performs exactly like the average injected item) reproduces the prior exactly; a degenerate all-fail or all-success sample carries no cross-tier contrast and is refused outright (the priors stand). The old fit calibrated "rate 0.5 → base weight", but the real judged-used rate is ~0.14, so every well-sampled cell landed on the lower branch and was crushed toward WEIGHT_MIN — the live weights.json had factual/graph at 0.1 (a 6.6x inversion of its 2.0 prior) and the whole intent matrix had collapsed into noise, leaving intent routing silently inert. The collapsed live weights.json was removed on deploy; it regenerates healthily on the next dream refit, which anchors on DEFAULT_INTENT_WEIGHTS.

Turn-normalised lift kills the items-per-turn confound. _per_turn_lift credits each tier's SHARE of a turn's judged-used set (turn-normalised) rather than a raw per-item used-rate. Measured live 2026-07-22 over 130 turn-batches, hydration injects ~11.6 items/turn but the per-tier counts are wildly uneven (vector 4.55/turn, graph 1.09, skill 0.92), so a raw per-item rate mechanically capped a verbose tier near 1.65/4.55 ≈ 0.36 while a sparse tier could reach 1.0 — the old fit measured tier VERBOSITY, not usefulness. Shares are immune to that. dream.py's _refit_rrf_weights now forwards the ledger's turn id so the turn-aware estimator activates; a legacy line without a turn id degrades to the pooled estimator (fit_intent_weights handles both shapes).

Consensus normalisation + lower fusion k. _reciprocal_rank_fusion gained normalize_consensus: it divides each source's accumulated scores by that source's REDUNDANCY (emissions ÷ distinct items) across the sub-query lists, so a keyword-stable graph edge returned identically by 4 sub-queries no longer banks 4x the reciprocal rank while vector — which by design returns different items per sub-query — banks 1x. INTRA-tier ordering (the genuine cross-sub-query consensus reward) is untouched; the flag is off for direct classmethod callers and on for hydrate_context. Separately, the hydration path's fusion k dropped to _RRF_K = 10: at 60 the intra-tier spread flattened to ~19% over the 5-15 item lists this bus fuses, collapsing fused order to tier_weight × consensus and discarding the vector store's own distance ranking; _reciprocal_rank_fusion keeps 60 as its own default for back-compat.

Adaptive budget honours the caller's context_budget as a base. A simple query gets it as-is; a medium query (>15 words) gets 2x it capped at 9000; a complex one (>30 words) 3x capped at 12000. The old form min(max(context_budget, 6000), cap) evaluated to 6000 in all three branches for the sole prod caller's context_budget=4000 — the adaptive path was dead code, the 12000 ceiling unreachable, and the operator's deliberate 4000 silently overridden.

Budget emission loop continues past an oversized item. When an item can't fit the remaining budget (and something is already emitted), _format_markdown_with_survivors now continues to the next item instead of breaking. Items are ordered by fused score, NOT length, so the old break was simply wrong: a 900-char vector item at rank 3 discarded a 23-char SKILL LESSON and a 6-char graph edge that both fit trivially in the remaining ~277 chars — and because headers are emitted lazily on first sight, the entire SKILL PLAYBOOK and TOPOLOGICAL KNOWLEDGE GRAPH sections (headers included) vanished from the prompt silently.

Intent is classified on the raw user text. hydrate_context gained a raw_user_text kwarg (agent.py passes last_user_content); _intent_source_text classifies it when supplied. The old path classified the whole expanded query — Context: <~200 chars of the PREVIOUS assistant reply> | User intent: <query> — so on a short follow-up the assistant's own prose (dense in classifier keywords) drove the tier weighting instead of the user's actual question. The _EXPANSION_MARKER tail-split remains as the fallback for call sites that don't pass the kwarg yet.

Off-topic vector gate no longer self-disables on an unrelated TypeError. The blanket except TypeError that used to catch a stub's missing min_relevance_dist also swallowed any TypeError raised INSIDE a real search_items and retried without the gate — silently injecting the weakly-related tail with no log at all. _accepts_relevance_floor now decides by signature introspection whether the callable takes the parameter; a genuine arity mismatch on an introspectable callable drops the gate with a WARNING instead of masking a real error. (Distinct from the 2026-07-15 _VECTOR_MATCH_FLOOR distance gate above — this fixes how that gate is passed, not the gate itself.)

Per-hydration instrumentation. _log_hydration emits one INFO line per hydration with per-tier candidate counts, per-tier survivor counts, survivor total, intent and budget — e.g. Hydration tiers: v=4 g=0 s=2 e=3 sess=1 -> 7 survivors (v4 g0 s2 e3 sess1) intent=contextual budget=4000. A wired-but-empty tier reads as x=0 and an unwired (None) store as x=-, distinguishing "dead for weeks" from "not configured" — the cheap guard that would have surfaced this whole finding class months earlier. INFO so it lands in the file log without touching the WARNING+ console/pretty stream.

Tests: tests/test_bus_rrf_fixes.py.

Dedup LRU

_DEDUP_LRU_MAX = 256. Each publish_fact hashes (event_type, fact_data) with MD5 before the fan-out — a recently-seen signature short-circuits and writes nothing. Prevents re-emission storms when the LLM repeats itself.
