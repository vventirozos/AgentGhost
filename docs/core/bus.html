core / bus.py — MemoryBus

In-memory pub/sub façade over Vector / Graph / Skill / Profile / Episodic stores. Hydrates context with Reciprocal Rank Fusion; publishes facts in parallel.

RRF (Reciprocal Rank Fusion)

For each ranker (vector, graph, skill, episodic) the bus computes:

score(doc) = Σ_ranker  weight_ranker / (k + rank_in_ranker + 1)

k = rrf_k defaults to 60 (line 130s). Per-source weights depend on a query intent classification:

Intentgraphvectorskillepisodic
factual2.01.00.50.3
procedural0.51.02.01.5
contextual1.01.51.01.0

Global budget + per-item emission (2026-07-07)

There are no fixed per-source budgets. _format_markdown_with_survivors emits items strictly in descending fused-score order under one global max_chars (headers printed lazily on first sight of a source), with a _PER_SOURCE_CAP (6) so one tier can't monopolise the budget. An earlier redesign moved off the old 25/40/20/15 % section split because fixed quotas discarded the fused ranking; this section previously still documented the dead quotas.

Per-item, not per-tier. The skill and episodic fetchers now return one RRF item per lesson / per episode (SkillMemory.get_playbook_items, EpisodicMemory.format_episode) instead of a single whole-tier blob. A blob was always rank 1 in its ranker regardless of any individual lesson's relevance, so the per-item relevance floor and _PER_SOURCE_CAP were no-ops on it, and under sub-query fan-out several near-identical multi-KB blobs each consumed the global budget. Per-item, each lesson/episode is ranked, gated, capped, and deduped on its own.

Episodic semantic recall is wired. _fetch_episodic passes the bus's vector store to search_similar, so the embedding-based episode retrieval path fires (previously it always fell back to substring matching over the 100 most-recent episodes, leaving ~half the collection unreachable). Stubs with a 2-arg search_similar still work (the bus catches the TypeError and retries without the kwarg).

Retrieval credit is deferred to post-fusion. The fetchers use side-effect-free reads (search_items, get_playbook_items); _credit_surfaced then bumps reinforcement counters exactly once per turn, only for the items that actually entered the prompt, via VectorMemory.bump_retrievals(ids) + SkillMemory.record_retrievals_bulk(triggers). Previously every candidate of every sub-query was credited inside the search call — inflating retrievals ~4× per turn while helpful_retrievals credits at most once per window, which structurally crushed hit_rate and mis-flagged useful lessons as stale-and-prunable, at a cost of up to ~20 full skills_playbook.json rewrites per turn.

Dedup LRU

_DEDUP_LRU_MAX = 256. Each publish_fact hashes (event_type, fact_data) with MD5 before the fan-out — a recently-seen signature short-circuits and writes nothing. Prevents re-emission storms when the LLM repeats itself.
