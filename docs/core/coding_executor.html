core / coding_executor.py

The real per-task coding executor for autonomous batches: build one leaf for real — write files, verify — instead of marking shallow work done.

Why

When advance_many runs a batch ("proceed with all remaining tasks"), each coding tick used to generate a SINGLE shell command. That produced theatrical completion: a leaf marked DONE having written nothing — observed live, where "create hello.txt" was classified as research, ran a web_search, and was reported complete with no file on disk. This module makes a coding tick actually build.

What it does

build_coding_task(context, description, *, tool_runner, ledger="") is bounded by design — the task tree already decomposes the work, so this just builds ONE leaf well:

  1. Spec — one LLM call returns a JSON object: files (each a path + FULL content), an optional verify shell command, a one-line summary, and a one-line ledger fact. The project's design ledger AND its current files (existing_files, gathered by project_advancer._gather_project_files) are fed in, with an instruction to EXTEND those files rather than recreate them — so a single-file app accretes across tasks instead of each task overwriting the last. The project's research briefs (research_context, gathered by project_advancer._gather_research_briefs) are fed in too, as a READ-ONLY PROJECT RESEARCH reference block (2026-06-22) — so the build follows the design decisions the agent researched instead of ignoring its own work; these are explicitly framed as "NOT files to edit" and kept out of existing_files.
  2. Write — each file via the file_system tool (operation="write"), capped at MAX_FILES (16) / MAX_CONTENT_CHARS (400k) per file.
  3. Verify — runs the verify command via execute; a failure (per project_advancer._looks_like_failure) fails the build.
  4. Returns a CodingResult (ok, summary, files, ledger_note, detail).

Don't clobber prior work (2026-06-20)

A later failure: on a single-file app each task REGENERATED the whole file with write and OVERWROTE the previous task — three tasks "completed" left a 2.7KB shell with neither the File Explorer nor the Snake game, and frontend tasks (no shell verify) had no gate to catch it. Three defences:

  • Non-regression guard — a full-content rewrite of an existing file is REFUSED if the new content is materially smaller (<85%) or drops the file's existing structural identifiers (HTML ids, function/class names, CSS selectors). It must be a SUPERSET.
  • Edit, don't clobber — a file entry may carry edits ([{"find":…,"replace":…}], {"after":…}, or {"before":"",…}) applied via file_system operation="replace", so a feature is ADDED to an existing file without re-sending it. A non-matching edit fails the task.
  • Append (the easy incremental primitive, 2026-06-20) — a file entry may carry "append":""; the executor writes old + new. It's a strict superset (can never regress), and the model emits ONLY the new snippet. This was added because re-emitting a whole ~8KB file choked the small model on task 2 of a live single-file build — it returned an EMPTY spec twice and the task failed. With append/edits the model only produces the delta. Existing files are now shown to the model as compact head+tail EXCERPTS (not full content), and the prompt steers hard toward append/edits over full content.

No per-task render (correction). An earlier version headless-rendered each HTML task and failed it on an uncaught JS exception. That was wrong for incremental builds: no intermediate task produces a complete working page — the shell exists before the apps, an app before the next — so rendering each tick failed legitimate in-progress work. Observed live: the Core-shell task built real files (a 799B index.html + script.js + 7.7KB style.css), then a render of the app-less shell "crashed" on a not-yet-defined reference, the retry returned an empty spec, and a good task was marked FAILED. The render gate was removed; the non-regression guard is the real protection, and a shell verify still hard-gates runnable code. Whole-app rendering belongs to a final verification task, not to every tick.

Retry with feedback. A refused write or a failed verify is regenerated ONCE with the exact failure reason fed back, then the task fails. Single-file projects (detected from the goal in advance_once) pass single_file=True to strengthen the "grow the one file" steer.

Recover the spec from the reasoning channel (2026-06-22)

A live failure mode killed 5 coding leaves in one project: every attempt logged coding_executor: no file spec parsed (len=0) with an EMPTY raw window, and advance_once marked the task FAILED with "model produced no file spec for the task". The cause was not the model — it was _generate_build_spec reading only message.content. Reasoning models (Qwen via llama.cpp) emit their chain-of-thought in a separate reasoning_content field; when the think block consumes the whole max_tokens budget without closing, the parser routes EVERYTHING — including the JSON build spec — into reasoning_content and leaves content empty. The executor saw "" and gave up.

The fix mirrors the established pattern in agent.py (~L2973) and project_research.py: when content yields no usable spec, fall back to reasoning_content (then to the two channels concatenated, for a spec split across the </think> boundary). A genuinely empty response (upstream contention) still fails — but only after the fallback was tried, and the diagnostic now logs BOTH channel lengths (content=N reasoning=M) instead of a misleading empty window. Covered by test_build_recovers_spec_from_reasoning_content and siblings in tests/test_coding_executor.py.

Think-ate-the-budget: one no-think retry (2026-07-06)

The reasoning-channel fallback above only helps when the JSON spec actually made it into reasoning_content. The worse shape (observed live 2026-07-06, twice in one project: content=0 reasoning=40–62k chars) is the model spending the ENTIRE 16384-token budget on think-prose and never starting the JSON at all — each occurrence a ~5-minute generation lost, logged as extract_json_from_text: malformed JSON-like content followed by no file spec parsed. _generate_build_spec now detects exactly this signature (no usable spec AND empty content AND non-empty reasoning) and retries ONCE with thinking disabled, using the established no-think recipe from project_research._llm_call/dream.py: the /no_think soft-switch, the chat_template_kwargs {"enable_thinking": false} hard-switch, and a system nudge. Attempt 1 keeps thinking ON (design depth is worth it when it works); prose-in-content failures do NOT trigger the retry (that's the normal feedback loop's job). Covered by test_think_only_response_retries_once_without_thinking and test_prose_content_does_not_trigger_nothink_retry.

Already-built leaf: empty files + a verify (2026-06-22)

When a task's deliverable was already written by a prior interactive turn, the autoadvance tick re-picks the task and the model — correctly — returns a spec with "files": [] and a "verify" command ("nothing to write; just check it runs"). The executor used to reject empty files as "model produced no file spec" and FAIL the task (observed live: the Model Architecture task FAILED in autoadvance though model.py existed and ran). Now an empty file list with a verify RUNS the verify: it passes → the task succeeds (no files written); it fails → the task fails with the real verify error fed back to the retry, not the generic no-spec message. Empty files AND no verify still fails as before. Covered by test_empty_files_with_passing_verify_succeeds and siblings in tests/test_coding_executor.py.

Capacity caps raised off the small-model era (2026-06-22)

An audit for leftovers tuned to a weaker model surfaced several caps that were starving or fragmenting code from the current strong model:

  • MAX_ATTEMPTS 2 → 4 and MAX_FILES 8 → 16. A capable model often fixes a verify failure on attempt 3-4; the hard stop at 2 turned a recoverable build into a batch-halting FAILURE. And a single "scaffold the app" leaf can legitimately emit a dozen+ files, which 8 silently dropped.
  • Append/edits steer rebalanced. The spec prompt previously pushed append as "STRONGLY PREFERRED" and told the model to "avoid edits" — biasing toward additive-only changes (you cannot append your way to a bug fix or refactor). It now picks by INTENT: adding → append; changing existing code → edits.
  • IIFE auto-wrap made conservative. _isolate_scripts wrapped EVERY appended inline <script> in an IIFE to prevent cross-block identifier collisions — but that SILENTLY scoped away intentional globals, breaking <button onclick="startGame()"> wired to a top-level function startGame() with no error the model could see. Wrapping is now SKIPPED when the body exposes a global (window.x = …) or declares a name referenced by an inline on*= handler; self-contained blocks are still isolated. Covered by tests/test_coding_executor_accretion.py.
  • Spec max_tokens 8192 → 16384. This is the one call that emits a COMPLETE file as embedded JSON content; at 8192 (~24-32 KB) it was 12× below the 400 KB a file may be, and a reasoning model spends part of the budget in reasoning_content before the JSON starts — so a new full file + think preamble could truncate into invalid JSON.
  • Existing-file excerpt 800/400 → 2000/1000, block cap 12 KB → 40 KB. An 800/400 head+tail of a grown index.html hid the middle, where an edits find/replace anchor often lives, so the model guessed an anchor that byte-failed. With a 65 K-token context the prompt budget easily affords real structural visibility.
  • Oversized full content now FAILS, not silently sliced. A file over MAX_CONTENT_CHARS used to be content[:cap] — a hard cut mid-code that wrote broken output. It now returns a "split across multiple files" failure, mirroring the append path's "never truncate" stance. Covered by test_oversized_content_fails_loudly_not_silently_truncated.
  • Single-command fallback max_tokens 1024 → 4096 (project_advancer.default_code_generator) — that command "may invoke python3 -c", so it can carry a whole inline program that 1024 tokens truncated mid-script.

Empty upstream responses ≠ "no file spec" (2026-06-22)

A fully empty completion (content=0 reasoning=0) is upstream contention/overload, not the model failing to produce a spec — feedback retries can't fix it. Previously each empty response consumed a full feedback-retry, so a 2-task autoadvance hammered the server with 8 instant empty calls and then mislabeled the tasks "model produced no file spec", dumping the agent into a long manual repair. Now _generate_build_spec returns (spec, was_empty); on an empty response the executor backs off (_EMPTY_BACKOFF_S) and retries, capped at MAX_EMPTY_RETRIES (2) separately from the normal MAX_ATTEMPTS, then fails with an honest "LLM returned empty responses (Nx) — likely upstream contention, not a code problem; retry later". Covered by test_empty_response_fails_with_honest_contention_message and test_empty_then_valid_recovers_after_backoff.

Fail loud, not shallow

If the spec is empty, a write is refused/rejected, or verify fails, the build returns ok=False. advance_once._finalize_coding then marks the task FAILED, which stops the batch loop so the user can take the hard task themselves — the opposite of silently marking it done. On success it registers the produced files as deliverable artifacts (so cleanup keeps them) and appends the ledger note. Covered by tests/test_coding_executor_accretion.py.

Wiring

Passed as coding_executor=build_coding_task into advance_once / advance_many from both the manage_projects autoadvance tool path and the idle autoadvancer (agent.py phase 2.95). It is NOT an open-ended agent loop — one spec call, N writes, one verify — so per-tick context stays bounded however many tasks the batch runs.

Background lane (2026-06-22). build_coding_task takes is_background, threaded into the spec-generation chat_completion. The IDLE autoadvancer sets it True (along with its classifier + code-generator calls) so its ~8K-token spec call goes through the client's background lane — waiting for foreground to clear and capped concurrency — and a user who starts typing mid-build isn't stuck behind it. The user-initiated manage_projects autoadvance tool leaves it False (foreground): the user is actively waiting on that batch. Covered by test_spec_call_is_foreground_by_default / test_spec_call_uses_background_when_requested.

Tests

tests/test_coding_executor.py — happy path (write + verify), verify-failure, write-rejection, empty spec, no-verify, markdown-fenced JSON, unavailable executor, and the advance_once integration (DONE + artifacts + ledger on success, FAILED on a broken build), plus the classification fix that routes file/build leaves here.

Post-build gates (2026-07-08)

Two gates run after the spec's own verify succeeds, both feeding the build-retry loop on failure (chess-session post-mortem: a build passed every mechanical check while violating the user's core constraint and shipping three crash-on-first-touch bugs):

  • Smoke gate (core/build_gates.smoke_gate): py_compile every written .py; when a Flask app is detected, sweep its routes with test_client (no server process): parameterless GET routes must not 5xx, POST routes get an empty JSON body and may 4xx but must not 5xx. The generated script self-bounds with SIGALRM(45s) so an import that starts a blocking server can't wedge the sandbox exec.
  • Constraint gate (core/build_gates.constraint_gate): one background LLM audit of the files the task produced (files_from_specs) against the project's stored constraints — judgement-based, unlike the evidence-based DONE-gate in tools/projects.py. A confirmed violation becomes retry feedback quoting the constraint + evidence. The same gate guards the interactive manage_projects task_update→DONE path (refusal with an explicit CONSTRAINT-OVERRIDE: escape hatch for user-approved exceptions).

Both gates fail open on infrastructure errors — a broken gate must never block builds. Covered by tests/test_build_gates.py (including an end-to-end run of the real smoke script against a Flask app with a 500-ing route) and tests/test_probe_before_hypothesis.py (wiring pins).
