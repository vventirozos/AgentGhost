core / coding_executor.py

The real per-task coding executor for autonomous batches: build one leaf for real — write files, verify — instead of marking shallow work done.

Why

When advance_many runs a batch ("proceed with all remaining tasks"), each coding tick used to generate a SINGLE shell command. That produced theatrical completion: a leaf marked DONE having written nothing — observed live, where "create hello.txt" was classified as research, ran a web_search, and was reported complete with no file on disk. This module makes a coding tick actually build.

What it does

build_coding_task(context, description, *, tool_runner, ledger="") is bounded by design — the task tree already decomposes the work, so this just builds ONE leaf well:

  1. Spec — one LLM call returns a JSON object: files (each a path + FULL content), an optional verify shell command, a one-line summary, and a one-line ledger fact. The project's design ledger AND its current files (existing_files, gathered by project_advancer._gather_project_files) are fed in, with an instruction to EXTEND those files rather than recreate them — so a single-file app accretes across tasks instead of each task overwriting the last.
  2. Write — each file via the file_system tool (operation="write"), capped at MAX_FILES (8) / MAX_CONTENT_CHARS (60k) per file.
  3. Verify — runs the verify command via execute; a failure (per project_advancer._looks_like_failure) fails the build.
  4. Returns a CodingResult (ok, summary, files, ledger_note, detail).

Don't clobber prior work (2026-06-20)

A later failure: on a single-file app each task REGENERATED the whole file with write and OVERWROTE the previous task — three tasks "completed" left a 2.7KB shell with neither the File Explorer nor the Snake game, and frontend tasks (no shell verify) had no gate to catch it. Three defences:

  • Non-regression guard — a full-content rewrite of an existing file is REFUSED if the new content is materially smaller (<85%) or drops the file's existing structural identifiers (HTML ids, function/class names, CSS selectors). It must be a SUPERSET.
  • Edit, don't clobber — a file entry may carry edits ([{"find":…,"replace":…}], {"after":…}, or {"before":"",…}) applied via file_system operation="replace", so a feature is ADDED to an existing file without re-sending it. A non-matching edit fails the task.
  • Append (the easy incremental primitive, 2026-06-20) — a file entry may carry "append":""; the executor writes old + new. It's a strict superset (can never regress), and the model emits ONLY the new snippet. This was added because re-emitting a whole ~8KB file choked the small model on task 2 of a live single-file build — it returned an EMPTY spec twice and the task failed. With append/edits the model only produces the delta. Existing files are now shown to the model as compact head+tail EXCERPTS (not full content), and the prompt steers hard toward append/edits over full content.

No per-task render (correction). An earlier version headless-rendered each HTML task and failed it on an uncaught JS exception. That was wrong for incremental builds: no intermediate task produces a complete working page — the shell exists before the apps, an app before the next — so rendering each tick failed legitimate in-progress work. Observed live: the Core-shell task built real files (a 799B index.html + script.js + 7.7KB style.css), then a render of the app-less shell "crashed" on a not-yet-defined reference, the retry returned an empty spec, and a good task was marked FAILED. The render gate was removed; the non-regression guard is the real protection, and a shell verify still hard-gates runnable code. Whole-app rendering belongs to a final verification task, not to every tick.

Retry with feedback. A refused write or a failed verify is regenerated ONCE with the exact failure reason fed back, then the task fails. Single-file projects (detected from the goal in advance_once) pass single_file=True to strengthen the "grow the one file" steer.

Fail loud, not shallow

If the spec is empty, a write is refused/rejected, or verify fails, the build returns ok=False. advance_once._finalize_coding then marks the task FAILED, which stops the batch loop so the user can take the hard task themselves — the opposite of silently marking it done. On success it registers the produced files as deliverable artifacts (so cleanup keeps them) and appends the ledger note. Covered by tests/test_coding_executor_accretion.py.

Wiring

Passed as coding_executor=build_coding_task into advance_once / advance_many from both the manage_projects autoadvance tool path and the idle autoadvancer (agent.py phase 2.95). It is NOT an open-ended agent loop — one spec call, N writes, one verify — so per-tick context stays bounded however many tasks the batch runs.

Tests

tests/test_coding_executor.py — happy path (write + verify), verify-failure, write-rejection, empty spec, no-verify, markdown-fenced JSON, unavailable executor, and the advance_once integration (DONE + artifacts + ledger on success, FAILED on a broken build), plus the classification fix that routes file/build leaves here.
