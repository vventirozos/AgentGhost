core / agent.py

The reasoning loop. Drives streaming chat completion, parses tool-call XML, dispatches tools, and re-injects results.

Purpose

GhostAgent is the orchestrator that the FastAPI route hands a chat payload to. It owns the turn loop: it asks the upstream LLM for a streaming completion, watches the stream for <tool_call> XML blocks, parses arguments, runs the tool against the registry, captures output, and folds the result back into the conversation as a tool message. Sampling temperature, top-p, and the size of the "thinking" budget are all derived dynamically from a query classifier.

Thinking budgets

Set via core/prompts.py and selected per turn:

  • tight — ≤5 sentences, no brainstorming. Default for short answers.
  • extended — ~15 sentences for debugging / SQL / algorithm work. Forbids drafting code inside the think block.
  • selfplay — ≤6 bullet points; cheap fast loop used in dream / self-play synthetic exercises.

Thinking-loop guards relaxed for a strong model (2026-06-22)

Several reasoning guards were tuned for a weaker model that looped easily, at an explicitly "acceptable false-positive risk." For a capable reasoning model those false positives — guillotining legitimate long derivations — outweigh the benefit, so they were loosened (the genuine-runaway backstops remain):

  • Hard length-abort 64 K → 200 K chars. Between 32 K (where loop-checking starts) and 200 K, the stream is aborted ONLY when _detect_thinking_loop actually fires — so a hard algorithm/proof/debug problem that legitimately needs >16 K thinking tokens can finish. 200 K is the pathological-runaway ceiling.
  • Loop detector (300/150/2) → (500/200/3). Now needs ~600 chars of genuine repetition (3× a 200-char window) before firing, vs the aggressive ~300 chars (2×150) that could trip on reasoning restating a constraint or invariant. The tool-call-collapse probe + the 200 K ceiling stay as fast backstops.
  • Thinking-cap force-stop 2 → 3 events — one more grounding nudge before aborting the attempt, now that detection is conservative.
  • Cross-turn repetition Jaccard 0.7 → 0.85 — focused iterative work (refining the same function, debugging the same test) opens consecutive turns with ~0.7 vocabulary overlap; only ~0.85+ is a true restart-the-same-derivation loop.

Covered by tests/test_think_enumeration_loop.py, tests/test_cross_turn_repetition.py, tests/test_smart_problem_solving.py.

Streaming loop: per-token cost bounded (2026-07-07)

The streaming consume loop did O(buffer) work on every token — over a long thinking stream that is O(n²) CPU plus GBs of transient string allocation on the single event loop (a plausible contributor to the RSS ratchet, since malloc_trim is Linux-only and a no-op on this Darwin host). Four sites, all bounded with zero change to detection behavior:

  • Stop-marker scan. The </think / <tool_call check lowercased the entire accumulated buffer per chunk. It now scans only a len(token)+16-char tail (_tail_has_stop_marker) — a marker straddles at most one chunk boundary, so the tail is sufficient, and the display latches stop_printing on first sight anyway.
  • Tool-call-collapse probe. _detect_tool_call_loop ran two full-buffer re.findall scans per content chunk; TOOL_CALL_LOOP_PROBE_EVERY existed but was never consulted. Now gated on a next_tool_probe cadence (every 200 chars).
  • N-gram loop detector. Once thinking passed 32 K, _detect_thinking_loop (an O(buffer) buf.count(tail) over up to 64 K chars) ran per token — the boundary branch ignored the next_loop_probe cadence that the periodic probe already used. The per-token boundary branch is removed; the detector runs once per THINKING_LOOP_PROBE_EVERY (500) chars at all sizes, so early loops are still caught within 500 chars and the 32–64 K "extend if diverse" window falls out implicitly. Only the extended hard cap keeps an explicit per-token length check (cheap).
  • Final-generation scrub. The XML-scrub re.sub re-ran over the whole answer per content chunk. It's now skipped entirely until a < actually appears (_scrub_seen_lt) — the common tag-free final answer emits raw tokens in O(1)/chunk; the full sub only runs once real XML is present.

Covered by tests/test_streaming_dequadratic.py; the unchanged detector functions and cap-escalation behavior stay green under tests/test_thinking_loop_guards.py and tests/test_thinking_cap_escalation.py.

Multi-turn reply smoothing (2026-07-17)

The turn loop appends every iteration's visible text (ui_content) to final_ai_content, so a multi-tool fix used to arrive as stacked working narration ("Let me fix both:", "Now add the resize logic in openWindow:") with the summary stated twice — once before a verify/restart step, once after (operator report: the WebOS minesweeper turn). The system prompt's execution-mode rule already forbids narrating during tool use (and was sharpened: no summaries until the whole task is done, never restate one); the model doesn't reliably comply, so core/reply_smoothing.smooth_reply() cleans the DELIVERED reply deterministically in _finalize_and_return, right after the adjacent-duplicate collapse and BEFORE the verifier gate (the verdict judges the text the user actually receives). The live stream is untouched — narration keeps serving as progress; only the persisted/delivered reply (session store, Slack, trajectories, banners) is smoothed.

  • Gate: only turns with ≥2 non-synthetic tool runs — conversational and single-tool replies are never rewritten.
  • Shape 1 — connective narration: a non-final paragraph ≤300 chars opening with a progress connective (Let me / Now / Good, / Okay / Next, / I'll / I need to / Time to …). Fences and list items are exempt.
  • Shape 2 — superseded summary group: a ':'-lead-in plus its list items whose content words are ≥55% Jaccard-covered by a later group of comparable size (≥60%) — the pre-verification summary; the last statement wins. Groups containing fences are never dropped.
  • Fail-open: fenced blocks are split-atomic (blank lines inside a fence don't split it), the final block is never dropped, and an over-scrub that would empty the reply returns the original. Idempotent.

Tests: tests/test_reply_smoothing.py (12 — including the verbatim WebOS transcript as the regression fixture). Note: the prompt edit changes the byte-stable system prefix, so the first boot after deploy re-prefills the main-node cache once.

Grammar-constrained tool calls (2026-07-17)

core/tool_grammar.py builds a llama.cpp GBNF grammar from the registry's tool schemas (memoized on the same frozen signature as the XML-schema serialiser) and the payload attaches it LAZILY on every non-final tool turn: grammar + grammar_lazy + a PATTERN-type trigger on \n<tool_call> (newline-anchored so a tag QUOTED inline in thinking can't arm it — the think-strip incident shape). Dormant through reasoning and prose; from the trigger on, only valid framing, known tool names, known parameter names, enum-legal values and numeric-legal integers can decode — the malformed-call strike class dies at the sampler. Validated live before wiring: full 39-tool grammar compiled server-side; action='bogus_action' was coerced to a legal enum (value rules are deliberately TIGHT — whitespace padding gave the sampler an escape hatch and it stalled on tabs). String values use a </parameter> prefix-exclusion chain so file content with markup keeps working; required-ness stays with the tool layer. Paired with --no-native-tools in the launcher: the upstream parser (two corruption shapes on VALID output: merged multi-tool args 2026-07-05/AF, value duplication AF/43) is retired and the agent's own robust XML parser consumes the output. The XML schema block returns to the prompt (~7K tokens, absorbed by the stable-prefix prefill).

Status: OPT-IN (GHOST_TOOL_GRAMMAR=1), default OFF — first production request (req 9f1c3173) exposed a thinking-turn blind spot the probes missed (they ran with /no_think): the model DRAFTS the literal <tool_call> inside its reasoning block, the pattern trigger armed mid-think, and generation died at <tool on every retry — a hard-fail loop. Re-enable only after the trigger is upgraded to llama.cpp's think-aware PATTERN_FULL semantics (capture-group marks where constrained text begins, anchored past </think>) and validated on a THINKING turn. --no-native-tools stays: the XML parser path was the pre-native default and is verified live post-restart. Tests: tests/test_tool_grammar.py.

Truncated-answer auto-continuation

The streaming consume loop records the last non-null finish_reason seen on the upstream stream. When a text turn (no <tool_call> / <function> in flight) ends with finish_reason == "length", the answer was cut off at the model's token cap — shipping it would hand the user a mid-sentence reply, which the Verifier correctly REFUTES (truncated output, and any explicit question left unanswered).

Instead of surfacing the fragment, handle_chat transparently continues the generation: it replays the request with the partial answer as a prior assistant turn plus a "your reply was cut off — continue from exactly where it stopped, don't repeat anything" instruction, and appends the result to full_content. This repeats until the upstream returns a non-length finish or MAX_TRUNCATION_CONTINUATIONS (3) round-trips are spent — the bound prevents a model that mis-reports length on every reply from looping forever. The continuation payload drops tools/tool_choice so it can only produce prose, and strips any <think> prelude the continuation emits before appending. Turns that produced a tool call are unaffected — those go through the parse/retry path, not this user-facing-prose path.

Turn flow

Trivial-request fast path

A bare greeting or acknowledgement ("hi", "thanks", "ok cool") cannot benefit from memory hydration, tool schemas, the planner gate, or KV-cache rebuild — yet it used to traverse the entire turn loop. handle_chat intercepts these via _handle_trivial_chat, which issues a single tool-less LLM call against a pared-down "lite" system prompt (persona + user profile only) and returns the reply directly, collapsing ~2.5 s of work into one short request.

The gate is deliberately strict: _is_strict_trivial_chat matches the message against a small allowlist of greeting/ack tokens and phrases, and hard-rejects anything containing ?, /, file extensions, or other action signals. Missing a real greeting costs one extra request; intercepting a tool-bearing request would break the response — so the gate errs toward falling through.

Reasoning-model handling. The upstream is a Qwen-style reasoning model. Left unconstrained on a 256-token budget it spends the whole budget inside a <think> block and returns an empty content field, which previously made every greeting decline and fall through to the slow path. The fast path therefore suppresses chain-of-thought on both switches the model honours — the chat_template_kwargs={"enable_thinking": False} flag and the portable /no_think soft-switch appended to the user message — and uses the NON_THINKING_GENERAL_PARAMS sampling profile.

Extraction is robust to either response shape: it reads message.content and folds in any separate message.reasoning_content field, then strips <think>…</think> (closed or unclosed) before deciding whether a visible reply survived. Only a genuinely empty result, a missing LLM client, or an upstream error makes the path decline and fall through to the full turn loop.

Concurrency

Single-task asyncio. The streaming generator is consumed inside the FastAPI request task; tool execution is awaited inline (the tool itself may use threads / subprocess). The agent does not spawn parallel turns — concurrency lives in MemoryBus (parallel hydration) and LLMClient (parallel pool dispatch).

Turns are serialized — agent_semaphore = Semaphore(1) (2026-07-07, was 10). Per-turn state lives on the singleton context: last_user_content (read mid-turn by the intent gates in tools.projects and tools.memory) and current_project_id (drives project-scoped sandbox writes). Two concurrent turns therefore clobber each other's scope — the concrete hazard being an APScheduler cron job that fires handle_chat mid-user-turn and switches the active project, landing the user's uploads/writes in the wrong sandbox with plausible-looking logs. There is one upstream llama slot, so concurrent turns mostly just interleave waits anyway; serializing costs almost nothing and closes the whole state-corruption class. Complementarily, a scheduled job defers (skips this firing, re-fires next tick) when a user request is live — tools.tasks.should_defer_scheduled_task — so an idle autonomous tick never makes a user wait behind it. (If real per-turn concurrency is ever needed, the correct fix is moving those two fields to contextvars, not raising the semaphore back up.) Covered by tests/test_turn_serialization.py.

Context compaction (per-turn payload)

Before every upstream LLM call, handle_chat assembles a transient block (tool header + persona + memory hydration + dynamic state) that rides at the top of the last user message. Two predicates decide how aggressively to trim that block before serialisation:

Byte-stable advertised tool set (2026-07-07, #7)

The acquired-skill semantic router filters the ADVERTISED tool set by query. Feeding it the per-turn query — which becomes the planner's thought mid-request — byte-changed the tool header block every turn, invalidating the upstream prompt-prefix KV cache and forcing a full re-prefill each turn. _RequestState.stable_tool_query() now pins the routing query to the request's FIRST substantive query, so the advertised set (and thus the tool header) stays byte-identical across the request's turns while topic relevance is preserved (the first user message sets the request's topic). This is the prerequisite that lets the GHOST_PIN_TOOL_SCHEMAS KV pin (IMPROVEMENTS.md #6) actually hold — with an unstable tool set the pin had nothing constant to anchor. Covered by tests/test_stable_tool_query.py.

Self-play control hooks

handle_chat carries two small but load-bearing hooks that coordinate with the self-play subsystem:

  • Continuous-loop interrupt — at the top of every turn, if context.selfplay_loop_task is active, the incoming user message flips context.selfplay_loop_stop. The background loop checks the event at each cycle boundary and during cool-off, so the loop pauses cleanly before answering the user.
  • Perfect-It skip during simulation — the Perfect-It follow-up LLM call (which writes optimisation suggestions into SkillMemory) is gated on getattr(ctx.skill_memory, "is_read_only", False) is True. During self-play the isolated context's ReadOnlySkillMemory class carries that marker; the block is skipped entirely so the ~15 s follow-up LLM call and its misleading "Saved optimization strategy to playbook" log don't fire on no-op writes.

Checklist nudge & read-only tool exemption

The agent's meta-task compliance nudge ("you mentioned learning but didn't call learn_skill") is intentionally suppressed when the turn's tools include a read-only surface tool — list_lessons, recall, manage_skills. Otherwise a user query like "what have you learned today?" would loop for 5+ turns as the nudge pushes the agent to write a redundant skill it just deduplicates. The exemption keeps "surface a lesson" distinct from "persist a new one".

Validator crash detector

When the self-play solver gets a validator rejection, handle_chat's attempt-loop runs a circuit-breaker check for validator-frame crashes — tracebacks whose innermost frame is .validator.py and that don't mention solution.py. Classified as crashes: SyntaxError, IndentationError, ImportError, ModuleNotFoundError, NameError, ValueError, TypeError, KeyError, IndexError, AttributeError. Matched crashes abort the cycle after attempt 1 instead of burning the remaining attempts on an unwinnable challenge. See dream_cycle for the upstream gates that catch most of these earlier.

Repeated-identical-failure breaker (2026-06-07)

The structural strike count (execution_failure_count, cap 6) decays by 1 on any successful tool turn. That decay let a specific loop run unbounded: an agent that oscillates "same failing read → auto sandbox-listing (succeeds) → same failing read …" never accumulates strikes, because each interleaved success cancels the prior failure. Observed in production — a fresh project whose stale workspace narrative kept asserting a prior project's index.html existed; the model re-read the missing path every other turn and burned all 40 turns without progress.

The fix tracks failures by a (tool | whitespace-normalised error) signature (_note_repeated_failure). When the SAME structural failure recurs ≥3 times in a request, the loop is recognised: the success-decay is frozen (persistent_failure_seen) so the recurring failure now accumulates to the strike cap and the loop-breaker fires, and the model is told ONCE — clearly — to stop retrying the identical action, that the live tool result and sandbox listing are authoritative over any stale context / memory / DYNAMIC SYSTEM STATE hint, and to pick a different action (create the missing file, or choose an existing one). Paired with the file_system loop-breaking "not found" message that gives the model the same exit at the tool layer. Covered by tests/test_not_found_loop_breaker.py.

The freeze is reversible (2026-06-11): 3 consecutive clean tool successes unfreeze the decay — a genuine pivot produces them, while the fail→auto-list→fail oscillation never can. Signature counts are kept, so the same failure recurring re-freezes on its very next occurrence. Previously the freeze was permanent for the request, which trapped multi-step tasks whose step 1 looped before steps 2..n succeeded. Same audit also made every tool-argument parse-failure site log a WARNING naming the tool (no more silent except: pass dispatching empty args), and bounded turn-loop journal appends with a 10s timeout (_journal_append_safe) so hung journal I/O can no longer park the event loop until the watchdog's opaque activity-timeout. Covered by tests/test_core_loop_defect_fixes.py.

This loop-detection state — the failure/action signature dicts, the freeze flag + warned set, and the clean-success counter — is encapsulated in StrikeLedger (core.strikes), so the turn loop holds one ledger instead of five interacting locals. The raw strike counters (execution_failure_count / transient_failure_count) stay in the loop because they are cross-cutting (strike caps, mid-loop caps, System-3 pivot triggers). The ledger's behaviour is covered by tests/test_strike_ledger.py.

2026-06-12 log-analysis fixes (SQL-migration session postmortem)

Four production defects surfaced by one chat session (a Postgres int→bigint migration request) were fixed together:

  • Conversation-scoped project binding. handle_chat now calls tools.projects.reconcile_conversation(context, conversation_fingerprint(messages)) at request start, BEFORE the ambient sandbox listing or any file op. A project activated in one conversation no longer captures other conversations' file writes into its workspace — see tools.projects for the binding model. Covered by tests/test_project_conversation_scope.py.
  • Coding-intent follow-up inheritance. The inline intent heuristic was extracted to module-level detect_coding_intent(lc, messages) → (has_coding_intent, is_meta_task). New rule: a correction of the prior assistant turn ("that is wrong…", "you are not using…") whose previous assistant message contains a fenced code block inherits coding intent. Previously such corrections carried no coding/DBA keyword, fell to the conversational profile (temp 1.0) and silently dropped the Ghost Specialist persona mid-task. Why-questions and acknowledgements deliberately do NOT inherit. Covered by tests/test_coding_intent_inheritance.py.
  • Hippocampus truncation. run_smart_memory_task's consolidation call was capped at max_tokens=1024 and regularly died mid-JSON (observed: output ended at "profile_update":). The cap is now 3072 with response_format={"type":"json_object"}, and the extraction opts in to truncation repair (below). Covered by tests/test_hippocampus_truncation.py.
  • Truncated-JSON salvage (opt-in). extract_json_from_text(text, repair_truncated=False) gains _repair_truncated_json: close an open string, complete a dangling "key": with null, append missing closers, parse; on failure chop back to the previous comma/opener and retry (bounded). Opt-in only — background extraction paths (memory consolidation, graph triplets) set it; tool-call ARGUMENT parsing must NOT, since repairing a truncated {"path": …, "content": … would execute a tool with half its payload instead of triggering the parse-error retry loop. Default-off behaviour is pinned by tests/test_syntax_healer.py.

The same postmortem fixed numeric false positives in the redactors (LIMIT 1000000 became <REDACTED_PHONE> in stored trajectories) — see selfhood / distill.redact, covered by tests/test_redact_numeric_false_positives.py — and deconflicted the specialist prompt's NATIVE TOOLS FIRST rule with rule 8b (return the source, not a file write) in core.prompts.

2026-06-12 churn-loop fixes (delete/create oscillation postmortem)

A test session where the model re-executed "delete it and make something else" on every turn (six hard deletes, five of self-created projects, nothing built in 20 turns) produced two more fixes:

  • Request-start project snapshot. handle_chat records context.request_start_project_id (the project active when the user's message arrived) right after conversation reconcile. The project tool's delete-eligibility gate uses it: a bare "delete it" can only target that project or one the user's message names — a project created mid-request is refused with an instruction to build it instead. This converts the success-oscillation loop (which the failure-keyed StrikeLedger structurally cannot see) into a loud tool error on its second iteration.
  • Contextual <think>-fragment filter. The streaming display filter dropped any token whose stripped form was "think" or ">" (meant to hide fragmented <think> tags split across SSE chunks) — and the model emits the literal word " think" as its own token, so every rendered thought lost it ("Let me think about…" displayed as "Let me about…"). _is_think_tag_fragment(token, accumulated) now suppresses those tokens only when the stream immediately before them ends with the matching tag opener (</</ for think, <think/</think for >). The filter was and remains display-only: reasoning_content/full_content always accumulated the raw token, so model context was never corrupted. Covered by tests/test_think_stream_fragment.py.

Process operability — RSS watchdog, health, resolved config (2026-07-07)

The single long-running asyncio process has no external supervisor, and a known leak grows RSS ~270 MB→2 GB over hours on a box already at ~94 % RAM (shared with the 35B llama-server), so an OOM can take out the LLM, not just the agent. Three operability additions:

  • RSS self-defense (_rss_watchdog_check, run once per 60 s at the top of biological_watchdog, before the tick's memory_system guard so a degraded boot still gets it). Opt-in via GHOST_MAX_RSS_MB (default 0 = off). When RSS crosses the limit AND no foreground work is in flight, it logs one WARNING, stops the sandbox cleanly, and restarts in place with os.execv(sys.executable, ["-m","ghost_agent.main", *argv[1:]]) — preserving flags + the exported GHOST_HOME, no external supervisor. Mid-request RSS is never interrupted (it defers to the next idle tick).
  • GET /api/health (api.routes, registered ABOVE the /{path:path} catch-all, behind the API key): rss_mb, rss_limit_mb, uptime_s, asyncio_tasks, foreground_requests/tasks, biological_watchdog_alive, memory_system_loaded, scheduler_jobs, and the resolved config. Explicit-constraint surfacing + post-write steer (2026-07-02)

From the chess-game incident: the model read "don't come up with some random AI, YOU will play against me", acknowledged it in thinking, and had rationalised it away within four seconds — the constraint appeared once, in turn 1, and nothing ever re-showed it. Three hooks in handle_chat, all driven by the deterministic extractor in utils/constraints.py:

  • Request-start extraction. Right after context.last_user_content is set, extract_constraints() runs once; any hits are rendered as an EXPLICIT USER CONSTRAINTS (CURRENT REQUEST) block into the DYNAMIC SYSTEM STATE of every turn of the request (works with or without an active project; project-stored constraints additionally arrive via the briefing).
  • One-shot post-write steer. After the FIRST successful file_system mutation of a constrained request, a SYSTEM ALERT (constraint check) user-message is injected (same mechanism as the strike/no-progress steers): re-read what you wrote, verify no constraint is violated — a coded stand-in for a role the user assigned to YOU counts as a violation — fix now. This lands at the earliest moment a violation is still cheap to fix (the chess monolith was generated in one 183 s turn; the constraint died during generation). ⛓️ Constraint Check log line; steer fires at most once per request.
  • Verifier replay. _active_constraint_note() prepends the active project's stored constraints to the request view passed to verify_claim/verify_code_output (prepended, not appended — the call sites truncate context to 1000 chars). The current message is often just "proceed"; the binding clauses live on the project record from an earlier message. Constraint satisfaction is already the verifier prompt's highest-priority rule, so a violation now REFUTES the final answer and triggers the existing auto-repair loop.

Covered by tests/test_constraints_extraction.py and tests/test_project_constraints.py::TestActiveConstraintNote.

Project-constraint replay + participant engine guard (2026-07-05)

Second chess regression: with all three 07-02 hooks on disk, the agent still shipped a random.choice(legal_moves) opponent. The post-mortem found two holes: the request that wrote the engine said just "proceed." — extract_constraints() sees only the current message, so the post-write steer never armed (the stored clauses reached only the verifier path, which was dead under --no-verifier) — and in a later request the steer did fire and the model misread it as being about an unrelated bug. Steers ask; only a guard refuses. Two additions in handle_chat:

  • Stored-constraint replay into the request. _merge_project_constraints() runs right after the conversation⇄project reconciliation (so the binding is trustworthy) and merges the bound project's stored constraints (via _active_project_constraints() / _project_constraints_for(), which _active_constraint_note() now shares) into _request_constraints — case-insensitively de-duplicated against the message's own clauses. Effect: every turn's dynamic-state block AND the one-shot post-write steer now see the binding clauses on a bare "proceed", independent of the verifier flag. ⛓️ Constraints — replayed N stored project constraint(s) log line. (Extended same day, request C5 post-mortem): the merge additionally replays constraints from any project referenced by path in the message text (projects/<id>/…, up to 3 ids) — a traceback pasted into a FRESH conversation has no binding, so the original merge found nothing even though the traceback named the project on every line, and the mock-server misdiagnosis ran unchecked.
  • Participant-mode engine guard (deterministic, pre-dispatch). When the request's constraint set contains a participant clause (has_participant_constraint), every file_system write/replace body is scanned by find_engine_patterns() (utils/constraints.py): minimax/negamax, alpha-beta, piece-square tables, stockfish anywhere; random.*/Math.random only when the content is recognisably move-generation code (legal_moves, import chess, chess.js, …) so benign randomness elsewhere stays writable. A hit ABORTS the call before dispatch (same mechanism as the empty-content guard: 🛑 Local Guard log line, synthetic tool-error, strike) and the SYSTEM BLOCK message names the violation and embeds PARTICIPANT_STEER, i.e. the two sanctioned designs including POST /api/game/move. Both content and replace_with are scanned — the replace→write auto-promotion can make either the full file body. Narrow by design: a false negative still meets the steer/verifier layers; a false positive would block a legitimate write outright.

Covered by tests/test_participant_constraint_steer.py (TestFindEnginePatterns, TestParticipantWriteViolation, TestProjectConstraintHelpers, TestProjectConstraintReplayInTurn — the last one replays the literal "proceed." session shape end-to-end through handle_chat).

Finish-line honesty guards (2026-07-14)

Two ways a turn's ending could silently misrepresent what happened, both observed live during the code-correction failure-chain session (journal 2026-07-14j):

  • Trailing-promise guard. The conversational-filler guard only fires when a tool name is mentioned, so a mid-repair turn that finalized with "…That's what's causing the error. Let me fix it." shipped as the answer — the fix never ran and the user believed it had. _ends_with_action_promise() now checks the reply's last sentence (≤120 chars) for an imminent-action opener (let me — with let me know excluded — I'll, I will, I am going to, gonna); on a match after a tool-running turn, a once-per-request steer tells the model to either emit the tool_call and DO it now, or rewrite the ending to state plainly the action was NOT done. Latched like the notify guard: never loops, pure conversation (has_run_tools false) is exempt.
  • Dropped-mutation honesty note. When force_final_response drops queued tool_calls (defense against post-terminal-tool hallucinated calls), a dropped mutating call — file_system, execute, manage_services, manage_projects, database (observed 2026-07-12: file_system dropped at the finish line, twice) — meant the reply implied work that never ran. _dropped_mutation_note() appends "⚠ … has NOT been applied yet" to the final reply. Terminal-tool re-drops (self_play etc.) stay silent — dropping those is the guard's purpose.

Covered by tests/test_pending_action_and_inline_rescue.py.
