Postcondition matching

Token overlap ≥ 60 % case-insensitive, with stop words filtered ({the, and, for, not, are, has}). A postcondition prefixed with HUMAN_GATE: is special-cased by project_safety to flip the task to NEEDS_USER regardless of tool output — and since 2026-07-20 it is excluded from the token-overlap check entirely (a gate can never be satisfied, or failed, by result-text similarity; see Plan-gate hardening below).

ProjectPlan

Same shape as TaskTree but persisted via memory.projects's SQLite store. ProjectPlan.update_status() wraps TaskTree.update_status() with a write-through to the database.

Durable fallback edges. The write-through diff tracks parent_id, alternatives and depends_on in addition to the status fields. The failure cascade (_check_parent_failure) consumes an alternative by popping it off the parent and re-parenting it onto the failed parent — mutations that are silent on status. A status-only diff never persisted them, so on the next hydrate the alternative was no longer linked and the fallback edge was lost. Tracking those keys makes the alternatives/fallback path durable across sessions.

Prerequisite-gated scheduling. next_ready_leaf() skips any leaf whose depends_on prerequisites are not all DONE (unknown/stale dep ids are treated as satisfied so a bad reference can't deadlock the plan). decompose(task_id, subtasks, sequential=True) chains each subtask to the previous via depends_on, so the autoadvancer runs them strictly in order; sequential=False (default) leaves every subtask independently eligible, the historical behaviour.

Plan-gate hardening (2026-07-20)

Three-stack review fixes — the plan gate is live now, so malformed planner output has to be survivable, human gates have to be enforceable, and a triggered replan has to actually write.

  • load_from_json coerces malformed planner fields. Everything it reads is LLM JSON: a bare-string postconditions/alternatives/depends_on is wrapped into a one-item list via _coerce_str_list (previously _check_postconditions iterated the string and graded it per-character), a null/non-string status or dependency_type falls back to PENDING/ALL instead of crashing on .upper() (which aborted the whole planning step), and null/blank list entries are dropped. _check_postconditions coerces again at read time — belt to load_from_json's braces, for nodes built outside it.
  • HUMAN_GATE: forces NEEDS_USER, never FAILED — and is never textually scored. Closing a gated task as DONE parks it at NEEDS_USER instead of grading the gate against result text, which either cascaded a spurious FAILED up the tree (the result never echoes the gate) or, conversely, let echoed text "satisfy" a human approval. TaskTree.update_status / ProjectPlan.update_status gained human_approved: bool = False: an explicit human_approved=True, or a close on a task already parked NEEDS_USER (the user has been asked and the close path is now being taken), is the approval and proceeds to the normal postcondition gate. HUMAN_GATE: postconditions are skipped by the token-overlap check entirely.
  • ProjectPlan.request_revision(task_id, failure_reason) -> bool PERSISTS. The metacog ReplanBridge prefers plan.request_revision and only falls back to plan.tree.request_revision — but its plan_getter builds a fresh ProjectPlan per event, so triggered replans mutated a THROWAWAY hydration that was never written back, while the replans_tried/replans_ok counters still reported "revised". The new override cascades through the tree, then writes every node the revision touched (the task itself plus a possible parent BLOCKED→IN_PROGRESS unblock) through the store — the trigger → ReplanBridge → replan pipeline is now durable.
  • Dangling alternatives are skipped, not blocking. The alternatives queue can hold ids the planner referenced but never created; _check_parent_failure now consumes dangling ids (with a warning) until a valid one resolves — blocking on the FIRST dangling id stranded valid alternatives still queued behind it.
Tests: tests/test_planning_gate_fixes.py.

Concurrency

Synchronous tree mutation. Persistence happens via the ProjectStore RLock; in-memory TaskTree is per-request only.

Load-bearing planning & router gating

Two changes make the plan and the router genuinely steer the turn rather than just advise.

  • Root postcondition gating. New method TaskTree.root_postconditions_unsatisfied(response_text) checks the FINAL agent response against the ROOT task's declared postconditions — gating the answer itself, not just internal task bookkeeping. At the verifier site, handle_chat rebuilds a TaskTree from the persisted current_plan_json and appends a visible "Plan check" note when the response misses a declared postcondition.
  • Router-gated reasoning depth. When the complexity router is confident a request is easy (label "easy", not escalated, confidence ≥ 0.75) the expensive strategic-planner LLM round-trip is skipped — the single most expensive step of the turn — making the router decision load-bearing on cost.
  • Startup bootstrap-train. The router ships untrained, and a model was previously only ever produced by an idle retrain — so a busy server or a benchmark (which never idles) stayed escalate-all forever. At boot, when no checkpoint exists yet, bootstrap_router() (in router/trainer.py) trains the classifier ONCE from the existing trajectory log, reusing the same labeling + feature extraction + fit as the idle path. It's gated on at least BOOTSTRAP_MIN_SAMPLES (50) labeled, multi-class samples and caps how much of the log it reads, so it can't stall startup. Any failure (too little data, single class, malformed/missing log, fit divergence) logs a warning and falls back to the safe escalate-all pass-through — boot never crashes.
Tests: covered by tests/test_planning_root_postconditions.py and tests/test_router_bootstrap.py.
