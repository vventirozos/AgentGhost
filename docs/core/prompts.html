Project working memory — DESIGN LEDGER & DONE SO FAR (2026-06-20)

The dominant cost on long projects was a fresh turn re-deriving the project's shape — re-listing the sandbox, re-reading files — because the briefing only carried the task tree and the last few events, never what had been built or how it worked (observed live: one "proceed" turn spent ~96 s reconstructing state before doing any work). The briefing now carries two compact, durable layers so the next turn inherits that knowledge instead:

  • DESIGN LEDGER — durable facts the agent records (file layout, key function/API names, conventions) via manage_projects action=ledger ledger="…" or a ledger="…" on a DONE task_update. Stored in project metadata, bounded (ProjectStore.LEDGER_MAX_LINES/MAX_CHARS), dedup'd against an identical trailing line, and written WITHOUT logging an event (_write_metadata) so it doesn't spam RECENT EVENTS. The briefing surfaces the last _LEDGER_BRIEFING_LINES (20).
  • DONE SO FAR — the max_done_tasks most-recently-completed tasks plus the one-line result_summary the agent wrote on completion (the DONE-gate already forces a result). "What exists and where", auto-derived — no new agent behaviour required.

The ONE TASK rule now tells the agent to read these before re-reading files, and to record durable decisions in the ledger as it goes. Covered by tests/test_project_working_memory.py.

Project pacing — ONE TASK AT A TIME (2026-06-20)

The project briefing previously opened with a PARALLEL TASK EXECUTION — HARD RULE that told the agent to scan the whole task tree and stack every finishable task into one reply ("project-of-N tasks completes in ~2-3 turns"). On large projects that floods the context window, and it steamrolled any user instruction about how to advance. It is replaced by a ONE TASK AT A TIME rule:

  • Advance exactly the NEXT TASK, one per turn — never stack several tasks' work into a single reply.
  • After create / task_decompose, STOP and present the plan; do not begin executing.
  • Wait for an explicit user go-ahead ("proceed", "next task", "continue") and honor the pacing/ordering the user asks for.
  • Each go-ahead does one task, closes just that id with task_update … status=DONE (plus its deliverables), reports, and stops.

The create and task_decompose actions of manage_projects reinforce this by returning a stop-and-await agent_instruction. The idle autoadvancer (agent.py phase 2.95) already runs a single advance_once tick per cooldown, so autonomous advancement is one-task-at-a-time too. Covered by tests/test_project_pacing.py.

Tool-orchestration routing rules (SYSTEM_PROMPT)

A block of MANDATORY TRIGGERS in SYSTEM_PROMPT maps user intents to specific tool calls so the LLM doesn't improvise. Current rules include:

  • SLEEP / REST → dream_mode
  • SELF-PLAY (one-shot) → self_play — for "practise", "train", "do self-play" (single cycle).
  • SELF-PLAY (continuous) → self_play_loop — for "continuously", "in a loop", "back to back", "until I tell you to stop". stop_self_play on explicit user stop while running.
  • LESSONS SURFACE → list_lessons — for "what have you learned today/so far/this week", "show me your lessons", "show me the lesson playbook". Picks scope ∈ {today, week, all, self_play_only} from phrasing. A LESSON is a mistake-and-fix the agent has internalized — distinct from a SKILL (a tool).
  • SKILLS SURFACE → manage_skills(action="list") — for "show me your skills", "list your skills", "what skills do you have", "show me your custom skills". A SKILL is a TOOL or set of tools, NOT a lesson. The historical "skill playbook" wording was the conflation that misrouted skill queries to list_lessons; both the tool description and this rule now disambiguate.
  • KNOWLEDGE & RAG → recall first.
  • WEB FACTS → web_search first; fact_check / deep_research only for complex verification.

Specialist rule deconfliction (2026-06-12)

SPECIALIST_SYSTEM_PROMPT carried two imperatives a small model resolved the wrong way: rule 8b ("give me the SQL" → the reply IS the code; do NOT save it with file_system) and the execution rule NATIVE TOOLS FIRST (use file_system write for native file tasks). Observed live: a "Just give me the SQL, no explanations" request produced an unrequested migration.sql write — twice. NATIVE TOOLS FIRST is now explicitly subordinated to 8b: it picks WHICH tool once tools are warranted, it does not make tool use warranted.

Self-play challenge-gen hardening (SYNTHETIC_CHALLENGE_PROMPT)

The challenge-generation system prompt carries explicit rules to prevent LLM-generated validators from being unwinnable:

  • Float formatting — always convert both sides to float() with tolerance before comparing; never compare round() output with f-string output directly.
  • Unit suffixes — if an expected field carries a %, $, ms, or comma-thousand separator, strip it before float()/int(). Includes a concrete WRONG/RIGHT example of the float('60.00%') bug.
  • One convention per field — either string equality with the suffix kept, or numeric tolerance with the suffix stripped. Never mix.
  • Mental self-test — the generator is told to trace the validator against a hypothetical solution.py that echoes its own expected output; every parse/comparison line must succeed.

When FrontierTracker reports all clusters saturated, the dream loop additionally injects a CURRICULUM DIVERSITY REQUIREMENT block telling the generator to pick from {concurrency, algo, regex_parse, sql, bash} and forbidding another data-analysis / CSV-groupby shape.
