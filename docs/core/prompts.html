Project working memory — DESIGN LEDGER & DONE SO FAR (2026-06-20)

The dominant cost on long projects was a fresh turn re-deriving the project's shape — re-listing the sandbox, re-reading files — because the briefing only carried the task tree and the last few events, never what had been built or how it worked (observed live: one "proceed" turn spent ~96 s reconstructing state before doing any work). The briefing now carries two compact, durable layers so the next turn inherits that knowledge instead:

  • DESIGN LEDGER — durable facts the agent records (file layout, key function/API names, conventions) via manage_projects action=ledger ledger="…" or a ledger="…" on a DONE task_update. Stored in project metadata, bounded (ProjectStore.LEDGER_MAX_LINES/MAX_CHARS), dedup'd against an identical trailing line, and written WITHOUT logging an event (_write_metadata) so it doesn't spam RECENT EVENTS. The briefing surfaces the last _LEDGER_BRIEFING_LINES (20).
  • DONE SO FAR — the max_done_tasks most-recently-completed tasks plus the one-line result_summary the agent wrote on completion (the DONE-gate already forces a result). "What exists and where", auto-derived — no new agent behaviour required.

The ONE TASK rule now tells the agent to read these before re-reading files, and to record durable decisions in the ledger as it goes. Covered by tests/test_project_working_memory.py.

Project working memory II — RECENT WORK LOG & STUCK TASKS (2026-07-18)

The 2026-06-20 layers cover work done inside tasks. The 2026-07-17 overnight session exposed the complement: once the game project rolled to DONE, an entire evening of debugging (6 requests) plus the next morning's root-cause fix wrote nothing to the store — the model has no open task to task_update, so the voluntary write-back path never fires — and the 06:20 turn re-read every file and re-derived the bug from scratch. Two new briefing sections close the gap:

  • RECENT WORK LOG — the last max_work_logs (5) automatic per-request records written by the finalize chain (work_log events: request head · files touched · tool counts · verifier-aware outcome · head of the final answer). Deterministic — requires no model cooperation — and rendered for DONE projects too, which is precisely when it is the only record. Header tells the model to trust it before re-reading files.
  • STUCK TASKS — FAILED/BLOCKED tasks with their failure_reason (≤max_stuck_tasks, 4). Previously invisible: the OPEN filter excludes both statuses and DONE SO FAR only shows DONE, so "work pending because it failed" never reached the model at all.

Write side and defect-reopening live in core.agent / memory.projects. Covered by tests/test_project_work_log.py.

Project working memory III — DELIVERABLES, RETROSPECTIVE, DREAM DIGEST (2026-07-18)

Same session, the remaining write-only records gained read sides in the briefing:

  • DELIVERABLES — the deduped file manifest from kind='file' artifacts (ProjectStore.list_deliverables), ≤max_deliverables (12) paths comma-joined + "(+N more)", with a pointer to manage_projects action=artifact_list for per-task detail. Previously these were registered on every DONE task purely as the cleanup keep-set — the model re-listed the sandbox to learn what its own project had built.
  • RETROSPECTIVE — rendered only for terminal-status projects (DONE/FAILED/ARCHIVED): TaskTree.generate_retrospective()'s summary line, up to 3 what_failed entries with reasons, and the measured-effort total from the (now written) actual_cost column. Computed lazily from task rows — nothing persisted, always current. The generator previously had no caller at all.
  • LAST DREAM DIGEST — one line from the newest dream_digest event ("what did I do last night"), which was reachable only via a manual event_log call — i.e. never.

Covered by tests/test_project_readside_plumbing.py.

Project pacing — ONE TASK AT A TIME (2026-06-20)

The project briefing previously opened with a PARALLEL TASK EXECUTION — HARD RULE that told the agent to scan the whole task tree and stack every finishable task into one reply ("project-of-N tasks completes in ~2-3 turns"). On large projects that floods the context window, and it steamrolled any user instruction about how to advance. It is replaced by a ONE TASK AT A TIME rule:

  • Advance exactly the NEXT TASK, one per turn — never stack several tasks' work into a single reply.
  • After create / task_decompose, STOP and present the plan; do not begin executing.
  • Wait for an explicit user go-ahead ("proceed", "next task", "continue") and honor the pacing/ordering the user asks for.
  • Each go-ahead does one task, closes just that id with task_update … status=DONE (plus its deliverables), reports, and stops.

The create and task_decompose actions of manage_projects reinforce this by returning a stop-and-await agent_instruction. The idle autoadvancer (agent.py phase 2.95) already runs a single advance_once tick per cooldown, so autonomous advancement is one-task-at-a-time too. Covered by tests/test_project_pacing.py.

ONE TASK AT A TIME — now ENFORCED, not just prompted (2026-06-22)

The rule above was advisory text only — nothing in the interactive turn loop enforced it. Observed live: "start task 1" ran the model through 8 tasks in a single request. Two things defeated the prompt: (1) build_project_briefing is rebuilt and re-injected on every loop iteration, so the instant a task closed, the briefing re-advertised NEXT TASK: [the next one] — the model read that as its new assignment; (2) the turn only ends when the model volunteers a tool-free answer, and nothing forced it to after a task closed. The model even wrote the correct plan ("mark Task 1 DONE and wait for direction") then overrode itself on the next iteration.

Two code-level teeth, both in agent.py's turn loop, gated on the user's go-ahead NOT being a batch ask (project_advancer.classify_advance_intent → one vs n/all):

  • Hard stop on close. When a tool result shows a task actually closed to DONE (_manage_projects_closed_a_task reads the store read-back {"updated":[{"status":"DONE"}]} — a DONE-gated/held update does NOT count), the loop sets force_final_response, so the turn converges to a summary and waits.
  • Stop re-dangling the next task. Once a task closed this request, the briefing is rebuilt with suppress_next_task=True: the NEXT TASK pointer is replaced by a hard "you completed a task this turn — STOP and wait" note (open tasks are still listed for awareness). Removes the contextual pull that overrode the rule.
  • Honest stop message (2026-06-22). A model that still emits a tool_call after the turn was finalized has it stripped by the text-only stream scrubber, which then emits a fallback. When the cause is this gate (a task just closed), _scrub_fallback_message returns "✅ Task complete — say proceed / next to continue…" instead of the generic "I prepared a tool call… please rephrase", which had misled the user into re-issuing a request that already succeeded; the operator log drops from ⚠ Stream Scrub Fallback to an informational One Task / Turn line. The generic message still applies to a genuine planner/model routing mismatch.

Net: "start task 1" / "proceed" / "next" advance exactly one task and stop; "do the next 3" / "finish the project" are batch intents that still run multiple (routed to the autoadvance tool). Covered by tests/test_one_task_per_turn.py and the suppress_next_task cases in tests/test_project_pacing.py.

Reliability around "start task N" (2026-06-22)

Two more fixes that made single-go-ahead project flow robust, both observed failing live in one session:

  • Enumerated batches. "proceed with task 3 and 4" now parses as a 2-task batch (classify_advance_intent), so the one-task gate doesn't stop after task 3 and leave task 4 half-done.
  • Conversation-scope escape hatch. The chat API ships no stable conversation id, so a project bound to one conversation got parked for later turns whose first-message fingerprint differed — the agent then spun (manage_projects get ×3 → loop breaker → scrub fallback). tools.projects.reconcile_conversation now keeps/activates a project when THIS request's user message names it by title or id (_message_references_project), overriding the mismatch.
  • Host-path scrub. A recalled memory / scratchpad note / workspace narrative could carry a host-absolute sandbox path (/Users/…/Data/AI/Data/sandbox/projects/<id>/…); the model used it in cd and ENOENT'd inside the container (host ≠ /workspace), burning strikes. agent._scrub_host_sandbox_paths rewrites such prefixes to the container /workspace form in the injected DYNAMIC SYSTEM STATE.

Covered by tests/test_one_task_per_turn.py, tests/test_project_conversation_scope.py, and tests/test_sandbox_path_scrub.py.

Tool-orchestration routing rules (SYSTEM_PROMPT)

A block of MANDATORY TRIGGERS in SYSTEM_PROMPT maps user intents to specific tool calls so the LLM doesn't improvise. Current rules include:

  • SLEEP / REST → dream_mode
  • SELF-PLAY (one-shot) → self_play — for "practise", "train", "do self-play" (single cycle).
  • SELF-PLAY (continuous) → self_play_loop — for "continuously", "in a loop", "back to back", "until I tell you to stop". stop_self_play on explicit user stop while running.
  • LESSONS SURFACE → list_lessons — for "what have you learned today/so far/this week", "show me your lessons", "show me the lesson playbook". Picks scope ∈ {today, week, all, self_play_only} from phrasing. A LESSON is a mistake-and-fix the agent has internalized — distinct from a SKILL (a tool).
  • SKILLS SURFACE → manage_skills(action="list") — for "show me your skills", "list your skills", "what skills do you have", "show me your custom skills". A SKILL is a TOOL or set of tools, NOT a lesson. The historical "skill playbook" wording was the conflation that misrouted skill queries to list_lessons; both the tool description and this rule now disambiguate.
  • KNOWLEDGE & RAG → recall first.
  • WEB FACTS → web_search first; fact_check / deep_research only for complex verification.
  • SYNTHESIS OVER LARGE DATA FILES → execute a digest script, not whole-file file_system(read) (see below).

Report-path steering — synthesise data files via a digest script (2026-06-23)

The per-batch read budget stops the context-overflow crash by refusing later whole-file reads once a turn's budget is spent — but the model only avoids the dead-end if it reaches for a script BEFORE hitting the limit. Observed live: asked to write report.md synthesising three experiment JSON files, the agent read them in parallel and overflowed a 131 K window at 136 K tokens. A new SYSTEM_PROMPT bullet (SYNTHESIS OVER LARGE DATA FILES) makes the right move explicit: when writing a report/summary/analysis that synthesises result files (JSON/CSV/logs), do NOT pull them in with file_system(operation="read") — execute a short script that loads them, computes only the digest needed (key stats, aggregates, top-N, anomalies; ≤~50 printed lines), and write the report from that digest. For a quick peek use operation="inspect"; to scan one genuinely large file use operation="read_chunked" / "search". The rule names the overflow it prevents and says to act before the read limit, so the model connects it to the budget refusal it would otherwise hit. Covered by tests/test_report_synthesis_steering.py.

Specialist rule deconfliction (2026-06-12)

SPECIALIST_SYSTEM_PROMPT carried two imperatives a small model resolved the wrong way: rule 8b ("give me the SQL" → the reply IS the code; do NOT save it with file_system) and the execution rule NATIVE TOOLS FIRST (use file_system write for native file tasks). Observed live: a "Just give me the SQL, no explanations" request produced an unrequested migration.sql write — twice. NATIVE TOOLS FIRST is now explicitly subordinated to 8b: it picks WHICH tool once tools are warranted, it does not make tool use warranted.

Self-play challenge-gen hardening (SYNTHETIC_CHALLENGE_PROMPT)

The challenge-generation system prompt carries explicit rules to prevent LLM-generated validators from being unwinnable:

  • Float formatting — always convert both sides to float() with tolerance before comparing; never compare round() output with f-string output directly.
  • Unit suffixes — if an expected field carries a %, $, ms, or comma-thousand separator, strip it before float()/int(). Includes a concrete WRONG/RIGHT example of the float('60.00%') bug.
  • One convention per field — either string equality with the suffix kept, or numeric tolerance with the suffix stripped. Never mix.
  • Mental self-test — the generator is told to trace the validator against a hypothetical solution.py that echoes its own expected output; every parse/comparison line must succeed.

When FrontierTracker reports all clusters saturated, the dream loop additionally injects a CURRICULUM DIVERSITY REQUIREMENT block telling the generator to pick from {concurrency, algo, regex_parse, sql, bash} and forbidding another data-analysis / CSV-groupby shape.

PARTICIPANT MODE — "you play against me" (2026-07-02)

Observed live: asked for a chess game "where YOU will play against me — don't come up with some random AI", the agent built a single-file HTML page with an embedded minimax engine. The only deliverable pattern it knew for "game" was artifact with an AI inside (the PROJECT MODE GATING rule even lists "a one-file game" as its canonical one-shot example); nothing framed the agent itself as a runtime participant, so "the AI opponent will be me" rationalised into "I'll code a small engine". The TOOL ORCHESTRATION section now has a PARTICIPANT MODE (YOU ARE A PLAYER) trigger, directly under PROJECT MODE GATING:

  • Coding a minimax/random/any algorithmic opponent when the user said YOU should play is named a FAILURE, in those words.
  • The prescribed shape: authoritative game state in a workspace file (chess/state.json, FEN + history) · validate each user move with python-chess via execute (pip install python-chess in the sandbox if missing) · the agent CHOOSES its own reply move by reasoning, one chat turn per move — it may list legal moves programmatically but never delegates the choice.
  • An HTML board is optional and render-only: it may display state and collect the user's move; move-selection for the agent's side happens at runtime. A UI that wants live agent moves calls POST /api/game/move (see api.routes).
  • Sandbox-loopback clause (added after a second live violation, 2026-07-02 evening): diagnosing the user's "Could not reach API (Load failed)" from inside the code sandbox, the agent probed 127.0.0.1:8000 (the container's loopback, not the host's), concluded no server existed, and wrote its own api_server.py — with random.choice(legal_moves), the exact forbidden engine, reframed as a "mock". The rule now states: the agent's own API is NOT reachable from inside the sandbox at 127.0.0.1:8000; unreachability there is never evidence the server is down; a mock that picks moves IS the forbidden engine regardless of name; and game UIs must be opened served by the agent (/api/download/projects/<id>/…/index.html, same origin as the API) — a file:// open cannot fetch the API.

Briefing: EXPLICIT USER CONSTRAINTS section

build_project_briefing renders the project's metadata.constraints (captured by manage_projects) as an EXPLICIT USER CONSTRAINTS (MUST HOLD…) block above the design ledger, every turn — a constraint stated once and never re-surfaced lost to reinforced skill priors within a single reasoning pass. Covered by tests/test_project_constraints.py.
