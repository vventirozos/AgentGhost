Durable sessions & real turn cancellation

core/sessions.py · core/turns.py — 2026-07-11. The two things that stood between this agent and being a daily driver: conversations that survive a device switch, and a way out of a wedged turn that doesn't involve restarting the process.

1. Durable server-side conversations

The gap. History was entirely CLIENT-carried — the web UI's localStorage, a Slack thread, or a manual workspace/save zip. Switch device or clear the cache and the conversation was gone; each client held a different fragment, so a conversation started in Slack could not be continued in the web UI. Ironic, given cross-session MEMORY is this agent's proven strength (Track B: 98% recall vs 0% control) — the memory substrate persisted while the conversations themselves did not.

The shape. One JSON file per session under $GHOST_HOME/system/sessions/: {id, title, created_at, updated_at, messages}. Sessions live entirely in the API layer — the chat route merges stored history into the request before handle_chat and appends the turn after it returns — so the agent's turn logic is untouched by this feature.

The merge contract (merge_history)

Pass session_id in the /api/chat body and the SERVER becomes the source of truth for history. It tolerates both client styles, which is what makes adoption incremental:

  • a thin client sends only the new message → stored + incoming;
  • a fat client (today's web UI) replays the whole conversation → the incoming list already carries the stored history as its prefix, so it is used as-is rather than appended. This is the check that stops a fat client from doubling the conversation on every turn.

Everything past the stored prefix is what this turn adds, and that slice (plus the assistant's reply) is what gets appended — after the turn completes, so a failed turn never leaves a dangling user message with no reply. Omit session_id and behaviour is exactly as before (fully client-carried), so every existing client keeps working untouched.

Bounds & safety

  • Session ids are opaque tokens ([A-Za-z0-9_-]{1,64}) — a path-traversal guard, since the id names a file.
  • 200 sessions (least-recently-updated evicted); 400 messages per session (leading system messages + the newest tail kept, mirroring the agent's own 500-message cap).
  • Malformed messages are dropped rather than persisted (a corrupt session would otherwise poison every future turn that loads it); tool-call plumbing (tool_calls / tool_call_id / name) IS preserved so a resumed conversation's tool pairing stays intact.
  • Every method degrades (None / [] / False) rather than raising into a request; a corrupt file reads as absent.
  • The streamed-final-generation path bypasses the finalize tail, so the route reconstructs what the user saw from the SSE deltas (_sse_delta_text) — without it a session would record the user's message and no reply.

Endpoints

GET /api/sessionssummaries (id / title / timestamps / message_count), newest first — no bodies, so the list stays cheap
POST /api/sessionscreate (optional title); a client may equally just POST /api/chat with a fresh session_id
GET /api/sessions/{id}full conversation — this is how a session is RESUMED on a different client than the one that started it
DELETE /api/sessions/{id}delete

1b. Hydration tier (2026-07-14)

Stored sessions are now a retrieval source, not just replay state: SessionStore.search_messages(query, limit=5) keyword-searches the most-recently-updated _SEARCH_MAX_SESSIONS (50) sessions and feeds MemoryBus._fetch_session as the fifth hydration tier (see bus). Bounded by design: per-file mtime cache (unchanged sessions are never re-parsed), a 2s summaries memo (one glob scan per hydration, not per sub-query), 240-char snippets, and a ≥2-distinct-term floor for multi-term queries so one shared common word can't surface an unrelated conversation. Fail-safe — returns [] on any error. Tests: tests/test_session_hydration_tier.py.

2. Real turn cancellation

The gap. Turns are globally serialized (agent_semaphore = Semaphore(1), #22), so ONE wedged turn blocks the web UI, Slack, and the autonomous project-advance ticks at once — and there was no way out: the interface's /api/chat/cancel only cancels the proxy's buffered stream, so the agent kept working and kept the lock. The only real remedy was restarting the process.

The registry. core.turns.TurnRegistry gives every in-flight turn an identity (its req_id), a handle on its asyncio task, and a cooperative cancel flag. A turn registers itself before acquiring the semaphore, so a request QUEUED behind a runaway turn is cancellable too.

GET /api/turnswhat is running and for how long, plus anything queued behind it — this is how you see WHY the agent appears unresponsive
POST /api/turn/cancel{request_id?, hard?} — request_id defaults to the running turn

Cooperative vs hard

  • Cooperative (default). The flag is set; the turn loop notices at its next boundary, raises TurnCancelled, and handle_chat catches it and returns whatever partial work exists (a clean stop, not a crash). Exiting through there unwinds the async with self.agent_semaphore, so the global turn lock is released — which is the entire point.
  • Hard (hard=true). The asyncio task is cancelled outright. This is the guaranteed release for a turn wedged inside a long upstream call that never reaches a boundary.
  • A queued turn is always hard-cancelled — it has no partial work and no boundary to reach, and killing it is strictly better than letting it start.

req_id collision handling (2026-07-14 bug hunt)

req_id can be a CLIENT-supplied X-Request-ID (the route reads the header), so two overlapping requests can carry the same id. The old registry overwrote the live entry unconditionally and popped by key, so /api/turns hid the first turn, cancel("X") flagged the SECOND turn (cancelling B killed running A), and A's finally evicted B's entry. Fixed: register() never clobbers a live entry — it uniquifies the new key (X#2, …) and handle_chat adopts the effective id; unregister() is IDENTITY-CHECKED (pops only if the stored object is this turn). Known residual (§4C): the STREAMING final-generation tail returns its generator inside the semaphore block, so the tail runs after the lock releases and after unregister — uncancellable and invisible to /api/turns (the single upstream slot is still protected by foreground-marking). Tests: tests/test_turn_registry_collision.py.
