core / strikes.py — StrikeLedger

Request-scoped loop-detection accounting for the turn loop: bundles the failure/action signature tracking and the decay-freeze decision the reasoning loop used to juggle as bare locals.

Why it exists

The reasoning loop in core.agent historically tracked its loop-safety state as five interacting locals — two signature dicts (repeated_failure_sigs, repeated_action_sigs), a persistent_failure_seen freeze flag, a persistent_warned_sigs set, and a consecutive_clean_successes counter — mutated across several hundred lines. StrikeLedger (2026-06-11) pulls that cohesive bundle into one object so the loop holds a single ledger.

Scope is deliberately narrow. The ledger owns the signature tracking and the freeze decision. The raw strike counters (execution_failure_count / transient_failure_count) stay in the loop because they are cross-cutting — strike caps, the mid-loop cap, and System-3 pivot triggers all read them, so folding them in too would touch dozens of sites for no clarity gain.

Pure helpers

note_repeated_failure(sigs, fname, error)Signature = tool | whitespace-normalised error head. Mutates the caller-owned dict; returns (sig, count, is_persistent). Stable across byte-identical repeats, distinct per tool/error.
action_result_fingerprint(result)Whitespace-normalised, bounded SHA1 of a tool result. Digits kept, so a result whose content genuinely changed looks different and does NOT count as a no-progress repeat.

These stay standalone (not just methods) because existing tests import them directly; core.agent re-exports them under their old private names (_note_repeated_failure, etc.).

No-progress remedy: read/write tools

Tripping the breaker is only half the story — the other half is the remedy. For an ordinary re-observation loop (re-screenshot, re-read the same file with nothing left to do) the first-trip remedy sets force_final_response: drop the toolset and finish the turn as plain text. For a tool that both reads and writes through one dispatch name that remedy is destructive — the loop is the agent re-reading to orient itself before performing the write it was actually asked to do, and forcing a text-only turn bars that pending mutation forever.

Regression fixed: a request to reconfigure a composed skill looped on manage_composed_skills(action="list"), the breaker force-finalised at 3×, and the model's follow-up action="define" was then scrubbed by the final-generation stream guard — the change silently never landed. For is_readwrite_loop_exempt tools the breaker now still steers the model off the wasteful re-read but keeps tools available so the write can complete; the ≥5 hard stop remains the backstop. The companion fix on the dispatch side adds manage_composed_skills's define/approve/delete actions to the is_mutating set so a real mutation is never mistaken for a no-progress read.

StrikeLedger API

MethodBehaviour
note_failure(fname, error)Record a structural failure. Freezes decay once the same failure recurs 3×. Returns (sig, count, is_persistent, is_first_warning) — is_first_warning is True exactly once per signature so the loop emits the "stop retrying" steer a single time.
reset_clean_streak()Break the consecutive-clean-success streak. Called on ANY failure (transient or structural) — decay only unfreezes after a run of UNINTERRUPTED clean successes.
note_clean_success()Record one clean, non-mutating success. Returns True if this unfroze a detected loop (3 consecutive clean results = a genuine pivot). Signature counts are kept, so the same failure re-freezes on recurrence.
note_action(fname, target, result_fp)Record a successful action's fingerprint; returns (sig, count, tripped).
note_world_changed()Clear every accumulated no-progress observation. Called by the dispatch pipeline when a file_system mutation SUCCEEDS — see below.
decay_frozen (property)True while success-decay should be suppressed. The loop gates its execution_failure_count decrement on not decay_frozen.

World-changed reset (2026-07-18)

A successful file_system mutation now clears the no-progress observation ledger (note_world_changed()) and the once-per-signature steer set. Rationale: re-observing after an edit is verification of the change, not thrash — even when the observation comes back byte-identical, because the fix may target a deeper page state than the one being re-observed (a landing page is unchanged while the level-select flow behind it was repaired).

Regression fixed: in the 2026-07-17 overnight session every fix-verify turn (requests 26/3B/72/1E/91) ended with the post-fix browser navigate killed by "repeated 2x with no new info", and in request 3B the verifier gate then REFUTED the turn precisely because the evidence only showed the pre-fix page load — the auto-repair round's evidence-gathering navigate was itself cut by the breaker. Two guards fighting. The reset is scoped to file_system mutations on purpose: execute probes stay exempt, so a probe loop that keeps returning identical results still trips the breaker (the steer message explicitly tells the model to probe — resetting on probes would nullify the trip).

A companion guard in the dispatch pipeline (core.agent) collapses byte-identical read-only calls within one batch: the tool executes once and the result is fanned out to every duplicate tool_call_id. The no-progress ledger still counts each repeat, so detection is unchanged — only the wasted executions go away (observed: one 2026-07-17 batch carried 144 identical file_system reads; the breaker aborted correctly, but only after the sandbox ran all 144). Mutating duplicates are never collapsed — a repeated write can be intentional.

The reversible freeze

Freezing the success-decay lets the recurring failure accumulate to the strike cap (otherwise an interleaved success — e.g. the auto sandbox-listing after a failed read — cancels the strike and the cap never fires). Before 2026-06-11 the freeze was permanent for the request, which trapped multi-step tasks whose step 1 looped before steps 2..n succeeded. Now note_clean_success unfreezes after UNFREEZE_AFTER_CLEAN_SUCCESSES (3) uninterrupted clean results — a real pivot produces them, the fail→auto-list→fail oscillation never can.

The no-progress / not-found breakers it feeds are covered by tests/test_no_progress_loop_breaker.py and tests/test_not_found_loop_breaker.py; batch dedup and the dispatch-side reset by tests/test_dispatch_pipeline_extraction.py.
