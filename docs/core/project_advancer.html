core / project_advancer.py

Single autoadvance tick for long-running projects. Classifies the next task, runs one constrained step, and bookkeeps budget / runtime / contradictions.

API

FunctionPurpose
classify_task(description) -> "needs_user" | "coding" | "research"Keyword-driven classifier (line 85).
async advance_once(context, project_id, tool_runner, llm_classifier) -> AdvanceResultOne full advancement tick — returns the result of running one task (line 126).

Classification keywords

  • needs_user: decide · approve · sign off · delete production.
  • coding: implement · build · fix · debug · deploy.
  • research (default): research · investigate · compare · analyze.

Tick flow

  1. Load project plan; find next READY/PENDING leaf.
  2. Classify task type.
  3. Mark IN_PROGRESS.
  4. If needs_user: flip to NEEDS_USER, log event, return.
  5. Pick tool: coding → execute with code from the injected code_generator (or, if none, degrade to web_search — never the old inert "# Autoadvance stub" comment that marked the task DONE having run nothing); research → web_search.
  6. Run tool through the injected runner.
  7. Store result as a project artifact.
  8. Stricter completion: if the tool ran but returned an error or no usable output (_looks_like_failure — empty, or a first line starting ERROR: / Traceback), flip the task to FAILED and log autoadvance_failed instead of marking it DONE. The old path recorded DONE on any output, so a failed web_search still counted as progress (theatrical completion). The no-runner classify-only path still marks DONE, as before.
  9. Check human-gate postconditions (project_safety) → flip to NEEDS_USER if any fire.
  10. Auto-research persistence (research tasks): when a research-classified task ran a real web_search and got usable output, that output is turned into a durable, summarised brief in the project workspace (research/<slug>.md) and indexed via project_research — no second search. The synthesised summary becomes the task's result_summary and the brief path is added to the step event + result. Best-effort: never breaks the tick.
  11. Detect contradictions with DONE siblings.
  12. Mark DONE; increment steps_used; record runtime + tool_calls; stamp last_autoadvance_ts (drives round-robin project selection — see phase 2.95).

Budget enforcement

_get_budget() reads steps_cap / steps_used from project metadata; _increment_budget() bumps after each tick. Checked via check_budget() across steps, runtime_seconds, and tool_calls.

AdvanceResult

AdvanceResult(ok: bool, task_id: str, classification: str, summary: str, artifact_id: Optional[str])

Async; tool_runner, llm_classifier, and code_generator are optional callbacks so the advancer can be tested without a real agent.

Autonomous idle advancement (phase 2.95)

advance_once was historically only reachable via the manage_projects tool or the HTTP /advance route — no idle phase ever called it, so the "self-advancing research loop" only advanced when a human or the model explicitly poked it. Under --autoadvance-idle, biological-watchdog phase 2.95 picks one ACTIVE project and runs a single tick during the 15–60 min idle window (30-min cooldown), building a registry-backed tool_runner, an LLM code_generator so coding tasks produce and run real code, and an LLM llm_classifier so task routing no longer depends on the brittle keyword buckets (a task like "design the schema" matches no keyword and used to default to research/web_search). Strictly bounded — one project, one tick per cooldown, on the same hard per-project budgets + human gates — and off by default.

Round-robin fairness. The project picked each tick is the one advanced least recently (by metadata.last_autoadvance_ts; never-advanced projects sort first). Previously the phase took actives[0], but list_projects orders by updated_at DESC and advancing a project bumps its updated_at, so the same project re-selected itself every tick and all other ACTIVE projects starved. This is what turns the dormant loop into genuine, fair background autonomy. Covered by tests/test_project_advancer.py and tests/test_project_mgmt_fixes.py.

Telling the user (autonomous-progress digest)

Background advancement updates the project event log (autoadvance_step, autoadvance_needs_user, human_gate_triggered) but on its own only logs to the operator stream — the chatting user was never told. core/project_digest.py closes that loop: on the next chat turn, core.agent.handle_chat builds a concise "while you were away" header from the events since the user last saw a digest and prepends it to the reply, flagging the tasks that now need their input. It is watermark-gated on the monotonic project-event id (so each batch shows exactly once; the first run baselines silently with no historical backlog), fail-safe (a digest error never breaks a turn), and triggered by background progress from any source — the idle phase, the tool, or the HTTP route. Covered by tests/test_project_digest.py.
