core / project_advancer.py

Single autoadvance tick for long-running projects. Classifies the next task, runs one constrained step, and bookkeeps budget / runtime / contradictions.

Pacing — single turn vs. bounded batch (2026-06-20)

A user saying "proceed" / "next" is handled as a full agent turn (the agent writes files, runs, verifies — highest quality for one task). The batch case — "do the next 3 tasks", "proceed with all remaining tasks", "finish the project" — instead routes (via manage_projects action=autoadvance count=<N|"all">) to advance_many, a bounded loop of advance_once ticks. classify_advance_intent maps the phrasing to a count (ambiguous → 1, the safe default).

Why a loop instead of one big turn: each tick claims one leaf, runs one constrained step, and checkpoints status+result to the store, so per-tick context stays bounded no matter how many tasks run — the antidote to one chat turn grinding the whole tree and flooding the context window. The loop stops at the FIRST of: project_done · count_reached · needs_user (a human-gate task) · budget_or_inactive (step budget exhausted) · failed (a task FAILED, when stop_on_fail) · hard_cap (ADVANCE_ALL_HARD_CAP=40, a backstop for an uncapped project).

Coding executor (2026-06-20). A tick's coding path is the real build_coding_task: spec → write files → verify, finalized by _finalize_coding (success registers the produced files as deliverables + appends a ledger note + marks DONE; a failed build marks FAILED, which stops the batch loop for the user). This replaced the original single-command path that produced theatrical completion — a leaf marked DONE having written nothing (observed live: "create hello.txt" ran a web_search). The single-command default_code_generator remains only as a fallback when the executor is unavailable/crashes.

Classification (hardened 2026-06-20 after a live failure). In a batch over a browser-OS project, 9 of 10 build leaves ("File Explorer app", "Snake game") were classified as research and web-searched, marked DONE with no code — the small LLM classifier reliably mislabels build work as research. Fixes:

  • CODING projects bypass the LLM classifier entirely and use the deterministic keyword classifier with default="coding" — a leaf is coding work unless it carries an explicit research/needs_user verb. Only GENERAL projects consult the LLM.
  • Kind is inferred at create time (tools.projects._infer_kind): a build-flavoured title/goal (build / app / website / game / OS / single-file / …) creates a CODING project, so its leaves are treated as coding instead of defaulting to research.
  • No theatrical research. A coding leaf with no build path (no executor handled it, no command generated) now FAILS (stopping the batch loop) instead of being web-searched and marked DONE. And research briefs are no longer registered as deliverables in CODING projects (they were polluting cleanup's "kept deliverables").
  • Cumulative builds. _gather_project_files feeds the project's current workspace into the executor (existing_files) so a leaf EXTENDS prior files (single-file apps) instead of regenerating and overwriting them.

Covered by tests/test_advance_classification_fixes.py, tests/test_advance_pacing.py, and tests/test_coding_executor.py.

Strong code-file signal beats a research verb (2026-06-22). A residual hole in the keyword classifier: classify_task checks research keywords BEFORE coding ones, so a build leaf whose description also contained a research verb was still mis-routed. Observed live (project 33e23d50): the leaf "analyze training logs (analyze_results.py) — produce metrics summary… saved as results/report.md" hit analyze (research) and was web_searched into a phantom DONE, never building the script — the model later flagged it as "a phantom task… auto-researched but not actually built." Fix: when a description names a concrete source/artifact file (_STRONG_CODE_FILE_RE: .py .js .html .md .json …) AND carries a build verb (produce · build · create · write · generate · save · …), classify it coding ahead of the research check. needs_user still wins (checked first), so "publish report.md" routes to the human, and a file mentioned without a build verb ("investigate what config.yaml controls") stays research. Covered by test_classify_strong_code_file_beats_research_verb in tests/test_project_advancer.py.

Classification keywords

  • needs_user: decide · approve · sign off · delete production.
  • coding: implement · build · fix · debug · deploy.
  • research (default): research · investigate · compare · analyze.

Tick flow

  1. Load project plan; find next READY/PENDING leaf.
  2. Classify task type.
  3. Mark IN_PROGRESS.
  4. If needs_user: flip to NEEDS_USER, log event, return.
  5. Pick tool: coding → execute with code from the injected code_generator (or, if none, degrade to web_search — never the old inert "# Autoadvance stub" comment that marked the task DONE having run nothing); research → web_search.
  6. Run tool through the injected runner.
  7. Store result as a project artifact.
  8. Stricter completion: if the tool ran but returned an error or no usable output (_looks_like_failure — empty, or a first line starting ERROR: / Traceback), flip the task to FAILED and log autoadvance_failed instead of marking it DONE. The old path recorded DONE on any output, so a failed web_search still counted as progress (theatrical completion). The no-runner classify-only path still marks DONE, as before. (2026-06-22) _looks_like_failure now strips leading quote/paren/bracket punctuation before the prefix test: a transport failure surfaced as a stringified exception tuple ('error sending request for url …', '…'), whose leading (' hid the error marker, so an erroring search still marked its task DONE (observed live on project 33e23d50). Covered by test_looks_like_failure in tests/test_project_advancer.py.
  9. Check human-gate postconditions (project_safety) → flip to NEEDS_USER if any fire.
  10. Auto-research persistence (research tasks): when a research-classified task ran a real web_search and got usable output, that output is turned into a durable, summarised brief in the project workspace (research/<slug>.md) and indexed via project_research — no second search. The synthesised summary becomes the task's result_summary and the brief path is added to the step event + result. Best-effort: never breaks the tick.
  11. Detect contradictions with DONE siblings.
  12. Mark DONE; increment steps_used; record runtime + tool_calls; stamp last_autoadvance_ts (drives round-robin project selection — see phase 2.95).

Budget enforcement

_get_budget() reads steps_cap / steps_used from project metadata; _increment_budget() bumps after each tick. Checked via check_budget() across steps, runtime_seconds, and tool_calls.

AdvanceResult

Autonomous idle advancement (phase 2.95)

advance_once was historically only reachable via the manage_projects tool or the HTTP /advance route — no idle phase ever called it, so the "self-advancing research loop" only advanced when a human or the model explicitly poked it. Under --autoadvance-idle, biological-watchdog phase 2.95 picks one ACTIVE project and runs a single tick during the 15–60 min idle window (30-min cooldown), building a registry-backed tool_runner from a pinned_project_context (a proxy that pins current_project_id to the target project — idle ticks carry no conversation, so the process-global id is parked and unpinned file writes landed at the sandbox ROOT, invisible to interactive sessions on the project; observed live 2026-07-08 with TinyAI's model/train/evaluate files), an LLM code_generator so coding tasks produce and run real code, and an LLM llm_classifier so task routing no longer depends on the brittle keyword buckets (a task like "design the schema" matches no keyword and used to default to research/web_search). Strictly bounded — one project, one tick per cooldown, on the same hard per-project budgets + human gates — and off by default.

Round-robin fairness. The project picked each tick is the one advanced least recently (by metadata.last_autoadvance_ts; never-advanced projects sort first). Previously the phase took actives[0], but list_projects orders by updated_at DESC and advancing a project bumps its updated_at, so the same project re-selected itself every tick and all other ACTIVE projects starved. This is what turns the dormant loop into genuine, fair background autonomy. Covered by tests/test_project_advancer.py and tests/test_project_mgmt_fixes.py.

Workspace pinning (2026-07-08): both autoadvance tool-runner construction sites — the idle phase-2.95 tick and the interactive manage_projects autoadvance batch — now build their tools via pinned_project_context(context, project_id), so file writes always resolve to projects/<id>/ regardless of what the racy process-global current_project_id says mid-run. Covered by tests/test_autoadvance_project_scope.py. Event-stamp pinning (2026-07-09): the sandbox pin covers only context.current_project_id; workspace events (command outcomes etc.) stamp from workspace_model.current_project_id, which an idle tick never set — so autoadvance outcomes were stamped with the LAST chat turn's project. Both sites now also wrap their runs in pinned_event_project(project_id) (a task-local ContextVar override in workspace.model). Covered by tests/test_workspace_event_stamping.py.

Telling the user (autonomous-progress digest)

Background advancement updates the project event log (autoadvance_step, autoadvance_needs_user, human_gate_triggered) but on its own only logs to the operator stream — the chatting user was never told. core/project_digest.py closes that loop: on the next chat turn, core.agent.handle_chat builds a concise "while you were away" header from the events since the user last saw a digest and prepends it to the reply, flagging the tasks that now need their input. It is watermark-gated on the monotonic project-event id (so each batch shows exactly once; the first run baselines silently with no historical backlog), fail-safe (a digest error never breaks a turn), and triggered by background progress from any source — the idle phase, the tool, or the HTTP route. Covered by tests/test_project_digest.py.
