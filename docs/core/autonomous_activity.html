core / autonomous_activity.py + utils/notify.py

The agent's "mouth" (2026-07-11): a structured ledger of autonomous outcomes + an all-phase "Background activity" digest + outbound push notifications. Closes the biggest open loop of the 2026-07 feature survey — the agent did real work while idle but had no way to tell anyone.

Why

Before this module, 12 of the 13 idle phases surfaced their outcomes only as pretty_log lines on the console; the post-mortem defect queue had no surfacing at all; and scheduled turns (cron jobs that fire a full handle_chat) discarded their final content — only pass/fail reached the workspace ledger. Only project autoadvance had a user-facing digest (core.project_digest), and even that was pull-only. Blocked needs-user project tasks accumulated silently until the operator happened to open a chat.

The ledger

ActivityLog — append-only JSONL at $GHOST_HOME/system/autonomous_activity.jsonl. One record per operator-relevant autonomous outcome: {ts, phase, summary, severity, meta}. Two severities:

  • info — digest-only (dream ran, PRM refit, …).
  • notify — digest plus immediate push + the external-consumer feed (needs-user project tasks, scheduled-turn conclusions).

Fail-safe by contract: record()/read_since() never raise into a caller — a broken ledger must never break a turn or an idle phase. read_since(offset) is byte-offset watermarked, only consumes complete lines (a partially-written tail is left for the next read), skips corrupt lines, and silently re-baselines on a stale offset.

Producers

  • Idle phases — GhostAgent._record_autonomous_activity is called at each phase's completion point in _biological_tick: dream, reflection, post-mortem (when defects were queued), skills_auto (when something graduated), PRM/router/calibration (on successful hot-swap), stale open questions, self-play.
  • Project advancer — _record_needs_user_activity pushes severity=notify records for autoadvance_needs_user / human_gate_triggered outcomes (all three sites). The digest renderer excludes the project phase (DIGEST_EXCLUDED_PHASES) because the project digest already renders those — the record exists purely for push.
  • Scheduled turns — main._run_proactive_task now captures handle_chat's returned content and sinks it via record_scheduled_result() (severity=notify — a cron job's whole point is "tell me what you found"). Failures are recorded too.
  • The model itself — the notify_operator tool (tools/notify_tool.py, added later on 2026-07-11) lets the agent DELIBERATELY push one short line (severity=notify, phase agent_message) when asked to "report back in Slack" / "notify me" or when long background work concludes. Rails: 500-char clamp, 12/hour rate limit, and an honesty contract — the confirmation names only the delivery channels that are actually live (push transport configured? has the Slack consumer ever polled?) instead of claiming "sent to Slack" blindly. Delegated sub-agents can NOT reach this tool (not in their allowlist — the main agent reports; a delegate paging the operator is spam surface). Tests: tests/test_notify_operator.py (14).
    Finish-line guard (2026-07-13, req 11fe11d8): the model can PLAN the notify call in its reasoning and then finalize without making it — the user's one explicit delivery requirement silently dropped (the verifier confirms such turns: the deliverable itself is fine). The turn loop now detects an explicit notify-me ask in the user text (_user_asked_for_notification — narrow by design; negations and questions about Slack don't arm it) and, when a turn is about to ship its final response with notify_operator never called, injects ONE steer toward the call (never against a force-finalising loop-breaker; one-shot per request). Tests: tests/test_notify_finish_guard.py.

Consumers

  1. Next-turn digest — _finalize_and_return renders unseen records as a "Background activity while you were away:" header (after the project digest), watermarked at $GHOST_HOME/system/activity_digest.json; first run baselines silently. Each item is clamped to 140 chars so the whole block stays under the 1500-char bound _strip_leading_banners peels — a longer block would break the correction-fingerprint peel (the 2026-07-07 bug class). The banner-peel loop allows up to 5 stacked blocks now (correction + clarifying + project digest + activity digest + headroom).
  2. Outbound push — utils.notify.OutboundNotifier, wired as the ledger's on_notify callback in main.py. Transports: --notify-webhook (JSON POST) and/or --notify-ntfy (ntfy text POST), env fallbacks GHOST_NOTIFY_WEBHOOK/GHOST_NOTIFY_NTFY. Egress posture: loopback/RFC1918/Tailscale-CGNAT/LAN-suffix targets connect directly (the egress guard allows them); PUBLIC targets are only reached through the Tor SOCKS proxy (upgraded to socks5h:// so DNS resolves inside Tor); with mandatory-tor and no proxy, a public target is skipped outright. Best-effort: one retry per transport, failures logged and swallowed — the durable record is already in the ledger.
  3. External deliverers — GET /api/notifications/pending?consumer=NAME returns undelivered notify-severity records + a watermark; POST /api/notifications/ack advances the per-consumer watermark (stored in $GHOST_HOME/system/notify_consumers.json). A consumer that never acked is baselined to end-of-ledger (no historical replay). The Slack bot (interface/externals/slack_bot) runs a notification_poller when GHOST_NOTIFY_SLACK_CHANNEL is set — poll → chat_postMessage → ack, so a crash re-serves rather than drops.
    Wedge fixed 2026-07-13 (both halves). The pipeline was silently dead 2026-07-11→13: (a) server — read_since's limit bounds scanned lines, so a poll window full of info-severity records (dream/self-play spam) returned [] without ever reaching the notify records beyond it; pending now scans forward in 200-line chunks (≤50/poll) until it has limit notify records or EOF, and limit is a soft bound — whole chunks are kept, since truncating mid-chunk would advance the watermark past records never returned; (b) bot — the poller only acked non-empty responses, so the empty-but-advanced watermark was never persisted and every poll re-scanned the same window forever. The bot now acks every response that carries a watermark (still after delivery, so at-least-once holds) and logs delivered N notification(s) so a silent pipeline is diagnosable. Backlog proof at fix time: 4 stranded records (2 days) arrived in one DM.

Internal-request gate

Turns the agent fires at itself (req_id prefixes sched-, job-, sub-; see INTERNAL_REQUEST_PREFIXES) skip BOTH digest blocks — an internal turn consuming the watermark would silently eat the operator's next "while you were away" report. This also fixed a latent bug where a cron turn consumed the project digest watermark.
