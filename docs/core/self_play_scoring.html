core / self_play_scoring.py

Correctness-weighted curriculum metric that replaces the naive compression-delta score (which used to reward wrong one-shot solves).

Functions

FunctionDescription
count_tool_errors(messages) → intCount tool-result messages that indicate tool failures (three-tier discrimination, below).
correctness_weighted_score(passed, compression_delta, tool_errors, alpha=0.4, beta=0.1, novelty=None, attempts_used=None, gamma_novelty=0.6, delta_attempts=0.3) → floatMulti-signal score consumed by the frontier tracker, the adaptive cooldown, and the skill-write gate.

Formula (post 2026-05-17)

if passed:
    base = 1.0 + α · compression_delta + γ · novelty + δ · attempts_efficiency
else:
    base = 0.0
score = base − β · tool_errors

α = 0.4   # reward for compressing the description after a successful solve
β = 0.1   # penalty per tool error
γ = 0.6   # reward for finding a structurally novel solution
δ = 0.3   # reward for solving on the first attempt (vs 2nd / 3rd)

When passed=False, the base term is 0 — compression / novelty / attempts are ignored because a failed run's "efficient plan" wasn't correct. The tool-error penalty still applies so a failed run with many errors is ranked below a failed run with few.

Back-compat: when both novelty and attempts_used are None the formula reduces to the pre-2026-05 shape so every legacy test still asserts the same numbers.

novelty ∈ [0, 1] comes from solution_novelty.jaccard_novelty — 1.0 when the cluster has no prior winning solution, 0.0 when the new solution.py AST-canonicalises to a shape already on file. Variable renames don't move the score.

attempts_efficiency is a small lookup table: {1 → 1.0, 2 → 0.5, 3 → 0.2}. First-try wins outrank third-try wins even when both pass with the same novelty.

Why the redesign

Pre-2026-05 the score collapsed to pass/fail because compression_delta (tool-call count vs. prior best) was pinned at +0.000 on deterministic templates — tool counts barely move between consecutive wins. The flat score starved the frontier tracker of gradient and the skill-write gate of any reason to fire. The 2026-05-17 redesign adds two independent signals (novelty + attempts efficiency) so a first- try win with a structurally new solution outranks one that reproduces a saturated AST shape. See the self-improvement pipeline doc for the full post-mortem and the matching reflector / write-gate changes.

Three-tier tool-error discrimination

The old heuristic substring-matched error markers anywhere in any tool content — which produced false positives whenever a fixture legitimately contained ERROR: / WARN / Traceback tokens (a log-counting challenge's log files, for example). The new count_tool_errors uses three tiers:

  1. Data-tool whitelist: tool-result messages whose name is one of {file_system, recall, knowledge_base, web_search, deep_research, fact_check, postgres_admin} are never counted. Their content is retrieved data, not failure output.
  2. Unambiguous patterns (counted anywhere in content): "Traceback (most recent call last)", "SYSTEM ERROR: Tool", "SYSTEM ALERT: You have failed", "CalledProcessError", "TimeoutExpired". These don't collide with fixture text.
  3. Line-start prefixes (counted only when they appear at the start of content, after optional whitespace): Error:, ERROR:, error:, AssertionError, SyntaxError, NameError, IndentationError, ImportError, ModuleNotFoundError. Matches the shape the dispatch layer synthesises for genuine tool failures.

The old heuristic would score a legitimate log-counting solve as tool_errors=6 (5 file reads + 1 execute whose stdout mentions ERROR); the new one correctly scores it as 0.

Why

The previous metric rewarded compression even when the agent solved the wrong problem. This formulation makes correctness mandatory and penalises noisy tool use — but only actual tool noise, not fixture content — so brittle clusters are surfaced accurately to the FrontierTracker and the adaptive cooldown.
