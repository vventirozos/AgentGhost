core / adversarial_generator.py

Per-prompt-fingerprint solver pass-rate tracker. Biases the next challenge-generation prompt toward families that have been stumping the solver.

Why

Pre-2026-05 the LLM that generated challenges was the same LLM that solved them, and there was no incentive for the generator to produce hard challenges. Production traces showed its creative range collapsing to a handful of CSV/JSON groupby shapes the solver aced first-try (traces 17:44, 18:46, 19:48). The 8-template bank dominated cold-start.

This module is the tiny state store for proposal G in the 2026-05-17 redesign. It tracks, per generator-prompt fingerprint, the recent solver pass rate on challenges produced under that prompt. The dreamer reads this score and biases the next generator prompt toward families with low solver pass rates — i.e. toward what's actually stumping the agent.

API

Function / methodDescription
fingerprint_prompt(prompt_fragment) → strStable 12-char sha1 of the variable part of the generator prompt (typically the frontier hint).
AdversarialGeneratorTracker(memory_dir)JSON-backed store at {memory_dir}/adversarial_generator.json.
.record(fingerprint, *, passed, cluster)Append one outcome to the fingerprint's ring (cap 10 outcomes per fingerprint).
.pass_rate(fingerprint) → float | NoneRecent solver pass rate in [0, 1], or None when fewer than 2 outcomes recorded.
.worst_fingerprints(limit=3) → list[(fp, rate, top_cluster)]The N fingerprints with the lowest pass rates.
.suggest_bias() → strA short prose block to inject into the challenge-generator prompt. Empty when no signal yet.

Wiring

core/dream.synthetic_self_play calls suggest_bias() at the top of the system prompt and appends the returned block before the saturation override. The challenge-gen LLM sees something like:

### ADVERSARIAL HINT (generator feedback)
The solver's recent pass rate on 'concurrency, sql'-shaped
challenges is only 25%. Prefer producing MORE challenges in that
family with subtle edge cases (empty inputs, off-by-one
boundaries, unicode payloads, concurrent writers) rather than
rotating to a different family the solver already aces.

After the cycle finishes, record() fires with the cycle's (passed, cluster) outcome against the same fingerprint, so the next bias suggestion reflects the most recent attempt.

Why a separate file (not on FrontierTracker)

  • The frontier tracker is keyed by cluster; the adversarial signal is keyed by generator-prompt fingerprint. Mixing them would couple two independent rotation policies.
  • Generator fingerprints are short-lived — a few dozen, not thousands. A small standalone JSON keeps the I/O footprint obvious and easy to reset.
  • No advisory-lock machinery — the dream loop is single-writer within one process, and cross-process self-play is not supported.

Coverage

tests/test_self_play_meaningful.py::TestAdversarialTracker (5 cases): fingerprint stability, minimum-sample gate on pass_rate, ordering of worst_fingerprints, bias-suggestion content, empty-state behaviour.
