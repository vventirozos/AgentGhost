sandbox / services.py + tools/sandbox_services.py

Supervised long-lived services inside the sandbox container (2026-07-11): the manage_services tool lets the agent start a web server or daemon that SURVIVES across turns — closing the "can build an app, can't host it" gap.

Why

execute wraps every command in timeout -k 5s (600 s cap) and the container is PID-isolated, so anything the agent started died with its exec. Yet the toolset strongly implies hosting: it can write web apps (file_system), drive a browser at them, and the container itself is persistent. The 2026-07 feature survey flagged "build and host me a dashboard" as implied-but-impossible.

How it works

The command ships as a script via the bind mount (/workspace/.services/<name>.cmd.sh — no quoting hazards), then launches with setsid nohup … &: the exec shell exits immediately (satisfying the timeout wrapper) and the detached process re-parents to the container's PID 1, surviving the exec, the turn, and agent restarts (the container outlives the agent process). setsid makes it a process-group leader so stop can kill the whole tree (kill -TERM -- -pid, escalating to KILL). Per-service pid, log (<name>.log), and a host-side registry.json live under .services/.

Tool surface

  • start — name + foreground command (+ recommended port): launches, verifies liveness after ~1 s (an immediate exit returns the log tail instead of registering), probes the TCP port for up to ~6 s, registers, and reports the in-sandbox URL.
  • stop / restart — group-kill; restart re-launches from the registered command/port/workdir.
  • status — liveness (kill -0) + port listening state + uptime for every registered service; a recreated container shows them DEAD (pids don't survive re-provisioning).
  • logs — host-side tail (default 60 lines, 8 KB cap).

Rails

  • Max 5 concurrent services; names [A-Za-z][A-Za-z0-9_-]{0,31}.
  • Ports 8000 / 8088 / 8080 / 9050 are refused (agent API / upstream LLM — the sandbox-loopback blind spot and its mock-server pathology — plus NetMon and Tor on a host-network Linux box), as is any command referencing 127.0.0.1:8000/:8088. Suggested ports: 8100-8104.

Reachability

  • In-sandbox (the browser tool's embedded runner, execute): http://127.0.0.1:<port>. The browser SSRF guard (host-side _browser_blocked_url AND the in-runner interceptor) admits explicit-loopback URLs whose port is in the service registry (active_service_ports, threaded through the op payload as allowed_local_ports) — literal hosts only, so no DNS/rebind surface; every other loopback / internal target stays blocked.
  • From the host (operator): in host network mode (Linux default) the service binds host ports directly. In bridge mode (macOS default) docker.py publishes the GHOST_SANDBOX_SERVICE_PORTS range (default 8100-8104, loopback-bound; empty disables) when the container is (re)created — an existing container picks this up on its next re-provision.

Fixed host ports must assume ≥2 agent instances. A second sandbox container — a throwaway agent for an ablation, or the test suite — cannot publish the same fixed host ports as the instance already running. Two compounding failures made this bite (caught by the suite, now fixed): containers.run fails outright on the taken port, AND a port-bind failure leaves the container created-but-not-started, so a naive retry-without-ports died with a 409 name-in-use — which propagated and bricked the sandbox entirely. So: publishable_service_ports() bind-checks each port on the host and publishes only the FREE ones (a second agent silently gets no published ports — the right degradation, since in-sandbox consumers reach the service on the container's own loopback regardless), and the residual bind-race retry removes the stale container before re-running.
