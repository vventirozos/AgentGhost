docker services tools.browser tools.execute

sandbox / services.py + tools/sandbox_services.py

Supervised long-lived services inside the sandbox container (2026-07-11): the manage_services tool lets the agent start a web server or daemon that SURVIVES across turns — closing the "can build an app, can't host it" gap.

Why

execute wraps every command in timeout -k 5s (600 s cap) and the container is PID-isolated, so anything the agent started died with its exec. Yet the toolset strongly implies hosting: it can write web apps (file_system), drive a browser at them, and the container itself is persistent. The 2026-07 feature survey flagged "build and host me a dashboard" as implied-but-impossible.

How it works

The command ships as a script via the bind mount (/workspace/.services/<name>.cmd.sh — no quoting hazards), then launches with setsid nohup … &: the exec shell exits immediately (satisfying the timeout wrapper) and the detached process re-parents to the container's PID 1, surviving the exec, the turn, and agent restarts (the container outlives the agent process). setsid makes it a process-group leader so stop can kill the whole tree (kill -TERM -- -pid, escalating to KILL). Per-service pid, log (<name>.log), and a host-side registry.json live under .services/.

Tool surface

  • start — name + foreground command (+ recommended port): launches, verifies liveness after ~1 s (an immediate exit returns the log tail instead of registering), probes the TCP port for up to ~6 s, registers, and reports the in-sandbox URL. The assigned port is exported to the command as PORT (and GHOST_SERVICE_PORT) so a well-behaved app binds it (port=int(os.environ.get("PORT", …))) rather than hardcoding one that would mismatch the probe (2026-07-12). If the process stays alive but the port never binds — a crash on import, a missing dependency, or a hardcoded wrong port — start now returns the log tail with the actual cause (e.g. ModuleNotFoundError) and a restart hint, instead of a vague "not listening yet"; before this the agent would browse the URL, hit ERR_CONNECTION_REFUSED, and only then discover the missing dep — a whole browse→fail→install→restart cycle for an error already captured.
  • workdir — start only: the directory the command runs IN, relative to /workspace (e.g. projects/30d5d5b65c38; defaults to /workspace). Pass this to run an app that lives in a subdirectory instead of prefixing the command with cd <dir> &&. The handler and supervisor always accepted it, but it was missing from the tool schema until 2026-07-12 — so the model couldn't see it and burned ~50 s hacking a path into the command (tripping the loop-breaker) when hosting a subdirectory app.
  • stop / restart — group-kill; restart re-launches from the registered command/port/workdir. If the tracked pid is dead but the port is still held (a mis-tracked orphan), stop reclaims it by killing the port's listener, found via ss (iproute2, baked into the image).
  • stop-all — the one-shot cleanup (2026-07-12): stops EVERY service, reclaims their ports, and clears the registry. For when services have accumulated or orphaned.
  • status — liveness (kill -0) + port listening state + uptime for every registered service; a recreated container shows them DEAD (pids don't survive re-provisioning), and the listing points at stop-all when any dead entry is present.
  • logs — host-side tail (default 60 lines, 8 KB cap).

Reliable pid tracking (2026-07-12). The launcher's & echo $! captured a transient wrapper pid, not the real service — so under setsid the actual process (re-parented to the container's PID 1) had a different pid, and stop/restart killed the wrong one, orphaning the real service (hung processes piled up across restarts). Now the generated cmd.sh records its own pid — echo $$ > <name>.pid as its first action; under setsid that shell is the session/group leader, so kill -- -<pid> reaps the whole tree — and start registers that pid. The port-reclaim fallback above is the belt-and-suspenders for any pre-existing orphan.

Zombies + workdir validation (2026-07-12, from a live 137s postmortem). The container's original PID 1 (sleep infinity) never reaped orphans, so every dead launcher became a permanent zombie — and zombies pass kill -0. A service that never started looked "already running", stop no-op'd against it, and the exited-immediately diagnostic never fired. Fixes: docker.py now runs the container with init=true (tini reaps), and _pid_alive additionally rejects state Z from /proc/<pid>/stat. Separately, start now validates the workdir exists before launching (a cd failure inside the async launcher was invisible — not even the service log got written), heals the common /projects/<id> → /workspace/projects/<id> mistake, anchors relative paths at /workspace, and strips a redundant leading cd X && when workdir already covers it.

Rails

  • Max 5 concurrent services; names [A-Za-z][A-Za-z0-9_-]{0,31}.
  • Ports 8000 / 8088 / 8080 / 9050 / 9051 are refused (agent API / upstream LLM — the sandbox-loopback blind spot and its mock-server pathology — plus NetMon and Tor SOCKS/CONTROL on a host-network Linux box), as is any command referencing 127.0.0.1:8000/:8088. Suggested ports: 8100-8104. 9051 added 2026-07-14 — a registered service on 9051 would otherwise enter active_service_ports, which the browser SSRF allowlist consults, and the browser guard explicitly names 127.0.0.1:9051 (Tor control) as must-block.

Known residual (§4B, 2026-07-14 bug hunt). is_published_port consults the CONFIGURED port range, not the ports docker actually published. In the ≥2-instance case (a throwaway/ablation agent alongside prod) a second instance publishes nothing yet is_published_port still returns true, so the operator is pointed at 127.0.0.1:<port> — which is the FIRST instance's forwarder. Fix direction: persist the actually-published set on the docker manager and consult that. Not yet fixed (needs cross-module plumbing; rare, non-security).

Reachability

  • In-sandbox (the browser tool's embedded runner, execute): http://127.0.0.1:<port>. The browser SSRF guard (host-side _browser_blocked_url AND the in-runner interceptor) admits explicit-loopback URLs whose port is in the service registry (active_service_ports, threaded through the op payload as allowed_local_ports) — literal hosts only, so no DNS/rebind surface; every other loopback / internal target stays blocked.
  • From the host (operator): in host network mode (Linux default) the service binds host ports directly. In bridge mode (macOS default) docker.py publishes the GHOST_SANDBOX_SERVICE_PORTS range (default 8100-8104, loopback-bound; empty disables) when the container is (re)created — an existing container picks this up on its next re-provision. The service must bind 0.0.0.0 inside the container for the bridge-publish to forward — the manager exports HOST=0.0.0.0 (and GHOST_SERVICE_HOST) alongside PORT for exactly this. An app bound to the container's own loopback is reachable by the in-sandbox browser but NOT from the host.
  • From another device (tailnet): a published port lands on host 127.0.0.1 only (by design — it's an agent sandbox). Put it on your tailnet with tailscale serve — see Remote access below.

Fixed host ports must assume ≥2 agent instances. A second sandbox container — a throwaway agent for an ablation, or the test suite — cannot publish the same fixed host ports as the instance already running. Two compounding failures made this bite (caught by the suite, now fixed): containers.run fails outright on the taken port, AND a port-bind failure leaves the container created-but-not-started, so a naive retry-without-ports died with a 409 name-in-use — which propagated and bricked the sandbox entirely. So: publishable_service_ports() bind-checks each port on the host and publishes only the FREE ones (a second agent silently gets no published ports — the right degradation, since in-sandbox consumers reach the service on the container's own loopback regardless), and the residual bind-race retry removes the stale container before re-running.

Remote access — reach a hosted service from another device (2026-07-12)

"The agent hosts something and I reach it remotely" needs two things, one automatic and one operator-run:

  1. Escape the container (automatic). Host on a published port (the 8100-8104 range) and bind 0.0.0.0 — the manager exports HOST=0.0.0.0 and PORT=<port> into the service command, so a well-behaved app (uvicorn --host $HOST, flask run, most Node servers, or app.run(host=os.environ["HOST"], …)) binds the forwarding interface. Docker then publishes it to host 127.0.0.1:<port>.
  2. Put it on the tailnet (operator, one command). Published ports bind host loopback only — deliberately, since the agent's own API runs authless on this host. Expose one to your tailnet with: 
    bin/serve-remote.sh 8100
# → https://<this-host>.<tailnet>.ts.net:8100/  →  127.0.0.1:8100
bin/unserve-remote.sh 8100    # tear down (or `… all` to reset every mapping)
    The helper self-discovers the tailscale CLI (including the macOS GUI app bundle, where it is not on PATH), maps the tailnet HTTPS port 1:1 onto the local port, and prints the URL. tailscale serve keeps it authenticated to your tailnet with a real TLS cert and nothing exposed publicly (that would be tailscale funnel).

The exposure step is a host action, run by the operator — the agent lives in the sandbox and can't (and shouldn't) flip network exposure itself. The manage_services report surfaces the exact serve-remote.sh command + URL whenever a service comes up on a published port, so the operator never has to remember the recipe.
