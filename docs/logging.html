Logging & monitoring

The agent is monitored by watching its live log stream. Every operationally-important event — and every failure — appears there in one aligned, per-request, icon-tagged format. This page is how to read it, and what the May-2026 overhaul changed.

pretty_log per-request tags failures visible distinct icons
The model. pretty_log(title, content, icon, level) in utils/logging.py writes the monitored console stream. Standard library logger.warning/error calls are now routed through the same renderer, so a failure never appears as a bare, easily-missed plain line interleaved with the pretty output. The on-disk file (system/ghost-agent.log) keeps a plain, grep-friendly format (now stamped with the logger name).

Reading a line

Each line has a fixed anatomy — once your eye learns the columns, a busy multi-request stream is scannable:

│  A7  💭  +0.42s  thinking            128 tokens · 0.9s
^   ^   ^    ^        ^                    ^
│   │   │    │        │                    └── content (one line; ERROR/WARNING get a larger budget)
│   │   │    │        └── title (lower-cased, bold, level-coloured)
│   │   │    └── elapsed since the request began
│   │   └── icon — picks itself per event type
│   └── 2-char request tag (deterministic colour; groups concurrent requests)
└── left frame edge

Requests are framed by ┌─ … request started / └─ … request finished +12.3s markers. These now fill the terminal width with a ─ rule, so each request is a visually distinct block even in a busy interleaved stream. The boot and shutdown sequences are framed by 🌅 System Boot … 🚀 System Ready … 💤 System Shutdown.

Guaranteed column alignment. The icon sits in a fixed 2-cell field, so the delta / title / content columns line up on every line. This holds because every registry glyph is East-Asian wide-base (renders as 2 cells on any terminal). Narrow-base or bare variation-selector symbols (⚠️ 🗣️ 🛡️) are terminal-dependent — 2 cells on some, 1 on others — and were the cause of columns drifting on certain lines; they are now banned from the registry (test_every_registry_icon_is_wide_base). _icon_cell still pads any stray narrow glyph as a backstop, and the content column is derived from one layout formula (_CONTENT_COL) so it can't silently drift.

May 2026 overhaul

An audit of all ~700 log sites surfaced three problem classes; all are fixed (with unit tests). See the audit page for the companion bug-fix work.

1 — Orphaned loggers (logs going nowhere)

Five first-party loggers — GhostSelfhood, GhostWorkspace, GhostOptim, GhostDistill, GhostReflect — were never given handlers; setup_logging only configured GhostAgent. Their debug/info was discarded and warning/error leaked to bare stderr — never the file, never the stream. setup_logging now configures all of them.

2 — Failures off the monitored stream

~150 logger.warning/error calls printed as un-iconed plain lines (or, for the orphaned loggers, nowhere). A new _PrettyLogHandler renders every WARNING+ record through pretty_log (⚠️/❌, with the request tag), so failures land in the channel you're already watching. Where an event had no log at all it was added with a semantic icon: API auth rejections (🛡️), proxy failures (❌), browser / sandbox-exec / Postgres failures, context compaction escalation (✂️, so you can see when answers start degrading from dropped context), a shutdown frame (💤), and profile-corruption resets (👤, which previously reverted the user's identity silently).

3 — Readability

  • Failure content survives. WARNING/ERROR lines get a larger content budget, so the exception text (the why) isn't truncated off the live stream.
  • Distinct icons. De-collided glyphs that were double-bound (📚/📓, ✂️ vs split, ⚡ boot vs edge-node) and gave the browser/image/PDF/worker/edge events their own icons instead of borrowing 🐚/literals.
  • Distinct titles. The 8-way overloaded Sandbox boot title was split into Sandbox Provision / Image / Chromium / Tor / Cache / Ready; node-failover messages that were crammed into the 18-char title (losing the error) are now a short title + full content; a failed verifier no longer renders a green ✅.

Beautification pass — alignment & icon accuracy

A follow-up pass took the stream "a step further" for the monitoring operator: aligned by construction, each request cleanly separated, and every icon accurately matched to its event.

  • Fixed-width icon column. The layout always assumed a 2-cell icon; a narrow or variation-selector glyph would shift every following column on that line. Icons now pass through _icon_cell, and the content column is computed from one formula (_CONTENT_COL) rather than a hard-coded constant.
  • All icons are wide-base. A real-terminal report showed lines like 💬 llm request and 📈 metacog conf drifting right — their old glyphs (🗣️, 🌡️) were narrow-base + VS16, whose width is terminal-dependent. All 17 such glyphs (registry + metacog) were swapped for East-Asian wide-base emoji that render as 2 cells everywhere: ⚠️→🔶 warn, 🛡️→🔒 shield, 🗣️→💬 llm, ⬇️→📥 download, ⚙️→🔧 worker, 🛰️→🛸 edge, ✂️→🔻 cut, ⛓️→🔗 constraint, ♟️→🎮 game, and more. A test now fails the build if any narrow-base glyph re-enters the registry.
  • Full-width request frames. ┌─/└─ markers fill the terminal width, turning each request into a clearly delimited block in an interleaved stream.
  • No more registry-bypassing icons. 18 inline emoji literals (e.g. icon="💤") across dream.py, skills.py, acquired_skills.py, vector.py were replaced with named Icons constants. Literals silently collided — dream was reusing 💤 SYSTEM_SHUT; it now has its own 🌙 DREAM. A guard test (test_no_inline_emoji_literal_icons_in_source) fails the build if any reappear, now catching single- and double-quoted literals (the 🌡️ metacog icon had slipped through as a single-quoted one).
  • Icon accuracy fixes. New constants 🌙 DREAM, ⏩ SKIP, 💪 MEM_REINFORCE; the previously-orphaned 🎓 SKILL_GRADUATE is now wired to the graduation call sites; skill reinforcement (was 🔄, read as "retry") and skipped/deduped work now read at a glance.

Covered by tests/test_logging_framework.py (registry-hygiene guard, icon-width normalization, derived content column, adaptive frames, new-icon presence/distinctness).

Verbosity knobs. --verbose lifts the content truncation entirely (full payloads on the stream); --debug drops the file/console threshold to DEBUG and mirrors every pretty_log into the file. --daemon suppresses the console handler (file only). NO_COLOR / a non-TTY sink disables ANSI so piped logs stay clean.

Thinking is never filtered — and never truncated (2026-07-08). Streamed 💭 thinking blocks used to be swallowed before they reached the logger unless --verbose was set. The emit/flush closures are no longer gated on VERBOSE_MODE, and the three thinking call sites pass pretty_log(..., no_truncate=True) — a per-call exemption from the content budget (newline flattening, redaction and column wrapping still apply, so the line format is identical). Result: the full reasoning stream is visible in EVERY mode; all other lines keep the standard 60-char budget in non-verbose. The post-stream 🧠 summary line is unchanged in both modes. Covered by tests/test_logging_thinking_nonverbose.py.

Stream redaction (anonymity)

The operator monitors this stream live, which made it the single largest cleartext sink in the system — redaction historically ran only at the JSONL trajectory boundary, never at the log boundary, so secrets, .onion addresses, full URLs, and absolute home paths reached the console verbatim. pretty_log now routes content through distill.redact.redact_text before rendering (post-truncation, so the regex cost is bounded to the visible line), and the WARNING+ auto-render path inherits it. On by default; --no-redact-logs opts out for debugging. redact_text only rewrites known sensitive patterns, so ordinary log lines stay byte-identical and readable. Covered by tests/test_quick_wins.py.

Fire-and-forget: spawn_bg (2026-07-07)

Background work that must not block a turn — memory writes, lesson retraction, PRM online updates, graph-triplet extraction, reflection — is scheduled through one primitive, utils.logging.spawn_bg(coro, *, name). It composes the three guarantees that four earlier ad-hoc conventions each had only some of:

  • contextvars propagation — the task logs under the spawning request's id (same as spawn_task).
  • strong reference — held in a process-wide registry (_BG_TASKS) until completion. asyncio keeps only a weak ref to a bare create_task, so an un-stored task can be garbage-collected mid-flight and the work silently never lands; two real sites (lesson retraction, PRM online update) were doing exactly this in violation of the codebase's own documented rule.
  • exception logging — a done-callback logs any non-Cancelled failure via logger.warning (auto-renders on the live stream). Previously a dying background coroutine vanished into an except: pass body — the root of the "built-but-silently-not-happening" pathology.

drain_background_tasks(timeout) awaits the registry at lifespan shutdown (bounded) so in-flight writes get a chance to land. The three superseded variants (bare spawn_task for fire-and-forget, the module-local _GRAPH_EXTRACT_TASKS set, and GC-unsafe bare create_task) are removed; the remaining legitimate strong-ref sites (the biological daemon task, the swarm task registry, the context _pending_background_tasks set, the self-play loop) are allow-listed. A grep-based regression guard (tests/test_spawn_bg.py::test_no_bare_create_task_in_hot_modules) fails the suite if a new bare create_task appears outside that allow-list.
